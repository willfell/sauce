## Lessons codified (so this class of accident doesn't recur)

1. **Always `git pull --ff-only` before reading a handoff doc.** A handoff doc captures state-at-write-time. If origin has advanced since, the doc may name closed work as "next".
2. **Pre-flight check the handoff's `gates:` frontmatter against actual state.** This file's original `gates:` named `HEAD = d7b7e80 on main` and `run-seed.js shows 16 pass · 0 fail` — by the time the second chat read it, HEAD was 30+ commits past d7b7e80 and `run-seed.js` had grown well past 16 asserts. Either signal alone should have triggered an abort-and-investigate.
3. **Subagent prompts should refuse-on-collision.** The implementer subagent that re-wrote `cmd-seed.js` "adjusted from existing implementation to match the S2.1 placeholder" — it saw the file existed, knew it was being overwritten, and proceeded anyway. Future implementer prompts for new-file tasks must include an explicit `if [ -e <target> ]; then HALT with NEEDS_CONTEXT` precondition.
4. **The recursive-handoff pattern works, but only if the chain reads the latest state.** This handoff's "Recursive handoff" section was correct in spirit — write a successor at a clean stage break and end with the slash-command block — but it did not protect against a stale handoff being executed on a repo whose state had moved past the handoff's premise.
