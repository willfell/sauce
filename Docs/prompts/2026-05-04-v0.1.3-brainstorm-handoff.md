## What v0.1.3 needs to design + ship

### Code surfaces (anchored by the sketch in v0.1.x result writeup §"Follow-up cycle")

> [!info] Manifest schema additions (additive, optional, applied to mechanism + blueprint manifests)
> ```json
> "templater_hotkeys": [
>   { "template": "Validate.md" }
> ]
> "slash_commander_bindings": [
>   { "keyword": "/validate", "command_id": "obsidian-templater:insert-Validate" }
> ]
> ```
> Schema bikeshed during brainstorming — what's the right field name + entry shape? Does `slash_commander_bindings[]` need extra fields like `icon` / `description`?

> [!info] Two new installer helpers (mirror `applyExternalPlugins` posture exactly)
> - `applyTemplaterHotkeys(tp, manifest, history)` — read `.obsidian/plugins/templater-obsidian/data.json`; additive-merge each entry from manifest's `templater_hotkeys[]` into `enabled_templates_hotkeys[]`. Idempotent (skip if already present). C4 malformed-JSON guard. Backup-on-edit (`<file>.beacon-backup`). Failure-loud Notice + history.
> - `applySlashCommanderBindings(tp, manifest, history)` — same pattern against Slash Commander's data.json. **Schema discovery required (T1.x):** read `../tmp-test-barebones-vault/.obsidian/plugins/slash-commander/data.json` after the user has created one binding manually so we have a reference shape.

> [!info] `.obsidian/` ban lift (scoped allowlist)
> CLAUDE.md non-negotiables currently say modifications to `.obsidian/` are ask-before-acting. v0.1.3 lifts this for **exactly two paths**:
> - `.obsidian/plugins/templater-obsidian/data.json`
> - `.obsidian/plugins/slash-commander/data.json`
>
> Everything else in `.obsidian/` stays ask-before-acting. New landmine entry codifies the allowlist + the safety mechanics:
> - **Additive merge only** — never strip user entries.
> - **Backup-on-edit** — write `<file>.beacon-backup` before merge.
> - **Malformed-JSON guard** — C4 parity; install never aborts on plugin-data corruption.
> - **Failure-loud history** — every merge / skip / error logged with full context.

> [!info] Manifest declarations (this is what the helpers exercise)
> - validator: `templater_hotkeys: [{ template: "Validate.md" }]` + `slash_commander_bindings: [{ keyword: "/validate", command_id: "<TBD>" }]`
> - audit: `templater_hotkeys: [{ template: "Audit.md" }]` + `slash_commander_bindings: [{ keyword: "/audit", command_id: "<TBD>" }]`
> - project: `templater_hotkeys: [{ template: "Create New Project.md" }]` + `slash_commander_bindings: [{ keyword: "/new-project", command_id: "<TBD>" }]`
>
> The `<TBD>` command IDs must be locked from disk during T1 discovery (read Templater's runtime command registry or read the data.json after one manual binding to learn the format).

## Open questions for brainstorming (Phase 1 clarifying questions)

These are the gates `de:brainstorming` Phase 1 should resolve before proposing approaches:

| # | Question | Why it matters |
|:--:|---|---|
| 1 | What's Slash Commander's `data.json` schema? Specifically the bindings array shape — single flat array, nested by category, includes icon/description fields? | Locks the `slash_commander_bindings[]` manifest schema + the merge primitive. |
| 2 | What's the EXACT command_id format for `Templater: Insert <Name>` commands? `obsidian-templater:insert-<Name>` or `templater-obsidian:insert-<Name>` or something else? | The `slash_commander_bindings[].command_id` must match the runtime command id verbatim. Lock from disk. |
| 3 | After the installer writes to a plugin's data.json, does Obsidian's runtime PICK UP the changes immediately, or only on the next plugin reload / Obsidian reload? | Determines the user's "reload Obsidian" step necessity. May be different for Templater vs Slash Commander. |
| 4 | Is there a way to trigger plugin reload programmatically (e.g., `app.plugins.disablePlugin(id)` + `app.plugins.enablePlugin(id)`)? | If yes, the user might not need to reload Obsidian; brings irreducible manual to ONE step (just install Slash Commander). If no, Obsidian reload stays. |
| 5 | Schema bikeshed: should `templater_hotkeys[]` entries be `{ template: "Validate.md" }` (basename) or `{ template: "{{templates_path}}/Validate.md" }` (resolvable path)? | Consistency with existing `files[]` substitution discipline (landmine #9: substitution variables only from `platform-config.json:variables`). |
| 6 | Cycle staging: 2 stages (S1 schema+helpers / S2 manifest declarations + smokes) or 3 stages (S1 schema / S2 helpers + harness / S3 declarations + smokes)? | Affects the implementation plan's checkpoints. |
| 7 | T2.6 deferred smokes' EXACT pass criteria when run against the automated flow: do they validate on the FIRST install run, or does the user need to reload Obsidian once between install + first slash command? | Matters for v0.1.3's S2 acceptance gate copy. |
