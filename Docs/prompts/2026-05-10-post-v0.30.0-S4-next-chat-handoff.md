### A.1 — Install cowork@0.1.0 + daily@0.2.5 into accuris-sauce

This is the FIRST per-vault rollout in S5 sequence. Pre-conditions:

1. Snapshot the accuris-sauce vault before any installer touch:
   ```bash
   VAULT=/Users/willfell/Documents/obsidian/sync/sauce/accuris-sauce
   TS=$(date +%Y%m%d-%H%M%S)
   cp -R "$VAULT" "${VAULT}.pre-cowork-${TS}/"
   echo "snapshot at ${VAULT}.pre-cowork-${TS}"
   ```
2. Run the sauce updater against accuris-sauce. From inside Obsidian, invoke `<vault>/ranch/templates/_install-platform` Templater template (the workshop's installer entry-point), or use the CLI:
   ```bash
   cd /Users/willfell/Documents/obsidian/sync/workshop/sauce
   node platform/cli/sauce-cli.js update --vault "$VAULT"
   ```
3. Verify post-install:
   ```bash
   find "$VAULT/.claude/skills/cowork" -name SKILL.md | wc -l   # expect 38
   ls "$VAULT/spice/cowork/"                                    # expect Cowork.md + context/
   grep -q "COWORK_CALLOUTS" "$VAULT/ranch/templates/Daily Note.md" && echo "anchor OK"
   ```

### A.2 — Invoke `cowork:bootstrap-vault` from inside accuris-sauce

Open accuris-sauce in Claude Code (or via the Obsidian MCP from a Claude session pointed at that vault). Invoke:

```
Use Skill cowork:bootstrap-vault
```

The skill is **interactive**. It will:
1. Probe MCP backends (gmail / google-calendar / brex / obsidian / imessage).
2. Read the vault surface (which `spice/<module>/` dirs exist, note counts).
3. Ask the user (one question at a time) about:
   - Vault scope (`life` | `work` | `both` | `none-yet`) — accuris is likely `work` since it's the client engagement vault. May also be `both` if the user wants their accuris-engagement-life-shadow there.
   - Owner identity (`owner_name`, `ero_role`, `ero_primary_client: Accuris`, etc.)
   - Finance posture (`ero_hourly_rate_usd`, `ero_ap_email`, `ero_invoice_cadence`, etc.)
   - Per-MCP scoped questions (which gmail account, which calendar, brex on/off, etc.)
4. Build a recommendation table per orchestrator (enable / skip).
5. Write `<vault>/spice/cowork/context/vault-config.md` with all answers.
6. Materialize per-scope context files (from `<scope>-context-templates/` into `<vault>/spice/cowork/context/`).
7. Seed `active-threads.md` + `weekly-snapshot.md` with empty schemas.
8. Emit `<vault>/spice/cowork/bootstrap-report.md` with:
   - What was discovered.
   - What the user said.
   - **For each enabled orchestrator: the literal cron SKILL.md body to paste into the user's cron infrastructure.**
   - What was skipped + why.
   - Template variables still needing manual fill in `<vault>/ranch/platform-config.json`.

### A.3 — Set up cron jobs from the bootstrap report

The bootstrap report's Section 3 contains copy-paste-ready cron SKILL.md bodies, one per enabled orchestrator. For each:

1. Open the user's cron infrastructure path (e.g., `/sessions/<scheduler-base>/mnt/.scheduled/<job-name>/`).
2. Paste the cron SKILL.md body verbatim into `SKILL.md` at that path.
3. Confirm the cron scheduler picks it up.

Result: scheduled jobs fire daily/weekly/monthly per each orchestrator's `schedule:` field; each job invokes Claude with `Use Skill cowork:<orchestrator-id> in <vault-name>`, which materializes the daily callout block / weekly summary / monthly review in the vault.

### A.4 — Iterate

Run a manual `Use Skill cowork:morning-briefing` against accuris-sauce after bootstrap to verify the full pipeline produces the expected callout block at the `<!-- COWORK_CALLOUTS -->` anchor. Tune as needed (e.g., expand the gmail filters, add inner-circle people, adjust thresholds).

## Deferred items (carry forward; NOT blockers for tag)

> [!warning] These are real issues, not perfectionism. Schedule into v0.30.x patches.

1. **Smoke harness DEFERRED.** Plan called for `platform/test/run-cowork-smoke.js` with 5 fixtures (date-context, gather-calendar, gather-gmail, write-callout-morning-briefing-life, update-active-threads). Build this in a v0.30.x patch — 8th harness, ~30-50 sub-asserts.

2. **`gather-threads` mode branching.** Outputs declare mode-specific fields (`resolved_today`, `still_open`, `auto_created_eod`, `opened_this_week`, `stale_over_7d`, `longest_running`, `average_resolution_days`) but Steps describe only `morning-surface` mode. Explicit branching needed for `eod-reconcile`, `weekly-audit`, `monthly-audit`.

3. **`gather-finance-cc-today` severity classifier.** Outputs declare `severity` per charge (consumed by midday-tripwire dispatch logic) but Steps don't show the RED/YELLOW/GREEN classification rules. Document the threshold logic or move classifier into a helper sub-skill.

4. **`gather-calendar` `ai_committee_status`.** Output declared for work scope; derivation logic not in Steps.

5. **`update-weekly-snapshot` phase branching.** Inputs document `phase: morning | eod | weekly-close | monthly-reset | weekly-reset | archive-and-reset | section-update`. Steps body covers 3 of them. Explicit branching for all 7.

6. **`ero-eod` doesn't dispatch `update-active-projects`.** Active-projects file goes stale from morning until next morning, even though EOD has fresh `gather-projects` data. Add `Use Skill cowork:update-active-projects with { phase: "eod-pass", ... }` to ero-eod's State section.

7. **`patch-daily-callouts` title-match idempotence.** Inputs say `id` is "used for idempotent replace-by-title" but Step 3 actually matches by the body's first line, not the `id`. If user renames a callout title, idempotence breaks. Either match by `id` (storing `<!-- cowork-callout:<id> -->` HTML comments) or document the title-fragility.

8. **`patch-daily-callouts` write tool selection.** Step 5 says "via `mcp__obsidian__patch_note` (preferred) or `mcp__obsidian__write_note` (fallback)". For multi-block buffer changes, `write_note` is the real path. Tighten.

9. **Brand-voice trim incomplete.** Two more spots may still have meta-prose: review `write-callout-morning-briefing-life` and `-ero` opening callouts after the polish pass landed.

10. **Bootstrap report Section 3** could include a `daily-template-marker-check` substep so the user runs `sauce update` automatically if marker is missing in their daily template (the cowork orchestrators rely on it).
