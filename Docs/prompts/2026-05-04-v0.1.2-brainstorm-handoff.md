## What v0.1.2 needs to design + ship

### Distribution model (per existing design draft — verify still accurate)

The existing design (`Docs/plans/2026-05-03-multi-vault-automation-design.md`) sketched:

- **Thin stub `platformInstall.js`** in each consumer's `Docs/Meta/Templater/` — short script (likely <50 lines) that:
  1. Reads the consumer's `platform-config.json` for the workshop's path.
  2. Optionally calls `git -C <workshop_path> pull` (gated; opt-in).
  3. Dynamically requires the workshop's canonical `platform/install.js` and calls its exported function with `tp`.
- **Workshop git remote** (already configured 2026-05-04: `git@github-personal:willfell/beacon.git`).
- **Optional `git pull` gating** — opt-in flag in `platform-config.json` (e.g., `"auto_pull": true`)? Or always-on with user approval? Or never-on (manual `git pull` in workshop, separate from installer)?

### Open trails / refresh items (Phase 1 questions for new session)

> [!info] Items to verify against existing design
> 1. **Does the existing design still cover everything?** v0.1.3 introduced landmine #12, two new helpers, ergonomic manifest schema. Does the design assume the OLD shape? If yes, refresh.
> 2. **Auto-pull posture.** Three options: (a) always pull on every install (could surprise users); (b) opt-in via config flag; (c) manual-only (separate command). Existing design's recommendation should be re-validated.
> 3. **Stub-only-bootstrap vs full-thin-bootstrap.** The stub could either ONLY do the dispatch (assume workshop is pre-pulled) or ALSO do the pull. Trade-off: complexity vs convenience.
> 4. **Failure modes.** What happens if `git pull` fails (network down, merge conflict, dirty working tree)? Stub must fail-loud + bail; not silently dispatch to a stale workshop.
> 5. **Cross-machine setup.** Each consumer machine still needs `git clone` of the beacon repo at the expected relative path before the stub works. Document the one-time bootstrap.
> 6. **Version-bump semantics.** Workshop version 0.4.1 → 0.5.0 for v0.1.2 (breaking distribution-model change)? Or 0.4.2 (additive — old bootstrap copies still work for legacy consumers)? Decide explicitly.

### Code surfaces

- **NEW thin stub** at `platform/installer-stub.js` (or similar) — gets copied into consumer's `Docs/Meta/Templater/platformInstall.js` on initial bootstrap. After that, the stub auto-updates via `git pull`.
- **Bootstrap-copy × 3 ritual retires** — once stubs are in all three vaults, future cycles never touch them again. The CLAUDE.md non-negotiable + landmines that reference the ritual get updated.
- **Workshop self-test** — must continue to dogfood. The workshop's own `Docs/Meta/Templater/platformInstall.js` becomes a stub too (workshop pulls from itself? or workshop uses the canonical directly?). Decide explicitly.
- **Onboarding doc rewrite** — `Docs/use.md` "Onboarding a new consumer vault" section becomes: clone beacon to a known path → run a one-shot bootstrap script → done. No more "copy install.js × 3."

### Non-goals (carry forward into v0.1.2 design)

- **Auto-installing community plugins** (Slash Commander) — Obsidian API limitation; manual install stays.
- **GitHub Actions / CI** — out of scope; single-developer-single-machine workflow is enough for now.
- **Migration of existing legacy bootstrap-copy consumers to stubs** — should the v0.1.2 cycle ALSO migrate barebones + tmp-acc, or leave that to a follow-up? Decide explicitly.
- **Mobile support** — desktop-only stays.
