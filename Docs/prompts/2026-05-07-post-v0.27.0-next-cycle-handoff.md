## v0.28.0 design considerations (open at brainstorm time)

1. **Migration tool location** — new CLI verb `sauce migrate` OR standalone Node script in `platform/migrate/`?
2. **Source vault detection** — auto-detect from filesystem layout (Extras/People vs Resources/People) OR per-vault config file?
3. **Dry-run + diff** — emit a per-file change report before writing? Required for confidence on first run.
4. **Rollback / .sauce-backup posture** — every overwrite preserved as `.sauce-backup` + a final `migration.log` summarizing every write.
5. **Body regeneration vs preservation** — for `## Notes` content (free-form): preserve verbatim. For `## Meetings` Dataview block: replace with Sauce template body. Need a per-section preserve/regenerate decision matrix.
6. **Per-blueprint migrators** — separate migrators for People / Daily / Meetings / Journal / Projects / Todos / Finance / Trips? Or one monolithic migrator with per-section switches?
7. **Fresh sauce vault bootstrap** — `<vault>-sauce/` is bootstrapped first via `sauce` install.sh, THEN migration tool populates the spice/ namespace.
8. **Cross-vault de-dupe** — Headspace's `Cowork/context/people.md` priority-hub data should propagate to ONE migrated vault (Headspace) only, not all three.
9. **Scope of source vaults** — strictly the 3 named (`accuris`, `ero`, `headspace`)? Or generalize to any N source vaults via config?
10. **Carry items from v0.27.0** — bundle v0.27.x carries (autocomplete, AS6 coverage, atomic-write tmp orphan, NIT cleanups) into v0.28.0 OR ship as a separate v0.27.x PATCH cycle before migration?
