## Bug catalogue

### BUG-1 — Lucide icon names render as literal text in AccentButton (HIGH severity)

**Symptom:** every entity-create button label is prefixed with the raw Lucide name string:
- `users-plus New Meeting`
- `user-plus + New Person`
- (presumed) `folder-plus New Project`
- (presumed) `wallet-plus New Budget` etc.

**Root cause:** API mismatch.

`platform/mechanisms/entity-create/entity-create.js` `render()` passes `icon: spec.icon || plusIcon` to `customJS.AccentButton.render()`. `spec.icon` is a Lucide name string (e.g., `"users-plus"`); the SVG fallback `plusIcon` is only used when `spec.icon` is missing.

`platform/mechanisms/accent-button/accent-button.js:42` literally does `btn.innerHTML = opts.icon + ...`. AccentButton expects `icon` to be raw inline SVG HTML, not a Lucide name. Stringified Lucide names get inserted as text nodes.

**Affected manifests** (every `new_entity_buttons[].icon` is a Lucide name string):
- `platform/blueprints/meetings/manifest.json`: `"icon": "users-plus"`
- `platform/blueprints/people/manifest.json`: `"icon": "user-plus"`
- `platform/blueprints/project/manifest.json`: `"icon": "folder-plus"`
- `platform/blueprints/scratch/manifest.json`: check
- `platform/blueprints/finance/manifest.json`: budget `"wallet-plus"`, paycheck `"wallet-plus"`, invoice `"file-plus"`

**Fix options:**
- **(A) Resolve Lucide names in entity-create.js's `render()`:** call Obsidian's `setIcon(tempEl, name)` on a throwaway element, capture its innerHTML, pass to AccentButton. Smallest change; entity-create-local.
- **(B) Make AccentButton accept either SVG or Lucide name:** detect whether `opts.icon` starts with `<svg` vs is a kebab-name; if name, resolve via setIcon. API contract widening; other callers benefit. Larger blast radius — check every AccentButton call site.
- **(C) Switch entity-create manifests to inline SVG strings:** verbose; gives up Lucide compositionality. Not recommended.

`platform/mechanisms/nav-buttons/space-nav-buttons.js` has a hardcoded `ICONS` map (briefcase / scratch added in v0.44.0 + v0.40.1) — a third pattern. Each renderer has resolved icons differently. Brainstorm should pick one canonical approach.

**Test gap that let this through:** `run-entity-create.js` covers the EntityCreate class surface but never asserts the rendered DOM for icon presence. `run-helper-cases.js` doesn't exercise AccentButton dispatch from EntityCreate. The render-time integration isn't covered by any static harness — that's why preflight stayed green.

### BUG-2 — `type: person` missing from person frontmatter (HIGH severity)

**Symptom:** Alice Testperson.md was created without `type: person`. People hub queries that filter `where type = "person"` will skip notes created via the New Person button.

**Root cause:** `platform/blueprints/people/manifest.json` `new_entity_buttons[0].frontmatter_template` doesn't set `type`. Only ships `company / location / title / email / website / aliases / phone / tags[]`.

**Fix:** add `"type": "person"` to the frontmatter_template. One-line manifest edit. PATCH bump people@0.2.0 → 0.2.1.

**Why this slipped past v0.46.0:** the workshop seed harness (`platform/test/run-seed.js`) for people declarative entries seeds existing Alice/Bob/Carol with `type: person`. But the NEW person flow's frontmatter_template was never asserted to include `type`. Add an assertion: every blueprint's `new_entity_buttons[*].frontmatter_template` MUST emit `type: <blueprint-singular>` to align with audit conventions.

### BUG-3 — Path A `body_template` not resolved to vault-relative path (CRITICAL severity)

**Symptom:** project atlas + Map + Board sidecars all created with EMPTY bodies. Only frontmatter present.

**Root cause:** `platform/blueprints/project/manifest.json` declares:
```json
"body_template": "Template, Project.md",
"extra_files": [
  { "filename_pattern": "Project Map.md", "body_template": "Template, Project Map.md" },
  { "filename_pattern": "{{prompts.slug}}-board.md", "body_template": "Template, Project Board.md" }
]
```

The installer's `resolveEntityCreateEntry` (install.js:1664–1687) runs `substituteLenient` on these values — but since they have NO `{{var}}` tokens, the substitution is a no-op. The registry stores them verbatim.

At runtime, `entity-create.js` `_readBody` calls `app.vault.getAbstractFileByPath("Template, Project.md")` which resolves against vault root, finds nothing (the actual installed location is `ranch/templates/Template, Project.md`), returns `""`. Body is empty.

**Fix:** the project manifest's body_template values must be `{{templates_path}}/Template, Project.md` (or use a different convention). Three sub-options:

- **(A) Manifest-side fix:** rewrite the three project manifest entries to include `{{templates_path}}/` prefix. Minimal change. PATCH bump project@1.7.1 → 1.7.2.
- **(B) Convention-shift:** make `body_template` ALWAYS resolved relative to the blueprint's templates dir; installer prepends `{{templates_path}}/` automatically. Tightens the contract; reduces foot-gun risk. Manifest writers just specify the bare filename. BREAKING change for any other Path A consumer (currently project is the only one).
- **(C) Runtime fallback:** make `_readBody` look up `ranch/templates/<basename>` as a fallback when the bare path fails. Heuristic; can mask other path errors.

(B) is the right long-term move (Lego principles — DRY); (A) is the immediate-ship fix.

**Test gap:** workshop dogfood install + audit didn't catch this. The audit walker counts an entry "aligned" if the manifest declaration is well-formed — it doesn't simulate the body-template resolution. Need an installer- or e2e-level invariant: every entry with `body_template` must reference a path that resolves to a real file post-install.

### BUG-4 — extra_files[].body_template has identical Path A regression (CRITICAL severity)

**Symptom:** Project Map.md + test-project-alpha-board.md created as empty files (1 line each).

**Root cause:** same as BUG-3 — extra_files inherit the bare-filename pattern. Installer substitutes-but-can't-substitute (no tokens to expand). `_readBody` reads against vault root, returns `""`.

**Fix:** same as BUG-3 — manifest paths need `{{templates_path}}/` prefix, OR convention-shift to auto-prepend.

### BUG-5 — Scratch UX (LOW severity / not strictly a bug)

**Symptom:** user reported "not entirely sure what's going on here" after the scratch flow created a note with SpaceNavButtons + ScratchLeafActions blocks. Frontmatter + body technically correct.

**Likely cause:** the leaf-actions block renders unfamiliar UI (probably a "Back to day" link + actions). Without prior context the user didn't know what action it offered.

**Action:** brainstorm whether scratch's leaf actions need a brief intro / label / illustration in the rendered output. Not a v0.46.3 blocker.

### BUG-6 — Audit walker counts "unverifiable" entries (LOW severity / documented limitation)

**Symptom:** audit shows `aligned=5 + unverifiable=2 = 7` entries — two entries marked unverifiable.

**Root cause:** `entity-create-walker.js` falls back to `unverifiable` when an entry uses `inline_body` with non-static content or `body_template` paths it can't statically resolve. This is documented in FLN-d.

**Action:** revisit the walker after BUG-3 / BUG-4 are fixed — once Path A entries can be reliably static-fingerprinted, the `unverifiable` cases should drop or get a more specific categorization. Low-pri compared to BUG-1..4.

## v0.46.3 brainstorm prompts

The brainstorm should decide:

### Q1 — Bundle into one v0.46.3, or sequence multiple patches?

Bundling BUG-1 + BUG-2 + BUG-3 + BUG-4 into v0.46.3 is small (manifest edits + 1 mechanism update). Sequencing as 2–4 patches gives finer-grained tap PRs. Recommendation: bundle as v0.46.3 PATCH; brainstorm to validate.

### Q2 — BUG-1 fix: (A) entity-create-local Lucide resolver, (B) AccentButton API widening, (C) inline SVG manifests?

Pick one. (A) is smallest scope; (B) is most ergonomic for future mechanisms. Note: nav-buttons has its own hardcoded ICONS map (third pattern). A unified Lucide-resolution helper (under `platform/mechanisms/icons/` or similar) could DRY all three. Out-of-scope dimension: should sauce have a single canonical icon-resolution helper used by every renderer? Worth a 5-minute discussion before committing to the smallest fix.

### Q3 — BUG-3 / BUG-4 fix: (A) manifest-side path expansion, (B) convention-shift to auto-prepend templates_path?

(B) reduces foot-gun risk for future Path A blueprints (boards / journal / etc. that may use body_template). (A) is faster to ship. If the brainstorm commits to (B), document the contract in the entity-create schema + add validator coverage. If (A), document the body_template path convention as a numbered landmine to prevent the bare-filename mistake recurring.

### Q4 — BUG-2 fix scope: just people, or audit every blueprint's new_entity_buttons[*].frontmatter_template for missing `type` field?

Check meetings (has `type: meeting` ✓), project (has `type: project` ✓), scratch (has `type: scratch` ✓), finance (budget `type: budget`, paycheck `type: paycheck`, invoice `type: invoice` — all ✓). So only people is missing. But add a validator rule (deep validator Layer 2) requiring every blueprint with a `<bp>` audit type to also have `type: <bp-singular>` in any new_entity_buttons[] frontmatter_template. Prevents future blueprints from forgetting.

### Q5 — Test harness deltas: what to add so v0.46.x bug class doesn't recur?

Candidates:
- **run-entity-create.js**: add a per-blueprint assertion that the materialized atlas note carries the expected `type` frontmatter + non-empty body when `body_template` is declared. Requires a test seed for Path A (project) — currently `run-integration-smoke.js` exercises 6/7 flows but not the body content.
- **run-renderer.js / new harness**: add a render-time DOM assertion that the AccentButton's icon HTML contains `<svg` (rejects raw Lucide name strings). Validates the icon-resolution contract.
- **audit walker**: extend `entity-create-walker.js` to verify `body_template` paths resolve to real files in the workshop blueprint dir (catches the bare-filename mistake at workshop time, before consumer install).

### Q6 — Pacing for v0.46.3

Subagent-driven? Controller-direct? Bug-1 needs design judgment (icon API direction). Others are mechanical once the design is set. Suggest: brainstorm for ~15 min on Q1–Q4 → write design doc → controller-direct execution for the mechanical fixes → subagent-spec for the icon refactor if (B) is chosen.
