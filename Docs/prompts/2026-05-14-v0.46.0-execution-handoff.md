## Execution batching: per-stage, not per-step

The plan has ~80 numbered steps clustered into 12 stages. **Batch by stage, not by step.** One implementer dispatch per stage (the implementer follows all steps in order, committing per the plan's pre-drafted commit messages — usually 1 commit per stage; S8/S9/S10 each commit separately under finance's single MINOR bump landing at S10). One spec-reviewer dispatch per stage. One code-quality-reviewer dispatch per stage. Total: ~36 subagent dispatches (12 stages × 3 roles) plus any review-loops.

This matches sauce's recent cadence (v0.36.0, v0.38.0, v0.45.0 all shipped 12 stages this way).

## Per-stage gates (in order)

After each implementer reports `DONE`:
1. Verify `npm run release:preflight` exits 0 in the post-stage state (only required at stages that touch testable code — S1 / S2 / S3 / S11 / S12; per-blueprint migration stages S4–S10 should still leave preflight green but the harness deltas land in S11).
2. Dispatch spec-compliance reviewer (`spec-reviewer-prompt.md` template).

5. If code-quality reviewer ❌: implementer fixes → re-quality-review → repeat until ✅.
6. Mark stage complete; proceed to next stage.
