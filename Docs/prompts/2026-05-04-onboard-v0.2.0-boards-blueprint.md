## Why each cycle exists (the architectural context)

### v0.1.x patch — validator/audit subsystem fixes (recommended NEXT)

> [!warning] Four findings surfaced by v0.1.1 T4.7 manual smoke
> v0.1.0 shipped the validator + audit-walker mechanisms but they were never end-to-end tested against the project blueprint's rule. v0.1.1's T4.7 manual smoke discovered:
>
> 1. **No slash-command UX for `/new-project` or `/audit`.** Templater templates exist + `commands/<name>.md` markdown files ship, but no slash-command plugin is wired up to register them. Users invoke via Templater's Insert Template modal only.
> 2. **No standard runner UX for validator / audit-walker.** Manual invocation requires one-shot Templater snippets via "Replace templates in active file." No `Validate.md` template, no `Audit.md` template, no `/audit` slash command.
> 3. **`validate.js` input resolution broken for current Templater.** `tp.user.validate(tp.file, ...)` (the documented usage) crashes with `e.lastIndexOf is not a function` because `tp.file` is a wrapper, not a TFile. Workaround: `{ file: app.workspace.getActiveFile() }`. Real fix: defensive input handling.
> 4. **`validate.js:checkRequiredBlocks` schema mismatch with project rule.** Validator expects `spec.content` (string snippet); project rule uses `spec.kind` + `spec.must_call` + `spec.via`. Result: `expectedSnippet.slice(0, 80)` crashes on undefined.
>
> Real fix scope: ship `Validate.md` + `Audit.md` runner templates; defensive input handling in `validate.js`; rewrite `checkRequiredBlocks` to handle the real rule schema (parse the dataviewjs block, check `must_call` appears via `via` mechanism); decide slash-command-plugin dependency strategy. Likely 1-2 stages. Critical because the validator hook fires on every template execution today.

### v0.2.0 — boards blueprint + retire project's Board

> [!warning] The "wrong primitive" finding (v0.1.1 S4 manual smoke)
> v0.1.1 shipped a Dataview-list-rendering "kanban" via the project blueprint. Manual smoke surfaced:
> 1. **Wrong primitive.** User's mental model + accuris reference is **Obsidian-Kanban-plugin board** with dated card notes — not a Dataview view.

> [!success] Decisions locked during 2026-05-03 brainstorming + 2026-05-04 refinement
> - **`boards` is the right module name** (not `to-do` — that's a future, distinct daily-checklist module).
> - **Module-directory invariant under `beacon/` namespace.** Each blueprint owns ONE directory at `beacon/<module_directory>/`. Boards blueprint materializes at `beacon/boards/`. ## Constraints (carrying forward — non-negotiable)

These apply throughout the cycle. From `CLAUDE.md` + all 11 landmines:

- **No personal content** in the workshop vault.
- **All blueprint metadata is JSON, not YAML** (landmine #6).
- **`template_source` in `nav_buttons[]` declarations is BASENAME ONLY** — installer prepends `${content_path}/${source-name}/` (or `${module_directory}/` once that semantics is enforced) automatically.
- **Module-directory invariant** (landmine #11) — every new blueprint manifest declares `module_directory`. Cross-module writes are illegal.
- **Substitution variables come ONLY from `platform-config.json:variables`** (landmine #9). variants.json + top-level `vault_identity` are NOT consumed. The new `{{module_directory}}` variable is sourced from the blueprint manifest, not platform-config — installer maps it per-blueprint.
- **Triple-bump protocol for any test that needs to re-run `applyNavButtons` / `applyRuleFragment` / etc.** (landmine #10) — bump item manifest + workshop manifest entry + consumer subscription transiently; restore all three.
- **Failure-loud posture** — every error path = Notice + history entry. No silent fallbacks.
- **Workshop dogfods every release.** Self-install must be green before any consumer promote.
- **Desktop-only.** Mobile (iOS) is unsupported. `require("fs")`, `child_process.execSync` only work on macOS/Windows/Linux.
- **Bootstrap-copy discipline** — superseded by v0.1.2's thin stub. If v0.1.2 has shipped, the resync ritual is gone. If not, you're in the wrong cycle order.
- **Ask before** — bumping `workshop_version`, editing `.obsidian/`, hand-editing `platform-installed.json`, removing files inside `platform/blueprints/*/` after a version is promoted.
