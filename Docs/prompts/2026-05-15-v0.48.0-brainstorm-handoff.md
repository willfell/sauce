## BUG-5 problem statement (verbatim from v0.47.0 FLN-i)

Per-project kanban-board task-creation flow does not prompt for workstream + atlas missing helper buttons:

**Symptoms (observed at headspace 2026-05-15 post-v0.47.0):**
- User creates a project (`test-project-alpha`). Atlas at `spice/projects/test-project-alpha/Test Project Alpha.md`.
- User opens the kanban board at `spice/projects/test-project-alpha/test-project-alpha-board.md`.
- User clicks "+ Add a card" → creates a new task note at `spice/projects/test-project-alpha/tasks/<TaskName>.md`.
- **Expected:** Templater prompts for workstream (via the `tp.system.suggester` in `Template, Kanban Card.md`).
- **Actual:** No workstream prompt fires. The raw `<%* %>` block is pasted as literal text into the new task note.
- **Also:** The project atlas page (`Test Project Alpha.md`) has no "+ Add Task" / "+ Add Task Board" buttons — those were stripped in v0.46.0 with the broader `_createTaskNote`/`_createTaskBoard` removal.

**Root cause:** v0.46.0 cycle removed ~184 LOC from `ProjectNavButtons` (`_promptForProjectName`, `_createProject`, `_renderProjectsHub`, **plus** `_createTaskNote` + `_createTaskBoard` and their button-render code). The replacement plan was the kanban plugin's "+ Add a card" UI calling its template setting (`"new-note-template":"ranch/templates/Template, Kanban Card.md"`). The kanban plugin uses **Obsidian's CORE template paste mechanism, NOT Templater** — so the `<%* %>` workstream picker syntax stays raw and never executes. project's manifest declares **zero `templater_folder_templates[]`** entries, so Templater's `trigger_on_file_creation: true` doesn't wake up on `spice/projects/<slug>/tasks/` file creation either.

**v0.47.0 innocence proof:** install at headspace did NOT touch `ranch/scripts/project/project-nav-buttons.js` (source bytes unchanged from v0.46.x; file timestamp May 15 13:37:11 predates the v0.47.0 install at 18:02), did NOT touch `.obsidian/plugins/templater-obsidian/data.json` (last modified May 13).

## Three design options for the BUG-5 fix (need to pick one or sketch a hybrid)

### Option A — Extend entity-create with a `pick-from-frontmatter` prompt type

**Shape:** add a new prompt-type to entity-create's JSON schema:

```json
{
  "key": "workstream",
  "label": "Workstream",
  "type": "select",
  "source": { "from_frontmatter": { "file_match": "atlas", "field": "workstreams", "label_key": "name", "value_key": "id" } }
}
```

Then declare a `+ New Task` entity-create entry in project's manifest, with `render_in.kind: "hub"` pointing at the project ATLAS page (or each per-project atlas via a runtime context lookup). The kanban board's "+ Add a card" UI gets retired in favor of the entity-create button on the atlas.

**Pros:** aligned with v0.46.0's mechanism-extraction philosophy. Declarative. Future-proof for other blueprints with sibling-frontmatter-derived selects.

**Cons:** entity-create's `target_path` is currently fixed per-blueprint (`{{module_directory}}/<file>.md`); per-project hub injection at `spice/projects/<slug>/...` needs new context plumbing. Atlas frontmatter lookup needs file-discovery logic. Kanban board's "+ Add a card" UI continues to NOT prompt for workstream — only the new entity-create button does. Retraining the muscle memory is a cost.

**Effort:** MEDIUM (~1-2 cycles). New prompt-type schema + entity-create runtime + installer wiring for per-project hub injection.

### Option B — Restore `_createTaskNote` + `_createTaskBoard` in `ProjectNavButtons` (revert + modernize)

**Shape:** bring back the ~184 LOC stripped in v0.46.0, modernized to use entity-create's substitution catalogue (so the helpers reuse `{{prompts.X}}` / `{{current_file.frontmatter.workstreams}}` tokens). The atlas page regains "+ Add Task" / "+ Add Task Board" buttons. Kanban board's "+ Add a card" UI continues to bypass the workstream prompt — users are pushed toward the atlas buttons.

**Pros:** Smallest design footprint. Code already exists in git history (`git show <v0.45.0-sha>:platform/blueprints/project/helpers/project-nav-buttons.js`). Restores user-observable parity with pre-v0.46.0 behavior.

**Cons:** Pulls back from the v0.46.0 mechanism-extraction direction. Two paths for task creation: atlas helper buttons (workstream-aware) + kanban-plugin (workstream-blind). Confusing UX.

**Effort:** SMALL (~1 cycle, mostly restoration + harness updates).

### Option C — Register Templater folder template for project tasks

**Shape:** add `templater_folder_templates[]` to project's manifest:

```json
"templater_folder_templates": [
  { "folder": "spice/projects", "template": "{{templates_path}}/Template, Kanban Card.md" }
]
```

Templater's folder-template feature uses most-specific-prefix matching. But the folder spec must be literal; we can't put `spice/projects/<slug>/tasks` with a variable slug. If we put `spice/projects`, it would fire on EVERY new file under spice/projects (atlas, map, board, tasks/) — too broad.

Templater modern versions (1.16+) may support glob patterns in folder_templates entries — needs verification against the brew-bundled Templater plugin version at headspace. If supported, `spice/projects/*/tasks` is the target.

**Pros:** Single-line manifest change if Templater supports the glob pattern. Restores the v0.46.0 design intent (kanban-plugin + Templater integration).

**Cons:** Depends on Templater plugin version support. May silently downgrade on older Templater. Doesn't restore the atlas helper buttons (which the user also expects per BUG-5).

**Effort:** TINY if it works; otherwise it's a non-starter.

## Brainstorm questions

This is a brainstorm-handoff (not execution-handoff). The next session must work through these BEFORE writing a plan:

**Q1 — Cycle scope.** Is the v0.48.0 scope:
- (a) BUG-5 fix only (smallest)
- (b) BUG-5 + FLN-a/b/c icons polish (recommended bundle)
- (c) BUG-5 + FLN-a/b/c + FLN-e schema-validator (medium)
- (d) something else

**Q2 — BUG-5 design option.** Which of A / B / C above (or a hybrid)? Recommend running this through `superpowers:brainstorming` first — the trade-offs are real.

**Q3 — Workstream picker mechanism (if Option A wins).** If extending entity-create with `pick-from-frontmatter`:
- How does the prompt locate the atlas note from the task-creation context?
- What if the atlas has no workstreams declared? (skip prompt? force-skip? error?)
- Does the prompt offer a "create new workstream" affordance, or strictly pick from existing?

**Q4 — Atlas helper buttons (Option B or A's "+ New Task" entity-create button).** Should the atlas regain the buttons that were stripped in v0.46.0?
- If yes: which ones — `+ Add Task`, `+ Add Task Board`, `+ Add Sub-Note`, or just `+ New Task`?
- If A's entity-create button: how does it render INSIDE the atlas (which is per-project, not a static path)? Likely needs per-project marker injection at install time OR a runtime context-aware EntityCreate.render call.

**Q5 — Kanban-plugin "+ Add a card" coexistence.** The plugin's UI is convenient. Do we:
- Leave it alone (users who don't need workstream tagging can still use it)
- Replace it (remove the `new-note-template` setting from the board file template)
- Augment it (post-creation Templater hook that retroactively processes the file)

**Q6 — Templater folder template hedge.** Even with Option A or B as the primary, should we add Templater folder_templates as defense-in-depth? Needs Templater version + glob support verification.

**Q7 — Acceptance gates.** What's the v0.48.0 done-definition for BUG-5? Suggested:
- New task created via the chosen flow has frontmatter `workstream: <id>` set correctly.
- Atlas page shows expected helper buttons (if applicable).
- Headspace re-validation: create at least 2 tasks under 2 different workstreams; verify both end up tagged correctly.

**Q8 — Pacing.** This will likely be a subagent-driven cycle (per v0.47.0's pattern). Estimate ~6-10 stages depending on Option choice.
