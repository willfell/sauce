## Cross-cutting gotchas codified during recent cycles (READ BEFORE STARTING)

> [!warning] Gotchas surfaced 2026-05-03 / 2026-05-04 — load-bearing for any cycle touching runTemplaterTemplate, CustomJS classes, or invoke_command
>
> **Gotcha 1: Bracket-escape gap on runTemplaterTemplate folder/filename.** RESOLVED at v0.4.2 S1 — split-field schema makes literal text never reach `moment.format()`. Manifests still using legacy `folder` / `filename` fields are silently ignored (BREAKING). Future blueprints inherit the safety for free.
>
> **Gotcha 2: Node-harness install + Obsidian's plugin auto-rescan.** `platform/test/run-install.js` writes via `fs` (Node), bypassing Obsidian's vault adapter. Obsidian's CustomJS plugin watches the vault adapter for change events to auto-rescan its jsFolder. After a Node-harness install adds new files OR rewrites class bodies, the user MUST Cmd+R Obsidian to force plugin re-init. Smoke checklists for any cycle adding OR rewriting CustomJS classes should explicitly call out the Cmd+R step.
>
> **Gotcha 3: T-FIX1 mkdir module_directory at install time** (v0.3.0 codified). Every blueprint with `module_directory` set gets that directory mkdir'd between `applyPreInstall` and `files[]` loop. Mechanisms exempt.
>
> **Gotcha 4: Force re-process via installed.json edit (CLAUDE.md ask-before-acting).** `install.js:111` short-circuits per-item install when `subscribed.version === installed.version`. Tests/cycles that need to exercise behavior INSIDE `installItem()` must triple-bump versions OR drop the entry from `<consumer>/Docs/Meta/platform-installed.json:installed[]` (CLAUDE.md ask-before-acting; user-approved).
>
> **Gotcha 5: User-override preserved on folder_templates + nav-button registry overwrites.** `applyTemplaterFolderTemplates` preserves user-customized template paths. `applyNavButtons` REPLACES the contributions[<source>] array on each install. Existing `folder_templates` entries from prior installs may persist as user-overrides — cosmetic clutter but functionally inert.
>
> **Gotcha 6: Mid-cycle redesign methodology.** When smokes surface a misalignment with the user's mental model, ACCEPT the redesign rather than ship the wrong primitive. Workflow: pause SDD → invoke `de:brainstorming` → present revised design → get approval → invoke `de:writing-plans` → resume SDD. Used twice now (v0.3.2 v1→v2; v0.4.2 v1→v2). Pattern is reproducible.
>
> **Gotcha 7: Mobile-aware visual design as platform value (v0.4.2).** Every renderer (CustomJS class) MUST be designed for both desktop AND mobile from now on. Adaptive layouts, touch-friendly targets, no hover-only affordances. The installer remains desktop-only (landmine #8 unchanged); the visual contract is what changes. Carry forward to all future blueprint CustomJS class authoring.
>
> **Gotcha 8: Filename pattern `<Name>-YYYY-MM-DD` invariant (v0.4.2).** All date-routed daily artifacts use date-suffix; hyphen-only separator; no spaces. Future blueprints with date-routed artifacts MUST follow this convention. Date-extraction in CustomJS classes uses regex `\d{4}-\d{2}-\d{2}` against filenames (handles both prefix-style legacy + suffix-style v0.4.2+).
>
> **Gotcha 9: In-cycle re-process bump rule (v0.4.2).** When reusing in-cycle work staged at version X, in-cycle revisions to that work must bump to version X+1 to force consumer re-process. Plan-time inheritance assumption (versions stay) breaks if any consumer was already installed at X. Pre-flight diff installed.json vs subscription before assuming a no-op.
>
> **Gotcha 10: Verify invoke_command command ids against runtime (v0.4.2).** When designing an invoke_command action, verify the actual command id via Obsidian DevTools (`Object.keys(app.commands.commands).filter(k => k.includes('<keyword>'))`). v0.4.2 design assumed `daily-notes:goto-today` for the Today button; actual id is `daily-notes`. Don't trust naming conventions or Obsidian docs; use the live runtime as truth. Note that `daily-notes:goto-prev` / `daily-notes:goto-next` exist as core commands but they create-on-demand rather than skip-to-nearest-existing — usable for some flows but not for the "skip-to-nearest" UX.
