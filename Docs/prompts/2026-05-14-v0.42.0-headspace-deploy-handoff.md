## Known obstacles you will hit (and how to power through)

### Obstacle 1 — Subscription pins lag the new catalogue

After `brew upgrade sauce` lands cowork@0.4.0 in the brew Cellar's `platform/manifest.json` catalogue, the Headspace subscription will still pin cowork@0.3.0. `sauce reinstall` will refuse to install cowork@0.4.0 if the subscription still says 0.3.0. **Fix:** Edit `<vault>/ranch/platform-subscription.json` and bump the cowork row from `"version": "0.3.0"` to `"version": "0.4.0"` directly. Same for any other pin that lags the catalogue (likely **convenience@0.2.0 → whatever the catalogue now has**, possibly **0.2.4 or higher**).

**Aggressive resolution policy for this handoff:** Bump every subscription pin to match the catalogue version exactly. The user wants the vault current. Don't ask permission — just match all pins to catalogue and proceed. Capture the diff for the result writeup.

### Obstacle 2 — Pre-existing project version skew (FLN-2 in v0.42.0 result doc)

The workshop's own subscription pins `project@1.5.0` but the catalogue has `project@1.6.1`. This same skew likely exists in headspace if headspace pins `project@1.5.0` (check). **Fix:** Bump headspace's project pin to match the catalogue. Apply the same aggressive bump-everything policy from Obstacle 1.

### Obstacle 3 — sauce CLI version may be older than what understands the new manifest fields

`sauce doctor` reported the brew binary is at v0.38.0 (pre-v0.42.0). After `brew upgrade sauce`, this should bump. If the brew tap hasn't picked up v0.42.0 yet (the Formula bump might lag the tag push by a few minutes), `brew info sauce` will still show 0.38.0. **Fix:** If brew tap is stale, run `brew update` first. If still stale after that, check the tap's GitHub Actions: `gh run list --repo willfell/homebrew-sauce --limit 3` — confirm the bump-tap workflow ran. If it didn't, you can manually trigger it OR fall back to keeping `active-pantry` linked to the dev repo (the dev repo IS at v0.42.0 with cowork@0.4.0 source; sauce will use it directly).

### Obstacle 4 — Templates resolve at create-time, not install-time

The new `Weekly Note.md` and `Monthly Note.md` templates ship with Templater placeholders (`<% tp.file.title %>`, `<% moment(...).startOf("isoWeek") %>`, etc.). These resolve at the moment a USER clicks "This Week" / "This Month" nav-button OR runs `/weekly` / `/monthly`. **No action needed during install** — install just materializes the literal template files. Don't be alarmed by the unresolved `<% ... %>` strings in `<vault>/ranch/templates/Weekly Note.md` after install; that's correct.

### Obstacle 5 — `sauce reinstall --vault` may show "1 error/skip history entry — exit 1"

This happens when the installer skips one or more blueprints due to subscription pin / catalogue mismatch (Obstacle 1+2). After fixing all pin skews per Obstacles 1+2, re-run; the exit-1 should resolve to exit-0.

### Obstacle 6 — Headspace's `Cowork.md` may already exist with v0.3.0 content

The installer's posture for `Cowork.md` is overwrite-on-install (no `skip_if_exists`). So the user's existing `Cowork.md` will be replaced with v0.4.0's body (which adds the Timeframes section + scaffold-status callout). If the user has hand-edited their personal `Cowork.md`, those edits will be lost. **For this deploy:** if `<headspace>/spice/cowork/Cowork.md` exists with hand-edits, copy it to `<headspace>/spice/cowork/Cowork.md.user-backup-2026-05-14` before reinstall as a safety net. Mention this in your final report so the user knows where their backup is.
