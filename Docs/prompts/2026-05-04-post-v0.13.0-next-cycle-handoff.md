## Cross-cutting gotchas codified during recent cycles (READ BEFORE STARTING)

> [!warning] Gotchas surfaced 2026-05-03 / 2026-05-04 — load-bearing for any cycle touching runTemplaterTemplate, CustomJS classes, invoke_command, new renderer action_types, or BeaconCards
>
> **Gotcha 1: Bracket-escape gap on runTemplaterTemplate folder/filename.** RESOLVED at v0.4.2 S1 — split-field schema makes literal text never reach `moment.format()`. Manifests still using legacy `folder` / `filename` fields are silently ignored (BREAKING). Future blueprints inherit the safety for free (journal@0.1.0 v0.13.0 confirmed).
>
> **Gotcha 2: Node-harness install + Obsidian's plugin auto-rescan.** `platform/test/run-install.js` writes via `fs` (Node), bypassing Obsidian's vault adapter. Obsidian's CustomJS plugin watches the vault adapter for change events to auto-rescan its jsFolder. After a Node-harness install adds new files OR rewrites class bodies, the user MUST Cmd+R Obsidian to force plugin re-init. Smoke checklists for any cycle adding OR rewriting CustomJS classes should explicitly call out the Cmd+R step. **v0.13.0 reinforced:** even one-line ICONS additions to `space-nav-buttons.js` need a Cmd+R for the renderer to pick up the new SVG.
>
> **Gotcha 3: T-FIX1 mkdir module_directory at install time** (v0.3.0 codified). Every blueprint with `module_directory` set gets that directory mkdir'd between `applyPreInstall` and `files[]` loop. Mechanisms exempt. **v0.13.0 reinforced:** journal blueprint has no install-time files under its module_directory (date-routed children land at runtime), but `beacon/journal/` still gets mkdir'd.
>
> **Gotcha 4: Force re-process via installed.json edit (CLAUDE.md ask-before-acting).** `install.js:111` short-circuits per-item install when `subscribed.version === installed.version`. Tests/cycles that need to exercise behavior INSIDE `installItem()` must triple-bump versions OR drop the entry from `<consumer>/Docs/Meta/platform-installed.json:installed[]` (CLAUDE.md ask-before-acting; user-approved).
>
> **Gotcha 5: User-override preserved on folder_templates + nav-button registry overwrites.** `applyTemplaterFolderTemplates` preserves user-customized template paths. `applyNavButtons` REPLACES the contributions[<source>] array on each install. Existing `folder_templates` entries from prior installs may persist as user-overrides — cosmetic clutter but functionally inert.
>
> **Gotcha 6: Mid-cycle redesign methodology.** When smokes surface a misalignment with the user's mental model, ACCEPT the redesign rather than ship the wrong primitive. Workflow: pause SDD → invoke `de:brainstorming` → present revised design → get approval → invoke `de:writing-plans` → resume SDD. Used four times now (v0.3.2 v1→v2; v0.4.2 v1→v2; v0.5.0 inline S2.x sub-stages; v0.12.0 S3.4.1 tasks-as-bullets revert). v0.13.0 did NOT need a mid-cycle redesign (clean single-stage cycle). Reinforced via v0.11.0's SIX-bump iteration. Pattern is reproducible.
>
> **Gotcha 7: Mobile-aware visual design as platform value (v0.4.2; reinforced v0.5.0 + v0.11.0 + v0.12.0 + v0.13.0).** Every renderer (CustomJS class) MUST be designed for both desktop AND mobile. Adaptive layouts, touch-friendly targets, no hover-only affordances. BeaconCards 0.2.x (chip-collapse on small screens; row-layout adapts; subtitle two-line-wrap word-break preserved) is the canonical pattern reference. Installer remains desktop-only (landmine #8 unchanged); the visual contract is what changes.
>
> **Gotcha 8: Filename pattern `<Name>-YYYY-MM-DD` invariant (v0.4.2; reinforced v0.13.0).** All date-routed daily artifacts use date-suffix; hyphen-only separator; no spaces. Date-extraction in CustomJS classes uses regex `\d{4}-\d{2}-\d{2}`. **NOTE:** project artifacts are slug-routed (NOT date-routed) — gotcha 8 does NOT apply to project files. **v0.13.0 reinforced:** journal blueprint's `Journal-YYYY-MM-DD.md` shape held; if v0.6.x date-aware nav buttons cycle ships, the active-file date-extraction MUST use this same regex.
>
> **Gotcha 9: In-cycle re-process bump rule (v0.4.2; reinforced v0.5.0 + v0.11.0 + v0.12.0).** When reusing in-cycle work staged at version X, in-cycle revisions to that work must bump to version X+1 to force consumer re-process. v0.5.0 was a SIX-bump cycle (project 0.3.0 → 1.2.2). v0.11.0 was a SIX-bump cycle. v0.12.0 was a TWO-inline-CF cycle (cards@0.2.1 + daily@0.2.1). **v0.13.0 reserved 0.13.x headroom UNUSED** — single-stage clean cycle. **Pattern is now strong evidence over THREE consecutive cycles + ONE counter-example:** any cycle with anticipated polish/feedback should reserve "Y.Z.x" headroom upfront; pure-additive blueprints near-cloning existing precedent can land single-stage. Pre-flight diff installed.json vs subscription before assuming a no-op.
>
> **Gotcha 10: Verify invoke_command command ids against runtime (v0.4.2).** When designing an invoke_command action, verify the actual command id via Obsidian DevTools (`Object.keys(app.commands.commands).filter(k => k.includes('<keyword>'))`). Don't trust naming conventions or Obsidian docs; use the live runtime as truth.
>
> **Gotcha 11 (v0.12.0; reinforced v0.13.0): Dual-source-of-truth for blueprint versions.** Per-blueprint version bumps require lockstep edits to BOTH (a) the blueprint's own `manifest.json` AND (b) the workshop catalogue line at `platform/manifest.json` (~line 20 — the `blueprints[]` entry's `version` field). The installer reads versions from the workshop catalogue (NOT the blueprint manifest) for subscription-pin compares. v0.12.0 S2.2 missed the catalogue edit; first barebones install run skipped meetings with `subscription pins meetings@0.2.0 but workshop has 0.1.2` — surfaced as a silent no-op. **v0.13.0 lockstep landed clean** — three blueprint manifests + three catalogue lines edited as paired commits with no skip-events; discipline reproduces. **Action item:** any "bump <blueprint>" task must include both edits as a single edit pair. Future systemic fix candidate (v0.12.x?): single-source-of-truth schema where workshop catalogue versions are derived from blueprint manifests at install time.
>
> **NEW pattern (v0.5.0, not yet a numbered gotcha): renderer dispatcher action_types must have matching install.js validateAndResolve branches.** v0.5.0 surfaced openLink as dead-code installer-side — `target` literal `{{module_directory}}` was reaching the registry unresolved because the openLink branch didn't substitute. Landed CF-1. **Lesson:** for any new renderer action_type, ALSO add a `validateAndResolve` branch in install.js OR placeholders won't substitute at install time.
>
> **NEW pattern (v0.11.0, not yet a numbered gotcha): BeaconCards-via-customjs-guard wrapping is the canonical dataviewjs callsite.** Direct `customJS.BeaconCards.render(dv, ...)` from a `dataviewjs` block violates landmines #1/#2 (cold-load TDZ on `customJS`). The cohesion pattern: ship a thin wrapper class per use-site that internally calls BeaconCards, then invoke it via `dv.view("Docs/Meta/Views/customjs-guard", { class: "<WrapperName>" })`. v0.11.0 examples: `ProjectNotesCards` + `ProjectReferencedByCards`. v0.12.0 example: `MeetingsHubCards` rewritten as canonical thin wrapper (~159 → ~111 LOC). Carry forward to every future BeaconCards consumer.
>
> **NEW posture (v0.12.0): BeaconCards = hubs only; bullet lists for tasks.** v0.12.0 S3.4.1 reverted SpaceDailyDashboard tasks panel from BeaconCards to a flat `<ul>`, validating the cards-mechanism-fit audit's "EXTEND with caution: card grid for tasks would be visually heavier than the current at-a-glance list" verdict. **Posture:** BeaconCards is the right primitive for hub-style listings where each item is a real file with title/subtitle/meta worth surfacing visually. Bullet lists remain the right primitive for compact at-a-glance task panels (text + click-to-jump). Don't reach for BeaconCards for inline task / checklist / queue surfaces. Audit-flagged "EXTEND with caution" verdicts warrant a brainstorm gate BEFORE migrating.
