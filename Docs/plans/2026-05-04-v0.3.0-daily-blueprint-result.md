## The codified module-directory mkdir mechanic — invariant tightened

> [!success] What changed in T-FIX1 (in-cycle installer fix)
> v0.2.0 codified the module-directory invariant at the **schema** level (`module_directory` field required on every blueprint; per-blueprint `{{module_directory}}` substitution overlay resolves to `beacon/<bare-name>`). But until v0.3.0, the directory `beacon/<module>/` was only **implicitly** created — as a side-effect of `files[]` writes that happened to land under `{{module_directory}}/...` (e.g., boards@0.1.0 writes `beacon/boards/To-Do-Board.md`, which auto-creates `beacon/boards/`).
>
> daily is the first blueprint with NO `files[]` destinations under its own `module_directory` — its template lands under `Docs/Meta/Templates/`, its helper script lands under `Docs/Meta/Scripts/daily/`, and the `beacon/daily/` directory only ever holds runtime-generated daily-note files. Result: pre-T-FIX1, `beacon/daily/` never existed at install time, and Obsidian's core Daily Notes plugin (which requires the configured `folder` to pre-exist) bailed at "Open today's daily note" with `Folder "beacon/daily" not found.`
>
> T-FIX1 makes the invariant **explicit** at the installer level: every blueprint with a `module_directory` gets that directory mkdir'd at `installItem` time, between `applyPreInstall` and the `files[]` loop. From "directory exists if files[] writes there" to "directory definitively exists post-install."
>
> Mechanisms remain exempt (the gate is `variables.module_directory` truthiness; mechanisms don't carry that overlay variable). Future blueprints whose content all lands outside `{{module_directory}}/...` (siblings of daily — meetings, to-do, etc.) inherit this guarantee for free.

## Surprises during execution

> [!warning] Surprise 1 (resolved): Smoke A failed — `beacon/daily/` never created
> First barebones install completed exit 0 — all events green: 2 file writes, 1 `core_plugin_settings applied`, 1 post_install notice. The `.obsidian/daily-notes.json` shape was correct (`folder: "beacon/daily"`, `format: "YYYY/MM-MMMM/YYYY-MM-DD-dddd"`, `template: "Docs/Meta/Templates/Daily Note.md"`). User then ran "Daily notes: Open today's daily note" → red Notice `Failed to create daily note. Folder "beacon/daily" not found.`
>
> Root cause: Obsidian's core Daily Notes plugin requires the configured `folder` to pre-exist before it'll write a new daily note. The Beacon installer materializes daily's `files[]` entirely under `Docs/Meta/...`, so `beacon/daily/` was never created. Historically `beacon/<module>/` was created as a side-effect of `files[]` writes there (e.g., boards@0.1.0 writes `beacon/boards/To-Do-Board.md`, which auto-creates the parent dir). daily is the first blueprint with NO file destinations under its own `module_directory`.
>
> Fix (T-FIX1): added a ~50-LOC mkdir block in `installItem` between `applyPreInstall` and the `files[]` loop, gated on `variables.module_directory` truthiness. Three outcomes under `step: "module_directory"`: `info/created`, `info/already_exists`, `error`. Implementation uses an exists-first check before mkdir to unify behavior between Obsidian's vault adapter (throws on existing) and Node `fs.promises.mkdir({recursive:true})` (silently idempotent). Codifies landmine #11's invariant at the installer level — see "The codified module-directory mkdir mechanic" callout above.
>
> Carried into the S2 commit rather than queuing as a follow-up. After fix: re-ran barebones install (after surgically removing daily entry from `installed[]` to force `installItem` re-entry per landmine #10's "force re-process a manifest" guidance). History recorded `info/module_directory, name: "daily", action: "created", path: "beacon/daily"`. Smoke A retry passed; Smokes B and C also passed.

## Pacing rhythm — what worked / what to improve

### What worked

- **Per-task subagent dispatch on T1.1 (helper authoring + TDD harness extension) + T2.1-T2.3 (manifest + template body + verbatim port) + T2.11 (this writeup).** Master-driven mechanical tasks (T1.2-T1.5 doc edits + T2.4-T2.7 manifest/subscription/install + T2.12-T2.13 close). Same cadence as v0.1.3 + v0.2.0 — keeps master context light + reviewer-quality on the substantive code.
- **TDD red→green discipline on T1.1.** Subagent wrote the five failing C1-C5 cases first, observed the red light, then implemented the helper, observed green. Same pattern paid off again at T-FIX1: M6 was added BEFORE the mkdir block code change so the test surfaced the missing behavior rather than rationalizing the post-hoc fix.
- **Mid-cycle installer fix landed in-stage rather than deferred.** T-FIX1 was caught at Smoke A, root-caused, fixed, harness-extended (M6), and rolled into the S2 commit. Same playbook as v0.2.0's applyNavButtons + validateAndResolve fixes (caught at T2.8, fixed, re-verified). Net result: a stronger installer than the plan promised, and the user's first encounter with daily is the working flow.
- **Verbatim port discipline on `space-daily-dashboard.js`.** Three surgical changes (JSDoc path + two hardcoded module paths) — no behavioral drift from accuris. The class's existing graceful-empty (`if (!hasContent) return;`) handled the "siblings absent" Smoke C case for free, no platform-layer special-casing needed.
- **`nav_buttons: []` decision-deferral on daily.** User raised a "Today" nav-button question during plan-writing; we agreed to defer until a `invoke_command` action type (or equivalent) lands in nav-buttons. Daily ships with empty `nav_buttons[]`; SpaceNavButtons still renders boards's "Board" button on daily-notes, which is the cross-module surface the dashboard was designed to anchor. Clean handoff to a future v0.4.x cycle.

### What to improve

- **Future plans for blueprints whose `files[]` doesn't write under `{{module_directory}}/...` should explicitly enumerate the directory-creation path.** The implicit-side-effect assumption ("the dir exists because boards writes there") was a v0.2.0-era gap that v0.3.0 surfaced via Smoke A. Carry forward as a writing-plans discipline note (similar in spirit to v0.2.0's note about applyNavButtons retirement path): when the plan covers a new blueprint, the plan-writer should ask "does any `files[]` entry write under `{{module_directory}}/...`? if not, who creates the dir?" — and answer in the plan, not at smoke time. Now that T-FIX1 codifies the mkdir at installer level, future plans inherit the answer for free, but the plan-level checklist should still call out the expectation.
- **Plan T1.1 specified a sub-assert floor of +20 but the implementer landed at +35.** Honest behavioral coverage, not padding (each delta is a real assertion). Future plans could either (a) raise the floor when the helper has 5+ failure paths, or (b) drop the floor and trust the TDD discipline. Either is fine; the current pattern of "spec a floor, allow overshoot if behaviorally justified" is working but slightly under-specified.
