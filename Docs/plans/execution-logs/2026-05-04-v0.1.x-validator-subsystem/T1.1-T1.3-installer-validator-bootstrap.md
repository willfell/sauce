## T1.1 ‚Äî `applyExternalPlugins` helper + `installItem` wire-up

> [!example]- New helper: `applyExternalPlugins`
>
> Inserted into `platform/install.js` at lines 656-746 (immediately after
> `validateAndResolve`'s closing brace at 654; immediately before
> `pruneNavButtonsRegistry` at 748). Placement keeps all `apply*` helpers
> (`applyRuleFragment`, `applyNavButtons` + its `validateAndResolve` companion,
> `applyExternalPlugins`) as a contiguous group, with the `prune*` helpers
> (`pruneNavButtonsRegistry`, `pruneInstalledLedger`) as a separate group after.
>
> Posture mirrors the C4 malformed-JSON guard pattern shared by `applyRuleFragment`,
> `applyNavButtons`, `pruneNavButtonsRegistry`, `pruneInstalledLedger`, and
> `enableSnippet`:
> - Short-circuit when `manifest.external_plugins` is absent or empty.
> - First gate: file absent ‚Üí `warning` history event + Notice (not `error` ‚Äî
>   absence of `community-plugins.json` means no community plugins are enabled,
>   which is a possible benign starting state, distinct from a broken file).
> - Read failure (adapter throw) ‚Üí `error` history + Notice + return.
> - Parse failure (malformed JSON) ‚Üí `error` history + Notice + return; do NOT
>   silently overwrite the on-disk file.
> - Shape guard: parsed-but-not-array ‚Üí `error` history + Notice + return.
> - Per-dep loop: skip entries missing `id`; for each `required` dep not present
>   in the enabled array, fire one user-facing Notice (10s timeout ‚Äî long enough
>   for the user to read + act) and one `warning` history event with `plugin_id`
>   and `reason` recorded for postmortem.
>
> Signature: `applyExternalPlugins(tp, manifest, history)` ‚Äî no `variables`
> parameter because external plugin IDs are not substituted (they're community
> plugin slugs like `obsidian-kanban`, not vault-relative paths).

> [!example]- Wire site in `installItem`
>
> Lines 268-269:
>
> ```js
>   await applyNavButtons(tp, mech, variables, history);
>   await applyExternalPlugins(tp, mech, history);
>
>   return true;
> ```
>
> Same call posture as the surrounding `apply*` helpers ‚Äî failures record
> history + Notice but never throw, so a missing dep in one item does not
> abort the install of subsequent items.

### Reasoning

- **Why a detection-only helper, not a remediation one?** Obsidian's plugin
  install API is not exposed to scripts; the installer can list what's enabled
  but cannot enable a plugin programmatically. The user-actionable signal is
  the Notice ("Install + enable in Settings ‚Üí Community plugins"); history
  preserves a forensic trail.
- **Why `warning` for absent file vs `error` for malformed?** Absence is a
  benign state for a fresh vault that has never enabled a community plugin ‚Äî
  the file just isn't created yet. Malformed JSON is a corrupted-state signal
  that warrants `error` so it surfaces in any history audit.
- **Why no `variables` arg?** External plugin IDs are not substituted. Keeping
  the signature narrow makes the helper trivially harness-testable without a
  variables fixture.

## T1.2 ‚Äî `validate.js` rewrite

> [!example]- USAGE comment refresh (lines 1-14)
>
> Old: "Usage from a template (manual): `tp.user.validate(tp.file, "project")`".
> New: "Usage from a runner template (preferred ‚Äî no Templater wrapper traps):
> `tp.user.validate({ file: app.workspace.getActiveFile() })`". Hook usage block
> kept verbatim. Reflects the design's input-contract decision: the canonical
> shape is `{ file: TFile }`, with `tp.file` accepted via the `resolveTFile`
> fall-through.

> [!example]- New helpers above `module.exports` (lines 16-33)
>
> ```js
> function resolveTFile(input, app) {
>   if (!input) return app.workspace.getActiveFile() || null;
>   if (input.file && input.file.path && typeof input.file.path === "string") return input.file;
>   if (input.path && typeof input.path === "string" && input.basename) return input;
>   return app.workspace.getActiveFile() || null;
> }
>
> function whenMatches(when, fm) {
>   const m = when.match(/^frontmatter\.(\w+)\s*(==|!=)\s*['"]([^'"]+)['"]$/);
>   if (!m) return true;
>   const [, key, op, val] = m;
>   const actual = fm[key];
>   return op === "==" ? actual === val : actual !== val;
> }
> ```
>
> `resolveTFile` accepts three input shapes (hook `{ file }`, raw TFile, or
> nothing ‚Üí activeFile fallback) and a Templater wrapper falls through to
> activeFile. `whenMatches` lifts the `frontmatter.X (==|!=) 'Y'` regex
> evaluator out of `checkRequiredBlocks` so it's a single source of truth and
> available for any future check that wants conditional applicability.

> [!example]- Input resolution prelude rewrite (lines 39-43)
>
> Old:
> ```js
> const file = tpFileOrObj?.file ?? tpFileOrObj;
> if (!file || !file.path) {
>   result.violations.push({ rule: "internal", severity: "error", message: "validate: invalid file argument" });
>   return result;
> }
> ```
>
> New:
> ```js
> const file = resolveTFile(tpFileOrObj, app);
> if (!file) {
>   result.violations.push({ rule: "internal", severity: "error", message: "validate: cannot resolve file" });
>   return result;
> }
> ```
>
> Removes the implicit assumption that the caller hands in an object whose
> `.file.path` is set. Now `resolveTFile` owns shape disambiguation, including
> the `app.workspace.getActiveFile()` fallback that lets a runner template work
> without explicitly forwarding `tp.file`.

> [!example]- `checkRequiredBlocks` rewrite (lines 147-173)
>
> The legacy implementation expected `spec.content` (a substring to search for
> in the file body) and used a bespoke inline `frontmatter.X (==|!=) 'Y'`
> regex evaluator. Both are gone:
>
> - `spec.content` substring path is **DROPPED ENTIRELY**. `_global.json`
>   was confirmed empty + only the `project` rule uses `required_blocks`
>   today, and that rule was already migrated to the new schema. No
>   backwards-compat surface remains.
> - The new schema is `kind: "dataviewjs" / must_call: "customJS.X" /
>   via: "customjs-guard" / when: "<predicate>"`. The check pre-extracts every
>   ` ```dataviewjs ` block in the file body, strips the `customJS.` prefix
>   from `must_call` to derive the class name, and constructs a regex that
>   matches the customjs-guard wrapper invocation pattern
>   (`dv.view("Docs/Meta/Views/customjs-guard", { ... class: "<className>" ... })`).
>   Any block that matches satisfies the requirement.
> - `whenMatches` is invoked first to gate applicability ‚Äî same predicate
>   semantics as before but extracted to the shared helper.
> - Schema-shape escape hatch: any spec that does not match the
>   `dataviewjs / must_call / via: customjs-guard` shape pushes a `warn`-severity
>   `required_blocks.schema` violation with a truncated JSON dump of the spec.
>   Future schema variants (e.g., a `kind: "kanban"` for a future blueprint)
>   surface as warnings, not silent skips.
