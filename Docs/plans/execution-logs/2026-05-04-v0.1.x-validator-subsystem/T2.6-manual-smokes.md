## §1 — Why deferred (not failed)

T2.6 manual setup prereq surfaced a real platform gap during T2.6 prep: Templater requires templates to be explicitly registered in its "Template Hotkeys" section (`.obsidian/plugins/templater-obsidian/data.json:enabled_templates_hotkeys[]`) before they appear as `Templater: Insert <name>` commands in the Obsidian command palette. Without that registration, Slash Commander cannot map slash keywords to them. The v0.1.x design assumed Templater auto-generates per-template commands — false. Captured in `T2.1-discovery.md:§8`.

The user's posture (2026-05-04): "no manual plugin configurations done by the user — install + run installer + reload Obsidian should be the entire user-facing flow per consumer vault." This rules out closing v0.1.x with manual Templater Template Hotkeys + Slash Commander mapping as the consumer setup ritual. Two paths to honor that:

1. **Pull automation into v0.1.x** as a new Stage 3 — scope creep mid-cycle (CLAUDE.md anti-pattern; v0.1.x design + plan + S1 already on `main`). Disfavored.
2. **Close v0.1.x cleanly with T2.6 deferred + open a follow-up cycle (v0.1.3) for plugin-data automation.** Recommended; user approved.

Path 2 selected.

## §3 — Follow-up cycle handoff

**Cycle name (working):** v0.1.3 — plugin-data automation
**Design doc target:** `Docs/plans/2026-05-04-v0.1.3-plugin-data-automation-design.md` (to be brainstormed + filed in a separate session per `de:brainstorming` workflow)

Scope sketch (committed in v0.1.x's result writeup §"Follow-up cycle"):

- **New manifest schema:**
  - `templater_hotkeys[]` — list of templates this manifest needs registered in Templater's Template Hotkeys.
  - `slash_commander_bindings[]` — list of `{ keyword, command_id }` pairs to surface as slash commands.
- **Two new installer helpers** (mirror `applyExternalPlugins` shape):
  - `applyTemplaterHotkeys(tp, manifest, history)` — read `.obsidian/plugins/templater-obsidian/data.json`, additive-merge into `enabled_templates_hotkeys[]`, write back. Idempotent. C4 malformed-JSON guard. Backup-on-edit.
  - `applySlashCommanderBindings(tp, manifest, history)` — same pattern against Slash Commander's data.json (schema TBD via T1.x discovery).
- **`.obsidian/` ban lift, scoped:** new CLAUDE.md non-negotiable allowing edits ONLY to `.obsidian/plugins/templater-obsidian/data.json` + `.obsidian/plugins/slash-commander/data.json`; everything else in `.obsidian/` stays ask-before-acting. New landmine entry codifying the allowlist.
- **Harness extension:** `run-install.js` mock surfaces plugin-data writes; new test cases for idempotent merges + malformed-data fallback.
- **Manifest declarations:** validator + audit + project gain `templater_hotkeys[]` + `slash_commander_bindings[]` entries.
- **Doc collapse:** `Docs/use.md` Slash Commander setup goes from 4 manual steps to: (1) install Slash Commander once, (2) run installer, (3) reload Obsidian. Two irreducible manual steps total (Obsidian's plugin install API limitation).
- **T2.6 smokes (deferred from v0.1.x) executed end-to-end** against the automated flow as v0.1.3's S2 acceptance gate.

**Non-goals (carry forward into v0.1.3 design):**
- Companion Obsidian plugin (future cycle if ever).
- Generic `plugin_config[]` schema for arbitrary plugins (YAGNI).
- Auto-installing Slash Commander or auto-reloading Obsidian (Obsidian API limitations — irreducible manual physics).
