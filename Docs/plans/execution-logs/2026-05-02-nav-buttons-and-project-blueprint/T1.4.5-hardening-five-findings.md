## What I did

Implemented all five hardening cases in `platform/install.js`, then re-synced the workshop's runtime bootstrap copy at `Docs/Meta/Templater/platformInstall.js`. Each failure mode now (a) fires a `Notice` naming the source/file, (b) records an `event: "error"` history entry, and (c) skips the offending step or item without clobbering data or aborting the entire install. The main flow is wrapped in `try/finally` so `platform-installed.json` is **always** persisted — partial state is now load-bearing.

## Findings addressed

### C2 — distinct skip reason for unrecognized range syntax

> [!example]- What changed
> - `satisfiesRange(version, range)` now returns a tagged object instead of a bare boolean:
>   - `{ ok: true }` — version satisfies range
>   - `{ ok: false, reason: "unsatisfied" }` — recognized syntax, version too low
>   - `{ ok: false, reason: "unrecognized", syntax: range }` — range syntax unknown
> - `checkDeps` branches on `result.reason` and writes a different skip reason for each case.

Lines: ~`satisfiesRange` and `checkDeps` (in the new file, roughly lines 380-440).

Before:
```js
function satisfiesRange(version, range) {
  if (range === version) return true;
  // ... >=N.N.N parse
  return false; // any unrecognized syntax falls into here
}
// caller emitted "subscription pins X@Y" regardless of cause
```

After:
```js
function satisfiesRange(version, range) {
  if (range === version) return { ok: true };
  const m = range.match(/^>=(\d+)\.(\d+)\.(\d+)$/);
  if (m) { /* compare; return { ok: true } or { ok: false, reason: "unsatisfied" } */ }
  return { ok: false, reason: "unrecognized", syntax: range };
}
// caller now emits:
//   "depends on X with unrecognized version range syntax \"<range>\". Supported: >=N.N.N or exact X.Y.Z."
```

### C4 — no silent rewrite on malformed JSON

> [!example]- What changed
> - `applyRuleFragment` and `enableSnippet` no longer reset `existing = {}` on `JSON.parse` failure.
> - On parse failure: fire a `Notice` naming the file + the parse error, push an `event: "error"` entry into `history`, and `return` early **without** writing.
> - Both helpers now also distinguish read failure from parse failure (each gets its own Notice + history entry).
> - Helper signatures grew: `applyRuleFragment(tp, frag, sourceName, variables, history)` and `enableSnippet(tp, snippet, approvalRequired, sourceName, history)`. `installItem` accepts `history` from the main flow and forwards it.

Lines: `applyRuleFragment` ~lines 442-498, `enableSnippet` ~lines 500-538.

Before:
```js
let existing = {};
if (await adapter.exists(rulePath)) {
  try { existing = JSON.parse(await adapter.read(rulePath)); }
  catch (e) { existing = {}; }            // <-- silent overwrite
}
// ... write fresh JSON, clobbering whatever was there
```

After:
```js
try {
  existing = JSON.parse(raw);
} catch (e) {
  new Notice(`applyRuleFragment: ${rulePath} is malformed JSON (${e.message}). Skipping fragment from ${sourceName}.`, 8000);
  history.push({ event: "error", step: "applyRuleFragment", name: sourceName,
                 message: `${rulePath} is malformed JSON: ${e.message}`,
                 attempted_at: new Date().toISOString() });
  return;
}
```

The `enableSnippet` change is symmetric (different filename, same shape).

### E1 — try/catch around install loop + try/finally for top-level writeJson

> [!example]- What changed
> - Entire body of `module.exports` from config read through the install loop is wrapped in `try { ... } catch { ... } finally { ... }`.
> - `installedNow` is constructed BEFORE the try block so the `finally` always has something to write.
> - `writeJson(installedNow)` is called from the `finally` block (also wrapped in its own try/catch — failure to persist still fires a Notice but doesn't throw).
> - Each `installItem` call inside the loop is wrapped in its own try/catch: on exception we push `{ event: "error", name, step: "installItem", message: e.message, attempted_at }` and continue with the next item.
> - Top-level catch surfaces a generic Notice + records `step: "top_level"` in history.
> - Each early-abort branch (config missing, subscription missing, manifest missing, dep cycle) now also pushes a history `event: "error"` before returning, so the persisted `installed.json` records the failure mode.
> - Final "complete" Notice only fires on successful pass; otherwise a "finished with errors" Notice fires from the `finally` block when `topLevelOk === false`.

Lines: ~lines 19-145 (the `module.exports` body).

Before:
```js
// resolver, manifest read, install loop all bare
// any throw aborts before writeJson(installedNow) runs
await writeJson(app, "Docs/Meta/platform-installed.json", installedNow);
new Notice("platformInstall: complete.", 4000);
```

After (sketch):
```js
const installedNow = { ... };
let topLevelOk = false;
try {
  // ... resolver, manifest, install loop with per-item try/catch ...
  topLevelOk = true;
  new Notice("platformInstall: complete.", 4000);
} catch (e) {
  new Notice(`platformInstall: top-level error — ${e.message}`, 8000);
  installedNow.history.push({ event: "error", step: "top_level", message: e.message, ... });
} finally {
  try { await writeJson(app, "Docs/Meta/platform-installed.json", installedNow); }
  catch (e) { new Notice(`failed to write platform-installed.json — ${e.message}`, 8000); }
  if (!topLevelOk) new Notice("platformInstall: finished with errors. ...", 8000);
}
```

### E3 — distinguish "subscribed-but-skipped" from "not-subscribed-at-all"

> [!example]- What changed
> - `checkDeps` now takes a fourth argument `missingItems` (the array of `{name, reason}` returned from `resolveDependencies`).
> - Builds `missingByName: Map<name, reason>` once at the top of the function.
> - When `subscriptionLookup.get(dep.name)` is `undefined`, checks `missingByName.has(dep.name)`:
>   - If yes → skip reason: `depends on <dep.name> which was skipped (<original reason>)`
>   - If no  → original message: `depends on <dep.name> <range> but it is not subscribed`

Lines: `checkDeps` ~lines 357-405; call site at ~line 67 updated to pass `missingItems` through.

Before:
```js
const sub = subscriptionLookup.get(dep.name);
if (!sub) {
  skipped.push({ name, reason: `depends on ${dep.name} ${dep.range} but it is not subscribed` });
  // wrong if dep WAS subscribed but resolveDependencies skipped it
}
```

After:
```js
if (!sub) {
  if (missingByName.has(dep.name)) {
    skipped.push({ name, reason: `depends on ${dep.name} which was skipped (${missingByName.get(dep.name)})` });
  } else {
    skipped.push({ name, reason: `depends on ${dep.name} ${dep.range} but it is not subscribed` });
  }
  ...
}
```

### L2 — detect mechanism + blueprint name collisions

> [!example]- What changed
> - Before building the `nodes` Map, `resolveDependencies` walks the four name sets (subscription mechanisms/blueprints, manifest mechanisms/blueprints).
> - Any name appearing in BOTH the mechanism and blueprint side of either source is added to `collisionNames`.
> - For each collision: a single skip entry is pushed: `name collision: "<X>" appears as both mechanism and blueprint`.
> - The main `for (const sub of subItems)` loop short-circuits (`continue`) on collisions so neither half ever lands in `nodes`.

Lines: `resolveDependencies` ~lines 313-355.

Before: blueprint with same name as mechanism would silently overwrite the mechanism in `nodes` and inherit blueprint kind/path.

After:
```js
const subMechNames = new Set((subscription.mechanisms || []).map(m => m.name));
const subBpNames   = new Set((subscription.blueprints || []).map(b => b.name));
// + same for manifest
const collisionNames = new Set();
for (const n of subMechNames) if (subBpNames.has(n)) collisionNames.add(n);
for (const n of manMechNames) if (manBpNames.has(n)) collisionNames.add(n);
for (const name of collisionNames) {
  skipped.push({ name, reason: `name collision: "${name}" appears as both mechanism and blueprint` });
}
// ...later...
if (collisionNames.has(sub.name)) continue; // hard-skip both halves
```

## Notes

- **Threading `history` vs. tagged returns.** The review allowed either passing `historyAccumulator` or returning a status tag. I chose passing `history` for `applyRuleFragment` / `enableSnippet` because both helpers are fire-and-forget (the caller doesn't otherwise care about their return value), and threading the array is the smaller diff.
- **`satisfiesRange`** got the tagged-union treatment because the caller (`checkDeps`) genuinely needs the discrimination; threading a `history` array there would have been awkward (the message construction belongs in `checkDeps`).
- **`installedNow` was hoisted out of the try block** so the `finally` writeJson always sees a populated object even when an early-abort branch fires before the resolver runs. Each early abort now also adds a history entry of its own so the persisted file records the failure cause.
- **The "finished with errors" Notice** is intentionally separate from per-step Notices so that a user who missed individual error popups still sees a summary on completion.
- **No changes to public manifest.json schemas.** All new behavior is internal to `install.js`.
