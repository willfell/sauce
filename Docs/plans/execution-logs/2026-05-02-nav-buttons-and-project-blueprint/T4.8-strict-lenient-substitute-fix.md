> [!abstract] Goal
> Stop the installer from aborting when file bodies contain legitimate runtime template strings (e.g. `{{DATE}}`, `{{TASK_NAME}}`, `{{ALIAS}}`) that JS helpers resolve at click-time via `replaceAll`. Keep `dest` path substitution strict.

---

## What broke

During Stage 4 onboarding, the user re-ran `tp.user.platformInstall(tp)` and the installer aborted on `project-nav-buttons.js`. That helper script's source body intentionally contains runtime placeholders like `{{DATE}}`, `{{TASK_NAME}}`, `{{ALIAS}}` — they are NOT installer-time variables; they're consumed by the installed JS at click-time via `replaceAll`. The original `substitute(text, variables)` was strict-everywhere and treated those as missing-variable errors, so the file was never written. The same issue would surface for any blueprint helper that templates content at runtime.

---

## Fix

Split the single `substitute(...)` into two functions in `platform/install.js`:

- `substituteStrict(text, variables)` — preserves the prior throw-on-missing behavior. Used ONLY for `f.dest` path resolution (every `{{var}}` in a destination path must resolve, or the file would land in the wrong place).
- `substituteLenient(text, variables)` — substitutes known variables, leaves unknown `{{X}}` placeholders intact. Used for file BODY content. Never throws.

`installItem` now wraps only the strict dest-path call in a try/catch (Notice + return false on failure). The lenient body call needs no try/catch.

## Notes

> [!warning] This is a design clarification, not a bug fix
> The Stage 1 hardening's "strict-everywhere" posture (Failure-mode case #1 in the design doc, "Strict substitution") was over-eager. It correctly demanded `dest` paths must resolve, but incorrectly extended that demand to file bodies — bodies legitimately contain `{{X}}` strings owned by the file's own runtime, not by the installer. The strict-everywhere posture would have eventually broken any mechanism whose installed code uses `replaceAll('{{...}}', ...)` patterns. Project-nav-buttons is the first such mechanism; it surfaced the issue. The fix preserves "fail loudly with attribution" for the things the installer is responsible for (dest paths) while ceding ownership of body-level placeholders to whoever installs them.

> [!tip] Going forward
> When designing future mechanisms whose runtime substitutes its own templates, no special handling is needed. Just write the helper, ship it through the installer, and trust that bodies pass through unknown `{{X}}` unchanged.
