> [!example]- After — `substitute()`

```javascript
function substitute(text, variables) {
  const missing = new Set();
  const result = text.replace(/\{\{(\w+)\}\}/g, (_, key) => {
    if (variables[key] === undefined || variables[key] === null) {
      missing.add(key);
      return `{{${key}}}`;
    }
    return variables[key];
  });
  if (missing.size > 0) {
    const err = new Error(`Unsubstituted variables: ${[...missing].join(", ")}`);
    err.missing = [...missing];
    throw err;
  }
  return result;
}
```

## Restructuring choice

Picked **Pattern 1** (read `sourceText` first, then a single try/catch around both `substitute()` calls).

Trade-offs considered:

- **Pattern 1 (chosen):** one try/catch, both substitutions atomic. Reads `sourceText` before the approval gate even fires. Side-effect: if the user rejects approval for a file whose `{{vars}}` happen to all be present, we did the substitution work and discarded it. Cost is negligible (string replace).
- **Pattern 2 (rejected):** two separate try/catches, one per call. More boilerplate, two return-false paths to keep aligned, and two distinct Notices for what is conceptually one "missing variables for this file" failure.

Pattern 1 also keeps the approval gate's prompt accurate: `destPath` is now the substituted path (it was already substituted in the original code, so this is preserved). If a missing variable affects `f.dest`, the user sees the failure Notice instead of being prompted for an unsubstituted path — strictly better UX.

Behavioral note: the approval prompt now appears **after** sourceText is read and substituted. Previously the gate fired before reading source. If the source file is missing AND the dest had unsubstituted vars, ordering used to be (1) prompt, (2) source-missing Notice. Now it's (1) source-missing Notice, no prompt. This is acceptable — the user can't approve installing a file whose source doesn't exist.
