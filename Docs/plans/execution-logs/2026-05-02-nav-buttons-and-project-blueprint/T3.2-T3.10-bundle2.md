## What I did

Authored the `create-new-project.md` Templater template and `new-project.md` slash-command wrapper for the project blueprint. The template is a real, runnable Templater script: it prompts for a project name, derives a kebab-case slug, idempotently creates `boards/planning/<slug>/`, writes `<slug>.md` with required frontmatter (`type: project`, `status: active`, tags including `vault_identity`) plus the two required dataviewjs nav-button blocks (SpaceNavButtons + ProjectNavButtons) and a ProjectWorkstreamManager block, then opens the new file. The slash-command wrapper calls `tp.file.find_tfile("{{templates_path}}/Create New Project.md")` so the installer's strict substitution bakes the consumer's templates folder per-vault. After authoring, registered `project@0.1.0` in `platform/manifest.json` (blueprints array) and added the same pin to both workshop and tmp-acc-vault `platform-subscription.json` files.

## Templater template choices

> [!example]- What v0.1.0 ships
> - **Single template** that fully automates atlas-note creation (folder + file + open).
> - Reads `vault_identity` from `Docs/Meta/platform-config.json` at runtime → frontmatter tag (`accuris`, `barebones`, etc.). Falls back to `[project]` only with a Notice.
> - Reads `views_path` from the same config → templates the `dv.view()` paths inside the dataviewjs blocks so it adapts per-vault (workshop & barebones use `Docs/Meta/Views`, tmp-acc-vault uses `Extras/Scripts`).
> - Idempotent: skips if folder exists; aborts (with Notice) if atlas file exists.
> - Wraps file writes in try/catch; surfaces errors as 8-second Notices.
> - Opens the new file in the active leaf on success.

> [!warning]- What v0.1.0 defers (to v0.1.1 or later)
> - **Kanban planning board.** The existing `Template, Project Board.md` workflow is not invoked. Users who want a board file must create it manually after running the template. Reason: keeping v0.1.0 minimal-viable; kanban frontmatter + the board file's relationship to the atlas isn't part of the rule.json contract yet.
> - **`variants.json` overrides.** v0.1.0 reads `vault_identity` directly from platform-config; it does NOT load `Docs/Meta/rules/project.variants.json` to honor per-vault `path_root` (e.g. ERO's `Projects/`) or `alias` (headspace's `side-quest`). All vaults currently get the canonical `boards/planning/<slug>/` layout. Acceptable for workshop/tmp-acc/barebones since all three use that root.
> - **`Referenced By` dataview block** from the legacy `Template, Project.md`. Easy to add later; not strictly required by rule.json.
> - **Project sub-tag** (`project/<slug>`) and date hierarchy tag (`YYYY/MM/DD`) from the legacy template. Not in the v0.1.0 rule contract.

## Notes

> [!info]- Workshop subscription `workshop_version` mismatch
> The workshop's own `platform-subscription.json` declares `workshop_version: 0.2.0` while `platform/manifest.json` is `0.3.0`. Bundle 1 / Stage 1 bumped the manifest but didn't bump the self-subscription. Not a Stage-3 blocker (the dep resolver doesn't gate on workshop_version), but mention to user during Stage 4 closeout.

> [!info]- Strict-substitution interaction
> The slash command body contains the literal string `{{templates_path}}`. When the installer copies `commands/new-project.md` from blueprint source → `{{commands_path}}/new-project.md` it will substitute that placeholder using the consumer's variables map. Result: the installed command file has the consumer's actual templates path baked in (e.g. `Extras/Templates/Create New Project.md` for tmp-acc-vault, `Docs/Meta/Templates/Create New Project.md` for workshop/barebones).

> [!info]- Template path-portability vs blueprint-path-portability
> The Templater template (`create-new-project.md`) does NOT use `{{templates_path}}`-style substitutions in its body. It does runtime config reads instead. Reason: at install time, only the dest *path* is substituted; the template body is the runtime entry point and reads vault state itself. Both approaches are valid; runtime reads are more flexible for variant overrides (future v0.1.1 work).
