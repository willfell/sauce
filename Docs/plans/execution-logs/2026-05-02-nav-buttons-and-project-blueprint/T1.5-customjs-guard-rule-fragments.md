## What I did

Updated the `customjs-guard` mechanism manifest so its `rule_fragments[0].fragment` field is now a JSON object literal (a nested dict) rather than a YAML-formatted string. This aligns with Task 1.4's installer change, which now expects fragments as native JSON objects instead of parsing them as YAML strings.

Also corrected the regex escaping. The previous YAML-string form double-escaped the dot (`\\\\.`) because that text was traveling through a JSON string layer first and then a YAML parse. As a native JSON object value, only the JSON-string layer remains, so the literal-dot regex is written as `\\.` (which JSON decodes to the two-character regex `\.`).

## Files changed

- `platform/mechanisms/customjs-guard/manifest.json`

## Before / after

Before:

```json
"rule_fragments": [
  {
    "target": "_global",
    "fragment": "forbid_dataviewjs_patterns:\n  - { pattern: \"await customJS\\\\.\", reason: \"use customjs-guard instead\" }\n"
  }
]
```

After:

```json
"rule_fragments": [
  {
    "target": "_global",
    "fragment": {
      "forbid_dataviewjs_patterns": [
        { "pattern": "await customJS\\.", "reason": "use customjs-guard instead" }
      ]
    }
  }
]
```

All other manifest fields (`name`, `version`, `description`, `files`, `post_install`) are untouched.
