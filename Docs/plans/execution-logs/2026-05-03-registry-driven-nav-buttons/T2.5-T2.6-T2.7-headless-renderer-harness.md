## Why this exists

The original v0.1.1 plan called for the user to manually verify SpaceNavButtons v2.0.0's renderer behavior in Obsidian: open a note, add a dataviewjs block, observe the rendered output, hand-write a malformed registry to confirm the error chip, hand-write a synthetic unknown-action.type entry to confirm the click-Notice. Same friction the install-side dogfood loop had before the install harness landed.

User feedback in-flight: *"can't you test it yourself?"* → built `platform/test/run-renderer.js`, a Node CLI that loads the renderer class via `new Function(...)`, stubs Obsidian's DOM + vault APIs + Notice, then asserts on the resulting tree shape and captured Notices.

## What the harness covers

The renderer's API surface that the harness fakes:

- `app.vault.adapter.read(path)` — proxied to `fs.promises.readFile`
- `app.vault.getAbstractFileByPath(path)` — `fs.accessSync` based existence
- `app.workspace.openLinkText(path, _)` — captured to an array
- `app.vault.create(path, body)` / `createFolder(path)` — no-ops (not exercised in T2.5/T2.6/T2.7)
- `app.isMobile` — fixed `false`
- Global `Notice` — capturing class, prints + records each `new Notice(msg, dur)` call
- `dv` object with `dv.el(tag, content, opts)`, `dv.container.querySelector(sel)`
- Element shape with `.style.cssText`, `.createEl(tag, opts)`, `.innerHTML`, `.onclick`/`.onmouseenter`/`.onmouseleave`, `.children`, `.cls`, `.text`, `.parent`, `.remove()`

The renderer file is loaded by reading its source as a string and wrapping in `new Function('app', 'Notice', code + '\nreturn SpaceNavButtons;')`. This gives us a callable that takes the stubbed globals and returns the class. No `vm` module needed.

## What each test verifies

### T2.5 — empty install

- Setup: no `Docs/Meta/nav-buttons-registry.json` present (registry sandbox unlinks any existing file for the test scope, restores afterward).
- Expected: renderer hits ENOENT → silent return.
- Asserts: zero buttons rendered, no error chip, no `vault-nav` container created, no Notices fired.

### T2.6 — malformed registry

- Setup: registry written with truncated JSON `{ "schema_version": 1, "contributions": [BAD`.
- Expected: renderer reads the body successfully but `JSON.parse` throws → `dv.el("div", ..., { cls: "nav-error" })`.
- Asserts: zero buttons, exactly one element with `cls === "nav-error"`, that element's text includes "parse error".

### T2.7 — unknown action.type

- Setup: registry written with one contribution `test → [{ id: "fake", label: "Fake", icon: "board", order: 100, action: { type: "fake" } }]`.
- Expected (render): one `<button>` element with the board icon SVG + a `<span>Fake</span>`.
- Expected (click): button's `onclick` calls `_dispatchAction(btn)`; the unknown-action branch fires a Notice naming the type and source.
- Asserts: exactly one button rendered, button's onclick triggers exactly one Notice, Notice text contains `unknown action.type`, `"fake"`, and `from test`.

## Registry sandbox semantics

`withTempRegistry(content_or_null, fn)` snapshots any existing registry, writes the test-specific content (or unlinks for the empty case), runs `fn`, then restores the original state in the `finally` block. Idempotent across runs — back-to-back invocations leave the workshop's actual registry state unchanged.

## Limitations

- **Doesn't run a real Dataview pipeline.** If Dataview itself ever changes how it invokes views (e.g., the `dv` object shape evolves), this harness wouldn't catch the breakage. Mitigation: the `dv` shape we shim is small and stable across Dataview's recent versions; if that changes, the harness's stub gets a one-time update.
- **Doesn't run a real CustomJS class registration.** The harness calls `new Cls()` directly. Real CustomJS calls the class lifecycle differently (instance pooling, per-render context). Behavioral tests at this level are still meaningful — the class's render/dispatch logic is what's being exercised, not its registration.
- **Doesn't exercise the createFromTemplate action path.** That action type is the project blueprint's "Board" button, which lands in S3. When S3 ships the template + button declaration, this harness can be extended with a `testCreateFromTemplate` case (exercising the full flow: click → registry resolution → template read → folder create → file create → openLinkText capture).

## What this enables going forward

- **S3 verification**: the harness can be extended to verify the project blueprint's "Board" button declaration produces the expected registry entry post-install AND that clicking it materializes a kanban file from the template — without Obsidian.
- **S4 regression**: every time SpaceNavButtons changes, the three smoke tests rerun in seconds.
- **v0.1.2 multi-vault**: the harness is per-vault by design — running it against barebones (after install) would exercise the same three scenarios in that vault's actual registry state.
