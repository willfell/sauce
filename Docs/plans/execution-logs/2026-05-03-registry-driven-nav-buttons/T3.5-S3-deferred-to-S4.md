## Why this can only run in S4

The malformed-`nav_buttons[]`-entry path inside `applyNavButtons` only fires
when an item that declares `nav_buttons[]` is being **installed** (i.e., is
in the current subscription). The workshop's `Docs/Meta/platform-subscription.json`
has `blueprints: []` — project is intentionally not subscribed (renderer-dogfood-only
posture, locked during S2). So no install path through workshop ever exercises
the validator on the project manifest's `nav_buttons[]`.

Barebones (`tmp-test-barebones-vault`) WILL subscribe to `project@0.2.0` in S4
and run install against the workshop's freshly-bumped project manifest. That
is the natural test point for this negative path: edit the workshop project
manifest's `nav_buttons[]` to inject a malformed entry, run barebones install,
observe the per-entry-skip + warning history event + valid-entry-survives
behavior, restore.
