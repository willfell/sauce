## What the harness does

> [!info] Path
> `platform/test/run-install.js` (new file, ~280 LOC, stdlib only — `fs`, `path`, `process`).

Loads `<vault>/Docs/Meta/Templater/platformInstall.js` (the byte-identical bootstrap copy of `platform/install.js`) via `require()` and runs it against a fake `tp` object that proxies every Obsidian API the installer touches into the real filesystem rooted at the given vault path.

```bash
node platform/test/run-install.js .                    # workshop self-install
node platform/test/run-install.js ../tmp-test-vault    # any consumer vault
node platform/test/run-install.js . --dry-run          # capture intended writes
node platform/test/run-install.js . --decline-all      # all suggester gates Esc
node platform/test/run-install.js . --verbose          # log every adapter call
```

Exit code: `0` when zero `error`/`skip` history entries are added during the run; `1` otherwise.

## Fake-API surface area

> [!example]- Exhaustive list (audited against `install.js` top-to-bottom)
> | Real API | Used by | Fake implementation |
> |---|---|:----:|
> | `app.vault.getAbstractFileByPath(p)` | `readJson`, `writeJson` | `fs.accessSync` → `{ path: p }` or `null` |
> | `app.vault.read(file)` | `readJson` | `fs.readFile(abs(file.path))` |
> | `app.vault.modify(file, text)` | `writeJson` | `fs.writeFile(abs(file.path))` (logs in dry-run) |
> | `app.vault.create(path, text)` | `writeJson` | `fs.writeFile(abs(path))` (logs in dry-run) |
> | `app.vault.createFolder(p)` | (defensive) | `fs.mkdir({ recursive: true })` |
> | `app.vault.adapter.read(p)` | rule-fragment, nav-buttons, snippet-enable, prune | `fs.readFile(abs(p))` |
> | `app.vault.adapter.write(p, c)` | same | `fs.mkdir + fs.writeFile` |
> | `app.vault.adapter.exists(p)` | many | `fs.access` → bool |
> | `app.vault.adapter.mkdir(p)` | `installItem`, `applyRuleFragment` | `fs.mkdir({ recursive: true })` |
> | `app.vault.adapter.basePath` | `resolveWorkshopPath` | `VAULT` (resolved positional arg) |
> | `tp.system.suggester(text, vals, throwOnCancel, placeholder)` | `approvalGate` | returns `vals[0]` (auto-approve) or `null` (decline-all) |
> | global `Notice` | everywhere | captures `{ message, durationMs, at }` to array + echoes to stdout |
> | `require("fs").promises` | `readAbsolute` | passes through to real fs (already pure Node) |
> | `require("path")` | `resolveWorkshopPath` | passes through |

`adapter.list` is not currently called by the installer, but a sensible implementation is provided (`{ files: [...], folders: [...] }`) so future installer changes don't immediately break the harness.

## Limitations

> [!warning] What this harness does NOT cover
> - **Renderer behavior.** Anything that runs inside an Obsidian Dataview/CustomJS view (e.g., `space-nav-buttons.js` rendering buttons from the registry) still needs Obsidian. The harness covers the install pipeline only.
> - **`reload` notices.** `enableSnippet` says "Reload Obsidian to apply" — we just log the Notice; there's no Obsidian to reload.
> - **`Templater` template execution.** `tp.user.platformInstall(tp)` is invoked from a template inside Obsidian. The harness skips the template layer entirely — it `require()`s the installer module directly, which is what the template calls anyway.
> - **CustomJS class registration side-effects.** Any post-install behaviour that depends on CustomJS classes being newly available is out of scope.

## Unexpected discoveries

> [!info] Surprises captured during build
> - **`adapter.list` is unused.** Faked it anyway as a defence against future installer changes.
> - **`tp.system.suggester` 4-arg signature.** `approvalGate` calls `tp.system.suggester(["Approve","Skip"], [true,false], false, message)` — i.e., (text_items, items, throw_on_cancel, placeholder). Documented in the spec as 3-arg. Fake matches the actual call shape.
> - **`platform-installed.json` had a stale gap.** When I ran the harness for the first time today, the `project` blueprint installed *fresh* — its files had been materialized in a prior session but the bucket entry was never recorded. The harness's first run correctly closed the gap (one `install` history entry added). Re-runs since then show 0 new history entries → confirmed idempotent.
> - **`writeJson` always writes once per run.** Even on a fully idempotent re-run, the installer writes `Docs/Meta/platform-installed.json` once (in the `finally` block). That's `1 would-be write` in `--dry-run` output for a clean run; expected behaviour, not a bug.
