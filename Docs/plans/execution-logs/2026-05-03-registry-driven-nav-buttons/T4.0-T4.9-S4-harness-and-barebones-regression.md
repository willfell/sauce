## T4.8 — malformed nav_buttons entry (negative test)

> [!info] Test mechanic deviation from prompt wording
> The prompt's wording ("transiently edit `platform/blueprints/project/manifest.json`") implies that adding a bad entry alone would trigger `applyNavButtons` against barebones. In practice the install loop short-circuits at line 111 of `install.js` (`if (installedEntry && installedEntry.version === node.sub.version) continue;`) — `applyNavButtons` is only called inside `installItem`, which is skipped when the project version already matches. To force re-processing of the manifest's `nav_buttons[]`, a coordinated **transient triple bump** was applied:
>
> - `platform/blueprints/project/manifest.json:version` 0.2.0 → 0.2.1
> - `platform/manifest.json:blueprints[].project.version` 0.2.0 → 0.2.1
> - `../tmp-test-barebones-vault/Docs/Meta/platform-subscription.json:blueprints[].project.version` 0.2.0 → 0.2.1
>
> All three were restored after the test ran. A subsequent harness run pulled barebones's installed.json back to project@0.2.0 by re-installing it (the version change in the subscription forces re-install regardless of direction). All three platform-side files end this log byte-identical to their state at the start of T4.8.
>
> **Recommended plan correction:** the plan's T4.8 wording should call out the version-bump requirement explicitly — without it the test silently never fires `applyNavButtons` (the symptom is "no Notice + no warning history entry, install ledger unchanged"). Adding this to landmines or to the v0.1.x harness recipe would prevent the next implementer from missing it.
