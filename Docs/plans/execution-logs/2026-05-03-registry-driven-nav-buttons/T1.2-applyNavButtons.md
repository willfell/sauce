# T1.2 — Implement `applyNavButtons` + `validateAndResolve`

## Function bodies

Both functions added to `platform/install.js` immediately after `applyRuleFragment` (the parallel structure is intentional — registry aggregation mirrors rule-fragment aggregation).

```javascript
async function applyNavButtons(tp, manifest, variables, history) {
  if (!manifest || !Array.isArray(manifest.nav_buttons) || manifest.nav_buttons.length === 0) return;
  const adapter = tp.app.vault.adapter;
  const registryPath = "Docs/Meta/nav-buttons-registry.json";

  let registry = { schema_version: 1, contributions: {} };
  if (await adapter.exists(registryPath)) {
    let raw;
    try { raw = await adapter.read(registryPath); }
    catch (e) {
      new Notice(`applyNavButtons: cannot read ${registryPath} (${e.message}). Skipping contribution from ${manifest.name}.`, 8000);
      history?.push({ event: "error", step: "nav_buttons", name: manifest.name,
        message: `read failed for ${registryPath}: ${e.message}`, attempted_at: new Date().toISOString() });
      return;
    }
    try { registry = JSON.parse(raw); }
    catch (e) {
      // C4: do NOT silently overwrite a malformed pre-existing registry file.
      new Notice(`applyNavButtons: ${registryPath} is malformed JSON (${e.message}). Skipping contribution from ${manifest.name}.`, 8000);
      history?.push({ event: "error", step: "nav_buttons", name: manifest.name,
        message: `${registryPath} is malformed JSON: ${e.message}`, attempted_at: new Date().toISOString() });
      return;
    }
  }
  registry.contributions = registry.contributions || {};

  const validated = manifest.nav_buttons
    .map((btn) => validateAndResolve(btn, manifest.name, variables, history))
    .filter(Boolean);

  if (validated.length === 0) {
    history?.push({ event: "error", step: "nav_buttons", name: manifest.name,
      reason: "all entries invalid", attempted_at: new Date().toISOString() });
    return;
  }

  registry.contributions[manifest.name] = validated;
  await adapter.write(registryPath, JSON.stringify(registry, null, 2));
}

function validateAndResolve(btn, sourceName, variables, history) {
  if (!btn || !btn.id || !btn.label || !btn.action || !btn.action.type) {
    new Notice(`nav-buttons: invalid declaration in ${sourceName} (missing id/label/action)`, 8000);
    history?.push({ event: "warning", step: "nav_buttons", name: sourceName,
      reason: `entry ${(btn && btn.id) || "<no-id>"} invalid`, attempted_at: new Date().toISOString() });
    return null;
  }
  if (btn.action.type === "createFromTemplate" && btn.action.template_source) {
    const contentPath = variables.content_path || "Docs/Meta/Content";
    return {
      ...btn,
      action: { ...btn.action, template_source: `${contentPath}/${sourceName}/${btn.action.template_source}` },
    };
  }
  return btn;
}
```

(Actual file uses explicit `if (history) { history.push(...) }` rather than `history?.push(...)` — matches the existing posture in `applyRuleFragment` and `enableSnippet`. Logic identical.)

## Deviations from the design sketch (and rationale)

### 1. Function signature — `(tp, manifest, variables, history)` instead of `(item, manifest, paths, history)`

The sketch in the design doc declares `applyNavButtons(item, manifest, paths, history)`, where `paths` is an object containing `meta_path`, `content_path`, etc.

Existing `install.js` does NOT have a `paths` object. The substitution map is `variables` (sourced from `config.variables`), and helpers like `applyRuleFragment` are called as `applyRuleFragment(tp, frag, mech.name, variables, history)`. To match the existing convention I:

- Renamed the parameter to `variables` (singular concept of "the substitution map").
- Pass `tp` (so we get `tp.app.vault.adapter` — same as `applyRuleFragment`).
- Dropped the `item` parameter from the sketch — `manifest.name` already provides the source name; there is no separate "item descriptor" object in the existing flow with extra info worth threading.

### 2. Registry path — hardcoded, not `${paths.meta_path}/...`

The design sketch uses `${paths.meta_path}/nav-buttons-registry.json`. There is no `meta_path` variable in `platform-config.json` for any vault, and the existing installer hardcodes meta-relative paths everywhere (e.g., `"Docs/Meta/platform-installed.json"`, `"Docs/Meta/platform-config.json"`).

I hardcoded `"Docs/Meta/nav-buttons-registry.json"` to match that convention. Introducing a `meta_path` variable would be a wider refactor that should be its own decision — the design doc does not call for it explicitly outside the sketch.

### 3. Helper — used `adapter.read` / `adapter.write` rather than `readJsonOrEmpty` / `writeJson`

The sketch references `readJsonOrEmpty` and `writeJson`. The existing installer has:

- `readJson(app, path)` — returns null on missing file or parse failure (collapses both into one signal — wrong for C4 hardening, where we MUST distinguish "missing file" from "malformed file").
- `writeJson(app, path, obj)` — writes via `app.vault.modify` / `app.vault.create`.

`applyRuleFragment` uses `adapter.read` + `JSON.parse` so it can distinguish those cases (and write via `adapter.write`). I followed the same pattern in `applyNavButtons` — gives correct C4 behavior (malformed file → skip + Notice + history; do NOT clobber). This matches the spirit of the sketch's "C4 hardening posture" comment.

### 4. Per-entry warning history — added a per-bad-entry warning push

The sketch only records `event: "error"` when `validated.length === 0` (i.e., every entry is bad). It surfaces a Notice on each bad entry (`new Notice("nav-buttons: invalid declaration in ${sourceName} ...")`) but doesn't push history per entry.

The plan's failure-loud table has both:

| One entry malformed in an otherwise-valid `nav_buttons[]` | ... | `{ event: "warning", step: "nav_buttons", name, reason: "entry <id> invalid" }` |
| Manifest declares `nav_buttons[]` but every entry is missing required fields | ... | `{ event: "error", step: "nav_buttons", name, reason: "all entries invalid" }` |

To honor the table I push **both**: one `warning` per bad entry, plus one `error` if every entry was bad. This is a strict superset of the sketch — no functional regression, plus the per-entry trace useful for debugging.

### 5. `template_source` rewrite — flagged inconsistency between sketch and S3 manifest example

This is the one real concern that needs human review.

**The sketch:**
```javascript
template_source: `${paths.content_path}/${sourceName}/${btn.action.template_source}`
```

**The S3 example manifest declares:**
```json
"template_source": "content/kanban-board.md"
```

**The S3 example post-install registry shows:**
```json
"template_source": "Docs/Meta/Content/project/kanban-board.md"
```

The design's stated rewrite produces `Docs/Meta/Content/project/content/kanban-board.md` (note the spurious `content/` segment) — NOT what the post-install example shows.

**Two possible interpretations:**
1. The S3 manifest declaration should drop the `content/` prefix (`"template_source": "kanban-board.md"`) — the rewrite then produces the expected result.
2. The rewrite should strip a leading `content/` prefix (or use some other normalization).

I implemented interpretation **(verbatim sketch)**: `${contentPath}/${sourceName}/${btn.action.template_source}`. This means S3's manifest declaration must be **either** `"kanban-board.md"` (preferred — registry path becomes `Docs/Meta/Content/project/kanban-board.md`) **or** the rewrite logic gets revisited in S3. The S3 implementer will see this immediately when they materialize the kanban template — the file lands at `Docs/Meta/Content/project/kanban-board.md` (per `files[]` dest), but the registry's `template_source` would point at `Docs/Meta/Content/project/content/kanban-board.md` (a path that does NOT exist), and the renderer's `app.vault.adapter.read(template_source)` will throw at click time.

**Recommended resolution at S3:** declare `"template_source": "kanban-board.md"` in the project manifest's `nav_buttons[]` entry (mirror the `files[].dest` basename, not the workshop-side `files[].source`). I'm flagging this here because it's a contract between S3 and the installer that needs to be locked. No code change in S1 — sketch implemented as written.
