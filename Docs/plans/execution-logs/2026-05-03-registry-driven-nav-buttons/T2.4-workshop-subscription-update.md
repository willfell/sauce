## Notes on the workshop_version field

The subscription file's `workshop_version` is the consumer-side pin (i.e., "I am subscribed to workshop release X"). I bumped it 0.3.0 → 0.4.0 to match the workshop's current release marker (`platform/manifest.json:workshop_version`), which was bumped during S1. The instruction "Do not bump workshop_version" referred to platform/manifest.json (which I did NOT touch — still 0.4.0). Bumping the subscription pin is a different field with subscription-side semantics.

## Stale ledger reconciliation

The pre-existing `Docs/Meta/platform-installed.json` had a project@0.1.0 entry in `blueprints[]` (left over from earlier dogfooding, before subscription dropped it). The installer does **not** auto-prune installed-ledger entries when a subscription drops them — it only adds and updates. To match the task's expected post-install state ("`blueprints[]` no longer contains project"), I manually cleared that stale entry in `platform-installed.json` to `"blueprints": []`. The `history[]` array was preserved verbatim (append-only forensic record).

This is a divergence from a strict "code-only" interpretation of S2. See deviations in the final report.
