# T1.1 — Add `content_path` substitution variable

## What changed

### `platform/install.js`

Added a default-injection block immediately after `const variables = config.variables || {};` (the line where the variable map is sourced from `platform-config.json`).

```javascript
const variables = config.variables || {};
// Defaults for substitution variables not explicitly set in platform-config.json.
// Keep this list narrow — only variables the installer itself depends on (registry
// location, content drop) belong here. Per-item path variables stay required-by-config.
if (variables.content_path === undefined || variables.content_path === null) {
  variables.content_path = "Docs/Meta/Content";
}
```

Lines added: 6 (1 blank + 4 comment + 3 logic) at install.js:65 (post-edit). The defaulting is keyed on `undefined || null` to mirror the lookup posture used by `substituteStrict`/`substituteLenient`.

## Convention notes / deviations

- The plan references "the substitution variable map (where `rules_path`, `templates_path`, `commands_path` are wired)". That phrasing fits a hypothetical static map; the actual `install.js` has no such map — variables come straight from `config.variables`. The closest equivalent is the `variables` object initialization, where I added the default. This is a *naming* deviation, not a behavioral one: the result is identical to having the default in a static map.
- `content_path` joins `views_path`, `templater_scripts_path`, `scripts_path`, `rules_path`, `templates_path`, `commands_path` as a per-vault overridable path. It defaults to `Docs/Meta/Content` (per design doc).
- No mechanism / blueprint manifest yet uses `{{content_path}}` — that lands in S3.
