The `commands/new-project.md` file ships, but barebones has no slash-command plugin (e.g., Slash Commander, QuickAdd) wired up to register it. ### Findings filed for T4.10

> [!warning] Finding #1 — No slash-command UX for `/new-project`

## T4.7 — Validator + audit walker

### Outcome: ❌ DEFERRED — three real bugs surfaced

T4.7 was meant to confirm the validator and audit-walker mechanisms still work after v0.1.1's changes. Instead, every attempt to invoke them manually surfaced a real bug. The mechanisms install correctly (verified by the install harness), but they do not function end-to-end on real notes.

### Test attempts and what each surfaced

| Attempt | Snippet | Result |
|---|---|---|
| 1 | `<%* await tp.user.validate({ file: tp.file }); ... %>` | `Templater Error: e.lastIndexOf is not a function` — `tp.file` is a Templater wrapper, not a TFile; the validator's `tpFileOrObj?.file ?? tpFileOrObj` fell through to the wrapper itself. Wrapper's `.path` is a function, not a string. |
| 2 | `<%* await tp.user.validate(tp.file, "project"); ... %>` | Same error — same root cause. The validator's documented usage example (`tp.user.validate(tp.file, ...)`) doesn't work in current Templater. |
| 3 | `<%* const file = app.workspace.getActiveFile(); await tp.user.validate({ file }, "project"); ... %>` | New error: `Cannot read properties of undefined (reading 'slice')` — `validate.js:checkRequiredBlocks` (line 146) calls `expectedSnippet.slice(0, 80)` where `expectedSnippet = spec.content`. The project rule's `required_blocks` has no `content` field — uses `kind` + `must_call` + `via` instead. Schema mismatch between validator and rule. |

### Three findings filed for T4.10

> [!warning] Finding #2 — No standard runner UX for validator / audit walker
> No `Validate.md` template, no `/audit` slash command, no `Audit.md` template ship with the validator/audit mechanisms. Manual invocation requires one-shot Templater snippets via "Replace templates in active file." Hook-based validation (via `hook-validate.js`) presumably works automatically when Templater runs templates, but there's no manual-trigger UX for ad-hoc validation or audit-walker invocation. This blocks observability on rule compliance.

> [!warning] Finding #3 — `validate.js` input resolution broken for direct `tp.file` invocation
> `validate.js` documents two usage patterns: hook-style `{ file }` (still works, presumably) and template-style `tp.user.validate(tp.file, ...)` (broken). In current Templater versions, `tp.file` is a wrapper object whose `.file` property is undefined; the validator's `tpFileOrObj?.file ?? tpFileOrObj` resolution falls through to the wrapper itself. Crashes downstream on `getFileCache`'s string handling. Workaround: pass `{ file: app.workspace.getActiveFile() }` explicitly. Real fix: defensive input handling — try `tpFileOrObj?.file`, then `tpFileOrObj?.basename ? tpFileOrObj : null` (TFile detection), then `app.workspace.getActiveFile()` as last resort.

> [!warning] Finding #4 — `validate.js:checkRequiredBlocks` schema mismatch with project rule
> `validate.js` expects `spec.content` (string) on every required-blocks entry. The project rule (and likely future rules) uses `kind` + `must_call` + `via` schema. Result: `expectedSnippet.slice(0, 80)` crashes with `Cannot read properties of undefined`. Three lines of code; rule was authored against a different schema than the validator implements. The validator was never end-to-end tested against the project rule before v0.1.0 closed. Real fix: rewrite `checkRequiredBlocks` to handle the rule's actual schema (look for the dataviewjs block, parse its body, check whether `must_call` (e.g., `customJS.SpaceNavButtons`) appears via the `via` mechanism (e.g., `customjs-guard`)).
