## Why the workshop-side recipe is unsafe

The plan's T3.6 calls for a transient three-step setup in workshop:

1. Re-add `{ "name": "project", "version": "0.2.0" }` to the workshop's
   `Docs/Meta/platform-subscription.json:blueprints[]`.
2. Edit the workshop project manifest's `depends_on.nav-buttons.range`
   from `">=2.0.0"` to `">=3.0.0"` (unsatisfiable against the workshop's
   installed `nav-buttons@2.0.0`).
3. Run install, observe skip event.
4. Restore BOTH mutations afterward.

Two issues:

1. **Violates the locked subscription decision.** Per the design doc
   (Section "Decisions locked during brainstorming") and S2 close: *"workshop
   drops project through v0.1.2."* Even a transient re-add of project to
   `blueprints[]` contradicts the locked posture and risks leaving workshop
   subscribed if a restore step is missed.
2. **Restore-discipline risk.** The T2.fix incident (this same exec-log
   directory) was triggered by exactly this class of mutate-and-restore
   pattern: a hand-edit of `platform-installed.json` that didn't get cleanly
   restored. Two simultaneous mutation sites multiplies that risk.

## Why S4 is the cleaner test point

In S4, barebones already subscribes to `project@0.2.0` (per plan T4.1).
The version-range negative test then becomes a **single mutation site**:
edit only the workshop's `platform/blueprints/project/manifest.json` to set
`depends_on.nav-buttons.range: ">=3.0.0"`, run barebones install, observe
the skip event, restore the manifest range. No subscription juggling at all.

Same coverage, half the moving parts, zero violation of the locked workshop
posture.
