# v0.54.0 — FA-2 Entity wave (plan)

Apply the FA-1 substrate to four light entity-shaped blueprints: meetings + people + products + teams. Project sits alone in FA-3.

Per-blueprint actions:
- **Templates** — adopt canonical fields (add `created_at`, drop discriminator tag entries, quote canonical wikilink lists)
- **Manifests** — bump version + update `new_entity_buttons[].frontmatter_template` (where present) + add `extends: "_canonical-vocab"` to `rule_fragments[*].fragment`
- **Rule fragments** — drop `required_tags: [{ tag: "<discriminator>" }]` since migration strips those; drop `required_frontmatter.created` (replaced by canonical `created_at` via extends); drop `required_frontmatter.tags.contains: ["<discriminator>"]` on people

Per-consumer migration:
- `sauce migrate-frontmatter --vault <vault> --blueprint <bp> --apply` for each of {headspace-sauce, barebones, ero-sauce} × {meetings, people} (products+teams not subscribed at any consumer; deferred until subscription happens)
- `.sauce-backup/` sidecars per touched file
- `sauce audit --frontmatter-alignment --vault <vault>` post-apply reports zero `legacy_key_used` for migrated blueprints

## Risks / FIX-LATER pre-notes

- **`date:` vs `created_at:` on meetings**: per §A.6 of the design, `date:` is retained as a meeting-time narrowing (per-blueprint format = ISO+TZ). Per §C the table says "date → created_at" rename. We choose §A's interpretation: keep `date:` (meeting-time semantic) and ADD `created_at:` (canonical audit triplet) on new note creation. Migration backfills `created_at:` from mtime when missing; does NOT rename `date:`.
- **Empty `tags:` after discriminator strip**: templates that previously emitted `tags: ["meeting"]` will emit `tags: []` after S1. This is fine — `tags:` is user-controlled categorization per §A.5.
- **products + teams subscription**: neither blueprint is currently subscribed at any consumer. The template + manifest changes ship but go live only at consumer subscription. Audit walker's `_canonical-vocab` template fallback path handles the no-vault-template case.
- **Consumer `--apply` writes**: 3 vaults × 2 blueprints = up to 6 apply runs. Each writes `.sauce-backup/` sidecars. Worst case ~1000 files touched (headspace has ~100+ people notes). Heavy but reversible via backup.
