## End goal (the "why" this all serves)

From `Docs/why.md`: *"Author once. Subscribe per vault. Update on demand. Audit drift always."* Workshop authors mechanisms (cross-cutting code) + blueprints (note-type bundles); each consumer vault declares a subscription; an installer materializes mechanisms into each consumer with vault-specific path substitution. The four real consumer vaults are accuris (work), headspace (personal), ero (consulting), plus this workshop. The end state replaces a 9-card-per-pattern manual rollout dance (3 vaults × 3 phases) with a one-command consumer-side update.

## Side-quests built mid-cycle (both keepers)

### `platform/test/run-install.js` — install harness (~280 LOC)

Built during v0.1.1 S1 in response to "these template refreshes and manual actions to test and validate sucks." Replaces:
- Templater "Reload User Script Functions" click
- Insert Template modal + template selection
- Approval gate clicks
- Notice popup squinting
- Obsidian-restart-after-CSS-snippet step

CLI:
```bash
node platform/test/run-install.js [vault-path]   # default: cwd
node platform/test/run-install.js . --auto-approve   # default
node platform/test/run-install.js . --decline-all    # simulate Esc on every gate
node platform/test/run-install.js . --dry-run        # capture intended writes without executing
node platform/test/run-install.js . --verbose        # log every adapter.read/write/exists
```

Faked APIs: `tp.app.vault.adapter.{read,write,exists,mkdir,basePath}`, `tp.app.vault.{getAbstractFileByPath,read,modify,create,createFolder}`, `tp.system.suggester` (4-arg), global `Notice`. Exit 0 on no error/skip history this run; 1 otherwise.

### `platform/test/run-renderer.js` — renderer harness (~270 LOC)

Built at S2 close in response to "can't you test it yourself?" Replaces the three Obsidian eye-checks (T2.5/T2.6/T2.7) for SpaceNavButtons. CLI:

```bash
node platform/test/run-renderer.js              # default: all
node platform/test/run-renderer.js empty
node platform/test/run-renderer.js malformed
node platform/test/run-renderer.js unknown-action
```

Stubs `dv` (with `el`, `container.querySelector`, child `createEl`), `app.vault.*`, `app.workspace.openLinkText`, `app.isMobile`, global `Notice`. Loads SpaceNavButtons via `new Function('app', 'Notice', code + '\nreturn SpaceNavButtons;')` — no `vm` module needed. Each test sandboxes the registry (snapshot → mutate → restore in `finally`), so workshop's actual state is preserved.

Limitations of both harnesses (combined):
- No real Dataview pipeline (renderer harness fakes `dv`'s shape).
- No real CustomJS class lifecycle (instantiates directly).
- No Templater template execution (Templater's `<%* … %>` machinery only runs in Obsidian — affects S4 T4.6/T4.7; future side-quest may build a Templater harness).
- No mobile (iOS) install flow (landmine #8 — desktop-only `fs` + `child_process`).

## Critical decisions locked across both cycles

### v0.1.1
- Registry-driven nav (each blueprint/mechanism declares `nav_buttons[]`; installer aggregates into `Docs/Meta/nav-buttons-registry.json` with subscription-aware pruning).
- Lazy scaffold via `createFromTemplate` action type (no eager scaffold concept; click materializes target from blueprint-shipped template).
- Hard strip of SpaceNavButtons (renderer-only; no detectNoteType, no prev/next, no kanban-card sniffing).
- Workshop drops project subscription; barebones is sole project regression target.
- Dataview-driven kanban template body (depends only on Dataview, already required by platform).
- `workshop_version` 0.3.0 → 0.4.0 in S1.
- `template_source` declared as **basename only** in manifest (installer prepends `${content_path}/${source-name}/` automatically — design doc + plan patched).
- Cosmetic nits applied inline; deferred-but-noted: function name `validateAndResolve` slightly imprecise; per-entry `event: "warning"` divergence from `event: "error"` in siblings.

### v0.1.2 (designed)
- Pull-based, git-versioned distribution. Workshop = publisher; consumers = autonomous subscribers.
- **Thin stub bootstrap** — eliminates the "re-sync 3 bootstrap copies" landmine permanently.
- Tags = authoritative versions. `manifest.json:workshop_version` field demoted to legacy human-readable mirror.
- Local-only git initially; remote (GitHub) deferred to Phase 4.
- Tag naming: `v0.4.0` style (semver + `v` prefix).
- Branch model: main-only.
- Dirty-install behavior: warn + proceed; record `git_dirty: true` in installed.json history.
- `.gitignore`: `tmp.md`, `_backups/`, `.obsidian/workspace*`, `.obsidian/cache`, `Docs/Meta/platform-installed.json`, `boards/`.
- Workshop's project subscription stays dropped through v0.1.2.

## Cross-cutting non-negotiables (carrying forward)

From `CLAUDE.md` + `Docs/landmines.md` — these apply to every cycle:

- **No personal content in workshop.** If you find yourself writing a daily note here, you're in the wrong vault.
- **Mechanisms are versioned in `platform/manifest.json`.** Bump on any change to a mechanism's source files.
- **All file paths in mechanism code use `{{template_variables}}`**, never hardcoded paths. Installer substitutes per-vault.
- **All platform metadata is JSON, not YAML** (landmine #6 — Templater can't reach `parseYaml`).
- **Five customjs-guard landmines** apply to every Dataview view written.
- **Workshop dogfoods every release** — workshop's own self-install must succeed before promoting to consumers.
- **Failure-loud posture** — every error path Notice + history entry; no silent fallbacks.
- **Desktop-only** — mobile (iOS) unsupported. `require("fs")`, `child_process.execSync` only work on macOS/Windows/Linux.
- **No git in vault yet** (changing in v0.1.2). Today: each implementer logs to `Docs/plans/execution-logs/<plan-name>/T<task>-<slug>.md` instead of committing.
- **Bootstrap-copy discipline** — when `platform/install.js` changes, all 3 runtime copies must be re-synced byte-identically (retired in v0.1.2 S3).
- **Ask before**: bumping `workshop_version`, editing `.obsidian/`, hand-editing `platform-installed.json`, removing files inside `platform/mechanisms/*/` after a version is promoted.

## How the rhythm has been working (worth preserving)

Pacing pattern that emerged across both v0.1.0 and v0.1.1:

1. **Brainstorm via `de:brainstorming` skill.** AskUserQuestion drives multi-choice clarifications. User picks recommended options ~95% of the time so far; exceptions happen and are decisive (e.g., "I don't like workshop holding state for other vaults" → pivoted v0.1.2 entirely to pull-based).
2. **Design doc lands at `Docs/plans/<date>-<topic>-design.md`.**
3. **Implementation plan via `de:writing-plans` lands at `Docs/plans/<date>-<topic>-plan.md`.** 24 tasks across 4 stages roughly.
4. **Subagent-driven execution via `de:subagent-driven-development`:** bundle implementer per stage; spec review (independent verify, never trust report); quality review (parallel where safe); fix-ups inline if cosmetic.
5. **Stage-level checkpoints with the user** — surface only at stage boundaries OR at decision points (CLAUDE.md "ask before" gates, design-doc bugs surfaced by implementers, etc.).
6. **Execution logs** under `Docs/plans/execution-logs/<plan-name>/T<task>-<slug>.md` — one per implementer dispatch.

Side-quests are welcome mid-cycle when they unblock tooling/testing pain. Two have paid for themselves in the same cycles they shipped (the install harness; the renderer harness).
