# Design — `runTemplaterTemplate` nav-button action + `applyTemplaterFolderTemplates` helper + `to-do` blueprint (v0.4.0)

> [!abstract] Goal
> Settle two cross-cutting Templater-integration foundations BEFORE shipping the next batch of date-routed runtime-creation blueprints (to-do, meetings, journal, retros, …), then deliver the `to-do` blueprint as the canonical first consumer of those foundations. The two foundations are: **(1)** a new `runTemplaterTemplate` nav-button action type that invokes Templater's API directly with declarative folder + filename moment-format strings, and **(2)** a new `applyTemplaterFolderTemplates` installer helper that auto-applies a blueprint's template to manually-created files in its module directory. Together these eliminate the boards-parity intermediate-file + `<%* tp.file.move %>` hack that v0.2.0 was forced into, and they generalize cleanly to every future date-routed blueprint with zero per-blueprint scaffolding cost.

> [!info] Why this design exists — the foundational-decisions framing
> v0.3.0 closed with a working daily blueprint anchored on Obsidian's core Daily Notes plugin + Calendar. The natural next blueprint (to-do) lights up SpaceDailyDashboard's tasks panel and is the first **runtime-create-from-nav-button** date-routed blueprint. The boards-parity pattern (intermediate file at a fixed path + Templater-trigger-on-create + `<%* tp.file.move %>` self-routing in the template body) would technically work for to-do, but it requires the user to keep Templater's "Trigger on file creation" setting enabled globally, ships boilerplate `<%* %>` move logic in every template, and gets re-implemented in every future date-routed blueprint (meetings, journal, retros). v0.4.0 invests in the foundations once so every future blueprint inherits a clean contract: declare a folder format string + a filename format string + a template, and let the renderer + Templater API do the rest. The `applyTemplaterFolderTemplates` helper is the matching backstop for manual file creation (right-click "New note" while in `beacon/<module>/`, drag-in, etc.) — without it, manual creation produces empty files that the dashboard query and module-shape contracts can't account for.

> [!success]+ Locked decisions
>

>
> 2. **Cycle picked: v0.4.0** — `runTemplaterTemplate` action type + `applyTemplaterFolderTemplates` helper + `to-do@0.1.0` blueprint as the canonical first consumer. Rejected runners-up: v0.3.2 meetings (richer schema; better deferred until to-do validates the foundations); v0.2.1 project migration (orthogonal architectural debt; defer); v0.3.x Calendar display preferences (YAGNI).
>
> 3. **Workshop version bump approved: 0.7.0 → 0.8.0 (MINOR).** User-confirmed during brainstorming. Justification: schema additions (`templater_folder_templates[]`; new nav-button `runTemplaterTemplate` action shape) + mechanism bump (nav-buttons 2.0.0 → 2.1.0) + new blueprint (`to-do@0.1.0`). All non-breaking; existing `createFromTemplate` and `openLink` action types preserved verbatim. Tag at close: `v0.8.0` annotated.
>
> 4. **Foundation A — nav-buttons gets `runTemplaterTemplate` action type.** Bumps mechanism 2.0.0 → 2.1.0. Renderer's `_dispatchAction` (in `Docs/Meta/Scripts/nav-buttons/space-nav-buttons.js`) gets a new branch alongside `openLink` + `createFromTemplate`. Action shape: `{ type: "runTemplaterTemplate", template_source: "<basename>", folder: "<moment-format>", filename: "<moment-format>" }`. Renderer resolves folder + filename via `window.moment().format(...)` at click-time, ensures parent folder exists, idempotency-checks if target file exists (opens existing if so), then calls `app.plugins.plugins["templater-obsidian"].templater.create_new_note_from_template(templateFile, folder, filename, true)`.
>
> 5. **Why runTemplaterTemplate beats the boards-parity intermediate-file pattern.** Three concrete wins:
>    - **No intermediate file flash.** File lands directly at the dated target.
>    - **No dependency on Templater "Trigger on file creation" setting.** create_new_note_from_template invokes template processing inline regardless of trigger setting.
>    - **No `<%* tp.file.move %>` boilerplate in template bodies.** Template body is pure content + `<% %>` data tokens. Renderer owns routing.
>    - **Idempotency at renderer level.** Re-click on same day → renderer's existence check opens existing target. No template-side check needed.
>
> 6. **`runTemplaterTemplate` template_source resolution lives under `{{templates_path}}/`, NOT `{{content_path}}/`.** validateAndResolve (in `platform/install.js`) gets a per-action-type branch: existing `createFromTemplate` keeps `${contentPath}/${sourceName}/${template_source}` (preserves boards back-compat); new `runTemplaterTemplate` resolves to `${templatesPath}/${template_source}` (basename-only, drops `<sourceName>/` prefix). Rationale: templates_path is where Templater's UI naturally surfaces templates (the user can also invoke them via Templater's command palette directly if they want). content_path is a v0.1.1-era legacy that boards inherited; we don't propagate it to the new action type.
>
> 7. **Moment format strings carry the date routing.** Action shape uses moment.format() syntax for both `folder` and `filename`. Examples: `folder: "{{module_directory}}/YYYY/MM-MMMM"` → `"beacon/to-do/2026/05-May"`; `filename: "YYYY-MM-DD-[ToDo]"` → `"2026-05-04-ToDo"` (square brackets are moment's literal-escape syntax for chars that are also format tokens, e.g., the `D` in "ToDo"). Install-time `substituteLenient` resolves `{{module_directory}}` and friends BEFORE the moment.format() runtime resolution; both layers compose cleanly.
>
> 8. **Foundation B — new manifest field `templater_folder_templates[]` + new installer helper `applyTemplaterFolderTemplates`.** Schema: `[{ folder: "<path>", template: "<path>" }, ...]`. Helper mirrors v0.1.3's `applyTemplaterHotkeys` posture exactly: read `.obsidian/plugins/templater-obsidian/data.json`; additive merge into top-level `folder_templates[]` array (skip-match by `folder` field — first-wins idempotent); apply substituteLenient to both folder + template values; backup-on-edit to `<target>.beacon-backup`; malformed-JSON guard (C4 parity); failure-loud history under `step: "templater_folder_templates"`. Wire-up in `installItem` between `applySlashCommanderBindings` and `applyCorePluginSettings`.
>
> 9. **Empty-default quirk for `folder_templates[]` accommodated.** Templater's default config ships `folder_templates: [{folder: "", template: ""}]` (a single empty placeholder entry that the Templater UI renders as a "+ Add new" hint row). The helper treats any entry with empty `folder` AND empty `template` as the empty-default and replaces it on first-write rather than appending alongside; subsequent writes are pure additive append. This avoids polluting the user's config with a stranded placeholder once real entries exist.
>
> 10. **No new landmine #12 allowlist path.** `applyTemplaterFolderTemplates` writes to `.obsidian/plugins/templater-obsidian/data.json` — already path 1 of 3 in the allowlist. The new helper extends the existing allowed-path's surface to a new top-level field (`folder_templates[]`) but doesn't add a new file. Landmine #12's text gets a small edit to add `applyTemplaterFolderTemplates` to the helper-list line; no count change (still 3 paths).
>
> 11. **Decision 3 — flat dir vs date-routed for to-do: LOCKED date-routed.** Originally locked during the v0.3.1-framing AskUserQuestion round; reopened by user during scope expansion; relocked. With runTemplaterTemplate, date-routed is now strictly equal-cost to flat at the code level (one moment format string vs another). Date-routed wins on consistency with daily, long-term scalability (1000+ flat files = navigation-hostile in Obsidian's file explorer), and visual organization. Final layout: `beacon/to-do/YYYY/MM-MMMM/YYYY-MM-DD-ToDo.md`. Same convention applies to every future date-routed blueprint.
>
> 12. **`to-do@0.1.0` is the canonical first consumer + the cycle's smoke target.** Manifest declares: `module_directory: "to-do"`; `depends_on` nav-buttons ≥2.1.0 + customjs-guard ≥1.0.0; `files[]` 1 entry materializing the template body to `{{templates_path}}/Today To-Do.md`; `nav_buttons[]` 1 entry using `runTemplaterTemplate`; `templater_folder_templates[]` 1 entry mapping `{{module_directory}}` to the same template; terse post_install notice. NO `external_plugins[]` / `core_plugin_settings[]` / `pre_install[]` / `templater_hotkeys[]` / `slash_commander_bindings[]` / `customjs_classes[]` / `rule_fragments[]`.
>
> 13. **Template body is clean — no `<%* %>` self-routing logic.** Frontmatter `created` token + `tags: [{{vault_identity_tag}}, todo, YYYY/MM/DD]` + `## Tasks` section + one starter `- [ ]` line. Substitution layers compose: install-time `substituteLenient` resolves `{{vault_identity_tag}}` and `{{module_directory}}` into the materialized body; click-time Templater processes `<% %>` data tokens during create_new_note_from_template's inline processing.
>
> 14. **Nav-button visibility: same scope as boards.** "+ Today's To-Do" renders everywhere SpaceNavButtons does (daily notes, project notes, anywhere with the dataviewjs block). No per-page filtering. Acknowledged future-cycle work; out of scope for v0.4.0.
>
> 15. **Re-click semantics: time-of-click anchored, not host-note anchored.** `tp.date.now()` returns the actual current date regardless of which note the user clicked from. User on a 2026-05-03 daily note clicking "+ Today's To-Do" on 2026-05-05 gets the 2026-05-05 to-do file. Locked behavior; matches v0.3.0 daily's similar contract.
>
> 16. **Two stages.** S1 = foundations (both action type + helper + harness). S2 = to-do blueprint authoring + smokes + cycle close. Bundling foundations into one stage rather than splitting (S1 = action type, S2 = helper, S3 = blueprint) keeps the cycle tight; both foundations are Templater-integration changes that share testing surface and review focus.

### Action-type taxonomy after v0.4.0

| Action type | Status | Use case | Resolution |
|---|---|---|---|
| `openLink` | unchanged (v0.1.1) | Open a fixed wikilink target | renderer calls `app.workspace.openLinkText(action.target)` |
| `createFromTemplate` | unchanged (v0.1.1) | Create a single-fixed-path file from a content template (boards' Board button) | renderer reads template body raw → `app.vault.create(target, body)`; existence-check opens-existing |
| `runTemplaterTemplate` | **NEW (v0.4.0)** | Create a date-routed runtime file via Templater API with declarative folder + filename | renderer resolves moment-format strings → `app.vault.createFolder` → `templater.create_new_note_from_template(templateFile, folder, filename, true)` |

The three action types compose cleanly. Existing boards manifest stays at `createFromTemplate`; no migration. Future runtime-create blueprints (to-do, meetings, journal, retros) use `runTemplaterTemplate`.

### `applyTemplaterFolderTemplates` mechanic — pseudocode

```
async function applyTemplaterFolderTemplates(tp, manifest, history, git, variables) {
  if !manifest.templater_folder_templates or empty: return
  
  adapter = tp.app.vault.adapter
  dataPath = ".obsidian/plugins/templater-obsidian/data.json"
  
  if !exists(dataPath):
    history.push(warning, "templater data.json absent")
    return
  
  raw = await adapter.read(dataPath)
  
  try: data = JSON.parse(raw)
  catch (e):
    history.push(error, "malformed JSON, not overwriting")
    Notice user
    return
  
  if !Array.isArray(data.folder_templates):
    history.push(warning, "folder_templates not an array")
    return  // C4 parity: don't overwrite an unexpected shape
  
  applied = []
  skipped = []
  
  for entry in manifest.templater_folder_templates:
    resolvedFolder = substituteLenient(entry.folder, variables)
    resolvedTemplate = substituteLenient(entry.template, variables)
    
    if !resolvedFolder or !resolvedTemplate:
      history.push(warning, "empty folder or template after substitution")
      continue
    
    // Match-by-folder semantics; first-wins idempotency
    existingIdx = data.folder_templates.findIndex(ft => ft.folder === resolvedFolder)
    if existingIdx >= 0:
      existing = data.folder_templates[existingIdx]
      if existing.folder === "" and existing.template === "":
        // Empty-default placeholder; replace it
        data.folder_templates[existingIdx] = {folder: resolvedFolder, template: resolvedTemplate}
        applied.push(...)
        continue
      if existing.template === resolvedTemplate:
        skipped.push(...) // already matches; skip
        continue
      // Folder matches but template differs — preserve user customization, skip
      skipped.push(...)
      history.push(warning, "user override preserved")
      continue
    
    // No match → append
    data.folder_templates.push({folder: resolvedFolder, template: resolvedTemplate})
    applied.push(...)
  
  if applied.length === 0:
    history.push(info, "skipped_existing")
    return
  
  // Backup-on-edit (one-deep, overwrite-on-edit)
  await adapter.write(dataPath + ".beacon-backup", raw)
  await adapter.write(dataPath, JSON.stringify(data, null, 2))
  
  history.push(info, "applied", {applied, skipped})
}
```

History event types added by this helper:
- `info, step: "templater_folder_templates", action: "applied"` — at least one entry written
- `info, step: "templater_folder_templates", action: "skipped_existing"` — all entries already matched
- `warning, step: "templater_folder_templates", message: "<reason>"` — non-fatal issues (data.json absent; folder_templates not an array; empty after substitution; user override preserved)
- `error, step: "templater_folder_templates", message: "<reason>"` — fatal issues (malformed JSON; backup write failed; data.json write failed)

### `validateAndResolve` extension for `runTemplaterTemplate`

`platform/install.js:validateAndResolve` (currently ~line 1021) gets a per-action-type branch:

```
function validateAndResolve(btn, sourceName, variables, history, git) {
  if (basic shape invalid) return null
  
  if (btn.action.type === "createFromTemplate" && btn.action.template_source) {
    const contentPath = variables.content_path || "Docs/Meta/Content"
    const resolvedTarget = substituteLenient(btn.action.target, variables)
    return {
      ...btn,
      action: {
        ...btn.action,
        target: resolvedTarget,
        template_source: `${contentPath}/${sourceName}/${btn.action.template_source}`,
      },
    }
  }
  
  // NEW v0.4.0
  if (btn.action.type === "runTemplaterTemplate" && btn.action.template_source) {
    const templatesPath = variables.templates_path || "Docs/Meta/Templates"
    return {
      ...btn,
      action: {
        ...btn.action,
        // target/folder/filename stay as moment-format strings (resolved at click-time)
        // BUT install-time substituteLenient still applies for {{module_directory}} etc.
        folder:   substituteLenient(btn.action.folder || "",   variables),
        filename: substituteLenient(btn.action.filename || "", variables),
        template_source: `${templatesPath}/${btn.action.template_source}`,
      },
    }
  }
  
  return btn  // openLink and unknowns pass through unchanged
}
```

### Renderer `_dispatchAction` extension

`Docs/Meta/Scripts/nav-buttons/space-nav-buttons.js` gets a new branch after the existing `createFromTemplate` block:

```js
if (type === "runTemplaterTemplate") {
  const tpPlugin = app.plugins.plugins["templater-obsidian"];
  if (!tpPlugin || !tpPlugin.templater) {
    new Notice(`nav-buttons: Templater plugin not enabled (from ${btn._source})`, 8000);
    return;
  }
  
  // Resolve moment-format strings at click-time (after install-time {{...}} already substituted)
  const folder = (action.folder || "").trim()
    ? window.moment().format(action.folder)
    : "";
  const filenameNoExt = (action.filename || "").trim()
    ? window.moment().format(action.filename)
    : "Untitled";
  const target = folder ? `${folder}/${filenameNoExt}.md` : `${filenameNoExt}.md`;
  
  // Idempotency: if target exists, just open it
  const existingTarget = app.vault.getAbstractFileByPath(target);
  if (existingTarget) {
    app.workspace.openLinkText(target, "");
    return;
  }
  
  // Ensure parent folder exists
  if (folder && !app.vault.getAbstractFileByPath(folder)) {
    try {
      await app.vault.createFolder(folder);
    } catch (folderErr) {
      if (!/already exists|exists/i.test((folderErr && folderErr.message) || "")) {
        new Notice(`nav-buttons: cannot create folder ${folder} — ${folderErr.message}`, 8000);
        return;
      }
    }
  }
  
  // Locate template file
  const templateFile = app.vault.getAbstractFileByPath(action.template_source);
  if (!templateFile) {
    new Notice(`nav-buttons: template not found at ${action.template_source} (from ${btn._source})`, 8000);
    return;
  }
  
  // Invoke Templater API
  try {
    await tpPlugin.templater.create_new_note_from_template(templateFile, folder, filenameNoExt, true);
  } catch (err) {
    new Notice(`nav-buttons: Templater create failed — ${err.message}`, 8000);
    return;
  }
  return;
}
```

## Trade-offs vs alternatives explicitly considered

> [!example]- Why these alternatives were rejected
>
> **A) Keep boards-parity intermediate-file pattern; ship v0.3.1 patch with no foundations.** Tightest scope; ships to-do in a single small cycle. Rejected: kicks the foundations decision down the road, forces every future date-routed blueprint (meetings, journal, retros) to copy-paste the boards `<%* tp.file.move %>` template body, and accumulates exactly the kind of cross-cutting tech debt the user specifically called out preventing.
>
> **B) Ship `runTemplaterTemplate` only; defer `applyTemplaterFolderTemplates` to v0.4.1.** Smaller cycle. Rejected: both foundations are Templater-integration changes and share testing surface; bundling them avoids the second-cycle overhead and locks the ecosystem story coherent (every file in `beacon/<module>/` is template-shaped, regardless of how it was created). User explicitly endorsed bundling.
>
> **C) Ship `applyTemplaterFolderTemplates` only; keep boards-parity intermediate-file pattern + rely on folder_templates to fire on the intermediate file's creation.** Equivalent functional result without `runTemplaterTemplate`. Rejected: still requires intermediate-file UX wrinkle, still requires "Trigger on file creation" globally enabled, still requires `<%* tp.file.move %>` boilerplate (folder_templates points to the template; the template still has to self-route since folder_templates can't dynamically pick a date-nested folder for a specific creation). The `runTemplaterTemplate` action lifts ALL of these constraints.
>
> **D) Add `invoke_command` action type alongside `runTemplaterTemplate` (the v0.4.0 candidate from the handoff doc).** Would address the smoke-A user observation from v0.3.0 ("did the daily note button make it into the nav buttons?") by adding a "Today" button firing `daily-notes:goto-today`. Rejected for v0.4.0: different problem space (firing arbitrary command IDs vs invoking Templater templates with declarative folder/filename); the per-action dispatch architecture laid down in v0.4.0 makes `invoke_command` a trivial v0.4.x or v0.5.x add-on. Carry forward as a candidate for the next foundation cycle.
>
> **E) Retroactively migrate boards' card pattern to `runTemplaterTemplate` or to `folder_templates[]`.** Boards' cards are created by the Kanban plugin (not nav-buttons), so `runTemplaterTemplate` doesn't apply. A `templater_folder_templates` entry for `beacon/boards/cards/` would auto-apply the card template to manually-created files in that dir; useful but not urgent (Kanban's own `new-note-template` setting already handles the kanban-driven creation). Defer; revisit if a real failure mode surfaces.
>
> **F) Renderer-level date token expansion (custom syntax e.g., `<YYYY>` rather than moment-format strings).** Rejected: moment is already loaded in the renderer's runtime context (`window.moment`), the format-token vocabulary is wide and well-known, and using moment.format directly avoids inventing a project-specific micro-DSL that future readers have to learn. Square-brackets-for-literal escaping (e.g., `[ToDo]`) is moment's native syntax, not a project invention.
>
> **G) Pre-compute folder + filename at install time and store the resolved date in the registry.** Would resolve "today" once at install, so re-runs on a different day would still route to the install-day's date. Rejected as obviously broken; raised only to confirm the runtime-resolution architecture.
>
> **H) Per-page nav-button visibility filters (button only renders on daily notes).** Boards parity says all buttons render everywhere SpaceNavButtons mounts. Adding per-page filters expands SpaceNavButtons mechanism scope significantly (page-context awareness; query API). Defer to a future cycle if cross-pollution becomes an actual ergonomics problem.
>
> **I) Flat dir layout `beacon/to-do/<TODAY>-ToDo.md`.** With `runTemplaterTemplate` making date-routed equal-cost to flat at the code level, date-routed wins on consistency with daily, long-term scalability, and visual organization. No remaining argument for flat.
