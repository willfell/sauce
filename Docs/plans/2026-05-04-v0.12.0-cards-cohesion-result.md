# v0.12.0 Result — cards cohesion (BeaconCards 0.2.0 + meetings + daily migrations)

> [!success] Cycle status
> v0.12.0 — workshop bumped 0.11.0 → **0.12.0 MINOR**. BeaconCards mechanism extended to **0.2.1** (subtitle polymorphism + `badges[].icon` API in 0.2.0; visual polish PATCH in 0.2.1). MeetingsHubCards rewritten as a thin BeaconCards wrapper with async pre-fetch adapter (**meetings@0.1.2 → 0.2.0**). SpaceDailyDashboard meetings panel migrated to BeaconCards; tasks panel reverted to bullet-list per audit-predicted regression (**daily@0.1.4 → 0.2.1** via 0.2.0 + revert PATCH). Two inline-CF sub-stages (cards@0.2.1 visual polish; daily@0.2.1 tasks-as-bullets revert). All S1/S2/S3 manual smokes user-confirmed. Renderer harness +6 BC cases all green; helper-cases held at 255. Tag `v0.12.0` annotated and pushed at C3 (post-this-writeup).

---

## Outcome summary

> [!abstract] What landed
> The cards-cohesion cycle unified the workshop's card-style surfaces on the BeaconCards mechanism. After v0.12.0, hub-style listings (project hub, meetings hub, daily meetings panel) all render through a single primitive — no blueprint hand-rolls card chrome. The cycle simultaneously discovered the boundary of where cards stop being the right primitive: the daily tasks panel was reverted to a bullet list mid-cycle, validating the v0.5.0 audit's prediction that cards-for-tasks would be a regression versus the compact at-a-glance bullet list. **The cohesion thesis was proven for hubs and disproven for tasks** — both findings codified in the post-cycle posture.

## Lessons learned

> [!warning]- 1. In-cycle re-process bump rule + dual-source-of-truth gap
> S2.2's commit (`b7ab2e3`) bumped only the blueprint's own `manifest.json` and missed the workshop catalogue line at `platform/manifest.json:20`. The installer reads versions from the catalogue (not the blueprint manifest) for subscription-pin compares — first barebones install run skipped meetings with "subscription pins meetings@0.2.0 but workshop has 0.1.2". Implementer caught + fixed in S2.3 commit (`1d897b7`).
> **Takeaway:** per-blueprint version bumps require lockstep edits to BOTH the blueprint's own `manifest.json` AND the workshop catalogue line in `platform/manifest.json`. Future cycles should bake this into manifest-bump tasks (e.g. as a single edit pair in any "bump X" task).

> [!warning]- 2. Audit prediction validated
> The cards-mechanism-fit audit (`Docs/plans/2026-05-04-cards-mechanism-fit-audit.md`) flagged tasks panel as **EXTEND** with the warning: "Bulleted `<ul>` is intentionally compact. Card grid for tasks would be visually heavier than the current at-a-glance list. Migration may be a regression unless we deliberately reshape the daily dashboard around cards." User confirmed at Smoke A.
> **Takeaway:** the audit's caution was load-bearing; "EXTEND" verdict warrants a brainstorm gate BEFORE a reshape that the audit itself flagged as risky. Future cycles that touch EXTEND-tier surfaces should open a brainstorming sub-step rather than directly migrating.

> [!info]- 3. BeaconCards is right for hubs, not for tasks
> Cycle-derived posture: BeaconCards is the right primitive for hub-style listings (meetings hub, project hub, daily meetings panel) where each item is a real file with title/subtitle/meta worth surfacing visually. **Bullet lists remain the right primitive for compact at-a-glance task panels** where each line item is just text + click-to-jump.
> **Takeaway:** codify this distinction in the post-cycle handoff. The cohesion thesis is proven for hubs and disproven for tasks. Don't reach for BeaconCards for inline task / checklist / queue surfaces.

> [!info]- 4. Inline-CF-extension precedent reinforced (third cycle)
> v0.5.0 + v0.11.0 + v0.12.0 all surfaced user polish at stage close that landed as `Y.Z.x` PATCH sub-stages with their own commits. v0.12.0 was a TWO-inline-CF cycle (cards@0.2.1 + daily@0.2.1).
> **Takeaway:** the reserved Y.Z.x headroom paid off; the pattern is now strong evidence for future cycles. Plans should always reserve patch headroom for the consumer blueprints AND any extending mechanism.

> [!info]- 5. TDD-first on renderer harness extensions worked
> BC1–BC6 cases were written + failing at `b72801c` BEFORE the implementation at `e1e01a8`. Cases caught the spec correctly; the backward-compat regression cases (BC2/BC3/BC5) prevented landing breakage on existing project blueprint callsites.
> **Takeaway:** for any future BeaconCards API extension, write the renderer-harness case first, commit it failing, then implement. The regression-guard cases are cheap insurance against breaking existing callsites.

> [!info]- 6. dv-shim shape spot-check
> Design called for a spot-check of BeaconCards' `dv.X` usage before SpaceDailyDashboard rewrite (R3 in design doc). Grep at S3 Task 3.1 confirmed only `dv.container.createEl(...)` is touched — shim `{container: subContainerEl}` was sufficient.
> **Takeaway:** the pre-empted risk was accurately scoped. Future shim-based wrapping should follow the same grep-first methodology to avoid hidden-coupling surprises.
