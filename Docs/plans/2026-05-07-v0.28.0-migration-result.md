### NEW `sauce migrate` CLI verb

5th verb in `platform/cli/sauce-cli.js` VERBS registry. Invoked as:

```
sauce migrate --from <src> [--commit] [--no-keep-backups]
```

Dry-run by default (writes `migration-plan.json` + emits stdout summary; exits 0 with no other writes). `--commit` triggers 5-phase atomic write. Verb is fully discoverable via `sauce help`; argv parse + arg validation tested via run-cli.js cases C23-C28.

## Plan deviations (resolved at design/build time; documented for forensics)

1. **CF-1 skip-list** — design Section 3 listed `.obsidian/`, `.git/`, `.DS_Store`, `node_modules/`, `*.tmp`, `Invalid date/` only. Real Accuris source contained Python venvs + `__pycache__` + `.smart-env` plugin cache totaling ~28k files. Skip-list extended in-cycle; M4 case + 3 fixture files added. Lesson surfaces a generalizable pattern: **dry-run against full real-data source is the only reliable surface for skip-list completeness**; synthetic fixtures can't anticipate every cache + cruft pattern in a 1.5GB+ vault.

2. **CF-2 legacy-config override** — design Section 4 phase-2 said "invoke shared bootstrap orchestrator phases ... using `--non-interactive --mechanisms=all --blueprints=all`". The implementation honored `ctx.config.variables` via `Object.assign(canonical, variables)` — defensible "respect consumer config" posture, but wrong for migration cycles where the goal IS to bring vault to current Sauce schema. Lesson: **migration phase 2 should always write canonical paths; legacy config preservation is the wrong default for cycles whose stated goal is schema modernization**.

3. **CF-3 .obsidian scrub** — design Section 4 phase-2 said "restore .obsidian/ from backup" as a way to preserve user's vault-level Obsidian config without re-fetching plugins. But the restore brought back legacy `data.json` plugin state (templater folder-templates pointing at non-existent paths) that the installer's first-wins additive merge couldn't overwrite. Lesson: **after .obsidian/ restore from backup, surgically delete allowlist-managed plugin data files so installer helpers write fresh canonical entries**. Codified as a v0.28.0 commit.js code comment + new landmine candidate amendment to #12.

## NEW lessons (v0.28.0)

(a) **Parallel-subagent dispatch with literal-contract prompts works clean at 5+ subagents.** Third data point after v0.21.1 (3 parallel) + v0.27.0 (5 parallel — same scale as here). All 5 implementer subagents (T2.1-T2.5) landed clean first-pass with zero merge conflicts and zero re-review cycles needed. The LITERAL contract from S1.4 step 1 quoted verbatim in each subagent prompt was the load-bearing surface; without it, subagents would have diverged on `plan()` / `migrate()` shape.

(b) **Two-stage subagent review caught 2 CRITICAL + 6 IMPORTANT pre-S2-commit.** **9 data points reaffirming v0.20.0 lesson d** (after v0.20.0 / v0.22.1 / v0.23.0 / v0.24.0 / v0.25.0 / v0.26.0 / v0.26.1 / v0.27.0). Spec-only review would have shipped: silent verbatim swallow, silent finalize swallow, non-crash-safe backup, unverified restore target, propagating walker errors, non-atomic plan writes, inconsistent target containment, non-unique tmp filenames. Quality review is non-negotiable — never skip.

(c) **Dry-run against full real-data source is the only reliable surface for skip-list completeness.** Synthetic fixtures can't anticipate Python venvs + `__pycache__` + `.smart-env` plugin caches in a 1.5GB+ source vault. CF-1 surfaced 28,500+ cruft entries that no design-time review would have caught. Generalize: any new migrator + skip-list cycle should include a real-data dry-run as a S3 mandatory step (already in v0.28.0 plan; reaffirm for ero/headspace).

(d) **Migration phase 2 must write canonical paths; legacy config preservation is the wrong default for schema-modernization cycles.** CF-2 root cause was "respect consumer config" posture — defensible for plain `sauce update` reinstall flow but wrong when the cycle's stated goal IS schema modernization. Pattern reusable for ero/headspace migration: `legacyVars` should be inspected for ONLY a small whitelist (here: `vault_identity_tag`); everything else uses canonical defaults.

(e) **After .obsidian/ restore from backup, scrub allowlist-managed plugin data files.** CF-3 root cause was first-wins additive merge in installer helpers — they couldn't overwrite legacy data.json entries from backup. Fix codified in commit.js phase 2c: `ALLOWLIST_MANAGED_PATHS` constant lists the 9 paths to delete after restore. Pattern reusable for ero/headspace + any future migration cycle that restores `.obsidian/`.

(f) **Crash-safety markers (`.in-progress` + sentinel JSON) are cheap insurance for multi-phase orchestrators.** v0.28.0's commit.js writes `.in-progress` at start of phase 1 and `.sauce-migration-meta.json` recording vaultPath; precheck can detect orphan in-progress markers and `restoreFromBackup` validates the sentinel before wiping. Total cost: ~10 LOC; total benefit: silent data loss prevention if SIGKILL hits mid-backup.

(g) **Plan-baseline-count drift continues to be harmless when commit-message reports actual reality.** Plan T1.6 predicted run-cli.js cli=48/0 (+6 sub-asserts); actual was 50/0 (+8). Each C-case has 1-2 sub-asserts; baseline drift +2 is normal cycle-time-accuracy noise. Fifth data point (v0.13.0 / v0.21.1 / v0.22.1 / v0.26.1 / v0.28.0).

(h) **`process.cwd()` fallback in plan.emitPlan was a harness-pollution bug latent in design.** When dispatcher.run is invoked from harness with `ctx={}`, the plan-emit step wrote `migration-plan.json` to the workshop root (cwd at harness-run time). Fix: don't fall back to cwd; require explicit `outDir` from caller; harness path skips file-write entirely. Pattern: any helper that writes-to-cwd-as-fallback risks polluting the test environment; require explicit outDir.

(i) **Reserved-headroom precedent extends to 17 data points; 7th MIXED case** (v0.22.1 / v0.23.0 / v0.24.0 / v0.25.0 / v0.26.0 / v0.26.1 / v0.27.0 / v0.28.0). First-of-its-kind orchestration cycles consistently draw 1-5 CFs in middle-of-range; pure-additive cycles draw 0; pure-rename cycles draw 1-3. **Multi-surface cycles multiply CF count** (v0.28.0 = 3 first-of-kind surfaces × ~3.3 CFs/surface = 10 total) — useful for sizing future cycles that bundle orchestrator + new migrator + multi-vault rollout in one window.

(j) **Silent file-drops during full-run can be intermittent / environmental (CF-10 lesson).** Two confirmed drops during accuris + headspace --commit runs (Planning-Board.md verbatim + Find passport.md boards-rewrite). Isolation tests (boards.migrate / verbatim.migrate / minimal-repro on synthetic 2-file dir) all succeed. Re-running each migration to scratch yields `verified=plan_count, missing=0`. Root cause not pinned — likely macOS APFS readdir / atomic rename race under load (1670+ files in <5s). Defensive fix: phase 4.7 verification + phase 4.8 auto-recovery via re-invocation. Pattern reusable for any future multi-write orchestrator: **always verify the post-write state against the planned state; auto-retry per-entry on missing rather than blanket-abort + restore**.

(k) **migration.log fields move from "what was planned" to "what was verified".** Pre-CF-10, `counts: {migrator/action: N}` was built from `plan.planEntries` and reflected planning intent, not actual writes. Post-CF-10, `verification: {verified, missing[], recovered}` is the source of truth for actual file presence. Pattern reusable for any tool whose log claims completion without verifying outcomes.
