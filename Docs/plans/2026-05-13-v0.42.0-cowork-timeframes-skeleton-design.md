# v0.42.0 cowork timeframes skeleton — design

**Date:** 2026-05-13
**Cycle:** v0.42.0 (MINOR)
**Headline blueprint:** `cowork@0.3.0 → 0.4.0` (MINOR)
**Mode:** pure-additive scaffolding
**Spec author:** brainstorm session 2026-05-13

## Purpose

Ship the **navigation + structural skeleton** for cowork to write into across daily / weekly / monthly timeframes. The cycle does not add cowork content writes, daily→weekly→monthly aggregation, or scheduling. Those are deferred to follow-on cycles. This cycle lands the surface — hubs, note shells, prompt-file convention, nav-buttons, slash commands, and a Claude-invokable scaffold skill — so the next two cycles ("aggregation pipeline" and "scheduling substrate") slot in with file paths already established.

The user-visible north-star after this cycle ships: click a "Cowork" nav-button anywhere in the vault → land on `Cowork.md` → see a Timeframes section with three cards (Daily / Weekly / Monthly hubs) → click any of them → see card-listed notes in that timeframe → click any card → land on the underlying note → navigate back via a breadcrumb row. Click "This Week" or "This Month" from the global nav row to create / open the current week or month's note.

## Scope

In scope:
- Three new hub notes (`Daily Hub.md`, `Weekly Hub.md`, `Monthly Hub.md`) under `spice/cowork/`.
- Two new note-type families: weekly notes at `spice/cowork/weekly/YYYY/YYYY-Www.md`, monthly notes at `spice/cowork/monthly/YYYY/YYYY-MM.md`.
- Four prompt-file stubs at `spice/cowork/prompts/{morning-briefing,eod-review,weekly-review,monthly-review}.md` — empty-bodied path-convention placeholders.
- Three new global nav-buttons: `cowork-hub` (openLink to Cowork.md), `cowork-weekly-this` (runTemplaterTemplate for this-week's weekly note), `cowork-monthly-this` (runTemplaterTemplate for this-month's monthly note).
- Two new slash commands: `/weekly`, `/monthly`. Existing `/cowork` body updated.
- One new sub-skill: `cowork:scaffold-timeframes`. Composed into existing `cowork:bootstrap-vault` as a final step; also invocable standalone.
- A scaffold-status dataviewjs callout on `Cowork.md` master hub that detects missing pieces and tells the user what to run.
- Three new CustomJS helpers (`CoworkDailyHubCards`, `CoworkWeeklyHubCards`, `CoworkMonthlyHubCards`) using the existing `BeaconCards` (cards@0.2.4) for layout.
- Six new rule_fragments validating the new artifacts.
- Two new templates: `Weekly Note.md`, `Monthly Note.md` under `ranch/templates/`.
- Workshop dogfood: `sauce reinstall --vault $PWD` + audit clean + manual Obsidian smoke.

Out of scope (deferred):
- **Aggregation pipeline.** Daily-tag rollup (`#decision`, `#shipped`, `#blocker`) into weekly sections; weekly → monthly. Cycle: "cowork aggregation."
- **Scheduling substrate.** Cron / launchd / headless-Claude / credential storage / log rotation. Cycle: "cowork scheduling."
- **Per-note prev/next inline navigation** on individual daily / weekly / monthly notes. The hub-to-hub global nav row covers the primary navigation use case for the skeleton.
- **Interactive prompt-stub customization** during scaffold (interview-style: "what should the morning briefing emit by default?"). Skeleton ships empty stubs; user edits files directly. Interactive flow lands when scheduling is wired.
- **Migration of existing cowork `summaries/weekly/*` and `summaries/monthly/*` content** to the new `weekly/` and `monthly/` paths. The `summaries/` directory stays where it is; deprecated path; later cycle migrates.

## Architecture

Two changes to the platform's mental model:

1. **`spice/cowork/` becomes the timeframe entry point**, in addition to its existing role as the engagement-aware automation layer. The cowork master hub gains a Timeframes section above the engagement table. Cowork remains a single blueprint with a single module_directory (`spice/cowork/`), honoring landmine #11.

2. **Cross-blueprint read-only data flow.** The new `CoworkDailyHubCards` helper reads `spice/daily/**/*.md` to render its cards — but cowork writes nothing to `spice/daily/`. The daily blueprint stays where it is and continues to own its directory. This is the platform's first generalized cross-blueprint read pattern; future cycles can extend it (e.g., cowork's monthly hub eventually reading weekly notes for aggregation).

No new mechanisms. No changes to the installer or to existing mechanisms (customjs-guard, nav-buttons, cards, validator, audit, etc.). The cycle is pure blueprint authoring: manifest additions, template materialization, helper authoring, rule fragments.

## Hub designs

### `Cowork.md` (master, evolves)

Adds a scaffold-status dataviewjs callout at the top (above the existing "What is cowork?" abstract) and a Timeframes section above the existing Engagements + cadences section. Otherwise unchanged.

The scaffold-status callout reads four conditions:
1. Are engagements present in `context/vault-config.md`? (existing check)
2. Does this-week's weekly note exist at `weekly/<this-year>/<this-week-label>.md`?
3. Does this-month's monthly note exist at `monthly/<this-year>/<this-month-label>.md`?
4. Do the four prompt stubs exist at `prompts/*.md`?

If anything's missing, render `> [!warning]+ Cowork scaffold incomplete` with a checklist showing missing pieces + a one-line Claude invocation to run (`cowork:bootstrap-vault` covers all four; `cowork:scaffold-timeframes` covers just #2–#4). When everything's present, the callout renders nothing.

The Timeframes section is a 3-card inline dataviewjs block (no dedicated CustomJS class) that constructs three BeaconCards items hardcoded for Daily Hub / Weekly Hub / Monthly Hub and calls `BeaconCards.render(dv, opts)` directly. Click any card → lands on the sub-hub.

### `Daily Hub.md` (new)

```
---
type: cowork-daily-hub
tags: [cowork-hub, daily-hub]
---

> [[Cowork|◀ Cowork]] · [[Daily Hub]] · [[Weekly Hub]] · [[Monthly Hub]]

# Daily Notes

```dataviewjs
await dv.view("{{views_path}}/customjs-guard", { class: "CoworkDailyHubCards" });
```
```

Body is the breadcrumb row, heading, and a single customjs-guard-wrapped dataviewjs invocation. `CoworkDailyHubCards` query: `dv.pages('"spice/daily"').where(p => p.file.name.match(/^\w+-\d{4}-\d{2}-\d{2}$/))`. Groups: by year (descending) → by month (descending: `2026-05`, `2026-04`, …) → notes (descending by date). Card face: title = `<dddd> · <YYYY-MM-DD>`; subtitle = relative ("Today" / "Yesterday" / "Wed 2 weeks ago"); body snippet = first 120 chars stripped from `## Notes` section if present, else empty. Empty state: `> [!info]+ No daily notes yet · Click **Daily** in the nav row to create today's.`

### `Weekly Hub.md` (new)

Same shape as Daily Hub. `CoworkWeeklyHubCards` queries `dv.pages('"spice/cowork/weekly"').where(p => p.file.name.match(/^\d{4}-W\d{2}$/))`. Groups: by year (descending) → weeks (descending by file.name). Card face: title = file.name (e.g., `2026-W20`); subtitle = derived date range from frontmatter (`May 11 – May 17`); body snippet = first 120 chars from `## Notes`. Empty state: `> [!info]+ No weekly notes yet · Click **This Week** in the nav row to create one.`

### `Monthly Hub.md` (new)

Same shape. `CoworkMonthlyHubCards` queries `dv.pages('"spice/cowork/monthly"').where(p => p.file.name.match(/^\d{4}-\d{2}$/))`. Groups: by year (descending) → months (descending by file.name). Card face: title = `MMMM YYYY` (e.g., "May 2026"); subtitle = `YYYY-MM`; body snippet = first 120 chars from `## Notes`. Empty state: `> [!info]+ No monthly notes yet · Click **This Month** in the nav row to create one.`

## Scaffold flow

### New skill `cowork:scaffold-timeframes`

Lives at `platform/blueprints/cowork/skills/skills/scaffold-timeframes/SKILL.md`. Materialized to `.claude/skills/cowork/skills/scaffold-timeframes/SKILL.md`. Sub-skill (under `skills/skills/`, not orchestrator).

Inputs: none required. Optional `{ dry_run: bool }` for diagnostic invocations.

Steps:
1. Use Skill `cowork:date-context` with `{}`. Capture today's date, ISO week label (`YYYY-Www`), and month label (`YYYY-MM`).
2. Compute target paths: `spice/cowork/weekly/<YYYY>/<YYYY-Www>.md` and `spice/cowork/monthly/<YYYY>/<YYYY-MM>.md`.
3. Check each target path via `mcp__obsidian__list_files` (or equivalent file-read). If missing, instantiate from `ranch/templates/{Weekly Note,Monthly Note}.md` via `mcp__obsidian__create_note` / template helper.
4. Verify the four prompt stubs exist at `spice/cowork/prompts/*.md`. (Installer materializes these so a missing file is a Notice + diagnostic in the return payload, not a hard fail.)
5. Return JSON receipt: `{ created: [...], existed: [...], missing_prompts: [...], date_context }`.

Idempotent: re-running creates nothing new if everything is present. Skill body is ~80 lines.

### `cowork:bootstrap-vault` composition update

The existing bootstrap-vault orchestrator gains one new step at the end of its SKILL.md body:

```
20. Use Skill `cowork:scaffold-timeframes` with `{}`. Capture the receipt; include in the final bootstrap report.
```

(Step number is illustrative — the actual placement is "after engagement bootstrap completes, before final audit-receipt embed.")

Existing engagement-bootstrap flow unchanged. The composition means the user's first run of `cowork:bootstrap-vault` also scaffolds the timeframes — they don't need to remember to run two skills.

The prompts/ directory's path convention is already locked in. The scheduling runner reads the file body at runtime; user edits the file, next run picks it up — no redeploy.
