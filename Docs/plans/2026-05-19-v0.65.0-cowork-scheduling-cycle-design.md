## Why now

Three things converged to unblock this cycle:

1. **v0.42.0 deferred a "scheduling cycle"** to wire runtime invocation against the 4 prompt-stub placeholders at `spice/cowork/prompts/{morning-briefing,eod-review,weekly-review,monthly-review}.md`. The path convention has been locked since then; bodies stayed empty.
2. **v0.45.0 closed cowork's self-contained refactor** (the path convention `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD.md` landed) but flagged FLN-3: orchestrator SKILL.md bodies still wrote to the old `spice/daily/` callout-patching surface. The atomic-note write contract here is what closes that gap.
3. **v0.62.0 shipped the ActivityFeed mechanism** (cross-blueprint reverse-query by canonical `type:` + `created_at:`). Without it, surfacing run-notes on Daily/Weekly/Monthly Hubs would need bespoke per-hub Dataview queries; with it, the wiring is a `_DEFAULT_BLUEPRINTS` widening.

The cycle is the **structure** that lets a consumer connect Claude Cowork to their sauce-installed vault, invoke any `cowork:*` orchestrator, and have the output land in a deterministic place that the rest of the platform already surfaces.

## Out of scope (deliberate)

- **Filling the 4 user-editable prompt bodies** at `spice/cowork/prompts/*.md`. They stay empty / user-owned. Orchestrators emit a no-op note with `warning: empty_prompt` when run against an empty stub — lets the user *see* the wiring works before customizing.
- **Scheduler-config code outside Claude Cowork.** No launchd/cron/GitHub-Actions templates ship. For Claude Cowork specifically, the new `cowork:onboard-scheduled-jobs` skill (Section 2.7) registers jobs programmatically via the `scheduled-tasks` MCP. For other schedulers (launchd/cron/etc.) the onboarding doc has config snippets, but registration stays consumer-owned.
- **Per-engagement aspect routing inside orchestrators.** The v0.31.0 engagement-model design contemplated branching on `engagement.type` (`personal` / `w2-fte` / `consulting`) inside each orchestrator. Deferred to a future v0.31-completion cycle — this cycle keeps orchestrators engagement-aware only via `engagement_id` resolution + frontmatter emission.
- **Sprint-sync (or any other consumer-specific job).** The 5th scheduled job in the user's current accuris setup (ADO sprint-sync) stays consumer-local. Sauce ships the **consumer-extension pattern** (worked example: copy `write-run-note-finance`) but not the sprint-sync skill itself.
- **Replacing the existing per-hub Cards panels with ActivityFeed.** Daily Hub keeps `CoworkDailyHubCards` etc. ActivityFeed is additive (existing pattern from v0.62.0).

## Section 1 — Path & frontmatter contract (the sauce-shape)

### Per-timeframe atomic-note paths

Each scheduled job writes to a deterministic path. Path is a pure function of `(orchestrator, day | week | month)`:

| Orchestrator | Path |
|---|---|
| `cowork:morning-briefing` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/morning-briefing.md` |
| `cowork:midday-tripwire` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/midday-tripwire.md` |
| `cowork:eod-review` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/eod-review.md` |
| `cowork:weekly-review` | `spice/cowork/weekly/YYYY/YYYY-Www/weekly-review.md` |
| `cowork:monthly-review` | `spice/cowork/monthly/YYYY/YYYY-MM/monthly-review.md` |
| Consumer-extension (e.g. `cowork:sprint-sync`) | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/<slug>.md` (same shape) |

The per-day folder (`spice/cowork/daily/.../YYYY-MM-DD/`) is the same folder the existing v0.45.0 cowork Daily Note co-locates with. This cycle just declares that folder as a contract and adds sibling run-notes to it.

### Frontmatter shape (canonical-vocab-aligned per FA-5)

```yaml
type: cowork-morning-briefing        # or cowork-eod-review / cowork-midday-tripwire / cowork-weekly-review / cowork-monthly-review / cowork-finance-snapshot
created_at: "2026-05-19T07:05:14-06:00"   # ISO+TZ at write time
engagement_id: "accuris"             # resolved from vault-config.md engagements[]
day: "2026-05-19"                    # daily / midday / eod / finance only
week: "2026-W20"                     # weekly-review only
month: "2026-05"                     # monthly-review only
generator: "cowork:morning-briefing@1.0.0"   # orchestrator + version (for forensic / FLN tracking)
prompt_source: "spice/cowork/prompts/morning-briefing.md"   # which prompt file body was read
```

Additional required field for `cowork-midday-tripwire`:
```yaml
severity: "yellow" | "red"
```
(orchestrator skips writing when severity would be `green` — no note = green; presence of a tripwire note = something to flag.)

### Body shape (loose)

Whatever the prompt body instructs Claude to emit — Markdown freeform. No required headings. ActivityFeed cards key off frontmatter + first H1/H2, not body structure. Consumer is free to define structure inside their prompt bodies.

### Idempotency

Overwrite (last write wins). Re-running `cowork:morning-briefing` at 8:00 AM after the 7:05 AM run replaces the file. Path is deterministic per `(orchestrator, day)` so there's no race-with-self. Daylight-savings / TZ-boundary edges resolve via `cowork:date-context` (single source of truth) not local clock.

### Engagement scoping

Engagement lives in **frontmatter (`engagement_id:`), not in the path.** Path stays simple even as multi-engagement vaults arrive. ActivityFeed already supports filtering by frontmatter; surfaces that want engagement-aware grouping add a `groupBy: "engagement_id"` option in a later cycle (out of scope here).

### Six new canonical `type:` values

Registered in the FA-5 canonical-vocab catalogue:

| Type | Owner | Required fields |
|---|---|---|
| `cowork-morning-briefing` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source` |
| `cowork-midday-tripwire` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source`, `severity` |
| `cowork-eod-review` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source` |
| `cowork-finance-snapshot` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator` (prompt_source optional — finance is gather-driven) |
| `cowork-weekly-review` | cowork | `type`, `created_at`, `engagement_id`, `week`, `generator`, `prompt_source` |
| `cowork-monthly-review` | cowork | `type`, `created_at`, `engagement_id`, `month`, `generator`, `prompt_source` |

Each gets a rule_fragment validating the required fields + path-prefix expectation.

### What stays untouched

- The cowork Daily Note (`spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD.md`) and its `## Notes` section remain user-owned. Orchestrators **never write to it**.
- The cowork Weekly Note (`spice/cowork/weekly/YYYY/YYYY-Www.md`) and Monthly Note (`spice/cowork/monthly/YYYY/YYYY-MM.md`) shells remain user-owned.

## Section 2 — Components

### 2.1 Six `write-run-note-*` sub-skills (replaces 5 `write-callout-*` + 2 `write-summary-*`)

The legacy `write-callout-*` sub-skill family was built to patch callouts inside the daily-blueprint's daily note (FLN-3 from v0.45.0 flagged this as residual technical debt). This cycle replaces them with `write-run-note-*` siblings that emit the Section 1 contract:

| Old (deletes) | New (replaces) | Writes to | `type:` value |
|---|---|---|---|
| `write-callout-morning-briefing` | `write-run-note-morning-briefing` | `…/YYYY-MM-DD/morning-briefing.md` | `cowork-morning-briefing` |
| `write-callout-eod-review` | `write-run-note-eod-review` | `…/YYYY-MM-DD/eod-review.md` | `cowork-eod-review` |
| `write-callout-tripwire-yellow` + `write-callout-tripwire-red` | `write-run-note-midday-tripwire` (severity in frontmatter) | `…/YYYY-MM-DD/midday-tripwire.md` | `cowork-midday-tripwire` |
| `write-callout-finance` | `write-run-note-finance` | `…/YYYY-MM-DD/finance.md` | `cowork-finance-snapshot` |
| `write-summary-weekly` | `write-run-note-weekly-review` | `…/YYYY-Www/weekly-review.md` | `cowork-weekly-review` |
| `write-summary-monthly` | `write-run-note-monthly-review` | `…/YYYY-MM/monthly-review.md` | `cowork-monthly-review` |

`write-run-note-finance` is the **reference implementation** the consumer-extensions doc points at. Its skill body is the most copy-pasteable starting point for a consumer-defined atomic-note writer (e.g. sprint-sync).

### 2.2 Six orchestrator SKILL.md rewires

The 6 existing orchestrators get their step lists updated to call the new sub-skill names:

- `cowork:bootstrap-vault` — calls `cowork:scaffold-timeframes` (unchanged) + emits stub finance snapshot on first run via `write-run-note-finance`
- `cowork:morning-briefing` — terminates in `write-run-note-morning-briefing` (was `write-callout-morning-briefing`)
- `cowork:midday-tripwire` — terminates in `write-run-note-midday-tripwire` (was branched yellow/red callout writes)
- `cowork:eod-review` — terminates in `write-run-note-eod-review` (was `write-callout-eod-review`)
- `cowork:weekly-review` — terminates in `write-run-note-weekly-review` (was `write-summary-weekly`)
- `cowork:monthly-review` — terminates in `write-run-note-monthly-review` (was `write-summary-monthly`)

No orchestrator step **count** changes (still ~5–7 steps each); only the final write-step's sub-skill reference + the read of `prompt_source` upstream of write.

### 2.3 ActivityFeed default-blueprint widening

`activity-feed@0.1.0 → 0.2.0` MINOR. The only code delta is `_DEFAULT_BLUEPRINTS` adding the 6 new `type:` values. Daily/Weekly/Monthly Hubs + Today.md (all of which call ActivityFeed with default blueprints) auto-pick up run-notes with zero hub-side edits.

### 2.4 SpaceDailyDashboard 3rd panel

`daily@0.4.0 → 0.5.0` MINOR. `space-daily-dashboard.js` grows a "Today's Cowork Activity" section under tasks + meetings. Implementation pattern (mirrors existing tasks + meetings panels):

```js
const getCoworkRuns = () => {
  const folder = `spice/cowork/daily/${moment(today).format("YYYY/MM-MMMM")}/${today}`;
  const pages = dv.pages(`"${folder}"`)
    .where(p => typeof p.type === "string" && p.type.startsWith("cowork-"))
    .sort(p => p.created_at, "asc");
  return pages.array();
};
```

Renders as compact bullet list — each item: `{HH:mm} <orchestrator label> → [[link]]`. Hidden when empty (matches the existing both-empty short-circuit pattern). Re-uses the same panel chrome (SVG icon + header + container styles).

### 2.5 `CoworkReadiness` CustomJS helper + Cowork.md embed

### 2.6 Six rule_fragments

One per new `type:` value, mirroring the existing `cowork-daily` / `cowork-weekly` / `cowork-monthly` fragment pattern. Each declares:
- `extends: _canonical-vocab`
- `required_frontmatter` (per Section 1 table)
- `path_pattern` regex matching the atomic-note location

### 2.7 `cowork:onboard-scheduled-jobs` interactive setup skill (the zero-friction door)

The `/cowork` readiness panel (Section 2.5) is a passive observer — it surfaces state but doesn't help the user *create* that state. Section 2.7 adds the active counterpart: a new orchestrator-tier skill that walks the consumer through scheduling-config setup interactively, asks the right questions, writes the prompt-body customizations the user dictates, and **registers the scheduled jobs directly via the `scheduled-tasks` MCP** when invoked from inside Claude Cowork.

**Where it lives:** `platform/blueprints/cowork/skills/orchestrators/onboard-scheduled-jobs/SKILL.md` — same tier as `cowork:bootstrap-vault` (orchestrator, not sub-skill).

**Lifecycle (the user experience inside Claude Cowork):**

1. User says "set up my scheduled jobs" (or invokes the skill directly).
2. Skill runs `cowork:check-vault-routing` — if engagement isn't bootstrapped, bails cleanly with "run cowork:bootstrap-vault first."
3. Skill reads `vault-config.md`, resolves engagement(s), loads each engagement's `default_cadences` map from the engagement-type manifest (personal/w2-fte/consulting).
4. Skill detects whether the `scheduled-tasks` MCP is reachable (mcp__scheduled-tasks__list probe). Two modes:
   - **direct-register mode** (preferred — Claude Cowork session): MCP available; skill calls `mcp__scheduled-tasks__create` / `update` / `list` to actually configure jobs.
   - **paste-mode fallback** (regular Claude Code session against the vault): MCP unavailable; skill outputs a paste-ready config block + writes `spice/cowork/scheduled-jobs.md` so the user has a vault-resident copy to paste from later.
5. Per orchestrator (5 of them: morning-briefing / midday-tripwire / eod-review / weekly-review / monthly-review), 3 fast questions:
   - **Enable?** (default: yes if engagement-type's `default_cadences` has it on)
   - **Cadence?** (default: Mon-Fri 07:05 / 12:30 / 17:05; Fri 04:00; 1st-of-month 04:00; in engagement's resolved TZ)
   - **Prompt body?** Three sub-options:
     - **(a) Sauce default** — skill writes a sauce-shipped default template into the prompt file.
     - **(b) Leave empty stub** — orchestrator runs will emit notes with `warning: empty_prompt` frontmatter; user customizes later.
     - **(c) Tell me what this should emit** — interactive sub-Q&A. User describes desired content in natural language; skill drafts a prompt body; user approves before write.
6. Once all 5 orchestrators are walked:
   - **direct-register mode:** for each enabled orchestrator, skill checks the existing `scheduled-tasks` task list (matched by task name like `cowork-morning-briefing` per engagement), updates if exists with new schedule/prompt, creates if missing. Returns a per-orchestrator summary (`created` / `updated` / `unchanged`).
   - **paste-mode:** skill prints a paste-ready config block formatted for Claude Cowork's Add-Job form.
7. Skill writes `spice/cowork/scheduled-jobs.md` with the canonical config (task name, cron expression, prompt invocation, scheduled-tasks-MCP task ID when known). This becomes the source-of-truth the readiness panel parses for "expected vs observed" comparison.
8. Skill emits Notice `cowork:onboard-scheduled-jobs complete — N jobs configured for <engagement.label>`.

**Re-runnable.** Calling the skill again diffs `spice/cowork/scheduled-jobs.md` (current config) against the MCP's reported tasks; surfaces additions/removals/changes; lets the user accept or override. Not a one-shot wizard.

**Multi-engagement.** Re-running with a different engagement walks that engagement's cadence map separately; tasks are scoped per `engagement_id` (task names include the engagement id).

**The new vault-resident config note** `spice/cowork/scheduled-jobs.md` carries frontmatter `type: cowork-scheduled-jobs` + a YAML body listing the configured jobs:

```yaml
---
type: cowork-scheduled-jobs
created_at: "2026-05-19T15:30:00-06:00"
updated_at: "2026-05-19T15:30:00-06:00"
engagement_id: "accuris"
mcp_backend: "scheduled-tasks"
---

# Scheduled jobs

| Orchestrator | Schedule (TZ) | Cron | Prompt invocation | Task ID |
|---|---|---|---|---|
| cowork:morning-briefing | Mon-Fri 07:05 America/Denver | `5 7 * * 1-5` | `Use skill cowork:morning-briefing with { engagement_id: "accuris" }` | sched-task-abc123 |
| cowork:eod-review | Mon-Fri 17:05 America/Denver | `5 17 * * 1-5` | `Use skill cowork:eod-review with { engagement_id: "accuris" }` | sched-task-def456 |
| cowork:weekly-review | Fri 04:00 America/Denver | `0 4 * * 5` | `Use skill cowork:weekly-review with { engagement_id: "accuris" }` | sched-task-ghi789 |
```

**CoworkReadiness panel grows a 5th row** (refining Section 2.5):

| Row | Source | Status renderings |
|---|---|---|
| 5 (NEW) | `spice/cowork/scheduled-jobs.md` parsed + cross-checked against ActivityFeed last-runs | "Expected jobs: 5 configured · 3 fired today · 0 gaps" or "⚠ 2 configured jobs haven't fired (last run > 1 day stale)" |

**New canonical type value:** `cowork-scheduled-jobs` joins the 6 from Section 1 → **7 new canonical types total**, **7 new rule_fragments**.

**Manifest deltas:** `platform/blueprints/cowork/manifest.json` adds the orchestrator to `claude_surface[]` (skill kind), adds `content/scheduled-jobs.md` template to `files[]`, adds the new rule_fragment, and bumps `depends_on` for any MCP capability declaration.

**Out of scope here:** the `scheduled-tasks` MCP's own setup. Consumers must have it available in their Claude Cowork environment before invoking this skill. The skill's paste-mode fallback covers consumers who don't have the MCP.

## Section 3 — Data flow (invocation lifecycle)

```
7:05 AM   Claude Cowork app fires scheduled invocation
          ──────────────────────────────────────────────
          Use skill cowork:morning-briefing { engagement_id: "accuris" }

orchestrator
   resolve engagement_id → vault-config.md engagements[]
   ├─ Step 1: cowork:check-vault-routing { required: [obsidian, gmail, ...] }
   │    └─ aborts cleanly if not ready (logs to spice/cowork/.routing-cache.json)
   ├─ Step 2: cowork:date-context (resolves day = 2026-05-19)
   ├─ Step 3: cowork:gather-* (calendar, gmail, projects, threads, weather, ...)
   ├─ Step 4: read prompt body
   │    └─ vault read of spice/cowork/prompts/morning-briefing.md
   └─ Step 5: cowork:write-run-note-morning-briefing
        ├─ compose frontmatter (Section 1 contract)
        ├─ render body from gather-step outputs + prompt instructions
        └─ mcp__obsidian write to
           spice/cowork/daily/2026/05-May/2026-05-19/morning-briefing.md

(later, on demand)
          ──────────────────────────────────────────────
user opens spice/cowork/Daily Hub.md
   └─ Recent Activity ActivityFeed (scope=today)
        └─ query: dv.pages().where(type IN cowork-* AND created_at IN today)
        └─ picks up morning-briefing run-note → grouped by hour

user opens spice/daily/.../Tuesday-2026-05-19.md
   └─ SpaceDailyDashboard
        ├─ Tasks panel (existing)
        ├─ Meetings panel (existing)
        └─ Cowork Activity panel (NEW) → bullet list, links to atomic notes

user opens spice/cowork/Cowork.md
   └─ CoworkReadiness panel
        ├─ Engagement ✓ accuris
        ├─ Prompts 4/4 present
        ├─ MCP routing ✓ ready
        └─ Last runs: morning-briefing 7:05 today, eod-review (never today), ...
```

### Error handling

- **`check-vault-routing` returns non-`ready`** → orchestrator records failure to `.routing-cache.json` and exits. **No partial run-note is written.** Surfaces in `/cowork` readiness panel.
- **Write step fails partway** → orchestrator retries the single MCP write call once, then bails. Frontmatter-only / empty-body notes are never produced.
- **Prompt body empty (stub state)** → orchestrator emits a no-op note with `body: "(prompt body empty — edit spice/cowork/prompts/morning-briefing.md)"` and a `warning: empty_prompt` frontmatter field. Lets the user *see* the wiring works before customizing.

### Idempotency / overlapping runs

- Re-run same day → deterministic path, overwrite.
- Two jobs at 7:05 AM and 7:08 AM → different file slugs, no contention.
- Daylight-savings / TZ edge → `day:` from `cowork:date-context`, not local clock.

### Read-side cadence

- Daily / Weekly / Monthly Hubs + Today.md auto-refresh via Dataview's reactive query (no manual reload).
- SpaceDailyDashboard cowork panel uses the same `dv.pages()` pattern as tasks/meetings — picks up new run-notes within a Dataview refresh cycle.
- `/cowork` readiness panel reads `.routing-cache.json` for routing status (cheap), live-queries ActivityFeed for last-run timestamps.

## Section 4 — Onboarding, consumer extensions, migration

### Onboarding ritual (the "structured manner")

4-step checklist for connecting Claude Cowork to a sauce-installed vault — surfaced both in `Docs/cowork-onboarding.md` AND mirrored live by the `/cowork` readiness panel. The interactive skill from Section 2.7 collapses what used to be three manual sub-steps (customize prompts, configure scheduler, register jobs) into one conversational pass.

| Step | Action | How readiness panel reflects it |
|---|---|---|
| 1 | Install sauce + subscribe to cowork blueprint | implicit (panel renders = blueprint installed) |
| 2 | Run `cowork:bootstrap-vault` to create `vault-config.md` + engagement | `Engagement: ✓ accuris` |
| 3 | **(Inside Claude Cowork)** Run `cowork:onboard-scheduled-jobs`. The skill asks per-orchestrator enable/cadence/prompt questions; writes prompt customizations to `spice/cowork/prompts/*.md`; registers jobs via the `scheduled-tasks` MCP; persists config to `spice/cowork/scheduled-jobs.md` | `Prompts: 5/5 present` + `Expected jobs: N configured` |
| 4 | First scheduled run lands → readiness panel shows per-orchestrator last-run timestamps | `morning-briefing: last run 7:05 today` |

The panel is idempotent — it's the same panel a user opens on day 90 to sanity-check that runs are still landing. Re-running `cowork:onboard-scheduled-jobs` is the same loop a user uses on day 90 to change a cadence or update a prompt body — not a one-shot wizard.

### Consumer-extension pattern (sprint-sync as worked example)

Documented in `Docs/cowork-consumer-extensions.md`. Steps for a consumer adding their own scheduled job (e.g. `sprint-sync` against ADO):

1. Define a new `type:` value (e.g. `cowork-sprint-sync`) in a consumer-local rule_fragment (`.local/` override per landmine #22).
2. Author a skill body at `.claude/skills.local/cowork-sprint-sync/SKILL.md`. Use `write-run-note-finance/SKILL.md` (shipped in sauce) as the copy-paste reference template.
3. The skill writes an atomic note at `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/sprint-sync.md` with the canonical frontmatter shape.
4. Register `cowork:sprint-sync` (or any consumer-namespaced ID) as a scheduled job in Claude Cowork app.
5. Optional: add `cowork-sprint-sync` to ActivityFeed via a per-vault config override (or accept the default fallback — ActivityFeed renders unknown types under a generic group).

The doc explicitly notes that `write-run-note-finance` ships in sauce as the canonical reference because finance is consumer-flexible (open-ended snapshot vs structured review).

### MCP setup recommendation

Current setup (`obsidian-mcp` + Claude Cowork project mounted at vault root) **stays**. Reasoning documented in the onboarding doc:

- `obsidian-mcp` understands Obsidian semantics (wikilinks, frontmatter, daily-notes folder); filesystem MCP would lose that.
- Mounting at vault root keeps Claude's working-directory matching the canonical `spice/cowork/...` paths the contract uses — no path translation layer needed.
- The platform's `cowork:check-vault-routing` already probes `obsidian` as a backend; no new MCP plumbing.

What the doc *does* document: how to register scheduled jobs in Claude Cowork app with the right `engagement_id` argument, and how to confirm the vault path is correct.
