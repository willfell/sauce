# v0.31.0 — cowork engagement-model refactor (design draft)

> [!abstract] Goal
> Replace cowork@0.1.0's binary `vault_scope: life | work | both | none-yet` schema with a richer `engagements: [{ id, type, ... }]` list, where each engagement declares its archetype (`personal | w2-fte | consulting | volunteer | ...`) and carries only the fields that archetype needs. Orchestrators become engagement-aware: one orchestrator dispatch per engagement per cadence, gated by the engagement's type and which MCPs it subscribes to. Bootstrap-vault becomes "how many engagements are in this vault, what type is each?" instead of the one-dimensional life/work binary. Same blueprint serves accuris-sauce (1× w2-fte), ero-sauce (1+ consulting), headspace-sauce (1× personal), and any future shape.

## 1. Problem statement

cowork@0.1.0 (shipped 2026-05-10) modeled vault-shape as a single axis:

| `vault_scope` | Orchestrators enabled | Hidden assumption |
|---|---|---|
| `life` | morning/midday/eod/weekly/monthly (×5 life) | One personal-finance + life shape |
| `work` | ero-morning/eod/weekly/monthly (×4 ero) | One consultant-style work engagement (hourly rate, AP email, invoice cadence, 25th-cycle billing) |
| `both` | All 9 | One life + one consulting work in same vault |
| `none-yet` | None | — |

Surfaced 2026-05-11 during accuris-tonight bootstrap-vault demo: the `work` scope's interview prompts (`ero_hourly_rate_usd`, `ero_ap_email`, `ero_invoice_cadence`, `ero_25th_billing_cycle`) don't fit a **W2-FTE** engagement like Accuris. The schema conflates "work engagement" with "consultant engagement". Accuris is W2-FTE; ero is consulting; headspace is personal-only. Same vault shape (`spice/cowork/...`) but three distinct engagement archetypes.

**Naming-level evidence of the leak:** `{{ero_owner_name}}`, `{{ero_role}}`, `{{ero_primary_client}}`, `{{ero_email}}`, `{{ero_hourly_rate_usd}}`, `{{ero_ap_email}}` are template vars in `spice/cowork/context/ero-context-templates/`. They're consultant-flavored. Substituting an FTE engagement into them produces `{{ero_hourly_rate_usd}}` = `null` and `{{ero_ap_email}}` = `n/a` — schema lies.

**Symptom 2:** Per-callout sub-skills are split `*-life` vs `*-ero` (`write-callout-morning-briefing-life`, `write-callout-morning-briefing-ero`, `write-summary-weekly-life`, `write-summary-weekly-ero`, etc.) — 11 of the 27 sub-skills carry the binary suffix. A third engagement type (FTE, or a volunteer board engagement, or a side-business) doubles the matrix.

## 2. Architecture (proposed)

### Engagement model

Each engagement is `{ id, type, label, ...type-specific fields }`. The vault's `spice/cowork/context/vault-config.md` carries the `engagements[]` list. Engagement IDs are lowercase-with-hyphens, vault-unique.

      "type": "w2-fte",
      "label": "Accuris (FTE)",
      "role": "Principal Software Engineer",
      "employer": "Accuris",
      "manager": "...",
      "stakeholders": ["Alex Palacios", "Stefan de Pagter", ...],
      "gmail_label": "Work",          // optional MCP scoping
      "calendar_id": "willfellhoelter@gmail.com",
      "brex_enabled": false,          // FTE: not finance-tracked here

      "id": "ero-acme",
      "type": "consulting",

### Engagement-type taxonomy (initial)

| Type | Required fields | Optional fields | Cadences supported |
|---|---|---|---|
| `personal` | `owner_name`, `home_city` | `cc_*`, `debt_*`, `inner_circle_people`, `wellness_prompts` | morning, midday, eod, weekly, monthly |
| `w2-fte` | `role`, `employer`, `manager`, `stakeholders` | `gmail_label`, `calendar_id` | morning, eod, weekly |
| `consulting` | `role`, `primary_client`, `hourly_rate_usd`, `ap_email`, `invoice_cadence` | `billing_anchor_day`, `secondary_clients`, `brex_enabled` | morning, eod, weekly, monthly (invoice cycle) |
| `volunteer` (forecast) | `org_name`, `role`, `stakeholders` | `meeting_cadence` | weekly (light-touch) |
| `side-business` (forecast) | `business_name`, `role`, `revenue_source` | `brex_enabled`, `tax_jurisdiction` | weekly, monthly |

Types are pluggable: a future engagement archetype adds (a) a schema entry in `platform/blueprints/cowork/engagement-types/<type>.json` describing required/optional fields + supported cadences, (b) one or more per-type write-callout sub-skills (or a generic dispatch path — see §3.3). Existing engagement records keep working when new types arrive.

### Orchestrator dispatch model

One **cadence × engagement** pair = one orchestrator invocation. The cron layer fires once per engagement+cadence:

```
Cron tick 7:03 AM ──▶ for each engagement E in vault where E.cadences.morning == true:
                       agent: "Use Skill cowork:morning-briefing with { engagement_id: E.id }"
```

Implementation choice (decide in §6 Q3): a single `cowork:morning-briefing` orchestrator that branches on `engagement.type`, OR per-type orchestrators (`cowork:morning-briefing@personal`, `cowork:morning-briefing@w2-fte`, etc.). Recommendation: **single orchestrator + per-type write-callout sub-skills** — keeps orchestrators thin, isolates type-specific formatting into the FAT sub-skill (matches v0.30.0 contract 3).

### 3.3 Per-engagement-type manifest (NEW)

Each `engagement-types/<type>.json` declares the type's contract:

```json5
{
  "id": "w2-fte",
  "label": "W-2 full-time employment",
  "required_fields": [
    { "id": "role",         "type": "string", "prompt": "Job title (e.g., \"Principal Software Engineer\")" },
    { "id": "employer",     "type": "string", "prompt": "Employer name (e.g., \"Accuris\")" },
    { "id": "stakeholders", "type": "string[]", "prompt": "Comma-separated names of people you regularly coordinate with" }
  ],
  "optional_fields": [
    { "id": "manager",      "type": "string" },
    { "id": "gmail_label",  "type": "string" },
    { "id": "calendar_id",  "type": "string" }
  ],
  "supported_cadences": ["morning", "eod", "weekly"],
  "default_cadences": { "morning": true, "eod": true, "weekly": true, "midday": false, "monthly": false },
  "context_templates_dir": "engagement-templates/w2-fte",
  "render_aspects": {
    "finance_block":     "skip",         // FTE doesn't surface daily-finance
    "invoice_prep":      "skip",         // no invoicing
    "ai_committee":      "include",      // FTE may have committee meetings
    "kanban_projects":   "include"
  }
}
```

`render_aspects` is a per-type knob set that orchestrator write-callout sub-skills consult to decide which sections to compose. Adding a new engagement type is now schema-only: drop a `.json` + a `context-templates/<type>/` dir + (optionally) extend write-callout sub-skills with a new type branch.

### 3.5 Bootstrap-vault rewrite (the user-visible refactor)

New interview shape, replacing v0.30.0 step 9-12:

1. **Step 9 — engagement count:** `How many engagements does this vault host?` (1, 2, 3, "I'll add one for now and more later")
2. **Step 10 — per engagement, repeat:**
   - `Engagement ID (lowercase-hyphens, e.g., "accuris", "ero-acme", "personal")`
   - `Engagement type` — pick from registered types (`personal`, `w2-fte`, `consulting`, ...)
   - For each `required_field` in the type's manifest: ask one at a time
   - For each `optional_field`: ask with blank-allowed (default applies)
   - For each MCP backend marked `connected` in mcp_map: ask the engagement-scoped MCP question (which gmail label / calendar / brex-on-off / etc.) — but ONLY if the engagement type's `render_aspects` references that backend
   - Cadence overrides: present `default_cadences`, let user toggle
3. **Step 11 — recommendation table:** for each engagement, for each enabled cadence, list the cron job needed
4. **Step 12 — drop mode:** unchanged (`report-only` default)

The "ero" terminology vanishes from the interview entirely. Acuris is `type: w2-fte`. The consulting LLC in ero-sauce is `type: consulting` (with the actual primary_client + hourly_rate + ap_email + invoice_cadence captured — those become consulting-type-required, not work-type-required).

### 3.6 Cron orchestrator update (out-of-vault, manual)

Cron SKILL.md body shape changes — adds an explicit `engagement_id` argument:

```markdown
# Cron orchestrator — accuris-fte-morning

Invoke skill `cowork:morning-briefing` in the accuris-sauce Obsidian vault with `{ engagement_id: "accuris" }`.
```

Per-engagement-per-cadence cron jobs. A vault with 2 engagements × 4 cadences = up to 8 cron jobs (not 9 vault-wide). Bootstrap-report Section 3 emits the exact body for each enabled (engagement, cadence) pair.

## 3-bis. Always-first mechanic (Topic 1)

cowork:bootstrap-vault is THE canonical entry point for any Sauce-shape vault. Enforced at three sites (defense in depth):

1. **Consumer `CLAUDE.md` top-line directive** — installer's CLAUDE.md generator (or hand-edit during onboarding) adds a top-of-file directive: `> First-time setup: invoke cowork:bootstrap-vault before any other cowork operation.` Catches cold Claude sessions starting in the consumer vault.
2. **`Cowork.md` hub `[!warning]` callout** — self-suppressing: hub renders the warning only when `<vault>/spice/cowork/context/vault-config.md` is absent OR contains `engagements: []`. Once bootstrap completes, the callout disappears (dataviewjs condition). Catches Obsidian-only users browsing the vault.
3. **Installer post-install Notice** — `sauce install cowork` (or `sauce update` first-time landing cowork) emits a Notice: `cowork installed but not bootstrapped — invoke "Use Skill cowork:bootstrap-vault" before relying on any orchestrator.` Catches the fresh-install moment.

**Bootstrapped-state detection + refusal:** every cowork orchestrator pre-flights via `cowork:check-vault-routing` with a NEW `bootstrapped: true` required-state assertion. If `vault-config.md` is absent OR `engagements[]` is empty/missing, the orchestrator emits Notice `cowork:<orchestrator-id> aborted — vault not bootstrapped; run "Use Skill cowork:bootstrap-vault"` and exits cleanly. Cron jobs fail loud and visible.

**Re-bootstrap semantics:** additive-merge by default — new engagements append, existing engagement fields update only as user-flagged, engagement removals require explicit confirmation prompt. Templates re-materialize idempotently (read-only template-source copies overwrite); user-managed per-engagement files (`spice/cowork/context/<id>/about.md` after first bootstrap edited the contents) are NEVER clobbered.

## 3-ter. Dynamic blueprint introspection via `bootstrap_contributions[]` (Topic 2)

Each blueprint declares a new optional `bootstrap_contributions[]` array in its `manifest.json` (sibling to `rule_fragments[]`). Bootstrap-vault reads `<vault>/ranch/platform-installed.json` to enumerate installed blueprints, then reads each blueprint's manifest to compute the interview + materialization set.

### Schema

```json5
{
  "name": "<blueprint-name>",
  "version": "...",
  "rule_fragments": [ /* existing */ ],
  "bootstrap_contributions": [
    {
      "kind": "engagement_field_offer",
      "engagement_field_id": "cc_active_cards",
      "type": "string[]",
      "prompt": "Cards you actively spend from (comma-separated)",
      "consumed_by_types": ["personal"]
    },
    {
      "kind": "context_file_offer",
      "path_template": "spice/cowork/context/{{engagement_id}}/finance-guide.md",
      "source_template": "engagement-templates/personal/finance-guide.md",
      "consumed_by_types": ["personal"]
    },
    {
      "kind": "vault_question",
      "id": "primary_calendar_id",
      "prompt": "Which calendar is the primary vault-wide calendar?",
      "vault_scoped": true
    }
  ]
}
```

Three kinds:

- **`engagement_field_offer`** — declares a field the blueprint can consume IF an engagement of a matching type opts in. `consumed_by_types[]` is the gate. Composes with engagement-type manifests (see §3-quater).
- **`context_file_offer`** — declares a per-engagement context file to materialize, with template source + path template. `{{engagement_id}}` interpolation. Same `consumed_by_types[]` gate.
- **`vault_question`** — vault-scoped (not per-engagement) interview question that this blueprint needs answered. Rare; used for cross-engagement defaults.

### Composition rule

For each engagement E, the interview question set =
- `(E.type).required_fields[]` + `(E.type).optional_fields[]` (per engagement-type manifest, §3-quater)
- ⋃ over installed blueprints B: `B.bootstrap_contributions[].engagement_field_offer` where `E.type ∈ consumed_by_types`
- ⋃ over installed blueprints B: `B.bootstrap_contributions[].context_file_offer` where `E.type ∈ consumed_by_types` (this is materialization, not interview)
- ⋃ over installed blueprints B: `B.bootstrap_contributions[].vault_question` (asked once per vault, not per engagement)

Cadence is OWNED by cowork + engagement-type manifests — blueprints do NOT declare cadences. Daily blueprint provides the `<!-- COWORK_CALLOUTS -->` anchor (existing v0.2.5 contract) but does not own the "morning briefing" cadence; cowork's orchestrators do.

**No-op signal:** absence of `bootstrap_contributions` field (or `bootstrap_contributions: []`) = passive blueprint. Styling, customjs-guard, validator, audit, nav-buttons, cards, accent-button, people-rendering, convenience are all passive by default. Boards, finance, journal, meetings, people, project, to-do, trips, daily get bootstrap_contributions[] populated in v0.31.0 S2 (or stay [] if they have nothing engagement-aware to offer — TBD per blueprint in S2 planning).

## 3-quater. Engagement-type registry composition + extensibility (Topic 3)

### Composition lives in the engagement-type manifest

Each `platform/blueprints/cowork/engagement-types/<type>.json` declares:

```json5
{
  "id": "personal",
  "version": "0.1.0",
  "schema_version": 1,
  "label": "Personal life",
  "consumes": ["finance", "to-do", "daily", "people", "journal"],
  "required_fields": [
    { "id": "owner_name", "type": "string", "prompt": "Your name" },
    { "id": "home_city",  "type": "string", "prompt": "City (used by gather-weather)" }
  ],
  "optional_fields": [
    /* type-specific options + slots the blueprints fill via engagement_field_offer */
  ],
  "supported_cadences": ["morning", "midday", "eod", "weekly", "monthly"],
  "default_cadences": { "morning": true, "midday": true, "eod": true, "weekly": true, "monthly": true },
  "render_aspects": {
    "finance_block": "include",
    "invoice_prep": "skip",
    "ai_committee": "skip",
    "kanban_projects": "include",
    "inner_circle_imessage": "include"
  }
}
```

The `consumes[]` array is the type → blueprint composition rule. Adding a new engagement type is purely additive: drop `<type>.json` + (optionally) extend write-callout sub-skills with a new type branch. Existing engagement records keep working.

### Consumer-vault user-defined types

Supported in v0.31.0. Loader path:

1. Read `platform/blueprints/cowork/engagement-types/*.json` (platform-shipped).
2. Read `<vault>/spice/cowork/engagement-types/*.json` (consumer-defined, if present).
3. Merge by `id` — vault entries WIN on conflict (lets a vault override platform defaults).

Consumer-defined types pass the same schema validator as platform-shipped types. Engagement-types dir is a new allowlist entry in landmine #12-equivalent posture? No — `spice/cowork/engagement-types/` is under the `spice/` namespace which is already sanctioned (landmine #11 + the explicit `spice/` carve-out in CLAUDE.md). No new allowlist expansion needed.

### Per-type versioning

Semver-flavored:

- **PATCH** — doc-only edits, no schema change.
- **MINOR** — additive `optional_fields[]` or new `render_aspects[]` keys. Backwards-compatible.
- **MAJOR** — `required_fields[]` additions/renames/removals, breaking `consumes[]` changes, breaking `supported_cadences` changes.

`schema_version` integer is bumped on MAJOR only. Installer reads each engagement's `type_schema_version` (NEW field on engagement records in vault-config.md) and compares to the current type's `schema_version`. Mismatch → installer prompts "engagement-type 'consulting' schema version changed from 1 → 2; re-bootstrap engagement '<id>' now? (yes/no)". Yes triggers re-bootstrap flow scoped to that one engagement. No leaves engagement record as-is with a `LEGACY` flag in vault-config.md until next attempt.

## 3-septies. Rule-fragment posture + NEW `items_schema` predicate (Topic 6)

### vault-config.md required-frontmatter

```json5
{
  "scope": { "path_glob": "spice/cowork/context/vault-config.md" },
  "required_frontmatter": {
    "type":        { "required": true, "type": "string", "equals": "cowork-vault-config" },
    "updated":     { "required": true, "type": "string", "matches": "^\\d{4}-\\d{2}-\\d{2}$" },
    "updated_by":  { "required": true, "type": "string" },
    "engagements": { "required": true, "type": "list", "min_length": 1, "items_schema": { /* see below */ } }
  }
}
```

Two NEW predicates introduced for v0.31.0 (additive; existing rule_fragments unchanged):

- **`min_length`** — integer; list-type fields must have at least N entries. Small, general.
- **`items_schema`** — discriminated-union per-item validator (see below).

### `items_schema` discriminator + per-type required-fields

```json5
{
  "items_schema": {
    "discriminator": "type",
    "by_type_source": "engagement-types/<type>.json#required_fields",
    "common_required": {
      "id":   { "type": "string", "matches": "^[a-z][a-z0-9-]+$" },
      "type": { "type": "string" }
    }
  }
}
```

- `discriminator` names the field on each item whose value selects the per-item schema (here, `engagement.type`).
- `by_type_source` is a NEW resolver: validator reads the engagement-type manifest at audit time and treats that manifest's `required_fields[]` as the per-type required field set. Single source of truth — avoids duplicating per-type schemas in the rule_fragment AND the engagement-type manifest.
- `common_required` applies to every item regardless of discriminator value (id + type validation).

This is a SUBSTANTIAL additive predicate. Implementation lives in `platform/mechanisms/validator/` + audit's rule-runner. Generic enough to reuse for any future discriminated-union schema.

### Per-engagement context dir presence — soft-audit

v0.31.0 ships a soft-audit (warn-only). Audit walker reads `vault-config.md`, enumerates `engagements[].id`, then checks for `spice/cowork/context/<id>/` directory presence. Missing dirs emit warnings, not errors. Hard-audit via declarative `iterate_over` rule_fragment deferred to v0.32+.

### validator@0.1.x bump

Predicate additions are MINOR per validator versioning convention. `validator@0.1.1 → 0.1.2` in v0.31.0.
