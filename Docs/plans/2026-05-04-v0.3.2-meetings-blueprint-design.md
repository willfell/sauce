# Design — `meetings` blueprint, hub-centric (v0.3.2)

> [!warning] Mid-cycle redesign
> The original v0.3.2 design (one nav-button click → meeting note with title + attendees prompts) shipped to T1.6 smokes successfully but was redesigned at the user's request to mirror accuris's full hub pattern. This is the v2 design. The v1 implementation files (manifest, Meeting.md template) will be revised in place; new files will be added (Meeting Hub.md, two CustomJS classes); daily blueprint will bump 0.1.0 → 0.1.1 for the meetingsPath update. Cycle still ships under tag `v0.8.1`.

> [!abstract] Goal
> Ship the `meetings` blueprint with the accuris hub-centric pattern: one **Meetings Hub** note per day at `beacon/meetings/hubs/YYYY-MM-DD-Meetings.md` aggregating all meetings for that day, with an inline "New Meeting" button on the hub that creates individual meeting notes at `beacon/meetings/notes/YYYY/MM-MMMM/YYYY-MM-DD <title>.md`. The hub renders today's meetings as interactive cards via a `MeetingsHubCards` CustomJS class. Lights up SpaceDailyDashboard's "Today's Meetings" panel via a small daily blueprint update (`meetingsPath` from `beacon/meetings` → `beacon/meetings/notes`).

> [!info] Why redesigned
> v1 collapsed the hub layer — one click made one meeting note. The user surfaced that accuris's pattern is fundamentally a hub-then-meeting two-step: click "Meetings" → land on the day's hub → click "New Meeting" inline → individual meeting created. The hub aggregates all of today's meetings as cards (with status badges for attendees/notes/open-tasks/done-tasks/summary) and provides a single navigation surface for the day's meeting context. Mirroring this pattern is what the user wanted from "settle these underlying critical decisions while we ship out these first modules" — better to land it correctly now than refactor later.

## Architecture

### Click-time flow (v2)

```
User clicks "Meetings" nav-button
  ↓
runTemplaterTemplate dispatches:
  folder   = "beacon/meetings/hubs" (literal; no date routing in folder)
  filename = "2026-05-04-Meetings"
  target   = "beacon/meetings/hubs/2026-05-04-Meetings.md"
  ↓
Existence check: file exists today? → openLinkText (idempotent re-click)
  ↓
First click of the day → templater.create_new_note_from_template(MeetingHubTemplate, folder, filename, true)
  ↓
Hub note opens; renders three dataviewjs blocks:
  → SpaceNavButtons       (existing, unchanged)
  → NewMeetingButton       (NEW; renders inline button)
  → MeetingsHubCards       (NEW; renders today's meetings as cards)
  ↓
User clicks "New Meeting" inline button
  → modal opens, prompts for title
  → user types "Standup", presses Create
  → button computes date-routed folder: moment("2026-05-04").format("beacon/meetings/notes/YYYY/MM-MMMM")
                                       = "beacon/meetings/notes/2026/05-May"
  → button computes filename: "2026-05-04 Standup"
  → templater.create_new_note_from_template(MeetingTemplate, folder, filename, true)
  ↓
Meeting note created at beacon/meetings/notes/2026/05-May/2026-05-04 Standup.md
  → Templater fires Meeting.md template body inline
  → attendees-prompt fires; user types "Alice, Bob"
  → frontmatter populated; person/* tags generated; sections rendered
  → meeting note opens in active editor
  ↓
User returns to hub note → MeetingsHubCards re-renders; new meeting appears as a card
User goes to today's daily note → SpaceDailyDashboard.getMeetings (path beacon/meetings/notes) finds the meeting → "Today's Meetings" panel populates
```

## Trade-offs vs alternatives explicitly considered (v2)

> [!example]- Why these alternatives were rejected
>
> **A) Two separate blueprints (meetings + meeting-hubs).** Cleaner separation; daily blueprint untouched. Rejected: doubles the manifest plumbing for marginal architectural gain; landmine #11 prefers ONE module_directory per coherent feature surface, and meetings + hubs are tightly coupled (hubs only useful in conjunction with meetings).
>
> **B) Filter at dashboard level (no path change).** Keep all meetings + hubs flat under `beacon/meetings/`; update space-daily-dashboard.js to filter out `*-Meetings.md` files. Rejected: couples the dashboard filter to a naming convention; the subfolder structure is more semantically clean and discoverable.
>
> **C) Keep v1's one-click-makes-meeting flow.** Simpler. Rejected: user explicitly chose hub pattern after v1 smokes; hub aggregation is the right primitive for cross-meeting context, especially as days accumulate multiple meetings.
>
> **D) Nav-button click PROMPTS for title (legacy v1 flow), then creates meeting AND a hub silently.** One click does double-duty. Rejected: too magical; hub pattern is more discoverable + provides a navigation surface separate from meeting creation.
>
> **E) Workshop bump 0.8.0 → 0.9.0 MINOR for the redesign.** Signals substantial scope expansion. Rejected: per-blueprint changes are still additive (meetings new at 0.1.0; daily 0.1.0 → 0.1.1; no schema, no installer, no mechanism bumps). 0.8.1 patch is semantically correct.
