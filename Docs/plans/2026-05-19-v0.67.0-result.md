# v0.67.0 result — Activity Drill-In + Polish

## What shipped (final state at v0.67.0)

**User-visible:**
- **Luxon time fix (FLN-v66-7 CLOSED):** `_renderActivityMeta` now duck-types Luxon DateTime via `typeof tsRaw.toFormat === "function"` before falling back to `window.moment`. Resolves the every-card 12:00 AM regression on the Activity panel where Dataview parsed ISO-frontmatter `created_at` into Luxon objects that `window.moment(luxonObj)` couldn't coerce.
- **Inline drill-in for project/trip rollup cards:** clicking the `N notes touched` breadcrumb expands a muted indented list of the rolled-up children directly under the card. Children sorted by `file.mtime` DESC, capped at 12 visible + `+N more` (non-interactive label). Each row links to the child file via `app.workspace.openLinkText`. Chevron rotates 90° on expand; state is ephemeral (resets on re-render).
- **Universal open-todo badge:** any Meeting or Activity card whose note contains unchecked `- [ ]` tasks renders a small `☐ N` pill in its meta line. Tasks section excluded (circular — Tasks IS the open-task surface). Tooltip says `N open task(s)`.
- **Activity icon swap:** Lucide `zap` lightning bolt → Lucide `activity` heartbeat line. More semantically accurate; matches Lucide's literal "Activity" icon.
- **Activity defaults to open** when content present. Existing `if (activityCount > 0)` gate handles the "non-empty" condition; flag flipped from `false` → `true`.
- **Segmented per-blueprint left accent on Activity section:** the section's left border becomes a vertical mini bar-chart sized proportionally to the per-blueprint counts inside. CSS custom property `--sauce-accent-segments` drives a `linear-gradient(to bottom, ...)` via `::before` pseudo-element. Tasks + Meetings unchanged (single-color border-left).
- **"N children touched" → "N notes touched"** breadcrumb wording. More user-meaningful.

**Architecture:**
- `activity-feed@0.3.1 → 0.3.2` PATCH — synthetic rollup page (and the decorated-existing-survivor branch) now carries `_rollUpChildrenPages: children`. Additive; existing callers ignore the new field.
- `daily@0.7.1 → 0.8.0` MINOR — `_renderActivityMeta` rewritten with 4-param signature `(p, parentEl, squareIcon, chevronSvg)`. Four new helpers: `_formatTime`, `_renderTodoBadge`, `_renderDrillInList`, `_buildAccentSegments`. `_renderSection` accepts optional `accentSegments`. `_getActivityCount` returns `{total, byBlueprint}`. CSS additions (~70 lines) in `sauce-daily-dashboard.css` for drill-in + todo pill + segmented accent + mobile pass extension.
- `cards@0.2.6` untouched (Approach A — drill-in lives inside the existing `meta` function-form slot, no `afterBody` hook needed yet).
- Workshop `0.66.2 → 0.67.0` MINOR.

## NEW FLNs surfaced during this cycle

- **FLN-v67-1 (Minor — test quality):** AF-V067-RUC-4 relies on a fixture's `created_at: "2026-05-19T08:00:00Z"` falling within the asOf window to trigger the decorated-existing-survivor branch. The branch identity is comment-only; a future maintainer who changes the timestamp could silently flip RUC-4 to test the synthetic branch instead. Add a pre-condition assert `// root IS in windowed set` for self-documentation.
- **FLN-v67-2 (Minor — observation):** `_renderTodoBadge`'s initial `Array.isArray(tasks)` guard would have silently no-op'd on every real Dataview page (`p.file.tasks` is a DataArray Proxy, not a native array). Caught by S2 code-quality review; fixed in `353e082` to use `.where()` with `.filter()` fallback. Test harness mocks used plain arrays and wouldn't have caught this — codebase has prior precedent for DataArray-aware checks (existing `getTasks()` + `cowork-readiness.js`).
- **FLN-v67-3 (Minor — observation):** S6 implementer initially diagnosed 3 integration-smoke FAILs as "pre-existing." Verification against pre-cycle baseline (`d93a16c`) confirmed integration-smoke was GREEN there — the FAILs were actually caused by S5's manifest `depends_on activity-feed >=0.3.2` bump landing before the S7 catalogue bump. Mid-cycle inconsistency, resolved naturally by S7. The plan should have noted this expected failure mode for S5/S6 just as it did for the FA6 pin. **Plan-template improvement candidate:** when a blueprint's `depends_on` is bumped before the catalogue, document the temporary install-failure window in the predecessor stage's commit message.
- **FLN-v67-4 (Minor — UX nuance):** `byBlueprint` counts in `_getActivityCount` reflect RAW page counts (e.g., a project with 7 task edits counts as `project: 7`), but ActivityFeed's rollup coalesces these into ONE displayed card. The segmented accent visualization shows project as a larger segment than the card distribution would suggest. This is a feature (large-activity projects get visually emphasized), not a bug — but worth a code comment at the `_buildAccentSegments` call site to prevent future confusion during debugging.
- **FLN-v67-5 (Minor — cosmetic):** `.sauce-bread` selector is declared twice in `sauce-daily-dashboard.css` — once on line ~83 (pre-existing one-liner from v0.66.0) and once on line ~86 (new block from S4). Both work; CSS allows re-declaration. Worth a future polish PR to consolidate into a single block.
- **FLN-v67-6 (Minor — S4 hygiene):** S4 implementer used `git add -A` and swept in `.claude/scheduled_tasks.lock` (per-session scheduler state). Fixed in `0907c81` via untrack + .gitignore addition. Pattern improvement: implementer prompts should remind to stage specific files, not the whole tree.
- **FLN-v67-7 (Important — operational workflow gap):** Consumer vault subscription pins do not auto-bump when running `sauce update --force` against a newer workshop catalogue. Skip-with-notice is the installer's behavior on version mismatch (per `install.js:1190`). Consumer reinstalls required manual edits to each consumer's `ranch/platform-subscription.json` before `sauce update --force` would actually install the new versions. This is a known seam — `sauce wizard` can re-pick subscription pins, but a faster `sauce update --bump-pins` flag (or auto-bump when subscription `workshop_version` < workshop catalogue `workshop_version`) would streamline consumer reinstalls during cycle close-out. Track as candidate for a future cycle.

## Lessons

1. **Sub-task-vs-stage decomposition matters for SDD model selection.** S2 (4 pure helpers — well-scoped, single file) → sonnet was right. S3 (8 sub-edits — multi-file effects, integration touch-points) → sonnet was right. S4 (mechanical CSS append) → haiku was right. S5 (single-key JSON value bump) → haiku was right. S6 (multi-file test harness deltas with regex patterns) → sonnet was right. Model selection by signal `(file count × integration complexity)` worked well throughout.
2. **Plan template gap: temporary mid-cycle install-failure windows are predictable but undocumented.** S5's manifest depends_on bump preceding the catalogue bump (S7) is a known mid-cycle inconsistency. The plan documented the FA6 pin fail but missed predicting integration-smoke fails from the same cause. The S6 implementer reasonably assumed pre-existing failures based on the chain abort. **Fix for future plans:** when a manifest depends_on bump precedes the catalogue bump, add an explicit "expected failure modes" section to the predecessor stage's commit message.
3. **Code-quality review catches what unit tests can't.** S2's `Array.isArray(tasks)` guard would have passed all REND-V067-TODO-1 sub-asserts (which use plain-array mocks) AND shipped a silent no-op on every real Dataview page. The code-quality reviewer's catch on this saved a post-ship PATCH cycle. The lesson: when mocking, mocks should match the runtime type structure, not the test convenience structure. Future test pattern: when a CustomJS class consumes Dataview proxies, mock with proxy-like shapes where feasible.
4. **`git add -A` is a foot-gun in SDD prompts.** S4 implementer accidentally committed a workspace lock file. Forward-only cleanup is fast and clean (untrack + .gitignore in one commit), but better to prevent by reminding implementers to stage specific paths.
