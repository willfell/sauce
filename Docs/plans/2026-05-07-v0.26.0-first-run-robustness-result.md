## Per-P0 result

### P0-1 — wizard "Edit subscription" double-wrap fix ✅

**Root cause:** at `bootstrap.js:344` (and the mirror at `cmd-wizard.js:48`), `payload.mechanisms.map(name => find(x => x.name === name))` treated each entry as a bare string, but `payload.mechanisms` arrived from `runReRunWizard` as already-objects (via `_buildSubscriptionEntries`). The `find` consistently missed → emitted `{name: <object>, version: "0.0.0"}` nested-wrap entries.

**Fix shape:**
- NEW exported `_normalizeSubscriptionFile(subPath, selection, manifestEntries)` helper in `wizard.js` — atomic tmp+rename write of flat `{name: string, version: string}` entries; reads existing file to preserve top-level fields.
- NEW private `_coerceSubscriptionEntry(entry, manifestEntries)` defensively heals legacy double-wrapped on-disk state on read; returns null when name has no resolvable version (mirroring `_buildSubscriptionEntries` skip-unknown behavior).
- Production writeback rewired in BOTH `bootstrap.js` (re-run flow, lines 248-269) AND `cmd-wizard.js` (sauce wizard verb, lines 25-46). Dead `mergeSubscription` removed from bootstrap.js; dead `_mergeSubscription` removed from cmd-wizard.js.

**Test:** `caseC15WizardEditSubscriptionFlatWrite` in `run-cli.js` — pre-populates subscription with legacy double-wrapped entries on disk, calls `_normalizeSubscriptionFile`, asserts after-state has flat entries with no nested wrap. PASS post-S2.

## NEW lessons (v0.26.0)

> [!info] (a) — Scaffold helpers must include EVERY field downstream helpers read
> CF-1 root cause: design Section 3 specified 4 minimal fields for the Templater scaffold. Production install ran applyTemplaterHotkeys + applyTemplaterFolderTemplates against the scaffolded file; both helpers expect `enabled_templates_hotkeys` and `folder_templates` to be arrays — without the array fields, the helpers errored. **Generalization:** when a helper produces a state file that downstream helpers additively merge into, the scaffold MUST include EVERY field those helpers read (not just the minimum needed for the helper's own contract). Future scaffold helpers (next foundational plugin, or extension to existing ones) should grep downstream helper bodies for every field they consume. Pattern reusable for any future foundational scaffolding.

> [!info] (b) — Numbers-only inputs need explicit handling at every layer
> CF-2 root cause: `vault_identity_tag` defaulted to `path.basename(vaultPath).toLowerCase()`. For numeric directory names (`scratch/11`), this produced `"11"` — VALID YAML string, but INVALID as an Obsidian tag (numbers-only tags are rejected at render time with strikethrough). Combined with the daily template's UNQUOTED `{{vault_identity_tag}}` placeholder, the substituted result parsed as YAML int and triggered Type Mismatch. **Two layers of fix:** (1) template-body YAML quoting around the placeholder so substitution always produces a string; (2) input-derivation logic that prefixes numbers-only inputs with `vault-` to ensure tag-validity. Either alone wasn't enough. **Generalization:** when a user-supplied or path-derived value flows into multiple consumers (YAML frontmatter, Obsidian tag system, file paths), each consumer's validation rules must be checked at derivation time. The `_deriveVaultIdentityTag` helper is reusable for any future tag-from-display-name derivation.

> [!info] (c) — Acceptance smoke against a literally-named vault dir surfaces edge cases
> The user named the smoke dir `scratch/11/` (a numeric basename). Without that, CF-2 wouldn't have surfaced — every prior cycle's smokes used `scratch/2`, `scratch/3`, etc. (all numeric basenames!) but the v0.25.x daily template was the FIRST to hit the type-mismatch combination. The bug had been latent since the daily template was authored in v0.3.0. **Generalization:** future smoke-target naming should explicitly include adversarial inputs (numbers-only, special chars, very long names, unicode). Maintain a "smoke directory naming rotation" so different edge cases get exercised in sequence.

> [!info] (d) — Two-stage subagent review continues to be load-bearing
> S2 spec reviewer found ZERO issues (PASS first try). S2 quality reviewer found 1 CRITICAL (cmd-wizard.js test-hook seam — would have shipped a latent test-mode TypeError) + 3 IMPORTANT + 1 NIT, all applied pre-commit. S3 spec reviewer found 1 NIT (`ranch/Content`) bundled in-cycle. S3 quality reviewer APPROVED zero CRITICAL/IMPORTANT. **Reaffirms 6 data points** (v0.20.0 + v0.22.1 + v0.23.0 + v0.24.0 + v0.25.0 + v0.26.0): spec reviewer catches the "is what's done what was specified" gap; quality reviewer catches the "is what's done sound" gap. Skipping either ships latent bugs. The two-stage flow is cheap insurance.

> [!info] (e) — Pantry-clone source mutation is a separate latent bug (carry-forward)
> Investigation of CF-2 surfaced that the pantry/ clone's daily-template.md gets mutated DURING install — Templater placeholders + variable substitutions get written back into the clone's source-of-truth file. Workshop source remains intact. The mutation is self-healing on next `sauce update --force` (git pull resets HEAD). **Carry to v0.26.x or v0.27.0:** investigate why install.js writes back to the source template inside pantry/ rather than only to the consumer's `<vault>/ranch/templates/` destination. May be related to Templater's `overwrite_file_commands` API call sequence or a stale write-side-effect.

> [!info] (f) — In-cycle re-process bump rule (#16) didn't fire on this cycle
> CF-1 + CF-2 are both in the same cycle (v0.26.0); workshop_version stays at 0.25.0 throughout S2/S3/S4 until cycle close. The in-cycle re-process bump rule applies when an in-cycle CF lands at the same VERSION an earlier-in-cycle install already wrote `installed_at` for. Since v0.26.0 hasn't been "installed" anywhere yet (workshop_version unchanged through S1-S4), no `installed_at` short-circuit could fire on the CF commits. The rule continues to apply for cycles that bump version mid-stream — but doesn't apply when the version stays static across all CFs in one cycle.

> [!info] (g) — Reserved-headroom precedent extends to 13 data points (3rd MIXED case)
> Cycle classification: MIXED (1-2 CFs surfaced from real-world acceptance smoke, not from harness coverage). Precedent: clean cycles (v0.13.0, v0.14.0, v0.18.0, v0.18.1, v0.18.2, v0.20.0, v0.21.1) — pure additive / mechanical-rename / scoped-bug-fix; mixed cycles (v0.22.1, v0.23.0, v0.26.0) — review-surfaced or smoke-surfaced 1-3 CFs, all caught + fixed pre-close; first-of-its-kind cycles (v0.5.0, v0.11.0, v0.12.0, v0.6.0, v0.16.0, v0.17.0, v0.21.0, v0.22.0) — 4-6 CFs from extensive smoke iteration. v0.26.0 lands in the MIXED bucket (CF-1 from the deterministic install verdict; CF-2 from the manual-Obsidian smoke). **Pattern:** when scaffolding helpers integrate with existing additive-merge helpers, the smoke surfaces field-shape mismatches (CF-1) that pure unit tests miss; when user-supplied paths flow into YAML-tag consumers, smoke surfaces edge cases (CF-2) that template authors didn't predict.

> [!info] (h) — Cycle-time accuracy posture vs in-cycle housekeeping
> v0.26.0's `git add -A` in S3 picked up some untracked drift (workshop self-state .bak files + accent-button.js dogfood material). Per CLAUDE.md cycle-time-accuracy posture, dogfood drift should NOT bundle into atomic-rename commits (v0.23.0 lesson g precedent). v0.26.0 introduced `dea7c41` follow-up commit (.gitignore additions + `git rm --cached` of the 3 affected paths) to honor the posture. **Pattern:** explicit `git add <specific-files>` is preferred over `git add -A` when staging atomic commits; for renames, `git mv` followed by explicit `git add -u` (only-modified, not untracked) avoids the drift-pickup. Hindsight-drift removal as a separate forward-fix follow-up commit is the canonical recovery (no history rewrite required).
