## The 2-step consumer onboarding flow — locked

> [!success] What consumers do, end-to-end (post-v0.1.2)
> 1. **Bootstrap once** — clone the beacon workshop into a sibling path; drop the 20-line stub into the consumer vault's `Docs/Meta/Templater/platformInstall.js`; create `platform-config.json` + `platform-subscription.json`; install Slash Commander.
> 2. **Run installer + Cmd+R reload** — Templater `Insert _install-platform` runs the stub → stub `git pull`s the workshop → stub dispatches into `platform/install.js`. Cmd+R registers Templater hotkeys + SC bindings.
>
> v0.1.3 framed onboarding as 3 steps; v0.1.2 collapses it to effectively 2 because the bootstrap-copy × 3 resync ritual disappears entirely. The stub never needs to change as `install.js` evolves — `git pull` + `tp.user` dispatch decouples the bootstrap from the platform code. Cross-machine the model is `git pull` on each consumer + run installer; no hand-edited `installPlatform.js` copies to keep in sync.

## Surprises during execution

> [!warning] Surprise 1 (resolved): TDD ordering required blanking installed.json to demonstrate the failing assertion
> Workshop and barebones are both idempotent on repeat harness runs (0 new history entries). To exercise the TDD red light, T1.4 had to back up + blank `Docs/Meta/platform-installed.json` before the failing run, then again before each smoke that wanted to demonstrate field-capture on a non-empty history. Documented in T1.4 log. Future cycles touching `installed.history` semantics should expect the same constraint.

> [!warning] Surprise 2 (resolved): two unexpected "materialization-drift" diffs surfaced during S1
> `Docs/Meta/Scripts/nav-buttons/space-nav-buttons.js` and `Docs/Meta/rules/_global.json` showed working-tree changes after T1.X harness runs. Root cause: canonical platform sources (under `platform/mechanisms/...`) had evolved in commit `c3011bd` ("catch up v0.1.0 + v0.1.1 cycle work") but their materialized copies hadn't been refreshed. Today's harness runs caught them up. Decision (per AskUserQuestion at S1 close): leave uncommitted at S1; out of scope for v0.1.2 narrative. They remain as working-tree changes for the S2 commit to sweep up, OR a follow-up `chore: re-materialize stale platform outputs`, OR v0.2.0.

> [!warning] Surprise 3 (resolved, with deviation note): tmp-acc T2.4 exits 1, not 0 as the plan expected
> Two skip events fire because tmp-acc deliberately pins lagging versions (`nav-buttons@1.0.0` while workshop has 2.0.0; `project@0.1.0` while workshop has 0.2.1). Harness exit-code logic in `run-install.js:326-327` treats both `event:"error"` AND `event:"skip"` as exit-1 triggers. **Pre-existing tmp-acc-lag behavior**, not v0.1.2-induced — verified by temporarily restoring tmp-acc's PRE-stub bootstrap copy and re-running, which produced the same exit-1 with the same 2 skips. The v0.1.2-relevant test (does the stub dispatch + does gitState capture git fields on tmp-acc) passes — both skip events carry real `git_commit: "cee5afd5..."` shas.

> [!info] Confirmation: top-level catch records explicit nulls because `git` is const-scoped inside try
> The T1.3 wire-up subagent flagged this — if the top-level try block throws BEFORE workshopPath resolution, the catch records `null/null/null` rather than the captured `git.commit/tag/dirty`. Acceptable per landmine #14: gitState() is best-effort; null is "unknown," not a failure. The harness assertion accepts nulls in non-git scratch tmpdirs (Smoke C) without error.

## Pacing rhythm — what worked / what to improve

### What worked

- **Subagent dispatch on mechanical-but-error-prone tasks.** T1.1 (helper), T1.3 (51-site threading + 10 helper signatures), T2.6 (5-file doc edits), T2.7 (this writeup) all got dedicated subagents. Master context stayed light; spec + code-quality two-stage review caught the const-scoping nuance in T1.3 before it bit harness assertions.
- **TDD red→green discipline on T1.4.** Blanking `installed.json` to force a non-idempotent run gave a genuinely failing test (30 missing fields, exit 1) that then passed cleanly post-wire (10 entries with real shas, exit 0). The red-light evidence is preserved in the T1.4 log.
- **Master-driven mechanical tasks bypassed subagent overhead.** T1.0 (git tag), T1.2 (single insert), T1.5/T1.6/T1.7 (smokes via shell), T1.8 (cp + md5), T1.9 (harness invocation), T2.0 (backup), T2.1 (stub deploy), T2.2/T2.3/T2.4 (smokes) — all done directly. Subagent dispatch would have multiplied wall-clock for no quality gain.
- **Disk-locked state at every checkpoint.** md5 verifications across canonical + 3 bootstrap copies (v0.1.0 ritual) caught any drift immediately throughout S1; the stub-md5 verification at S2 close took its place going forward.
- **Pre-flight checks at session start caught the prophecy-hash discrepancy in the handoff doc** (top commit was `f7eaee8` not `c1c64b4`) without false-flagging — distinguishing "doc references its pre-commit hash" from a real state issue.

### What to improve

- **Plan T1.4.2 specified `installed.history` (full history) for the assertion.** Pre-cycle history entries (20 in workshop) lack git fields and would have failed the assertion forever. Scoped the assertion to `newHistory` (entries written this run) to match landmine #14's intent. The plan should be patched to reflect this — future cycles will inherit the same constraint.
- **Plan T2.4.1 expected exit 0 from tmp-acc smoke.** Actual exit is 1 (pre-existing skip behavior). Plan was overly optimistic — should have predicted the skip-event exit code given tmp-acc's deliberate lag posture. Carry forward as a writing-plans discipline note: when a plan touches tmp-acc, enumerate per-vault expected exit codes.
- **Master-driven smokes T1.5/T1.6 collapsed somewhat** because the working tree was already dirty mid-cycle (gitState() returned `dirty: true` for both). A strictly clean-state smoke would require stashing the in-flight cycle's own changes. Accepted the constraint and verified shape rather than absolute clean-vs-dirty discrimination.
