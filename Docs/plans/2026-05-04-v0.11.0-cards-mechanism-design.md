# v0.11.0 Design — `cards` mechanism + project visual cohesion

> [!abstract] Goal
> Introduce a shared `cards` mechanism (`BeaconCards` CustomJS class) so every blueprint that lists "related notes" renders the same mobile-aware card UI. Project blueprint becomes the dogfood consumer: `ProjectsHubCards` refactored to delegate, Project Structure renamed to Map and its Dataview tables converted to cards, and `ProjectWorkstreams` enhanced with per-workstream progress bars + counts. Set up styling cohesion before any v0.6.0 new blueprints land. Migration of `MeetingsHubCards` + `SpaceDailyDashboard` deferred to follow-up cycles after a fit audit.

---

## Why this cycle

User has been driving the project blueprint in barebones since v0.5.0 close. Pain points surfaced during use:

- Project Map (currently `Project Structure.md`) ships ugly Dataview tables for sub-page listings; visually inconsistent with `ProjectsHubCards`'s polished card UI on the Projects hub.
- Three card-style classes already coexist (`ProjectsHubCards`, `MeetingsHubCards`, `SpaceDailyDashboard` panels), each with its own implementation. Cohesion lives only in author discipline; drift will accelerate as more blueprints land.
- Workstream visualization on the Atlas only lists workstreams; doesn't show progress at a glance.
- The word "Structure" is dry; "Map" pairs naturally with "Atlas" (atlas = territory, map = navigation aid).

**Strategic framing:** establish a unified visual standard NOW, before v0.6.0+ blueprints inherit drift. The cards mechanism becomes load-bearing platform value — every future blueprint that lists notes calls it.

## What ships

### 1. NEW MECHANISM `cards@0.1.0`

| Aspect | Decision |
|---|---|
| Mechanism id | `cards` |
| CustomJS class | `BeaconCards` |
| Materialization | `{{scripts_path}}/cards/beacon-cards.js` |
| Depends on | `customjs-guard@>=1.0.0` |
| Installer mechanics | NONE (pure CustomJS class; no manifest schema additions; no helper additions) |
| Mobile-aware | YES (gotcha 7 carry-forward — adapts grid columns by container width; touch-friendly target sizes; no hover-only affordances) |
| Inline styles | YES (consistent with sibling mechanisms; no separate CSS file) |
| Helper-cases harness | unchanged at 255 sub-asserts (no installer-side schema; mechanism is renderer-only) |

#### API

```js
await customJS.BeaconCards.render(dv, {
  pages,                              // Array — Dataview pages OR hand-rolled objects
  title:    (p) => p.file.name,       // required: string title
  subtitle: (p) => null,              // optional: string subtitle line under title
  badges:   (p) => [],                // optional: Array<{label, tone?}> chip-style metadata
  progress: (p) => null,              // optional: {done, total} → renders progress bar + count
  target:   (p) => p.file.path,       // optional: wikilink target on click; default p.file.path
  sort:     (a, b) => …,              // optional: comparator; default mtime desc
  group:    (p) => null,              // optional: string group key → cards grouped under headings
  empty:    "Nothing here yet",       // optional: empty-state copy
  columns:  "auto",                   // optional: number | "auto" — default auto-fit by width
  onClick:  null,                     // optional: (page, ev) => void — overrides default openLinkText
});
```

**Decoupled from Dataview semantics.** Caller supplies `pages` array; mechanism doesn't query. Works with `dv.pages(...)` results AND hand-rolled arrays from CustomJS classes (matches how `ProjectsHubCards` already builds its list today).

**Visual contract starting point.** Inherits ProjectsHubCards's existing card layout:
- 1-column grid on narrow viewports (< 600px container width)
- Auto-fit grid (~280px min card width) on wider viewports
- Card body: title (bold) → subtitle (muted) → badges row → progress bar (if provided)
- Hover: subtle elevation; click: openLinkText (or onClick override)
- Empty state: dashed-border placeholder w/ `empty` copy

## Risks + mitigations

| Risk | Mitigation |

| BeaconCards API gaps surface during audit (T3.1) — meetings or daily can't fit | Cycle still ships; gaps documented; v0.5.x.y migration cycles get extension specs. Mechanism API extensions are MINOR-bump territory; no breakage of project consumer. |
