## Scope — Migration step (NEW `applyWikiToDocsMigration` in install.js)

**Order:** runs in the per-blueprint pipeline for `project`, after `installItem` but before `applyDocsBackfill`. Wrapped in try/catch; per-vault errors become history-warnings, not install-fails.

**Algorithm (per vault):**

```pseudo
for slug in listProjectSlugs(vault):
    wikiDir = vault/spice/projects/<slug>/wiki
    docsDir = vault/spice/projects/<slug>/docs

    if not exists(wikiDir): continue        # nothing to migrate
    if exists(docsDir):
        // Idempotency: prior migration already happened, OR user-created docs/ pre-migration.
        // Skip; emit history note ("docs/ already present; skipping migration for <slug>").
        continue

    // 1. Backup
    backupDir = vault/.sauce-backup/<slug>/wiki/<timestamp>/
    cpRecursive(wikiDir, backupDir)
    historyLog: "migrated_wiki_backup", slug, backupDir

    // 2. Rename directory
    fs.renameSync(wikiDir, docsDir)

    // 3. Rename Wiki.md → Docs.md
    wikiHub = docsDir/Wiki.md
    docsHub = docsDir/Docs.md
    if exists(wikiHub):
        fs.renameSync(wikiHub, docsHub)

    // 4. Rewrite frontmatter in each .md
    for file in docsDir/*.md:
        rewriteFrontmatter(file, {
            "type": {"wiki-hub": "docs-hub", "wiki-note": "doc-note"},
            "tags": {"wiki-hub": "docs-hub", "wiki-note": "doc-note"},
        })

    historyLog: "migrated_wiki_to_docs", slug
```

**Backup posture:** `.sauce-backup/<slug>/wiki/<timestamp>/` (timestamped per migration run; multiple runs accumulate). Mirrors the `.sauce-backup` posture used elsewhere in install.js.

**Idempotency:** the `if exists(docsDir): continue` gate is the primary mechanism. Second-and-subsequent install runs see `docs/` and skip. The backup is only written on the migration run itself.

**Failure mode:** any thrown error during a single project's migration: catch, write a history warning entry (`migration_failed`, slug, error.message), continue to next project. Install does NOT fail-loud — per existing `applyVendoredThemes` / `applySnippets` posture.

**Cross-reference rewriting:** explicitly out-of-scope. User-authored `[[Wiki Hub]]` or `[[wiki/Wiki]]` references in OTHER notes (daily notes, scratch, etc.) will dangle after migration. Manual fix via Obsidian's rename-with-backlinks UX or "Find and replace in files". Risk acknowledged.

## Backwards-compat shim

A consumer-side dataviewjs block that has already been materialized (in `ranch/templates/Template, Wiki Hub.md` or any user-authored Wiki.md) references `customJS.ProjectWikiCards`. After this cycle:

- The materialized `Template, Wiki Hub.md` is **renamed** to `Template, Docs Hub.md` (no longer present at old path on a clean reinstall).
- Migrated `Docs.md` files retain whatever dataviewjs block their original `Wiki.md` had — which references `ProjectWikiCards`.

**Fix:** the migration step (step 4 above) ALSO rewrites any `customJS.ProjectWikiCards` reference in migrated `.md` bodies to `customJS.ProjectDocsCards`. One additional regex pass per .md.

**Alternative (rejected):** ship a thin shim class `ProjectWikiCards = ProjectDocsCards` to preserve old references. Adds maintenance burden + leaves a deprecation breadcrumb that grows stale. Direct rewrite is cleaner.

## Risks

1. **Migration of user-authored content.** Migration runs against consumer vaults containing user-written wiki notes. Bug = potential user data loss. Mitigation: `.sauce-backup` per-project pre-migration; idempotency gate (`docs/` exists → skip); failure-loud history; per-project try/catch (one project's failure doesn't cascade).
2. **Dangling cross-references.** `[[Wiki Hub]]` links in non-blueprint notes will dangle after migration. Explicitly out-of-scope. Mitigation: user can run Obsidian's "Search and replace in files" or rename-with-backlinks UX.
3. **Workshop dogfood self-migration.** If the workshop itself has wiki/ dirs (it shouldn't — workshop is platform-only, no user content; but the seed.js test data MIGHT generate some), S7 will exercise the migration on itself. Verify pre-S7 whether the workshop has wiki/ dirs to migrate (`ls spice/projects/*/wiki/ 2>/dev/null`).
4. **Plan-execution drift on rename surface area.** Mass renames touch ~15+ files. Plan must enumerate every file. Mitigation: S5 includes a final grep-for-"wiki"/"Wiki" sweep before S6.
