# v0.14.0 Design — date-aware nav buttons

> [!abstract] Cycle goal
> Patch `nav-buttons@2.4.0 → 2.5.0` so `runTemplaterTemplate` actions read the active file's date from its basename and use that date in `folder_date_pattern` / `filename_date_pattern` substitutions. Falls back to today's date when the active file has no date pattern. Three RTT consumers (to-do / meetings / journal) inherit transparently — zero blueprint manifest edits.

## Architecture

> [!info] Wiring (right-to-left)
> 1. User on a date-routed note (e.g., `Monday-2026-05-10` daily, `Journal-2026-05-10`, `ToDo-2026-05-10`, `Meetings-2026-05-10`) clicks an RTT-action button (to-do / meetings / journal).
> 2. `space-nav-buttons.js` `_dispatchAction` runTemplaterTemplate branch (L300-358) calls `this._resolveActionDate(dv)` (NEW helper) BEFORE composing folder + filename.
> 3. `_resolveActionDate(dv)`:
>    - Reads `dv.current?.()?.file?.name` (string or undefined).
>    - Matches `/(\d{4}-\d{2}-\d{2})/` — if it captures a `YYYY-MM-DD` group, validates with `window.moment(captured, "YYYY-MM-DD", true).isValid()`.
>    - Valid → returns the captured ISO string verbatim (no further moment manipulation needed; the dispatcher will format).
>    - Invalid OR no match → returns `window.moment().format("YYYY-MM-DD")` (today).
> 4. Dispatcher uses the resolved ISO string by parsing it via `window.moment(actionDate, "YYYY-MM-DD", true)` and calling `.format(folder_date_pattern)` + `.format(filename_date_pattern)` exactly as before.
> 5. Existing race-tolerance + folder-creation + Templater-create paths UNCHANGED.

> [!info] Why a string, not a moment instance
> Returning a `'YYYY-MM-DD'` string makes the helper trivially testable without a moment dependency in the harness for DA1 (active-file-with-date) — the regex match group IS the assertion target. Only DA2 (no match → today fallback) needs a `window.moment` shim, and a one-line stub returning `{ format: () => '<known-test-date>' }` covers it. Keeps the helper's contract narrow and the harness zero-dep.

## File deltas

### Modified files

> [!example]- `platform/mechanisms/nav-buttons/space-nav-buttons.js`
> Two surgical edits:
>
> **1. NEW helper method on `SpaceNavButtons` class** (insert near the existing `_readDailyNotesMeta` helper):
> ```js
> // Resolve the date used by runTemplaterTemplate folder/filename substitutions.
> // If the active file's basename matches /(\d{4}-\d{2}-\d{2})/ AND the captured
> // string is a valid ISO date, return it verbatim. Otherwise fall back to today.
> // Returned shape: 'YYYY-MM-DD' string. Caller parses via window.moment(s, "YYYY-MM-DD", true).format(pattern).
> _resolveActionDate(dv) {
>   const currentFile = dv && dv.current && dv.current();
>   const fileName = (currentFile && currentFile.file && currentFile.file.name) || "";
>   const dm = fileName.match(/(\d{4}-\d{2}-\d{2})/);
>   if (dm) {
>     const m = window.moment(dm[1], "YYYY-MM-DD", true);
>     if (m.isValid()) return dm[1];
>   }
>   return window.moment().format("YYYY-MM-DD");
> }
> ```
>
> **2. Refactor `_dispatchAction` runTemplaterTemplate branch (L314, L319)** to use the helper. Plumb `dv` through `_dispatchAction`'s call site (the `el.onclick` handler at L235-240) since the helper needs the dataviewjs scope object.
>

## Risk register

> [!warning] Pre-identified risks + mitigations
> | ID | Risk | Mitigation |
> |---|---|---|
> | R1 | `dv.current()` semantics in dataviewjs | Verified — already used at L107-108 of same renderer for daily-nav top-row; identical access pattern |
> | R2 | False-positive date matches (e.g., `Project-Notes-1234-56-78`) | `window.moment(captured, "YYYY-MM-DD", true).isValid()` strict-parse rejects invalid date components; combined with the regex requirement gives correct semantics |
> | R3 | Multiple date matches in one filename (e.g., `Meetings-2026-05-04-archive-2025-12-31`) | Regex without `g` flag returns FIRST match → 2026-05-04 wins; acceptable + predictable |
> | R4 | Pre-existing T4.0 fatal on naked `run-renderer.js` | Out of scope; carried forward from v0.12.0/v0.13.0; do NOT bundle into this cycle |
> | R5 | Harness has no `window.moment` global; existing tests don't hit `_dispatchAction`'s moment calls | Install minimal `global.window = { moment: ... }` stub at top of `date-aware` test cases; restore after; zero-dep moment-mock for DA2 only (DA1's regex-match path bypasses moment entirely) |
> | R6 | `_dispatchAction` signature change ripples | Single call site at L235-240 (the click handler); update in lockstep with helper introduction |
