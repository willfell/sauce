# v0.49.2 â€” Kanban Card vault-root + workstream-picker fix (CLOSED 2026-05-16, validated at headspace)

## Summary

v0.49.0 shipped L2 listener registration + BUG-7 + BUG-8 (closed CLI-level at headspace 2026-05-16). v0.49.1 attempted a follow-up fix for a separate symptom reported by the user (new kanban card files landing at vault root with no workstream picker) but was based on a wrong assumption about the obsidian-kanban plugin's flow. v0.49.2 ships the real fix.

**Symptom:** clicking "+ Add a card" on a project's kanban board produced new task files at vault ROOT (e.g., `nono.md`, `another no.md`) instead of under `spice/projects/<slug>/tasks/`, with `source_board: <filename>.md` in frontmatter (meaning source-board detection failed) and no workstream picker firing.

**Root cause (verified by reading kanban-plugin main.js):** obsidian-kanban v2.0.51's "+ Add a card" flow with `new-note-template` set:
1. Create empty new file (at vault root OR `new-note-folder`)
2. Open the new file as the active leaf
3. Call `templaterPlugin.append_template_to_active_file(templatePath)`

By template-run time, BOTH `tp.config.active_file` AND `app.workspace.getActiveFile()` point at the NEW file. The kanban board's source identity is LOST in this flow. The kanban-plugin's per-board `new-note-folder` is ALSO ignored when `new-note-template` is set â€” kanban relies on Templater for placement, and Templater defaults to vault root.

**v0.49.2 fix:** `Template, Kanban Card.md` detects the source board by SCANNING THE VAULT for kanban-plugin board files containing `[[<new-card-title>]]` (which the kanban plugin appends when creating the card), picking the one with the most recent mtime. Robust to:
- vault-root file placement
- kanban-plugin's active-leaf switching to the new file
- multiple open kanban boards (mtime tiebreaker)

Both the workstream picker (top `<%* %>`) and the auto-promote `tp.file.move` (bottom `<%* %>`) use this vault-scan detection.
