# v0.51.0 — Project status widget (Bump Status on project hub)

**Date:** 2026-05-17
**Card:** [[Project status updates]] (Projects Blueprint sub-board)
**Cycle target:** MINOR v0.50.5 → v0.51.0

## Symptom / motivation

The project blueprint exposes a `status` frontmatter field (7-state enum: `idea | planning | in-progress | blocked | superseded | cancelled | done`) and the Projects.md hub filters/groups projects by status (shipped in the v0.39.0 S6 codepath that landed under later cycles). But there is **no in-vault UI** to change a project's status. Today a user must hand-edit the frontmatter — and remember to update `status_changed_at` in lockstep.

User card body asks for: a button on the project hub, "designed like the workstream edition and subtraction stuff", that updates the status and (implicitly) keeps `status_changed_at` in sync.

This is the missing piece of v0.39.0's `ProjectActionButtons` Tier-1 design that never shipped (S7 of v0.39.0 was never executed; v0.46.2 later removed the dangling manifest references — see CLAUDE.md status notes).

## Decision: scope of this round

**Just Bump Status.** v0.39.0's Tier-1 bundle also included Set Teams + Set Products; those defer to a follow-up cycle when needed. YAGNI.

## Design

### Class: `ProjectStatusWidget` (NEW)

Single-responsibility CustomJS class. Renders on the project hub (main project note) as a colored status chip with an "Updated YYYY-MM-DD" hint, click → custom popup with 7 status buttons → write back via `app.fileManager.processFrontMatter`.

**Why a new class, not an extension of `ProjectWorkstreamManager`:** keeping single responsibility per class is the established sauce pattern. `ProjectWorkstreamManager` is workstream-specific (renders workstream cards with per-workstream progress bars, parses the kanban board to count tasks). Adding status logic into it would make the name lie and tangle two independent concerns.

### File: `helpers/project-status-widget.js` (NEW)

~80 LOC. Public surface: `render(dv)` (called by `customjs-guard`). Module-local constants for the 7-status list + color palette (DRY copy from `projects-hub-cards.js _statusPill`; cross-class shared-helper extraction is out of scope this round).

### Layout on project hub

```
[ Map ]  [ Project Board ]  [ Wiki ]    ← existing ProjectNavButtons row
---
## Status
Status: [ in-progress ▼ ]    Updated 2026-05-16    ← NEW block (this cycle)
---
## Workstreams
[ + Add Workstream ] [ Remove ]          ← existing ProjectWorkstreamManager block
…workstream cards with progress bars…
```

The new `## Status` H2 section sits between nav-buttons and the existing `## Workstreams` section. User's chosen placement (Phase B): "inline with workstreams row" — interpreted as visually adjacent / same area. The H2 heading mirrors the existing template's typography.

### Picker UX

Click the chip → custom full-screen overlay (matches entity-create's overlay pattern at `entity-create.js:100-101`): semi-transparent backdrop, centered dialog box, 7 vertically-stacked buttons (one per status), each colored with the same palette used by the chip. Click any → write + close → automatic dataview re-render shows the new chip color.

Cancel: click overlay backdrop (not the dialog) or press Escape — matches entity-create's overlay close pattern.

### Data flow

- **Read:** `dv.current().status` (default `"idea"` if absent); `dv.current().status_changed_at` for the "Updated" label
- **Write:** `app.fileManager.processFrontMatter(file, fm => { fm.status = newStatus; fm.status_changed_at = todayISO(); })`. Same pattern as `ProjectWorkstreamManager:96`. Safe YAML serialization; Obsidian's metadata cache picks up the change and the dataview block re-renders automatically.

### Error handling

- `dv.current().file` missing → silent return (matches sibling helpers).
- Write throws → `new Notice("Failed to update project status: " + e.message)` + `console.error`. Matches `ProjectWorkstreamManager`'s posture.

## Testing

`run-helper-cases.js` grows by 6 sub-asserts (PSW-1..6, in a new `casePSWStatusWidget` block grouped near `casePNBWrap*`):

- **PSW-1:** `project-status-widget.js` source declares `class ProjectStatusWidget` and an `async render(dv)` method.
- **PSW-2:** Source defines a STATUSES array containing all 7 expected values: `idea`, `planning`, `in-progress`, `blocked`, `superseded`, `cancelled`, `done`.
- **PSW-3:** Source calls `app.fileManager.processFrontMatter` (the write-back API).
- **PSW-4:** Source writes BOTH `status` and `status_changed_at` keys (regex check on the processFrontMatter callback body).
- **PSW-5:** `manifest.json` `customjs_classes` includes `ProjectStatusWidget` AND `files[]` includes the new helper path.
- **PSW-6:** `Template, Project.md` contains the new dataviewjs block invoking `class: "ProjectStatusWidget"`.

Source-string asserts are sufficient — no behavior to test beyond contract presence; the runtime DOM behavior is in Obsidian's hands, and our renderer test harness doesn't simulate dataview re-render or `fileManager.processFrontMatter`.

## Risks

- **Low.** New isolated class + new isolated render surface. No mutation of existing helpers. No changes to existing manifests beyond the project blueprint's own.
- The `app.fileManager.processFrontMatter` API is the same one workstream-manager has used reliably for the entire blueprint's lifetime.
- Worst case: the popup misbehaves (visual glitch) and the user keeps hand-editing frontmatter — falls back to today's state.
