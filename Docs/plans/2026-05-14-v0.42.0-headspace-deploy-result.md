These are NEW findings surfaced by the headspace deploy. Numbered `FLN-D<N>` to distinguish from the v0.42.0 cycle-close FLNs in `2026-05-13-v0.42.0-result.md`. Each carries a proposed cycle shape so a planning session can promote them.

### FLN-D1: Pre-v0.36.0 vaults with stale `workshop_relative_path` can't `sauce reinstall` without a manual config edit

**Symptom (observed at headspace):** Iteration 1 of `sauce reinstall --vault <headspace>` crashed with `HARNESS CRASH: Cannot find module '/Users/willfellhoelter/notes/workshop/sauce/platform/install.js'`. Root cause: headspace's `ranch/platform-config.json` had `"workshop_relative_path": "../../workshop/sauce"`, a path correct on a previous machine but dead on this one.

**Why it matters:** Any consumer vault that was bootstrapped against a dev-repo path (pre-v0.36.0 distribution model) carries a machine-specific relative path. After `brew install sauce` becomes the canonical distribution channel, these stale paths block reinstall. `sauce migrate-layout` only handles the `<vault>/pantry/.git/` case (legacy in-vault checkout); it doesn't detect or repair a dead `workshop_relative_path`.

**Manual workaround used in this deploy:** Edit `workshop_relative_path` to `"/opt/homebrew/opt/sauce/libexec"` (the brew `opt` symlink is version-stable across `brew upgrade sauce` — `/opt/homebrew/opt/sauce → ../Cellar/sauce/<current-version>`).

**Proposed cycle shape (likely PATCH on `migrate-layout` or new verb):**
- Option A: `sauce migrate-layout` detects dead `workshop_relative_path` as an additional trigger (alongside `<vault>/pantry/.git/`), rewrites to `<brew --prefix sauce>/libexec`, then runs reinstall. Single verb; cohesive UX.
- Option B: New `sauce migrate-stub` verb dedicated to rewriting `workshop_relative_path`. Cleaner separation; smaller blast radius per cycle.
- Option C: `sauce doctor --vault <path>` mode that resolves `workshop_relative_path` and warns (or auto-repairs with `--fix`) on dead paths. Lower friction; doesn't require user to know the right repair verb.
- The bin shim's `active-pantry` resolution logic (`platform/cli/resolve-sauce-dir.js`) ALREADY contains the correct resolution semantics. The fix is to teach the in-vault stub (`<vault>/ranch/templater/platformInstall.js`) to honor the same resolution chain — but landmine #13 says the stub is content-static. Resolution: bump the stub schema (content-DYNAMIC migration), OR keep stub static and migrate `workshop_relative_path` per-vault.

**Severity:** Medium — single-pin manual edit fixes it; affects only pre-v0.36.0 vaults; once migrated, stable forever. But silent failure for users.
**Best cycle slot:** Standalone PATCH (e.g., v0.42.1) bundling FLN-D1 + FLN-D3 below. Small, low-blast-radius.

### FLN-D2: Audit's homegrown YAML parser doesn't read inline-flow tag arrays

**Symptom (observed at headspace):** `sauce audit` flagged 9 `required_tags.missing` violations on the v0.42.0 cowork hub notes (`Cowork.md`, `Daily Hub.md`, `Weekly Hub.md`, `Monthly Hub.md`). On inspection, the materialized files DO contain the required tags as inline-flow arrays (e.g., `tags: [cowork-hub, daily-hub]`). The audit's `frontmatter_parse` step is also flagging "nested mapping" errors on user-content vault-config files — same parser, same limitation surface.

**Why it matters:** Inline-flow YAML is the workshop's documented convention for short tag lists (the cowork v0.42.0 content sources use it; the daily blueprint uses it; multiple other blueprints use it). The audit parser disagreeing with the content convention generates noise that buries real findings. In this deploy, 877 audit hits had to be hand-classified to confirm 0 genuine v0.42.0 issues — most of those were unrelated daily-note naming-pattern violations, but the 9 cowork false-positives are particularly bad because they LOOK like a v0.42.0 install bug.

**Proposed cycle shape (audit mechanism PATCH or MINOR):**
- Replace the homegrown YAML parser in `platform/mechanisms/audit/...` with `js-yaml` (or equivalent battle-tested parser) — handles inline-flow + block-flow + nested mappings uniformly.
- Per-blueprint regression: add audit-parser-fixture tests covering inline-flow tag arrays + nested-mapping frontmatter so the parser swap is provably non-regressing.
- Bumps `audit@0.2.1 → 0.3.0` (MINOR — parser change is a behavior change even if the public contract holds).

**Severity:** Medium — actively misclassifies content as broken. Causes wasted human cycles classifying findings on every audit run.
**Best cycle slot:** Standalone MINOR for audit mechanism. Could bundle with FLN-D4 if both routes through the audit mechanism touch overlapping code.

### FLN-D3: Tap auto-bump PRs don't auto-merge

**Symptom (observed at handoff time):** The `bump-tap` workflow on `willfell/sauce`'s v0.42.0 tag push correctly created `willfell/homebrew-sauce#18` (URL+SHA bump from 0.41.5 → 0.42.0). But the PR remained OPEN. `brew update` didn't surface 0.42.0 until the deploy session squash-merged the PR manually. Prior 0.41.x bumps (PRs #11/#12/#14/#15/#16/#17) all show the same OPEN-then-manually-merged pattern.

**Why it matters:** Every release currently requires a manual click-to-merge in the tap repo to land on consumers. The auto-bump workflow is correctly emitting the PR but stopping short of merge. This is the kind of thing that's tolerable when active development is on a single machine but breaks down when the user is multi-machine (the tap pin lags the workshop tag by however long it takes the user to notice + click).

**Proposed cycle shape (release workflow PATCH):**
- Option A: Add `--auto-merge` flag to the `peter-evans/create-pull-request@v6` action in the tap-bump job. Requires branch-protection-aware auto-merge to be enabled on `homebrew-sauce`'s default branch. Simplest fix.
- Option B: Add a separate `auto-merge-bump` workflow on `homebrew-sauce` that triggers on `pull_request` labeled `automerge` (the bump action adds the label), waits for required checks, then merges. More opaque but defensible against accidental merges of non-bump PRs.

**Severity:** Low for single-developer single-machine, but blocks `brew upgrade sauce` from "just working" on every machine without human intervention.
**Best cycle slot:** Standalone PATCH on release workflow (`.github/workflows/release.yml` + possibly a new tap-side workflow). Could bundle with FLN-D1 in a "deploy hardening" PATCH.

### FLN-D4: Brew-shipped `project@1.6.1` references a missing helper

**Symptom (observed during install):** Install notice during step 6 (materialize):

```
[Notice] installItem: source missing: /opt/homebrew/opt/sauce/libexec/platform/blueprints/project/helpers/project-action-buttons.js
```

The path is referenced by `project@1.6.1`'s manifest but the file doesn't exist in the brew-installed tarball. Install proceeded to exit 0 because the installer treats `installItem: source missing` as a warning notice, not a fatal error. Verified the file exists in the dev repo: `/Users/willfellhoelter/projects/repos/sauce/platform/blueprints/project/helpers/project-action-buttons.js` is present — so the missing artifact is brew-tarball-specific.

**Why it matters:** Either:
- The manifest references a path that was renamed/removed between the dev tree and v0.42.0's tagged source, OR
- `npm pack` / git-archive (whichever the brew formula uses to construct the tarball) is excluding `helpers/*.js` from the project blueprint's content. Suggests a `.gitattributes export-ignore` or `npm` `files[]` posture issue.

**Proposed cycle shape (project blueprint PATCH or release workflow PATCH):**
- Step 1: Diff `git archive HEAD --prefix=sauce-0.42.0/ | tar tvf -` vs the project blueprint's directory contents. Identify which file(s) are excluded.
- Step 2: Repair (probably by adding the path to `files[]` in `package.json`, OR removing an `export-ignore` line in `.gitattributes`).
- Step 3: Re-tag if the issue is in the v0.42.0 tarball specifically, OR ride the fix into the next release. The `installItem: source missing` notice means the absent file is non-fatal — but the install is silently degraded for project consumers.

**Severity:** Medium — install proceeds, but project blueprint functionality is partially broken in brew-installed vaults. Dev-repo consumers (active-pantry linked to dev tree) don't see the bug.
**Best cycle slot:** Standalone PATCH on project blueprint or release workflow. Could roll into a "deploy hardening" PATCH alongside FLN-D1 + FLN-D3.
