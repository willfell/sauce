# v0.46.0 — entity-create mechanism + 7-site New<X>Button big-bang migration (CLOSED 2026-05-15)

## Summary

v0.46.0 extracts the "create a new entity" pattern — previously duplicated across 7 per-blueprint CustomJS helpers — into a single NEW mechanism `entity-create@0.1.0`, then migrates all 7 creation sites in a coordinated big-bang across 5 blueprints (meetings, people, project, scratch, finance × 3).

**Mechanism design:** `EntityCreate` CustomJS class (~470 LOC) reads a declarative per-blueprint `new_entity_buttons[]` spec and drives the entire create-flow: frontmatter scaffolding, body population, extra_files materialisation, nav prompt types (text / select / month / number), validate predicates (gte:<key>, min:<n>, max:<n>, safe-filename), `|number` substitution pipe, and a `derive` mini-DSL (slugify / lowercase / sanitize-filename). Schema covers 7 extensions: `extra_files[]`, `{{current_file.frontmatter.X}}` token, `month`/`number` prompt types, `inline_body` peer to `body_template`, `validate` predicates, `|number` substitution pipe, `derive` fields.

**Installer mechanics:** NEW `applyNewEntityButtons` + `resolveEntityCreateEntry` + `injectAccentButtonBlock` (~373 LOC) in `install.js` handle idempotent marker-anchored block injection into hub files. `pruneEntityCreateRegistry` parallels nav-buttons pruning. `render_in.kind: "nav_buttons"` declared reserved (no Cycle 1 site uses it). Each blueprint's `new_entity_buttons[]` entries are embedded into `ranch/platform-installed.json` per-blueprint at install time; the entity-create audit walker reads them directly from there.

**Audit integration:** NEW `platform/audit/entity-create-walker.js` with 3 severity rules (`manual_implementation_at_risk` HIGH, `escape_hatch_used` INFO, `dead_path` MEDIUM); NEW `--entity-create` flag in `cmd-audit.js` + report formatter with JSON summary footer.

**Migrations (5 blueprints, 7 sites):** All 7 `New<X>Button` helper files deleted; `EntityCreate` drives all create flows at runtime. Body-template compatibility required a mid-cycle design correction (commit `59219a1`): Templater-style templates (meetings, people, scratch) dropped `body_template` in favour of `frontmatter_template` + `inline_body`; project's legacy `{{NAME}}/{{SLUG}}/{{DATE}}/{{DATE_TAG}}` tokens were translated to entity-create's substitution catalogue. Finance's three sub-flows (budget / paycheck / invoice) all migrate; `finance@0.2.10 → 0.3.0` is the only MAJOR blueprint consumer of the new `month` prompt type, `validate` predicates, `|number` pipe, and `extra_files[]` for kanban sub-folder.

**Acceptance gate:** All 7 New<X>Button files deleted; `sauce reinstall --vault $PWD` exit 0; `sauce audit --entity-create` zero findings (after deleting stale orphan `ranch/scripts/meetings/new-meeting-button.js`); whole-suite preflight GREEN at every stage close; tag `v0.46.0` annotated + pushed; release.yml preflight → bump-tap chain auto-bumps `Formula/sauce.rb` in `willfell/homebrew-sauce`.

## Plan vs reality

### Two critical bugs surfaced by testing

**C1 (S3 follow-up `4fd5fec`):** The entity-create audit walker originally read `bp.path` + `bp.manifest` from `platform-installed.json` — fields that were never written by the installer. The fix embeds a `new_entity_buttons` field per-blueprint into `installed.json` at install time; the walker reads it directly from there. This required both a walker update and an installer change to persist the field.

**target_path critical bug (`85d315c`, post-S11):** All 7 blueprint manifests initially used source-relative `target_path` values (e.g., `"content/Meeting Hub.md"`). The installer treated these as vault-relative paths, so the canonical AccentButton fence was never injected into the correct hub files. Fixed by changing all 7 `target_path` values to vault-relative forms using `{{module_directory}}/<path>` or `{{templates_path}}/<path>`, matching the pattern used in `files[].dest`.

### Body-template gap (mid-cycle design correction `59219a1`)

Discovered post-S6: body_template files in meetings, people, and scratch used Templater `<%* %>` syntax; project used legacy `{{NAME}}/{{SLUG}}/{{DATE}}/{{DATE_TAG}}` tokens — both incompatible with entity-create's substitution catalogue.

- **Path A (project):** templates token-translated to entity-create's substitution catalogue; `Project.md` frontmatter stripped (manifest's `frontmatter_template` owns it).
- **Path C (meetings + people + scratch):** manifests drop `body_template`; populate `frontmatter_template` + `inline_body` instead. Templater templates remain in each blueprint's `files[]` for manual "Insert template" invocation.

### Sub-assert counts vs plan

S11 +2 sub-asserts added to `run-helper-cases.js` ahead-of-stage by S7's implementer (proactive update kept preflight green during S7). Audit cases AU-EC-2 + AU-EC-6 emit 2 sub-asserts each (counts + findings), so actual +8 vs plan's +6 target. All asserts load-bearing.

### Orphan stale file at S12

`ranch/scripts/meetings/new-meeting-button.js` lingered from pre-v0.46.0 install (no longer in meetings `files[]`). S12 deleted it manually before `sauce audit --entity-create` reported zero findings. The installer does not currently prune orphan materialized files when a blueprint's `files[]` entry is removed — captured as FLN-g.

### Subscription drift reconciled

`ranch/platform-subscription.json` was pinned at workshop_version `0.41.5` — four cycle hops behind. S12 reconciled it forward to `0.46.0` + added `entity-create` mechanism entry + 4 subscribed-blueprint bumps.

### Pacing vs estimate

~24 total subagent dispatches vs handoff estimate of ~36. Savings came from: parallel reviews, combined-review shortcut for simple per-blueprint migration stages, and controller-direct sequencing for S12 cycle close.

## FIX-LATER-NOTEs

a. **S1 — `_loadSpec` error masking:** `_loadSpec` silently treats "missing file" and "malformed JSON" identically — both return `null`. Should `console.error` + emit a Notice for parse errors to distinguish the two failure modes. Also: `_promptText` empty-check has a missing type-guard on the optional-branch path (line 174). Both minor, non-blocking.

b. **S1 — polish / M-items (deferred):**
   - M1: array-item YAML quoting (`- "finance"` vs unquoted `- finance`) — both valid; quoted form is safer for keys that could be parsed as non-string YAML types.
   - M2: nested-object emission via `JSON.parse(JSON.stringify())` reviver is brittle for nested arrays.
   - M3: `_substitute` non-string short-circuits (asymmetry with `_emitScalar` numeric path).
   - M4: predicate splitter regex tolerates trailing separators but worth documentation.
   - M5: plus-icon SVG inline in `render()` — extract candidate for easier theming.
   - M6: `_runOneValidate` silently passes unknown predicates (intentional graceful fallback, but worth documenting).
   - M7: `_promptSelect` doesn't validate non-empty options array.
   - M8: overlay scaffolds `_promptText` + `_promptSelect` are nearly-duplicate; refactor candidate.

c. **S2 — regex edge cases (deferred, all bounded to malformed inputs):**
   - I1: stray-fence corruption on truncated-fence input (mid-edit user-state edge).
   - I2: CRLF-mixed line endings produce mixed-EOL output (Windows-only).
   - I3: mid-line marker produces split output (hand-authoring edge).
   - I4: leading-whitespace marker keeps phantom indent (cosmetic).
   - M-items M1 (DRY recompute of entries[]), M2 (fence-opener regex tolerance), M3 (`writeJson` vs `adapter.write`), M4 (canonical body backtick invariant), M5 (magic marker prefix), M6 (Notice fire-rate).

d. **S3 — audit walker limitations (post-C1 fix):**
   - Walker's `staticPart` empty-template case counts as `unverifiable` instead of `aligned` — surfaces correctly in summary but worth a design-doc note.
   - `bp.path + "/manifest.json"` fallback in the walker is now unused in production (installer embeds `new_entity_buttons` directly); dead code candidate for removal.
   - `sauce audit --help` requires being inside a vault (context-resolver runs before flag parsing) — dispatcher-level help routing is a UX improvement candidate.

e. **Body-template gap fix posture:**
   - The Templater templates (Meeting.md, People.md, Scratch.md, etc.) remain in each blueprint's `files[]` for manual "Insert template" invocation in Obsidian. The user-facing flow change (entity-create drives creation; Templater template is for manual use only) is not yet documented in any user-facing doc.
   - `inline_body` literal-path approach (`"ranch/views/customjs-guard"`) means future reorganization of `ranch/views/` would need lockstep edits across 5 manifests.

f. **project@1.7.0 pre-existing defect retained:** manifest's `files[]` references `helpers/project-action-buttons.js` which doesn't exist. Installer correctly degrades (Notice, not fatal — confirmed at S12 dogfood). Small patch-cycle candidate; pre-dates v0.46.0.

g. **Orphan file pruning gap:** `ranch/scripts/meetings/new-meeting-button.js` lingered from a pre-v0.46.0 install (no longer in meetings `files[]`). S12 deleted it manually before `sauce audit --entity-create` reported zero findings. The installer does not currently prune orphan materialized files when a blueprint's `files[]` entry is removed. Candidate for a future installer step analogous to `pruneEntityCreateRegistry`.

h. **S11 proactive sub-assert additions:** S7's implementer added 2 sub-asserts to `run-helper-cases.js` ahead of S11's stage — kept preflight green during S7. Not a defect; noted for plan-vs-reality completeness.

i. **S11 sub-assert consolidations:**
   - Sub-assert 8 implemented end-to-end via `_renderFrontmatter` rather than `_substitute` alone (per implementer's discretion — a sounder test boundary).
   - Audit cases AU-EC-2 + AU-EC-6 each emit 2 sub-asserts (counts check + findings check), producing +8 vs plan's +6 target — all load-bearing.
