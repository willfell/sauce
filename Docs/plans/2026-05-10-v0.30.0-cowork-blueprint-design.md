# v0.30.0 — cowork blueprint design

> [!abstract] Goal
> Productize the ad-hoc `Cowork/` automation layer (8 cron-driven prompts + 11 context files across 2 vaults) as a proper sauce blueprint at `spice/cowork/`, with skills materialized as native Claude Code skills under `<vault>/.claude/skills/cowork/`. Both orchestrators and sub-skills are real Skill-tool-invokable artifacts. Single discovery surface, symmetric architecture, parallel-dispatch authoring. Adds a 9th orchestrator (`ero-monthly`). Forward-compatible with a v0.32+ "sauce as Claude Code plugin" migration.

### Six load-bearing contracts

> [!example]+ Contract 1 — Skill ID namespace
> Skill IDs use `cowork:<id>` form for both orchestrators and sub-skills. Both resolve via Claude Code's Skill tool. Flat namespace under `cowork/` parent — orchestrator IDs and sub-skill IDs share the same dispatch surface, no prefix collision because IDs are unique within `cowork/`.

> [!example]+ Contract 2 — Orchestrator skill body
> 25-60 lines. YAML frontmatter: `name`, `description`, `schedule` (human-readable cron), `scope` (life | work | shared). Body is a numbered list of `Use Skill cowork:<sub-id> with <args>` calls. NO raw MCP tool calls in orchestrator bodies — all data gathering routes through sub-skills.

> [!example]+ Contract 3 — Sub-skill body (FAT)
> YAML frontmatter: `name`, `description`, `inputs`, `outputs`, `tags`. Body owns full normalize-and-format logic. Inputs typed minimally (e.g., `{ range: "today+next-2-days" }`). Outputs are markdown-ready strings or JSON snippets the orchestrator pastes in. The orchestrator never sees raw MCP returns.

> [!example]+ Contract 4 — Refresh contract
> Blueprint `manifest.files[]` lists what `sauce update` overwrites. Blueprint-managed: orchestrator skills (9), sub-skill bodies (25), `Cowork.md` hub, `context/README.md`, `context/obsidian-vault-guide.md`. User-managed (untouched by update): `context/about-me.md`, `context/active-threads.md`, `context/weekly-snapshot.md`, all per-vault context files.

> [!example]+ Contract 5 — Daily blueprint coordination (PATCH bump 0.2.4 → 0.2.5)
> Daily blueprint template gains a single comment-marker line `<!-- COWORK_CALLOUTS -->` directly above the `## Notes` heading. All cowork orchestrators target this marker for callout insertion. Graceful fallback: if marker absent (older daily templates), orchestrators emit a Notice and fall back to `## Notes` heading-parse.

> [!example]+ Contract 6 — Audit hooks
> Cowork blueprint's `rule_fragments[]` covers: (a) `Cowork.md` `type+tag`; (b) `prompts/*.md` legacy detection (fail if any cowork prompt remains in old `Cowork/prompts/` location post-migration); (c) skill-presence at `.claude/skills/cowork/<id>/SKILL.md` for the 9 canonical orchestrator IDs; (d) `active-threads.md` schema (type/updated/updated_by + per-thread H3 schema).

### 4.3 Error handling

- **Missing MCP server (e.g., gmail not connected):** `cowork:check-vault-routing` (Step 1) fails fast with a Notice; orchestrator aborts cleanly. Cron logs the error; no retry — next scheduled run takes over.
- **Missing daily note:** `cowork:ensure-daily-note` creates from canonical sauce daily template. If template missing, fails loud (sauce install incomplete).
- **Anchor marker absent:** orchestrator falls back to `## Notes` heading-parse + emits a warning Notice ("daily template upgrade required — run `sauce update`"). Graceful degradation across daily@<0.2.5 vaults.
- **Sub-skill returns malformed payload:** orchestrator treats sub-skill outputs as opaque markdown strings — no validation at the orchestrator level. Validation lives inside each FAT sub-skill. If a sub-skill genuinely fails (no MCP data), it returns an empty string or `> [!warning] data unavailable` callout — orchestrator pastes in regardless.
- **Migration failure (S5):** snapshot exists at `<vault>.pre-cowork-<ts>/`. Restore: `rm -rf <vault>/spice/cowork/` + `cp -R <snapshot>/Cowork <vault>/`.

## 5. Long-term direction

> [!info]- v0.31.0 — threads as standalone blueprint
> Promote `active-threads.md` from cowork-internal state to a sauce blueprint (`threads@0.1.0`). Hub note + dataviewjs cards rendering. Cowork's active-threads.md becomes content the threads blueprint owns. Decouples thread schema evolution from cowork's skill catalogue.

> [!info]- v0.31.0+ — cron orchestrators in vault
> Move `/sessions/.../mnt/.scheduled/<job>/SKILL.md` to `<vault>/ranch/scheduled/<job>/SKILL.md`. Brings the cron layer into the dogfood + audit envelope. Cron just executes `<vault>/ranch/scheduled/<job>/SKILL.md` directly — out-of-vault scheduler is path-agnostic.

> [!info]- v0.30.x — finance-migrator
> Existing CLAUDE.md carry. Cowork prompts continue referencing legacy `Finance/` paths until that cycle lands. Small sweep cycle that updates cowork prompts + migrates `Finance/` to `spice/finance/`.

> [!info]- v0.30.x or v0.31.0 — skill-author template
> Ship `platform/blueprints/cowork/templates/Template, Cowork Skill.md` for adding new skills outside the canonical 34. Standardizes input/output frontmatter conventions. Future blueprints that adopt the skills pattern reuse this template.

> [!example]+ v0.32+ — sauce as Claude Code plugin
> The architectural pivot. Vault-local `.claude/skills/cowork/` skills migrate to `<plugin>/skills/cowork/<id>/SKILL.md` shipped via Claude Code plugin marketplace (parallel to the `de` plugin model). Vaults consume the plugin instead of materializing skills locally. **Skill bodies don't change; only location.** v0.30.0's architecture is forward-compatible:
>
> - Skill IDs (`cowork:<id>`) survive the move.
> - Orchestrator-to-sub-skill dispatch via Skill tool survives.
> - FAT-skill contract survives.
> - Per-vault context content (`spice/cowork/context/`) stays in-vault — only skills relocate.
>
> Migration cycle would (a) extract `platform/blueprints/cowork/skills/` to a plugin repo, (b) ship plugin via marketplace, (c) update `sauce` installer to skip `materializeSkills` when plugin detected, (d) document plugin install + vault-content-only mode. Net: vaults shrink (no `.claude/skills/`), plugin distribution standard.
