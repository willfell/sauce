## The mid-cycle redesign — v1 → v2

v1's plan landed S1 cleanly at `319ddf9` (pre-session work): split-field schema mechanics + nav-buttons 2.2.0 BREAKING + bracket-escape gap fix in `validateAndResolve` + harness 220 → 235. v1 S2 (in progress, partially staged) was about migrating to-do + meetings to the new schema while keeping their existing v1 filename shape (`<YYYY-MM-DD>-ToDo` / `<YYYY-MM-DD>-Meetings`).

During the v1 S2 staging gate the user surfaced a mental-model misalignment: the date-prefix filename pattern (`2026-05-04-ToDo`) doesn't match the daily-rhythm intuition. Daily artifacts should sort + read as `<Name>-YYYY-MM-DD` — ToDo, then date; Meetings, then date; Monday, then date. Hyphen separator, no spaces. While the question was open, two more came along: meetings hub should be date-routed under year/month (mirroring to-do, not the flat `hubs/` from v0.3.2); and the daily blueprint needs both a Today nav-button AND a universal prev/next-day arrow row that skips to nearest-existing (with grey-out when no target). The arrow row should render on EVERY date-extractable note, not just daily notes.

Per gotcha-6 (mid-cycle redesign methodology, established in v0.3.2's v1→v2): pause SDD; invoke `de:brainstorming` to explore v2; invoke `de:writing-plans` to commit the v2 plan; resume SDD on the v2 plan. v1 S1's already-committed work (the split-field schema mechanics) was carried forward; v1's partial S2 staging was EVOLVED rather than discarded — same to-do + meetings manifest files, refined to the date-suffix shape + meetings layout migration in the same edits. v2 ended up substantially expanded scope (filename pattern + meetings layout + daily-nav + mobile-aware platform value) but the per-task carry forward kept the cycle linear rather than restart-style.

## Why this cycle (recap)

v0.4.0 introduced the `runTemplaterTemplate` action type for to-do as a single-field `folder` + `filename` schema where each field went through `substituteForMomentFormat` (a helper that bracket-wraps substituted `{{var}}` values so `moment.format` treats them as literals). The bracket-escape gap surfaced at v0.3.2 FIX-2 (literal `hubs` in the meetings manifest's `folder` got token-decomposed by `moment.format` into `11ub1` because `h` and `s` are tokens). The fix at v0.3.2 was a manual `[hubs]` in the manifest. v0.4.2 v1 generalized the fix at the schema level: split each field into `_prefix` (literal, never moment-formatted) + `_date_pattern` (moment-formatted with no literals other than `/` and `-`). Blueprint authors can never again accidentally trigger the bracket-escape gap because the literal segments don't pass through `moment.format` at all. The gap surfaced at v0.4.0 FIX-1 (substituted-value escape) + v0.3.2 FIX-2 (non-substituted-literal escape) — two times establishes a pattern; the schema bump at the third occurrence threshold was justified.

> Design v2 specified `daily-notes:goto-today` for the Today nav-button. Smoke C (T4.4) clicked the Daily button → renderer Notice fired "command not found 'daily-notes:goto-today'". User verified actual command id at runtime via the dev console:

> Object.keys(app.commands.commands).filter(k => k.includes('daily'))
> // ['daily-notes', 'daily-notes:goto-prev', 'daily-notes:goto-next']
> ```
>

>
> Note: `daily-notes:goto-prev` and `daily-notes:goto-next` DO exist as core commands, but they create-on-demand rather than skip-to-nearest-existing — so the custom prev/next arrow row implementation in SpaceNavButtons is still needed for the user-visible behavior (user explicitly wants nearest-existing-only, never create-on-click; design v2 §"Daily-nav" Q1 lock).
>
> Carry forward as a brainstorming + writing-plans checklist item: "for any `invoke_command` action_type at design time, verify the command id against `app.commands.commands` keys before the design is approved — don't trust the namespace pattern from the plugin's settings UI labels."

## Pacing rhythm — what worked / what to improve

### What worked

- **Parallel subagent dispatch on T3.1 + T3.2 (disjoint files).** T3.1 owned `space-nav-buttons.js` (renderer) + nav-buttons manifest; T3.2 owned `install.js` (validateAndResolve) + `run-helper-cases.js` (R5 case). Zero merge conflicts; both subagents landed clean on first review.
- **AskUserQuestion gates kept commits user-approved.** S2 commit, S3 commit, in-cycle bumps for to-do/meetings/daily, S4 cycle-close commit + tag — every commit went through an explicit user gate. Each one caught at least one staging issue (extra `.bak` files, unintended `Docs/Meta/Scripts/` dirty path, etc.) before commit.
- **Per-smoke confirmations caught the daily-notes id bug fast.** Smoke C failed in <30s of user testing; the dev-console check (`Object.keys(app.commands.commands).filter`) gave the exact wrong-id evidence in the same session; in-cycle patch + re-test took ~5 minutes total. Compare with how this would have looked if smokes were batched: an "all 8 smokes" gate would have surfaced the same bug but the context-switching between failed-smoke debug + remaining-smokes execution is more expensive.
- **Mid-cycle redesign per gotcha-6 — second clean instance.** v0.3.2 v1→v2 established the methodology; v0.4.2 v1→v2 confirms it works for substantial scope expansion (3x the surface area). Pause SDD, brainstorm, write plan v2, resume SDD; carry forward partial work via per-task evolution rather than restart.

### What to improve

- **Design phase should verify command ids against runtime.** Surprise A could have been caught at design v2 review if the command id had been verified via `app.commands.commands` keys. Add to the brainstorming + writing-plans checklist: "for any `invoke_command` at design time, dev-console verify the command id before approving the design."
- **Pre-flight should check installed.json vs subscription deltas.** Surprise B (stale 0.1.1 install) would have surfaced at S4 T4.1 pre-flight if the installer printed a `subscription says X.Y.Z; installed says X.Y.Z` line per subscribed item. As-is the diff is implicit (silent no-op when versions match). Carry forward: add a verbose pre-flight check that prints subscription→installed deltas BEFORE running install actions.
- **The v1→v2 staged-work-carrying pattern is now established and can be trusted.** Two clean instances. Worth codifying in `Docs/use.md` (explicit "redesigning mid-cycle" section) — but not a blocker; per the existing handoff carry-forward.
