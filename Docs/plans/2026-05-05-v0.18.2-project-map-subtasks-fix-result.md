## NEW v0.18.2 lessons (carry forward to v0.19.0+)

> [!info] Belt-and-suspenders predicate stacking is cheap insurance
> 1. **When a filter has multiple independent signals to choose between, stacking two of them is zero-cost belt-and-suspenders.** Either alone would close this bug. Together they survive: a future template authoring slip that drops `task_parent`; a future layout reorg that drops `/board/`; a Dataview indexing edge case where one signal is unavailable. Pattern: prefer 2 cheap independent predicates over 1 cleaner predicate when the surfaces are independent and the cost is one extra `&&` clause.

> [!info] Reserved-headroom-UNUSED precedent extended (5 data points)
> 2. **v0.13.0 + v0.14.0 + v0.18.0 + v0.18.1 + v0.18.2 (reserved-headroom-UNUSED-or-near-unused) precedent now has 5 data points.** v0.18.2 was a clean first-pass with zero CFs. The pattern that reliably lands clean: **single-helper bug fix with quoted failing-state + quoted fix-shape in the design + master-driven mechanical edits + non-novel mechanism surface.** Decision rule reinforced: bug fixes scoped to one existing helper → expect clean; new mechanism / new UI surface → expect CFs.

> [!info] Master-driven cycles can be very fast
> 3. **The entire S1 was master-driven (1-line filter edit + 3 manifest version bumps + harness runs + commit) and took fewer turns than any prior cycle.** No subagent dispatches needed because every edit was mechanical and the design was specific enough that no judgment calls remained at execution time. Pattern: **master-driven shape is right when (a) edits are mechanical, (b) version-bump-lockstep is the only multi-file orchestration, and (c) no novel CustomJS class or template authoring.** Subagent dispatch reserved for blueprint/widget authoring or anything that benefits from fresh-context ideation.

> [!info] Carry-forward bugs: queue them with line numbers
> 4. **The fix took ~30 minutes from brainstorm-open to S1 commit because `blueprints i want.md` line 68-69 had a clear repro and clear scope.** Pattern: when the user surfaces a bug mid-cycle that's NOT this cycle's scope, file it in `blueprints i want.md` with a line number + clear repro instead of carrying it as memory. Future cycles inherit a checkable wishlist; the brainstorm starts at "which one?" instead of "what was the bug we talked about?"
