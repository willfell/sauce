## Why this cycle

The project blueprint's Projects-hub listing (ProjectsHubCards, shipped in v0.5.0 / S2.2) was a single class hard-coded against Dataview's pages API and a stacked auto-fit grid layout. Two downstream blueprints — meetings (MeetingsHubCards) and daily (SpaceDailyDashboard's meetings + tasks panels) — implement the same visual primitive (date-aware activity-sorted card listings) by independent ports of the same accuris source. Three classes, three slightly-divergent re-implementations, three places to fix when the visual contract evolves.

v0.11.0 extracts the shared primitive as the `cards` mechanism (BeaconCards CustomJS class) and ports the project blueprint's ProjectsHubCards to delegate against it. The mechanism is decoupled from Dataview semantics — callers pass a pre-resolved page array — so future non-Dataview callers (e.g. fs.readdir-sourced lists) can opt in. Mobile-awareness is owned at the mechanism level (gotcha 7: visual contract honored once, not three times). MeetingsHubCards and SpaceDailyDashboard panel migrations are AUDITED as ready (T3.1 audit doc) but DEFERRED to v0.5.x.y BEFORE any v0.6.0 next-blueprint cycle so the mechanism gets exercised by all three known consumers before its API ossifies.

The cycle also pays down two project-blueprint UX debts surfaced in v0.5.0 dogfooding: (1) "Structure" was an awkward rename of accuris's "Project Structure" — Map is the user-facing language, sweep "Structure" out of source; (2) atlas's "Project Notes" + "Referenced By" sections grew unboundedly noisy — wrap in auto-collapsed admonition. ProjectWorkstreams gaining per-workstream progress bars completes the visual-contract loop the cards mechanism opened.

The "iron everything out before moving on" framing extended scope inline: three CF (carry-forward) concerns surfaced at S2 close were sequenced as sub-stages CF-1..CF-3 rather than deferring to v0.11.x. The result is one MINOR bump from cards@0.1.0 to 0.1.1 with full landing across all in-cycle consumer code rather than a multi-cycle migration.

## Surprises during execution

> [!warning]- CF-1 (resolved): atlas dataviewjs called BeaconCards directly inline; violates landmines #1/#2 cold-load TDZ
> Initial S2 implementation had atlas dataviewjs blocks calling `customJS.BeaconCards.render(dv, {...})` directly inline. Per landmines #1 + #2, bare `customJS.X.Y(dv)` callsites cause cold-load `ReferenceError` during the customjs-plugin module-evaluation TDZ window: dataviewjs runs as soon as the page renders, but `customJS.BeaconCards` may not yet be bound. Dataview tracebacks the error inline; the user sees a "ReferenceError: BeaconCards is not defined" stack inside the atlas note — exactly the cold-load failure mode landmine #1 codifies.
>
> Tactical fix (in-cycle bump 1.3.0 → 1.3.1): ship two thin CustomJS wrapper classes — `ProjectNotesCards` + `ProjectReferencedByCards` — each ~25 LOC; delegate to BeaconCards with project-folder-derived queries; callable via `dv.view("Docs/Meta/Views/customjs-guard", { class: "ProjectNotesCards", ctx: { ... } })` from atlas dataviewjs. customjs-guard handles the TDZ retry; the wrapper class IS the customjs-guard call-target.
>
> Carry forward: any future use of BeaconCards inside dataviewjs MUST go through customjs-guard wrappers — never inline `customJS.BeaconCards.render(dv, ...)`. Document this as the canonical pattern in `Docs/how.md` next cycle (v0.11.x docs candidate).

> [!warning]- CF-2 (resolved): atlas "Project Notes" + "Referenced By" lists grow unboundedly noisy
> Atlas's "Project Notes" + "Referenced By" listings could grow to 50+ files for an active project (every Map child-note + every backlinking note) — drowning out the rest of the atlas. v0.5.0 left them as unbounded Dataview tables; v0.11.0 user-feedback flagged the noise.
>
> Tactical fix (in-cycle bump 1.3.1 → 1.3.2 partial; bundled with CF-3): wrap both sections in a single auto-collapsed `[!example]-` callout in `Project.md` template. Pure markdown change; no class changes. Default-collapsed hides noise by default; click-to-expand reveals when needed.
>
> Carry forward: atlas-style hub notes with multiple unbounded list contributions should default-hide via foldable callouts. Pattern is reusable for future blueprint atlas templates.

> [!warning]- CF-3 (resolved): ProjectsHubCards delegation produced stacked auto-fit grid; user wanted accuris Planning-Board row stack
> Initial ProjectsHubCards delegation to BeaconCards (S2 base, 1.3.0) produced a stacked-card auto-fit grid — visually different from accuris Planning-Board's row-stack pattern. User flagged: rows not grid; title+icon left, mtime+counts right, progress bar full-width across the row.
>
> Tactical fix (in-cycle bump cards 0.1.0 → 0.1.1; project 1.3.1 → 1.3.2 final; bundled with CF-2): extend BeaconCards (PATCH ADDITIVE — stacked layout default unchanged) with three new options: `icon` (string-or-getter for per-card icon SVG), `meta` (HTML-getter for per-card metadata block), `layout` (`"stacked"` default | `"row"`). New `_renderRowCard` + `_renderStackedCard` methods split out from the old single-render path; row layout is single-column flex with title+icon left / meta right / progress bar full-width below. ProjectsHubCards passes `layout: "row"` + briefcase icon getter + mtime/counts meta getter.
>
> Carry forward: future blueprint card-listings can opt into row layout for richer per-card metadata; stacked is still default for simpler lists. Reinforces gotcha 7 (mobile-aware visual contract owned at the mechanism, not duplicated per consumer).

> [!warning]- CF-A (deferred): Workstreams section briefly duplicates during _createProject before settling
> During `_createProject`, the Workstreams section in the projects atlas duplicates briefly (renders twice) before settling to a single section. Visual flicker only; no data corruption; settles within ~200ms.

## Pacing rhythm — what worked / what to improve

### What worked

- **Stage-close AskUserQuestion gates surfaced CF-1 + CF-2 + CF-3 inline.** Each addressed as S2.x sub-stage in the same session per gotcha 6 (in-cycle redesign methodology). Kept cycle continuity; no scope-creep into v0.11.x.
- **Per-task subagent dispatch on the Map rename sweep (T2.4).** Subagent ran across 4 files (`Project Map.md`, `Kanban Card.md`, `project-nav-buttons.js`, `project-workstream-manager.js`) and returned with zero "structure" hits remaining. Mechanical rename work parallelizes cleanly.
- **Master-driven mechanical edits interleaved efficiently with subagent class-authoring work.** Manifest version bumps + subscription bumps + workshop-side dogfood materialization handled by master while subagents authored class bodies; no sequencing bottleneck.
- **Bundling S2 + S2.2 + S2.3 + S2.4 into a single S2 commit (`bf7e565`)** preserved cohesion of the cycle — no per-CF micro-commits polluting history. Result writeup + audit doc reconstruct the sub-stage chronology when needed.
- **Audit doc (T3.1) BEFORE next-blueprint cycle.** Locking down which downstream classes are READY vs EXTEND vs MIGRATE before v0.6.0 starts prevents the cards mechanism API from ossifying without all three known consumers exercising it.

### What to improve

- **Initial scope under-counted in-cycle bumps.** Plan expected single project bump 1.2.2 → 1.3.0; actual was 1.3.2 (3 in-cycle bumps) plus cards 0.1.0 → 0.1.1. Six total bumps. Future plans should reserve "Y.Z.x" headroom upfront and not specify exact target version.
- **Pre-flight should have caught the workshop's own platform-subscription.json bump.** Self-install dogfood requires the workshop's `Docs/Meta/platform-subscription.json` (not just `platform/manifest.json`) to bump for the new mechanism to land in the workshop's installed ledger. Surfaced AT FIRST install run with "0 new history entries" — wasted ~30 sec round-trip. Pre-flight checklist should include "bump workshop self-subscription if introducing a mechanism the workshop hasn't subscribed to yet".
- **CF-1 (customjs-guard wrap requirement) should have been caught at design time.** Landmines #1 + #2 are well-documented; the design pass for ProjectsHubCards delegation should have noted "atlas dataviewjs cannot call BeaconCards directly — wrappers required". Caught at S2 smoke instead of design — round-trip cost ~one sub-stage cycle.
- **Mechanism-extraction cycles should ship with downstream-consumer audit BEFORE close.** v0.11.0 did this (T3.1 audit doc), but only because the user explicitly requested it. Codify as a standing pattern: any cycle that introduces a new shared mechanism must include an audit doc enumerating downstream consumers + their READY/EXTEND/MIGRATE state.
