# v0.64.x result — Daily-Note Activity Dashboard arc

## What shipped (final state at v0.64.3)

**User-visible features:**
- Every new daily note reliably contains `SpaceDailyDashboard` (collision with cowork's competing template resolved by daily blueprint reclaiming canonical ownership).
- `SpaceDailyDashboard` renders three collapsible sections:
  - **Today's Tasks** — bullet list with markdown-link + wikilink rendering (clickable inline anchors; LI click still opens the source note).
  - **Today's Meetings** — meeting cards.
  - **Today's Activity** — cross-blueprint activity feed grouped by blueprint type, anchored to the daily note's date (not `now()`), catches both created-today AND edited-today notes via `file.mtime` OR-in predicate. Each blueprint sub-group is its own collapsible `<details>` with a color stripe.
- Each main section has a colored left-border accent (cyan / blue / purple). Each Activity sub-group has its own blueprint-specific color (scratch=orange, project=green, person=purple, etc.).
- Smart title resolver — Activity cards prefer `title:` frontmatter → `aliases[0]` → first heading → filename fallback.
- `CoworkHubNav` row gone from per-day notes.
- `<!-- COWORK_CALLOUTS -->` anchor switched to `%% COWORK_CALLOUTS %%` — invisible in BOTH edit AND reading mode.
- Obsidian "Properties in document" now defaults to Visible (installed via `applyAppSettings` from workshop `app_settings.propertiesInDocument: "visible"`).

**Architecture:**
- Daily blueprint reclaimed `ranch/templates/Daily Note.md` ownership (cowork's `files[]` entry deleted). Daily template absorbed cowork-flavor frontmatter shape + Templater preamble + H1 + `%% COWORK_CALLOUTS %%` anchor.
- `SpaceDailyDashboard` extended with: third Activity panel; `_renderTaskHTML` markdown-link helper; `_getActivityCount` mtime-aware pre-count; `_DEFAULT_DASHBOARD_BLUEPRINTS` (10-type allowlist excluding meeting/daily/cowork-daily/cowork-weekly/cowork-monthly/scratch-day/to-do); `_BLUEPRINT_COLORS` (CSS-var map); `_resolveTitle` (defensive frontmatter → alias → outline → filename resolver wrapped in try-catch).
- `activity-feed@0.1.2` API: `render(dv, opts)` accepts 5 additive opts (`asOf`, `includeMtime`, `getTitle`, `collapsible`, `colorByType`) — all backwards-compatible.
- Cowork's prior `Daily Note.md` template + `files[]` entry + 5 SKILL.md files + 3 content files + 2 manifest descriptions all migrated to the `%% COWORK_CALLOUTS %%` syntax in one synchronized PATCH (v0.64.1).

### v0.64.3 — PATCH (shipped 2026-05-19T~22:10Z, closed final at user smoke)
**Title:** `_resolveTitle` defensive + drop `meeting` from Activity allowlist
**Versions:** workshop `0.64.2 → 0.64.3`; daily `0.5.2 → 0.5.3`
**Tap PR:** [#68](https://github.com/willfell/homebrew-sauce/pull/68) merged at user's pace

Two fixes from v0.64.2 headspace smoke:
- **BUGFIX** `_resolveTitle` was crashing with `aliases.values is not a function` on scratch + meeting notes lacking `aliases:` frontmatter. Dataview's `p.file.aliases` is a Proxy/DataArray where `.values` is a non-callable property (not Array.prototype.values), so the prior fallback `aliases.values && aliases.values()[0]` threw and aborted BeaconCards mid-render — activity-panel cards appeared as empty boxes. Fix: wrap resolver in try-catch (any throw falls back to filename) + simplified aliases probe to length-only (no `.values()` call).
- Allowlist drops `meeting` — already has its own dedicated top-level "Today's Meetings" panel; duplicate inside Activity was visual noise.

Single commit: `6d11d6c`. +1 sub-assert (DD-A6); DD-A4 widened to assert `meeting` absent.

## FIX-LATER-NOTEs

- **FLN-v64-1** (Important — code-quality reviewer flagged at S1): `_resolveTimeWindow(scope, asOf)` does not validate that `window.moment(asOf)` returns a valid moment. If `asOf` is garbage (e.g. `"banana"`), `.format()` yields `"Invalid date"` strings, the time window is non-null, `!timeWindow` guard doesn't fire, predicate silently filters out everything. Currently dormant: `SpaceDailyDashboard` extracts `asOf` from a filename-regex-validated date string. Worth `if (!now.isValid()) throw` guards in both branches.
- **FLN-v64-2** (Suggestion): `caseFA4TimelineManifests` in `run-helper-cases.js` uses `m.version >= floors[bp]` — string-lexicographic compare. Will silently false-negative once any timeline blueprint reaches a two-digit minor (e.g., `"0.10.0" < "0.5.0"` lexicographically). Replace with semver-aware comparison helper.
- **FLN-v64-3** (Suggestion): `run-activity-feed.js` shim's `textContent` recursive getter prepends `" "` between children; real DOM concatenates without separators. Irrelevant given `indexOf` substring asserts; worth a one-line comment for future maintainers.
- **FLN-v64-4** (Observation): all four consumer vaults were 5+ workshop versions behind at v0.64.0 cycle start (`workshop_version: 0.59.11`). The plan's S10 assumed one-version-bump cascades. Future cycles should call out "vault is N versions behind" as a baseline check.
- **FLN-v64-5** (Observation): No tap-PR auto-merge automation exists in `willfell/homebrew-sauce`. Past PRs merged manually by user within 1-10 minutes. Consider a tap-side workflow that auto-merges PRs from `bump-*` branches when CI is green.
- **FLN-v64-6** (Suggestion — v0.64.2 polish gap): Smart title resolver currently falls through to filename when frontmatter has no `title:` / `aliases:` / outline headings. For timestamp-named scratches like `Scratch-2026-05-19-11-58`, the user could add `aliases:` per scratch but it's manual. A future enhancement could parse the first `> [!abstract]` callout title from the body — but this requires reading the file body via `dv.io.load(p.file.path)` per page (expensive). Defer until users complain.
- **FLN-v64-7** (Observation — v0.64.3 root cause): The crash-discovery cycle (v0.64.0 → .1 → .2 → .3) was driven by per-stage smoke-on-user-data discovering edge cases the unit harnesses didn't catch. Pattern worth codifying: every CustomJS class that touches Dataview proxies should be wrapped in try-catch at boundaries, with filename-fallback semantics for any title-resolution path. The smoke surfaced two such gaps in three releases (`dv.pages` missing in v0.64.0; `aliases.values` throwing in v0.64.2).
- **FLN-v64-8** (Polish — deferred): Project blueprint notes weren't surfacing in the Activity panel during user smoke. Likely cause: their notes lack canonical `type: project` frontmatter OR `created_at:` in ISO format. Diagnose by inspecting a Denali project note's frontmatter shape. Defer to a future polish cycle.
