# v0.21.0 Consumer Bootstrap Implementation Plan

> **For the implementing agent:** REQUIRED SUB-SKILL: Use `de:executing-plans` (parallel session) OR `de:subagent-driven-development` (this session) to implement this plan task-by-task. Each task is bite-sized (2-5 minutes) with explicit surface, action, verification, and exit criterion. Cross-link the design at every section.

**Goal:** Add a Node-based interactive bootstrap orchestrator at `platform/bootstrap.js` so a fresh consumer vault goes from `git clone + cd` to "fully loaded, themed, configured, ready to use" with one shell command. Fetches required community plugins from GitHub releases, vendors them into `.obsidian/plugins/<id>/`, walks the user through subscription configuration via an interactive wizard, then invokes the existing Node installer.

**Architecture:** Five stages. S1 = this plan + design (already committed). S2 = schema + harness scaffolding (TDD-first). S3 = bootstrap implementation (parallel subagent dispatch per module). S4 = dogfood barebones + manual smokes in Obsidian. S5 = docs + cycle close. Workshop bump 0.20.0 → 0.21.0 MINOR. Allowlist 6 → 9 paths (USER APPROVED). New landmine #17.

**Tech Stack:** Node.js (built-in `https`, `fs`, `path`, `readline`), `@inquirer/prompts` (first npm dep in workshop), JSON (manifests + ledgers), Markdown (docs), Git.

**Design source:** `Docs/plans/2026-05-06-v0.21.0-consumer-bootstrap-design.md` (committed at `2b53089`).

// ---- HTTPS mock ----
function withMockedHttps(routes, fn) {
    const original = https.get;
    https.get = (url, opts, cb) => {
        if (typeof opts === "function") { cb = opts; opts = {}; }
        const u = typeof url === "string" ? url : url.href;
        const route = routes[u];
        const res = {
            statusCode: route ? (route.status || 200) : 404,
            on: (ev, h) => {
                if (ev === "data" && route && route.body) h(Buffer.from(route.body));
                if (ev === "end") setTimeout(h, 0);
            },
            headers: {}
        };
        cb(res);
        return { on: () => {}, destroy: () => {} };
    };
    try { return fn(); } finally { https.get = original; }
}

Common failure surfaces to anticipate (in order of likelihood):
- **HTTPS mock signature mismatch** — `https.get` callbacks have multiple shapes; align mock with what each lib module actually uses.
- **Backup-on-edit semantics** — distinguish `.bak` (overwrite, v0.2.0 boards) vs `.beacon-backup` (additive merge, v0.1.3+). Plugin files use `.beacon-backup` per allowlist expansion design.
- **Skip-if-present check granularity** — only `manifest.json` existence, not all 3 assets (per fetch-plugin contract).
- **Atomic write race** — write to `<file>.tmp` → rename. If process dies mid-write, no partial file.
- **runInstall integration** — child_process call must propagate vault path correctly + inherit stdio.
