# Design — multi-vault automation via thin-stub dispatch (v0.1.2)

> [!info] Predecessor relationship
> This design supersedes-in-part `Docs/plans/2026-05-03-multi-vault-automation-design.md`. The earlier design's **architectural decisions are retained** — pull-based distribution, thin-stub posture, tag-as-authority versioning, the `gitState()` helper concept. The earlier design's **execution path is re-cut** against post-v0.1.3 reality (workshop already a real git repo with GitHub remote; baseline manifest 0.4.1; landmine #12 codified; v0.1.3 helper patterns canonical) AND a deliberate **scope reduction** (lean scope; defer pinning, sync ritual, drift detection, cross-machine docs to follow-up cycles).
>
> Cite this doc — not the 2026-05-03 one — when implementing v0.1.2.

---

## Goal

Retire the bootstrap-copy × 3 resync ritual that's been load-bearing through v0.1.0 / v0.1.1 / v0.1.x / v0.1.3. Each consumer's `Docs/Meta/Templater/platformInstall.js` becomes a content-static **thin stub** (~12 lines) that dispatches at runtime to the workshop's canonical `platform/install.js`. Workshop becomes the single source of truth at runtime; cycles touching `install.js` no longer require md5-verified resync × 3.

## Lean scope (locked)

> [!success] In scope
> - **Stub-only dispatch** deployed to all 3 consumer vaults (workshop, barebones, tmp-acc).
> - **`gitState()` helper** in `platform/install.js` records `git_commit / git_tag / git_dirty` into `platform-installed.json` history at every install/update.
> - **CLAUDE.md + landmines updated** to formally retire the bootstrap-copy ritual; new landmines #13 (stub content-static) + #14 (gitState best-effort).
> - **Workshop_version 0.4.1 → 0.5.0**; tag `v0.5.0` annotated and pushed.
> - **Retro-tag `v0.4.1`** on the v0.1.3-close commit (`25ef276`) to establish baseline.

> [!warning] Out of scope (deferred to follow-up cycles)
> - `pinned_to` schema (head/tag/commit/branch) on subscription.
> - `auto_pull` config flag and automatic `git pull` from any code path.
> - `/sync-platform` slash command (or template-based sync ritual).
> - Drift detection harness mode.
> - Cross-machine setup doc.
> - accuris / headspace / ero onboarding.
> - GitHub Actions / CI on the workshop repo.
> - Mobile (iOS / iPadOS) support.
> - Backwards-compat shim for old bootstrap copies.

The CORE deliverable — retire the bootstrap-copy ritual — is achieved by stub deployment alone. Everything else is accretive and lands when actual demand surfaces (likely tied to non-dogfood consumer onboarding or sync-ritual UX work in v0.2.x).

## Architecture

### One-line summary

Stub dispatches; canonical `install.js` does everything else (including capturing git state into installed.json history). No `git pull` or `git checkout` happens automatically — pulling stays manual until v0.2.x sync ritual lands.

### File-level delta

```
WORKSHOP                                CONSUMERS (workshop, barebones, tmp-acc)
─────────────                           ──────────────────────────────────────
platform/install.js                     Docs/Meta/Templater/platformInstall.js
  + gitState() helper (~40 lines)         REPLACED with thin stub (~12 lines)
  + git_commit/tag/dirty in every          Bootstrap-copy md5 ritual RETIRED
    installed.history.push() site          Stub never re-edited after S2

platform/test/run-install.js
  (existing assertions unchanged;
   gitState() exercised by harness)

CLAUDE.md
  - "Bootstrap copies × 3 STILL IN FORCE"  (delete)
  + "Distribution model: stub-dispatch +
     canonical install.js (post-v0.1.2)"

Docs/landmines.md
  + #13 stub content-static; never re-edit
  + #14 gitState() best-effort; never throws
```

### What this kills

- Bootstrap-copy × 3 resync ritual.
- md5 verification step in every cycle that touches `install.js`.
- The cognitive overhead of "did I sync × 3?" before each commit.
- Workshop-as-implicit-subscriber-tracker (we don't need to know who's running which copy).

### What this preserves

- All existing install behavior (mechanism + blueprint loops, registry merge, all 4 v0.1.3 helpers including `applyTemplaterHotkeys` + `applySlashCommanderBindings`, failure-loud posture, six v0.1.0 hardenings, four v0.1.3 plugin-data safety mechanics).
- All existing consumer state files (`platform-config.json`, `platform-subscription.json`, `platform-installed.json`) — schema unchanged in lean scope EXCEPT history entries gain 3 git fields (additive; null-tolerant for pre-v0.5.0 entries).
- Workshop dogfood discipline — workshop self-installs via its own stub against its own canonical `install.js` (the meta-loop continues).

### Decisions locked

| Decision | Choice | Rationale |
|---|---|---|
| Stub posture | Stub-only dispatch (~12 lines) | Aligned with v0.1.3's "explicit user actions, no surprise side effects." Fewer failure modes baked into per-install path. Stub never re-edited. |
| Auto-pull behavior | None (no automatic pulling anywhere) | Lean scope; pulling is a separate ritual deferred to v0.2.x. |
| `pinned_to` schema | Not introduced in this cycle | No enforcement = no value in carrying the field. Lands when sync ritual lands. |
| `gitState()` failure mode | Best-effort; returns nulls; never aborts install | Install correctness must not depend on git correctness. |
| workshop_version bump | 0.4.1 → 0.5.0 | Distribution-model shift warrants minor bump. v0.5.0 sets clean line for future real-consumer onboarding. |
| Tag baseline | Retro-tag `v0.4.1` on `25ef276` (v0.1.3 close) | Establishes the missing tag history before v0.1.2 begins. |
| Migrate barebones + tmp-acc this cycle | Yes (S2 T2.1 swaps all 3 stubs uniformly) | Stub is content-static across consumers; uniform deployment is simpler than partial. tmp-acc's deliberate-lag is preserved at the *subscription* level, not the *bootstrap* level. |

## Schemas

### `platform-config.json` — unchanged

No `auto_pull`, no new fields. Existing schema (`workshop_relative_path`, `variables`) carried forward verbatim.

### `platform-subscription.json` — unchanged

No `pinned_to`. Existing schema (`workshop_version`, `mechanisms`, `blueprints`) carried forward verbatim.

### `platform-installed.json` — history entry gains 3 fields (additive)

```json
{
  "history": [
    {
      "event": "install",
      "kind": "mechanism",
      "name": "validator",
      "version": "0.1.1",
      "installed_at": "2026-05-04T...",
      "git_commit": "c1c64b4abc...",   // NEW — workshop HEAD sha at install time
      "git_tag": "v0.5.0",              // NEW — closest annotated tag at HEAD, or null
      "git_dirty": false                // NEW — true iff workshop had uncommitted changes
    }
  ]
}
```

**Backward-compat:** pre-v0.5.0 history entries don't have these fields. Any code that reads history must tolerate their absence — treat missing as `null` (semantically: "unknown," not "in sync" or "out of sync").

## The bootstrap stub (set once, never re-edited)

```javascript
// Docs/Meta/Templater/platformInstall.js  (~12 lines, content-static across all consumers)
module.exports = async (tp) => {
  const path = require('path');
  let cfg;
  try {
    cfg = JSON.parse(await tp.app.vault.adapter.read('Docs/Meta/platform-config.json'));
  } catch (e) {
    new Notice(`platformInstall: failed to read platform-config.json (${e.message})`, 10000);
    return;
  }
  const workshop = path.resolve(tp.app.vault.adapter.basePath, cfg.workshop_relative_path);
  const installer = path.join(workshop, 'platform', 'install.js');
  try { delete require.cache[require.resolve(installer)]; } catch {}
  return require(installer)(tp);
};
```

**Properties:**
- ~12 lines. Content-static across all 3 consumers — `diff` between any two stubs returns empty (the load-bearing invariant; codified in landmine #13).
- Reads only the consumer's `platform-config.json` to resolve workshop path; nothing else from consumer side.
- The `delete require.cache[...]` line is the load-bearing reload mechanism — without it, edits to canonical `install.js` won't take effect until Cmd+R Obsidian reload. Wrapped in try/catch (path-resolution edge cases tolerated).
- Failure modes:
  - Malformed `platform-config.json` → Notice + abort (no installer call).
  - Missing workshop dir / missing `install.js` → `require()` throws → Templater surfaces error.
  - All other errors → propagate from canonical installer (existing behavior).

## The `gitState()` helper

> [!info] Mirrors v0.1.3 helper posture
> `applyTemplaterHotkeys` and `applySlashCommanderBindings` (v0.1.3 S1) established the canonical helper pattern: failure-loud, additive, try/catch around every external call, never aborts the install on best-effort capture. `gitState()` follows the same template.

```javascript
// platform/install.js — new helper, ~40 lines including comments
function gitState(workshopPath) {
  const { execSync } = require('child_process');
  const result = { commit: null, tag: null, dirty: null };
  try {
    result.commit = execSync(`git -C "${workshopPath}" rev-parse HEAD`, { encoding: 'utf8' }).trim();
  } catch { /* not a git repo, or git unavailable; leave null */ }
  try {
    result.tag = execSync(`git -C "${workshopPath}" describe --tags --exact-match HEAD 2>/dev/null`, { encoding: 'utf8' }).trim() || null;
  } catch { /* HEAD has no exact tag; leave null */ }
  try {
    const status = execSync(`git -C "${workshopPath}" status --porcelain`, { encoding: 'utf8' });
    result.dirty = status.length > 0;
  } catch { /* leave null */ }
  return result;
}
```

**Wire-up at every `installed.history.push(...)` site:**

```javascript
const git = gitState(workshopPath);
installed.history.push({
  event: 'install',
  kind, name, version,
  installed_at: new Date().toISOString(),
  git_commit: git.commit,
  git_tag: git.tag,
  git_dirty: git.dirty
});
```

**Behavior matrix:**

| Workshop state | `commit` | `tag` | `dirty` |
|---|---|---|---|
| Clean, on tagged commit | sha | `v0.5.0` | `false` |
| Clean, between tags | sha | `null` | `false` |
| Uncommitted changes (no commit yet) | sha | tag-or-null | `true` |
| Not a git repo (legacy / scratch) | `null` | `null` | `null` |
| `git` binary missing on PATH | `null` | `null` | `null` |

Recording all-nulls is acceptable — the semantic is "we tried, no info available," not a hard failure.

## Failure modes (full table)

| Failure | Surface | Behavior | Recorded |
|---|---|---|---|
| Malformed `platform-config.json` | stub | Notice with parse error; abort before dispatch | (no installed.json write — too early) |
| `workshop_relative_path` resolves to non-existent dir | stub | `require()` throws ENOENT; Templater error path | (no write) |
| `workshop_relative_path` exists but no `platform/install.js` | stub | `require()` throws MODULE_NOT_FOUND; Notice surfaces | (no write) |
| `delete require.cache` throws (path-resolution edge) | stub | Try/catch silent; continues with possibly-cached module | (no log — best-effort) |
| Workshop is not a git repo | `gitState()` | All sub-calls throw; helper returns all-nulls | history entries record nulls |
| `git` binary missing on PATH | `gitState()` | Same as above — all nulls | history nulls |
| HEAD has no exact tag (between releases) | `gitState()` | `describe --exact-match` exits non-zero; `tag: null` | history records `tag: null` |
| Workshop has uncommitted changes | `gitState()` | `status --porcelain` non-empty; `dirty: true` | history records `dirty: true` (install proceeds) |
| Network down during `git pull` | N/A in lean scope | No automatic pulling happens | N/A |
| `pinned_to` mismatch | N/A in lean scope | No pin enforcement happens | N/A |
| All v0.1.0 / v0.1.1 / v0.1.3 hardenings | install.js | Carried forward unchanged | as before |

**Key posture:** `gitState()` never aborts the install. Worst case is all-null history fields. Aborting on git issues would couple install correctness to git correctness, which violates the lean principle that pulling is a separate ritual.

## Landmine updates

### Add: #13 — Bootstrap stub is content-static; never re-edit

> Each consumer's `Docs/Meta/Templater/platformInstall.js` is a ~12-line dispatcher set once during v0.1.2 S2. It MUST be byte-identical across all consumers (`diff` between any two stubs returns empty). The stub never re-syncs with `platform/install.js` — that file is now canonical-only and reached at runtime via `require()`. If a future cycle wants to change the stub's contract (config-file path, error telemetry, etc.), every consumer's stub must be updated in lockstep AND the change documented as a distribution-model bump.
>
> **Why:** the stub IS the new distribution mechanism. Drift in the stub breaks consumers silently — they'd dispatch to a different install.js path, or skip the require-cache clear, or read a different config. The stub's content-static invariant is the load-bearing replacement for the old md5-verified bootstrap-copy ritual.
>
> **What you can change without lockstep:** nothing. The stub is the smallest possible interface; if it needs change, every consumer needs change.
>
> Codified in v0.1.2.

### Add: #14 — `gitState()` is best-effort; must never throw

> The `gitState()` helper in `install.js` records workshop git revision into installed.json history. It wraps every `execSync` in try/catch and returns `{commit:null, tag:null, dirty:null}` on any failure. Install proceeds regardless of git state — even on a non-git workshop, even on a missing `git` binary.
>
> **Why:** install correctness must NOT depend on git correctness. The lean v0.1.2 model is "stub dispatches; install runs; we record what we can about workshop state." Coupling install success to git availability would block desktop-no-git scenarios and break workshop dogfood if anyone wiped `.git/`.
>
> **What this means for callers:** code that READS `installed.history[].git_commit` etc. must tolerate `null` for entries written before v0.5.0 OR by a non-git install. Drift-detection (future cycle) treats `null` as "unknown," not "in sync" or "out of sync."
>
> Codified in v0.1.2.
