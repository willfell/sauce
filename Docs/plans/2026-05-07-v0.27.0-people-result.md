## What shipped

### NEW `people-rendering@0.1.0` mechanism

Single class `PeopleRendering` (~302 LOC) with four-method API:
- `renderChip(parent, personLink, opts) → HTMLSpanElement` — inline chip with hover tooltip from frontmatter; mobile-aware initials when narrow viewport; red-tinted "Unknown person" fallback for unresolved links.
- `renderCard(dv, personLink, opts) → void` — per-person BeaconCards row card via synthetic-page composition.
- `renderMentionList(dv, query, opts) → void | Promise<void>` — two modes: `mentioning_person` (Dataview chain `dv.pages(scopePath).where(...).sort(...).slice(0, limit)`) or `mentioned_in_note` (reads body via `app.vault.adapter.read` + filters via `extractMentions`). Three render styles: `chips` / `cards` / `list`.
- `extractMentions(markdownBody, opts) → Array<{display, target}>` — regex-extracts `[[Person]]` wikilinks; filters to `spice/people/` resolved targets only; `unique=true` dedupes by canonical target.

Path-prefix `spice/people/` hardcoded per landmines #11 + #19. Mobile-aware per v0.4.2 gotcha 7.

### NEW `people@0.1.0` blueprint

Module directory `people` → materializes at `spice/people/` per landmines #11 + #19. Per-person template (`Template, People.md`) + canonical hub (`spice/people/People.md`) + 3 CustomJS classes:
- **`PeopleHubCards`** (37 LOC) — alphabetical cards grid; empty-state placeholder; embed-deduped.
- **`NewPersonButton`** (110 LOC) — AccentButton + overlay-dialog with name validation; Templater `create_new_note_from_template` with race-tolerance try/catch (`/already exists/i` swallow → `openLinkText`); embed-deduped; mobile-aware overlay sizing.
- **`PersonNavButtons`** (62 LOC) — identity row (Lucide user icon + name + tag-mode chip) + back-link AccentButton; embed-deduped + previous-instance cleanup via `:scope > .pnb-root`.

Templater folder-template registered for `spice/people` → `Template, People.md`. Global "People" nav-button (Lucide users icon, order 60). Migration-faithful frontmatter minimum: `company / location / title / email / website / aliases / phone / tags`.

## In-cycle CFs (review-surfaced; fixed pre-S2-commit)

**IMP-1: `_renderMentionedInNote` chip target bug** — chips were rendered into `dv.container` directly instead of the intended flex-wrap `row` div. User-visible on every meeting note's `## Attendees` block. **Fix:** changed `this.renderChip(dv.container, ...)` → `this.renderChip(row, ...)` at people-rendering.js:243.

**IMP-2: MeetingsHubCards over-extracted attendees** — `peopleAttendeeLinks` derived from `extractMentions(content)` over the entire meeting body, not just the `## Attendees` section. Caused (a) People mentioned in `## Notes` / `## Action Items` to render as attendee chips, and (b) `unregistered = p.attendees.length - p.peopleAttendeeLinks.length` to go negative when more People-references existed in the body than in `## Attendees`. **Fix:** scoped extraction to `attendeesMatch[1]` substring + clamped `unregistered` to `Math.max(0, ...)` at meetings-hub-cards.js:55-58 + 106.

**IMP-3: meetings depends_on coupled to `people` blueprint** — forced People hub + Templater folder-template + nav-button install on every consumer subscribing to meetings, even those who don't want a People hub. Pilot only needs the rendering mechanism. **Fix:** dropped `{ "name": "people", "range": ">=0.1.0" }` from meetings depends_on; kept `people-rendering >=0.1.0` + `cards >=0.2.4`. meetings-hub-cards.js already gracefully falls back when PeopleRendering is unavailable, so the dep was over-committal.

## NEW lessons (v0.27.0)

(a) **Two-stage subagent review caught 3 IMPORTANT findings pre-commit** — 8 data points reaffirmation. The single most-load-bearing review pattern in the project's cycle history. Spec reviewer found 6 NITs (none blocking); quality reviewer found 3 IMPORTANT bugs that would have shipped silent UX failures: (a) chips escaping flex-wrap container, (b) over-extracted attendees + negative unregistered count, (c) meetings over-coupled to people blueprint. Spec-only review would have shipped all three.

(b) **5 parallel implementer subagents landed clean first-pass with literal-contract prompts** — second cycle confirming v0.21.1 lesson a (after v0.19.0 styling 3-parallel). Pattern: when N independent helpers share posture (CustomJS class + customjs-guard reach + per-blueprint module_directory) and insertion points are far apart, parallel dispatch saves wall-time vs sequential without quality cost. Five subagents totalling ~611 LOC of new code; zero merge conflicts; zero re-review cycles needed.

(c) **Manifest schema alignment with project precedent matters** — first barebones-install caught two manifest fields drifting from canonical shape (`depends_on` strings vs `{name, range}` objects; `src` vs `source`; missing `kind` + `customjs_classes`). The v0.5.0 / v0.18.0 / v0.19.0 / v0.21.0 / v0.21.1 / v0.22.0 / v0.26.1 manifests are the schema reference. Every NEW manifest in a future cycle must `diff` against an existing manifest of the same kind before commit.

(d) **Plan-design drift on `space-nav-buttons.js`** — design assumed SpaceNavButtons had a `context` parameter routing system based on the v0.5.0 `projects-hub` cycle, but that context routing actually lives in `project-nav-buttons.js` (a per-blueprint class), not in the shared `space-nav-buttons.js` mechanism. Build-time discovery; resolved by mirroring `Projects.md` hub pattern (invoke `NewPersonButton` directly above `PeopleHubCards`) instead of adding a context branch. Lesson: design's "mirror v0.X.0 context branch precedent" must be VERIFIED by reading the cited file before locking the build path.

(e) **Bounded-pilot integration scope was the right call** — wiring meetings as the only pilot consumer (not also daily / journal / projects in v0.27.0) bounded the CF surface to one cross-blueprint integration. Future integration cycles (v0.27.x) inherit the proven API contract for cheap PATCH adoptions. Validates Q2 of brainstorm (Section 1 architecture).

(f) **Migration-faithful frontmatter minimum was the right call** — locking schema = exactly what Accuris/Ero/Headspace already have (`company / location / title / email / website / aliases / phone / tags`) keeps v0.28.0 migration to a 1:1 frontmatter translation. No invented fields means no schema-evolution risk at migration time.

(g) **Quality review IMP-2 surfaced a "section-scoped extraction" pattern worth codifying** — extracting from the FULL body instead of a specific markdown section is a reliable way to misattribute references. Future renderers that aggregate body-references should default to section-scoped extraction (passing only the relevant `## <Heading>` substring to the extractor) unless full-body context is intentional.

(h) **Quality review IMP-3 surfaced "blueprint vs mechanism dep" decision rule** — when a pilot consumer needs the RENDERING surface from a new blueprint+mechanism pair, depend ONLY on the mechanism. Adding the blueprint as a dep forces install of the blueprint's full ecosystem (hub + templates + nav-button) onto every downstream consumer, even those that don't want it. Future cross-blueprint integration cycles: depend on `*-rendering` mechanism, not the blueprint itself, unless the consumer genuinely needs the blueprint's data namespace materialized.
