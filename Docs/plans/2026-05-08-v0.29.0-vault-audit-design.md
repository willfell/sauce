# v0.29.0 — Vault audit cycle (design)

## 1. Goal

Ship a programmatic `sauce audit` CLI verb that surfaces per-blueprint structural conformance gaps + untracked top-level directories across all 3 migrated consumer vaults (`accuris-sauce`, `ero-sauce`, `headspace-sauce`). The audit is **detection-only**: violations are reported, not auto-fixed. Once shipped + tagged, a separate post-cycle session-series uses the audit reports as a work-list to manually edit vault content into a clean baseline. Once that baseline lands, future cycles (v0.30.0+) ship new functionality (journal-migrator, full Sauce-shape project ecosystem, etc.) against vaults known to be conformant.

## 2. Non-goals

- **No auto-fix tooling.** No `sauce audit --fix`, no `sauce conform` interactive verb. Fixes happen by hand (or by Claude in subsequent sessions making file edits directly). Auto-fix is a separate feature surface that can be designed once we know what real-world violations actually surface.
- **No coverage of journal/to-do/boards/finance blueprints this cycle.** Their `rule_fragments[]` stay empty. Deferred to v0.29.1 PATCH cycles after the v0.29.0 plumbing proves out on the 5 high-content blueprints.
- **No vault-cleanup work as part of v0.29.0 cycle close.** The cleanup is multi-session, post-tag, tracked separately in `Docs/plans/2026-05-08-vault-baseline-rollup.md` (skeleton ships at v0.29.0 close; sessions append to it).
- **No new migrators.** journal-migrator, finance-migrator, full Sauce-shape project ecosystem (Atlas/Map/Board/tasks subdir generation) deferred to v0.30.0+.
- **No allowlist mechanism.** v0.29.0 hard-flags every top-level dir outside the sanctioned set. Per-vault allowlist (`ranch/audit-allowlist.json`) deferred to v0.29.1 if signal-to-noise across the cleanup sessions warrants it.

## 4. CLI surface

### Verb registration

`sauce audit` slots in `platform/cli/sauce-cli.js` VERBS registry between `migrate` (#5) and `help` (#6).

### Invocation

```
sauce audit                            # run against cwd if it's a sauce vault
sauce audit --vault <path>             # run against a specified vault
sauce audit --blueprint trips          # filter to one blueprint (helpful for incremental cleanup)
sauce audit --output-file <path>       # write markdown report to file (default: stdout)
sauce audit --no-untracked-check       # skip top-level untracked-dir scan
sauce audit --quiet                    # exit-code only, no output (for CI / scripting)
```

### Exit codes

- `0` — zero violations + zero untracked dirs
- `1` — one or more violations / untracked dirs found
- `2` — invocation error (vault is not a sauce vault, can't read filesystem, missing platform-installed.json)

### Output mode

- **Default:** markdown to stdout (pipeable to `>` a file).
- **`--output-file <path>`:** writes markdown report to that path AND prints a one-line summary to stdout (`audit: <V> violations, <U> untracked dirs — written to <path>`).

### Vault validation

Audit refuses to run unless target dir contains `ranch/platform-installed.json`. Mirrors `cmd-status.js` posture. Prevents accidental run against arbitrary directories.

### Read-only contract (NEW landmine #21)

`cmd-audit.js` makes zero filesystem mutations to the audited vault. The only exception: `--output-file <path>` if the path falls inside the audited vault — and that's user-explicit. Code review must reject any `writeFileSync` / `appendFileSync` / etc. in `platform/audit/*` whose target is rooted in the audited vault path. Mirrors landmine #20's posture for `sauce migrate` source vaults.

**Two surfaces per blueprint covered this cycle:**
- **Entity rule** — fires on the canonical "this is a <thing>" note (e.g., `Trip Atlas.md` per trip slug, `Project.md` per project slug)

**Untouched this cycle:** sub-section notes (Trip Flights/Stay/etc.) and task notes (under `<slug>/board/`). Could surface their own rule fragments in v0.29.x; deferred.

## 7. Validator rule schema extensions (NEW this cycle)

The existing validator (`validate.js:84-100`) supports `required` + `type` per key. To express the rules above, we extend the schema **additively** (no break to existing rule files):

| New field | Used on | Behavior |
|---|---|---|
| `equals` | `required_frontmatter[key]` | Exact-value match (e.g., `equals: "trip"`) |
| `matches` | `required_frontmatter[key]` | Regex match against scalar string value |
| `contains` | `required_frontmatter[key]` (type=list) | All listed values must appear in array |
| `scope.path_glob` | rule_fragment | Glob the rule applies to (instead of `module:` frontmatter) |
| `scope.exclude_basenames` | rule_fragment | Excluded basenames within scope |
| `frontmatter_branch[]` | rule_fragment | Branch on `when` predicate (frontmatter shape) — picks first match |

**Compatibility:**
- **Audit walker (Node):** uses these directly via `rule-runner.js`.
- **In-vault validator (Templater):** `validate.js` patched to read these new fields. **Read-side compatible:** old rules without `equals`/`matches`/`contains`/`scope`/`frontmatter_branch` still work (the patches are additive checkers).
- **Risk:** validator extensions tested via `run-audit.js` synthetic fixtures; in-vault validator regression check in S3 dogfood (run `/validate` slash command on workshop dogfood note; verify pre-cycle + post-cycle behavior identical).

## 8. Data flow & error handling

### Pipeline

```
cmd-audit.js
  ├─ parse args                     (--vault, --blueprint, --output-file, --no-untracked-check, --quiet)
  ├─ validate vault                 (stat <vault>/ranch/platform-installed.json; exit 2 if missing)
  ├─ read platform-installed.json   → installed blueprints
  ├─ read ranch/rules/_global.json (optional) + ranch/rules/<bp>.json (per installed bp)
  ├─ walker.js                      → iterator of {path, relPath, frontmatter, body, blueprint}
  │    ├─ for each installed bp: walk spice/<bp>/**/*.md
  │    └─ scan top-level dirs (skipped if --no-untracked-check), flag any not in sanctioned set
  ├─ rule-runner.js                 → for each file, apply matching rule_fragments (path_glob match + frontmatter_branch resolution)
  ├─ report.js                      → format by-blueprint sections + untracked-dirs section + summary
  └─ output                         → stdout default, OR file (--output-file <path>) with stdout summary line
```

### Error handling

| Condition | Behavior |
|---|---|
| Vault missing `platform-installed.json` | Exit 2: `"not a sauce vault: <path>"` |
| Blueprint installed, `ranch/rules/<bp>.json` missing | Warning in report; continue (audit doesn't crash, just notes "rules file missing — install via `sauce update`") |
| Malformed YAML frontmatter in a file | Record as violation `rule="frontmatter_parse"`; continue walking |
| File read error (permissions) | Record as violation `rule="file_unreadable"`; continue |
| Glob pattern syntax error in a rule fragment | Exit 2 (programming error in workshop) |
| Unrecognized predicate field in rule fragment | Warning; rule still applies recognized fields (forward-compat for v0.29.x extensions) |
| `--output-file <path>` parent dir missing | Exit 2: `"output dir doesn't exist: <dir>"` |

## 12. Post-cycle: vault baseline rollup phase

After v0.29.0 tag, the cleanup work begins in subsequent sessions. **Not part of v0.29.0 cycle close.**

### Protocol per cleanup session

1. **Snapshot first:** copy target vault to `<vault>.pre-cleanup-<YYYYMMDD-HHmmss>/`. Mirrors v0.28.0 `.pre-migration-<ts>/` discipline.
2. **Run `sauce audit <vault> --output-file <vault>/ranch/audits/<YYYY-MM-DD-HHmmss>-audit.md`** to capture starting state.
3. **Walk the audit report top-down**, fix violations one by one. Claude makes file edits directly. Each fix is committed (if vault is git-managed) or simply persisted.
4. **Re-run audit** at session end. Capture closing report. Append delta-summary to `Docs/plans/2026-05-08-vault-baseline-rollup.md` (NEW per-session entry with: vault, before-violation-count, after-violation-count, what-was-fixed, what-remains).
5. **Iterate across sessions** until all 3 vaults report zero violations + zero untracked dirs (or every untracked dir is intentional + accepted as user-owned residue).

### "Done" criterion for the rollup phase

All 3 vaults pass `sauce audit --quiet` with exit 0. (Or: untracked-dirs are explicitly accepted per-vault and would be allowlisted in a future `audit-allowlist.json` if v0.29.1 ships that mechanism.)

After rollup is done, v0.30.0 opens with new functionality: journal-migrator, full Sauce-shape project ecosystem, etc. — exercised on barebones first, then propagated to the 3 conformant vaults.
