# v0.53.0 — FA-1 Foundation cycle (result)

Foundation cycle for the Frontmatter Alignment workstream. Ships cross-cutting infrastructure only — zero blueprint manifest changes. After this cycle, the canonical frontmatter vocab is documented + enforced (in the audit walker + Layer 2 install rule) + measurable, even though no blueprint has migrated yet.

## What shipped

**NEW artifacts:**
- `platform/rules/_canonical-vocab.json` — shared rule template consumable via `extends: "_canonical-vocab"`. Declares the canonical audit triplet (`type` + `created_at`) with ISO-8601-with-TZ pattern, plus walker/migrator config fields (`legacy_keys`, `wikilink_keys`, `discriminator_tags`, `temporal_tag_patterns`, `preserve_tags`).
- `platform/migrations/v0.53-frontmatter.json` — declarative migration spec consumed by `cmd-migrate-frontmatter`. Encodes renames (`created`→`created_at`, `attending`→`people`, `product`→`products` plural-wrap, `month`→`month_label` on cowork-monthly), date reformat keys, tag cleanup rules, wikilink quote keys, backfill heuristics.
- `platform/cli/cmd-migrate-frontmatter.js` (~530 LOC) — NEW `sauce migrate-frontmatter` verb. Default dry-run writes `<vault>/sauce-migration-report.md`. `--apply` performs in-place rewrites with `.sauce-backup/<rel>/<timestamp>/` sidecars per touched file. Reuses audit walker's minimal YAML parser scope.
- `platform/audit/frontmatter-alignment-walker.js` — NEW audit walker with 6 finding severities (legacy_key_used HIGH / non_iso_timestamp HIGH / unquoted_wikilink MEDIUM / missing_canonical_key MEDIUM / discriminator_tag_present INFO / temporal_tag_present INFO).
- `platform/test/run-migrate-frontmatter.js` — 50 sub-asserts covering migration verb correctness.
- `platform/test/run-validator.js` — 9 sub-asserts covering extends loader.

**Mechanism bumps:**
- `validator@0.2.0 → 0.3.0` MINOR — NEW `extends:` field on rule fragments. Per-fragment loader in `platform/audit/rule-runner.js` + top-level loader in `platform/mechanisms/validator/validate.js`. Failure-soft.
- `audit@0.2.1 → 0.3.0` MINOR — NEW `frontmatter-alignment-walker.js` + `sauce audit --frontmatter-alignment` CLI flag (third pass alongside `--claude-surface` and `--entity-create`).

**Installer rule:**
- NEW `_validateCanonicalVocab` Layer 2 rule in `platform/install.js` — checks each blueprint manifest for either `rule_fragments[*].fragment.extends="_canonical-vocab"` opt-in OR `canonical_vocab_opt_out: { reason: "..." }`. Forward-defense at v0.53.0: emits warning row into install history but does NOT fail install. Promotes to hard fail in a post-FA-7 cycle once all consumer vaults are migrated.

**Workshop version:** `0.52.1 → 0.53.0` MINOR.

## FIX-LATER notes

1. **FLN-1 — Canonical-vocab template materialization.** At v0.53.0 the canonical-vocab template lives only at `<workshop>/platform/rules/_canonical-vocab.json`. The rule-runner's vault-first → workshop-fallback resolution means consumers reach the template via the brew-installed workshop without per-consumer materialization. FA-2 onward, if a blueprint's rule_fragments declare `extends:`, the installer SHOULD materialize the template into `<vault>/ranch/rules/_canonical-vocab.json` (per landmine #11 consumer-vault namespacing). Plumbing this requires either: (a) validator mechanism `files[]` references the platform-level file via `../../rules/_canonical-vocab.json`, (b) install.js adds a `applySharedRuleTemplates` helper that materializes platform-level rule templates as a pre-step. Decide at FA-2 design.

2. **FLN-2 — In-vault validate.js extends loader untested.** The mirror at `platform/mechanisms/validator/validate.js loadRule` was extended to follow top-level `extends:` but is not exercised by `run-validator.js` (node-only harness; no Obsidian app stub). Inspection-tested only. FA-2 lights it up when the first blueprint opts in; if defect surfaces, patch-cycle.

3. **FLN-3 — `unquoted_wikilink` detection scans raw fm lines.** The minimal YAML parser strips outer quotes on parse, so a parsed value `[[Alice]]` could come from either `- [[Alice]]` or `- "[[Alice]]"`. The walker scans the raw fm lines for the unquoted form. Block-form (`key:\n  - [[X]]`) is well-covered; inline-form (`key: [[[X]]]`) is not detected (inline-array shape is outside the minimal parser scope anyway). Reasonable scope limit.

4. **FLN-4 — `missing_canonical_key` simplification.** The walker fires on the narrow case "type is set but created_at absent." The design spec suggested broader "blueprint emits canonical key by template but this note omits it" — that would require per-blueprint template-emission expectations encoded somewhere. Deferred; the narrow form is load-bearing for the v0.53.0 + FA-2 use case (entity notes should have audit triplet).

5. **FLN-5 — `_validateCanonicalVocab` warning posture.** At v0.53.0 the rule fires a warning row for every blueprint (none opt in). Warning rows are not visible in normal install output (they go into install history JSON). Promote to a hard fail in a post-FA-7 cleanup cycle once all consumers are migrated. Until then, the warning serves as a deferred trigger.

6. **FLN-6 — Date coercion uses local TZ at install machine.** `coerceIsoWithTz` infers the local TZ offset at coercion time. Re-running the migration on a different machine (or after a TZ change) would produce different offsets for partial-form inputs (e.g. `2026-05-15` without time). This is acceptable for a one-shot migration but worth noting if FA-2..FA-7 introduce per-blueprint re-runs.

7. **FLN-7 — `BLUEPRINT_MODULE_DIRS` duplicated between walker.js + walker.js (audit) + cmd-migrate-frontmatter.js.** Three files each carry their own `project → projects` map. Should be extracted to a shared helper. Cosmetic.
