# v0.63.x result — to-do All-To-Dos + Migrate dialog

## What shipped (final state at v0.63.2)

**User-visible features:**
- `spice/to-do/All-ToDos.md` hub note — live Dataview aggregator showing uncompleted tasks from prior days (excludes today), grouped by source-date desc, with `(from ToDo-YYYY-MM-DD)` source links.
- Inline `← Back to Today's To-Do` AccentButton on the hub note (via `ToDoHubActions`).
- Inline `All To-Dos` + `Migrate` AccentButton row on every daily `ToDo-YYYY-MM-DD.md` note (via `ToDoLeafActions`).
- Migrate-to-tomorrow modal — multi-select picker for unchecked tasks; selected tasks (plus indented children) move to tomorrow's note, creating tomorrow from template if absent; modal auto-opens tomorrow in the workspace after submit.
- Auto-recovery: `ToDoLeafActions.openAllToDos()` detects a deleted/empty hub file and re-writes the canonical body inline before opening.

**Architecture:**
- 5 new CustomJS classes: `ToDoHubActions`, `ToDoLeafActions`, `ToDoAllList`, `ToDoMigrateModal`, `ToDoMigrateInit`.
- 1 new template: `All To-Dos.md` materialized at `spice/to-do/All-ToDos.md`.
- Today To-Do.md template updated to embed `ToDoLeafActions` dataviewjs block.
- Pure-blueprint cycle — zero platform mechanism changes; leans on existing infra (nav-button `openLink` / `runTemplaterTemplate` action types, `customjs_startup_scripts[]` (v0.49.0), AccentButton mechanism).
- Manifest schema additions for to-do: `customjs_classes[]`, `customjs_startup_scripts[]`, `depends_on += accent-button`.
- Rule-fragment scope tightened: `spice/to-do/**/*.md` → `spice/to-do/**/ToDo-*.md` so the new hub file isn't validated against the date-naming pattern.

## The arc — three releases in one afternoon

### v0.63.0 MINOR (CLOSED, 2026-05-19 ~17:12 UTC)

Original cycle. Shipped per design + plan. Tag: `v0.63.0` → tap PR #61 merged → brew upgrade verified.

**Found at headspace smoke:**
- `customJS.ToDoMigrateModal is not a constructor` — TypeError on Migrate command invocation. Root cause: customjs plugin AUTO-INSTANTIATES each registered class, so `customJS.ToDoMigrateModal` is the singleton instance, NOT the class constructor. `new customJS.X()` fails.
- Global nav-button scope undesirable — `All To-Dos` + `Migrate` surfaced on every note (correct per the design's stated choice, but the user's preference reversed during smoke).

### v0.63.1 PATCH (CLOSED, 2026-05-19 ~17:45 UTC)

Two-fix bundle. Tag: `v0.63.1` → tap PR #62 merged → brew upgrade verified.

- `todo-migrate-init.js`: `new customJS.ToDoMigrateModal().open()` → `customJS.ToDoMigrateModal.open()` (singleton method call).
- NEW `ToDoLeafActions` CustomJS class renders All-To-Dos + Migrate as inline AccentButtons in `Today To-Do.md`. Both removed from manifest `nav_buttons[]` (where they were global). The `todo-today` "create today's note" button stays global (correct quick-create semantics).
- `Today To-Do.md` template gets a second dataviewjs block calling `ToDoLeafActions`.

**Found at headspace smoke:**
- When tomorrow doesn't exist yet, the modal creates it inline via `_renderTomorrowTemplate` — but that template body was missing the new `ToDoLeafActions` block, so auto-created tomorrow notes were missing the inline buttons.
- Modal silently emitted a Notice toast after migration; user couldn't see WHERE the migrated tasks went. Felt like a failure.
- `openAllToDos` called `app.workspace.openLinkText` directly, which creates an empty placeholder if the target is missing. Installer treats `{{module_directory}}` destinations as idempotent (skip-if-exists), so deleted-then-re-clicked hub stays empty forever.

### v0.63.2 PATCH (CLOSED, 2026-05-19 ~18:03 UTC)

Three-fix bundle. Tag: `v0.63.2` → tap PR #63 merged → brew upgrade verified → headspace smoke PASSED end-to-end.

- `_renderTomorrowTemplate`: add the `ToDoLeafActions` dataviewjs block between SpaceNavButtons and `## Tasks`.
- `_submit`: after both `vault.modify` calls, await `openLinkText(tomorrowFile.path)` so the workspace switches to tomorrow's note.
- `openAllToDos`: pre-flight check for missing / empty / aggregator-block-missing file; write canonical body inline; emit Notice on recovery; then open.

## Lessons learned

1. **Singleton-vs-constructor pitfall is a real platform gotcha** — the v0.63.0 design + plan + spec body all baked in `new customJS.X()` without anyone catching it. Code-quality review noted the lazy lookup as "intentional, confirmed" without noticing the constructor mismatch. Worth promoting to CLAUDE.md or Docs/how.md.
2. **Headspace smoke ≠ workshop preflight.** The customJS singleton bug + the nav-button scope concern + the "where did tomorrow go" UX issue all passed CLI-level preflight cleanly. Only manual smoke surfaced them. Continuing the v0.48.0/v0.49.0 lesson: scriptable verification is necessary but not sufficient.
3. **Idempotent installer destinations can become trap doors.** `{{module_directory}}/All-ToDos.md` correctly resists overwriting user-customized hub notes, but the same behavior prevents recovery from accidental deletion. Helpers can paper over this (per v0.63.2's `openAllToDos`); a deeper fix needs installer support.
4. **Three-way cycle-number collisions happen in multi-session workflows.** Detected by checking git log + grep before committing. Cheap to resolve early; expensive to resolve after implementation has landed under the wrong number.
5. **`git add <specific-file>` doesn't prevent already-staged-from-another-session bundling.** The S5 implementer's commit accidentally bundled a 1980-line cowork-scheduling plan because it was in the index from a parallel session's `git add`. Workaround: `git status --short` before commit to confirm what's staged.
