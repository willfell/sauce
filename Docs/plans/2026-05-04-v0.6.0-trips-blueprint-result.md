## Outcome summary

> [!abstract] What landed
> A `trips` blueprint that scaffolds a per-trip directory tree from a "New Trip" overlay dialog (name + start_date + end_date + location). Each trip gets:
> - **Atlas note** at `beacon/trips/<slug>/<TripName>.md` with frontmatter (`type: trip`, `start_date`, `end_date`, `location`, `tags: [trip]`, `cssclasses: [wide]`).
> - **Five baseline sections** (Flights, Stay, Packing List, To Do, Notes) — bare-shape (frontmatter + nav-button blocks; no body text).
> - **Trip Board** at `<slug>/board/<slug>-board.md` — Kanban-plugin board with To Do / In Progress / Completed lanes; cards added to lanes auto-promote to folder-style (`<slug>/board/<TaskName>/<TaskName>.md`) via Templater (mirrors project blueprint's task-folder pattern, simplified).
>
> The hub at `beacon/trips/Trips.md` lists all trips as BeaconCards row-layout cards grouped into Current Trip / Upcoming Trips / Past Trips. Per-trip atlas renders a TripSectionsCards grid grouping sections into Default Sections (Flights, Stay, Packing List, To Do, Notes, Trip Board) and Additional Sections (anything user-added via the New Section button).
>
> Per-trip nav buttons auto-discover sibling sections at render time so user-added sections get nav coverage with zero manifest edits. Multi-row layout: atlas big button (row 1, only on non-atlas), default sections + Trip Board (row 2), additional sections (row 3, only on non-atlas — atlas suppresses to avoid redundancy with the cards grid).

> [!abstract] Why this matters
> First cycle to ship a "per-entity ecosystem" blueprint with a substantial CustomJS surface — three classes, eight templates, an overlay dialog with native `<input type="date">` fields, and a per-entity Kanban board. The cycle exercised cloning the project blueprint's pattern (per-slug-subfolder + atlas + sections + Kanban board with auto-promote cards) into a NEW domain, validating that the platform's per-entity-blueprint shape generalizes.

## In-cycle CFs — chronological

> [!example]- CF-1 (commit 006cc0b): TripsHubCards BeaconCards API + atlas filter
> **Smoke trigger:** Smoke B failed — TripsHubCards rendered "Nothing here yet." instead of trip cards.
>
> **Root cause:** TripsHubCards was calling `customJS.BeaconCards.render(dv, items, opts)` (3-arg shape with hand-built items array) but BeaconCards' actual API is `render(dv, opts)` with field-functions reading from `opts.pages`. The items array was silently treated as `opts`; `opts.pages` undefined → defaulted to `[]` → BeaconCards' empty-state rendered.
>
> **Fix:** rewrote render() to use the field-function API matching ProjectsHubCards precedent (title/icon/subtitle/meta/target/sort/empty as functions; pages opt holds the dataview pages array). Same upcoming-first-then-past sort logic preserved; past-trip dimming via meta-text color rather than card opacity (BeaconCards has no per-page dim primitive).
>
> **Bonus:** dataview query was `'"beacon/trips" AND #trip'` which would also match section files (Flights/Stay/etc. all had `tags: [trip]`). Tightened to `.where(p => p.type === "trip")` so only atlas notes (frontmatter `type: trip`) appear as cards.
>
> **Bump:** trips 0.1.0 → 0.1.1 PATCH.
>
> **Spec lesson:** Should have read BeaconCards' interface during design phase, not just inferred from a sibling helper (ProjectsHubCards). Codified for future cycles touching new mechanisms — RTFM the API surface.

## In-cycle surprises (4)

> [!warning]- Surprise S-1 (CF-1): BeaconCards API mismatch
> **What:** TripsHubCards called `render(dv, items, opts)` (hand-built items array as 2nd arg) but BeaconCards' actual API is `render(dv, opts)` with field-functions and `pages` opt. items array was silently treated as opts; opts.pages undefined → empty state rendered.
>
> **Why missed in design:** I described the field shape inside the design doc (title/subtitle/meta as field-functions per BeaconCards' contract — see ProjectsHubCards pattern) but the implementer subagent generated an items-array shape because the contract wasn't restated as concrete code in the spec — the prompt allowed inference. Should have provided a literal API contract example or pointed the subagent at the file directly.
>
> **Codification:** future blueprints touching BeaconCards (or ANY existing mechanism) should include a literal `customJS.<MechanismName>.render(...)` call shape in the design doc, not just describe field semantics. RTFM > infer.

## Lessons (NEW)

> [!info]- 1. Existing mechanism API contracts must be quoted, not described
> CF-1 root cause was inferring BeaconCards' API from a sibling helper rather than reading the actual signature. Future cycles touching existing mechanisms should include a literal `customJS.<X>.render(...)` call shape in the design doc, not "field-function API matching ProjectsHubCards precedent." Quote the contract.

> [!info]- 2. Per-entity-ecosystem blueprints exercise inline-CF pattern heavily
> When shipping a NEW entity-ecosystem (trips, finance, future), expect 3-5 inline CFs at minimum. The user's mental model for the entity's UX surfaces only when they actually click around in the smokes. Design phase locks the SHAPE; CF iterations refine the FEEL. Don't fight this — plan for the headroom.

> [!info]- 3. Migration friction is real for in-cycle CFs that change template content
> CF-3 + CF-4 changed the atlas template body and section template bodies. Existing trips created via earlier CFs retain the old body content (templates are read at trip-creation time, not retroactively applied). Smoke instructions had to call out "delete and recreate" as the cleanest path. For future cycles: design templates as bare-as-possible from the start; reduce the surface area that subsequent CFs would need to rewrite.

> [!info]- 4. Per-entity sub-folders (e.g. <slug>/board/) live within landmine #11's invariant
> Trip Board lives at `beacon/trips/<slug>/board/<slug>-board.md`. That's a sub-folder under a per-entity sub-folder under the module directory. Landmine #11 says "every blueprint owns ONE directory at `beacon/<module_directory>/`" — this still holds. Sub-folders inside `beacon/trips/` are per-entity instances, not sub-modules. The project blueprint already established this pattern at `beacon/projects/<slug>/tasks/<task>/`. Trips extends with `beacon/trips/<slug>/board/<task>/` — same shape, no new invariant.

> [!info]- 5. Native Kanban-plugin auto-promote is "free" via Templater integration
> The Trip Board Card template uses the same Templater `tp.file.move` trick from project's Kanban Card template (~simplified — no workstream picker). The Kanban plugin invokes the Templater template at card creation; Templater moves the file to folder-style. No new mechanism, no installer extension. Future entity blueprints that need per-task folders can reuse this pattern.

> [!info]- 6. Cmd+R discipline reinforced (gotcha 2)
> User's H/I failures were stale class registration — Cmd+R hadn't been done after the CF-3 install. Smoke instructions explicitly call out Cmd+R as a prereq, but humans miss steps. Future smoke checklists for cycles introducing NEW CustomJS classes should make Cmd+R a numbered step (not just a callout). The new-class-introduction case is the highest-risk for stale registration.
