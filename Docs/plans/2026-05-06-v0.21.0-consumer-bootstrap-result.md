## NEW lessons (v0.21.0)

> [!example]+ Six new platform values codified

**(a) First-of-its-kind orchestration cycles draw 4-6 in-cycle CFs at integration boundaries.** Harness mocks can't fully simulate real-world vendor behavior: 302 redirects (CF-1), missing thin-stub (CF-4), depth-mismatched relative paths (CF-3), schema-default expectations (CF-5), output volume (CF-2). The 7-data-point UI precedent (1-6 CFs for first-of-its-kind UI cycles) extends to non-UI orchestration cycles too — call it 8-data-point first-of-kind precedent. Plan reserved 0.21.x headroom upfront; used CF-1 through CF-5 + a styling PATCH unstick.

**(b) `process.exit` truncation when piped is a real footgun in CLI tools.** Children using `console.log` + `process.exit(N)` lose buffered stdout when stdio is a pipe (Node's default async stdout buffering doesn't flush on exit). Canonical fix: `process.exitCode = N` + let main return naturally so the event loop drains. Surfaced in CF-2 when bootstrap's spawn pipe captured ~1900 of ~4100 lines of run-install.js output. Worth grepping `platform/test/run-*.js` for other `process.exit` callsites if a future CLI wrapper needs full output capture.

**(c) Async `spawn` over `spawnSync` when capturing child stdout.** `spawnSync` collects stdout chunks but the close event timing differs from async `spawn`. With async, the `close` event fires AFTER stdout EOF; with sync, races on child exit can drop final chunks even with the `process.exitCode` fix in place. Pattern: capture child stdout via `child.stdout.on("data", ...)` + resolve on `close` event.

**(d) Atomic write + backup-on-overwrite + path-traversal validator are the canonical posture for any new state-materializing helper.** Quality-review findings #1/#3/#4 distilled the v0.1.3 helper posture into 3 reusable invariants:
- `_writeAtomic(target, content)`: write `<target>.tmp` + renameSync (rename is atomic on POSIX)
- `_writeWithBackup(target, content)`: copyFileSync to `.beacon-backup` BEFORE write; backup-failure throws (mirrors `applyTemplaterHotkeys`); calls `_writeAtomic`
- `_validatePluginPath(root, dir, id)`: id matches `/^[a-z0-9][a-z0-9._-]*$/i` AND `path.relative(root, dir)` does not escape with `..` AND result is not absolute
Future bootstrap-lib additions (or any helper that vendors files outside `.obsidian/plugins/<id>/data.json`) should adopt this stack from the start, not retrofit at quality-review time.

**(e) Defer-then-validate-then-load is the right wizard architecture for chicken-and-egg path resolution.** When a config field gates downstream prompts/data loads, gate the load behind the field's prompt+validator. CF-3 root cause: bootstrap.js used to pre-read the workshop manifest using a hardcoded `../beacon` default BEFORE the wizard prompted for the actual path. From `/workshop/scratch/1`, the right relative is `../../beacon`; the manifest read failed and the wizard never got to ask. Fix: pass `workshopManifest=null` to the wizard for first-run; wizard prompts + validates path + loads manifest itself. Pattern reusable for any future wizard prompt that yields a path used by subsequent prompts.

**(f) Defensive augment-on-every-run for new schema fields makes patches retroactive.** CF-5: older configs lacking `views_path` / `templates_path` etc. would never get them via wizard re-runs (re-run path doesn't re-prompt). Fix: bootstrap.js's per-run setup includes a `CANONICAL_VARIABLES` augment-if-missing pass that adds new keys without overwriting user-supplied values. Pattern: any future schema extension where backward compatibility is desired can ship its own augment-if-missing pass alongside the schema bump. Avoids forcing manual hand-edits AND avoids a heavyweight migration tool.
