# v0.63.0 ‚Äî Cowork Scheduling Cycle ‚Äî Design

**Date:** 2026-05-19
**Cycle:** Cowork scheduling cycle (closes deferred work from v0.42.0 + FLN-3 from v0.45.0)
**Workshop baseline:** v0.62.0
**Workshop target:** v0.63.0 (MINOR ‚Äî atomic-note write contract + skill rewires + /cowork readiness panel)
**Upstream context:**
- `Docs/plans/2026-05-13-v0.42.0-cowork-timeframes-skeleton-design.md` (declared the deferred scheduling cycle and locked the prompt-stub path convention)
- `Docs/plans/2026-05-11-v0.31.0-cowork-engagement-model-design.md` (engagement-id resolution + orchestrator contract)
- `Docs/plans/2026-05-14-v0.45.0-result.md` (FLN-3: orchestrator SKILL.md bodies still reference daily-blueprint paths ‚Äî closed here)
- `Docs/plans/2026-05-18-v0.62.0-fa-9a-design.md` (ActivityFeed mechanism ‚Äî the read surface this cycle plugs into)

## Why now

Three things converged to unblock this cycle:

1. **v0.42.0 deferred a "scheduling cycle"** to wire runtime invocation against the 4 prompt-stub placeholders at `spice/cowork/prompts/{morning-briefing,eod-review,weekly-review,monthly-review}.md`. The path convention has been locked since then; bodies stayed empty.
2. **v0.45.0 closed cowork's self-contained refactor** (the path convention `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD.md` landed) but flagged FLN-3: orchestrator SKILL.md bodies still wrote to the old `spice/daily/` callout-patching surface. The atomic-note write contract here is what closes that gap.
3. **v0.62.0 shipped the ActivityFeed mechanism** (cross-blueprint reverse-query by canonical `type:` + `created_at:`). Without it, surfacing run-notes on Daily/Weekly/Monthly Hubs would need bespoke per-hub Dataview queries; with it, the wiring is a `_DEFAULT_BLUEPRINTS` widening.

The cycle is the **structure** that lets a consumer connect Claude Cowork to their sauce-installed vault, invoke any `cowork:*` orchestrator, and have the output land in a deterministic place that the rest of the platform already surfaces.

## Out of scope (deliberate)

- **Filling the 4 user-editable prompt bodies** at `spice/cowork/prompts/*.md`. They stay empty / user-owned. Orchestrators emit a no-op note with `warning: empty_prompt` when run against an empty stub ‚Äî lets the user *see* the wiring works before customizing.
- **Scheduler-config code.** No launchd/cron/GitHub-Actions templates ship as code. The onboarding doc includes config snippets, but registering scheduled jobs in Claude Cowork app (or any scheduler) is consumer-owned.
- **Per-engagement aspect routing inside orchestrators.** The v0.31.0 engagement-model design contemplated branching on `engagement.type` (`personal` / `w2-fte` / `consulting`) inside each orchestrator. Deferred to a future v0.31-completion cycle ‚Äî this cycle keeps orchestrators engagement-aware only via `engagement_id` resolution + frontmatter emission.
- **Sprint-sync (or any other consumer-specific job).** The 5th scheduled job in the user's current accuris setup (ADO sprint-sync) stays consumer-local. Sauce ships the **consumer-extension pattern** (worked example: copy `write-run-note-finance`) but not the sprint-sync skill itself.
- **Replacing the existing per-hub Cards panels with ActivityFeed.** Daily Hub keeps `CoworkDailyHubCards` etc. ActivityFeed is additive (existing pattern from v0.62.0).

## Section 1 ‚Äî Path & frontmatter contract (the sauce-shape)

### Per-timeframe atomic-note paths

Each scheduled job writes to a deterministic path. Path is a pure function of `(orchestrator, day | week | month)`:

| Orchestrator | Path |
|---|---|
| `cowork:morning-briefing` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/morning-briefing.md` |
| `cowork:midday-tripwire` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/midday-tripwire.md` |
| `cowork:eod-review` | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/eod-review.md` |
| `cowork:weekly-review` | `spice/cowork/weekly/YYYY/YYYY-Www/weekly-review.md` |
| `cowork:monthly-review` | `spice/cowork/monthly/YYYY/YYYY-MM/monthly-review.md` |
| Consumer-extension (e.g. `cowork:sprint-sync`) | `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/<slug>.md` (same shape) |

The per-day folder (`spice/cowork/daily/.../YYYY-MM-DD/`) is the same folder the existing v0.45.0 cowork Daily Note co-locates with. This cycle just declares that folder as a contract and adds sibling run-notes to it.

### Frontmatter shape (canonical-vocab-aligned per FA-5)

```yaml
type: cowork-morning-briefing        # or cowork-eod-review / cowork-midday-tripwire / cowork-weekly-review / cowork-monthly-review / cowork-finance-snapshot
created_at: "2026-05-19T07:05:14-06:00"   # ISO+TZ at write time
engagement_id: "accuris"             # resolved from vault-config.md engagements[]
day: "2026-05-19"                    # daily / midday / eod / finance only
week: "2026-W20"                     # weekly-review only
month: "2026-05"                     # monthly-review only
generator: "cowork:morning-briefing@1.0.0"   # orchestrator + version (for forensic / FLN tracking)
prompt_source: "spice/cowork/prompts/morning-briefing.md"   # which prompt file body was read
```

Additional required field for `cowork-midday-tripwire`:
```yaml
severity: "yellow" | "red"
```
(orchestrator skips writing when severity would be `green` ‚Äî no note = green; presence of a tripwire note = something to flag.)

### Body shape (loose)

Whatever the prompt body instructs Claude to emit ‚Äî Markdown freeform. No required headings. ActivityFeed cards key off frontmatter + first H1/H2, not body structure. Consumer is free to define structure inside their prompt bodies.

### Idempotency

Overwrite (last write wins). Re-running `cowork:morning-briefing` at 8:00 AM after the 7:05 AM run replaces the file. Path is deterministic per `(orchestrator, day)` so there's no race-with-self. Daylight-savings / TZ-boundary edges resolve via `cowork:date-context` (single source of truth) not local clock.

### Engagement scoping

Engagement lives in **frontmatter (`engagement_id:`), not in the path.** Path stays simple even as multi-engagement vaults arrive. ActivityFeed already supports filtering by frontmatter; surfaces that want engagement-aware grouping add a `groupBy: "engagement_id"` option in a later cycle (out of scope here).

### Six new canonical `type:` values

Registered in the FA-5 canonical-vocab catalogue:

| Type | Owner | Required fields |
|---|---|---|
| `cowork-morning-briefing` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source` |
| `cowork-midday-tripwire` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source`, `severity` |
| `cowork-eod-review` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator`, `prompt_source` |
| `cowork-finance-snapshot` | cowork | `type`, `created_at`, `engagement_id`, `day`, `generator` (prompt_source optional ‚Äî finance is gather-driven) |
| `cowork-weekly-review` | cowork | `type`, `created_at`, `engagement_id`, `week`, `generator`, `prompt_source` |
| `cowork-monthly-review` | cowork | `type`, `created_at`, `engagement_id`, `month`, `generator`, `prompt_source` |

Each gets a rule_fragment validating the required fields + path-prefix expectation.

### What stays untouched

- The cowork Daily Note (`spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD.md`) and its `## Notes` section remain user-owned. Orchestrators **never write to it**.
- The cowork Weekly Note (`spice/cowork/weekly/YYYY/YYYY-Www.md`) and Monthly Note (`spice/cowork/monthly/YYYY/YYYY-MM.md`) shells remain user-owned.

## Section 2 ‚Äî Components

### 2.1 Six `write-run-note-*` sub-skills (replaces 5 `write-callout-*` + 2 `write-summary-*`)

The legacy `write-callout-*` sub-skill family was built to patch callouts inside the daily-blueprint's daily note (FLN-3 from v0.45.0 flagged this as residual technical debt). This cycle replaces them with `write-run-note-*` siblings that emit the Section 1 contract:

| Old (deletes) | New (replaces) | Writes to | `type:` value |
|---|---|---|---|
| `write-callout-morning-briefing` | `write-run-note-morning-briefing` | `‚Ä¶/YYYY-MM-DD/morning-briefing.md` | `cowork-morning-briefing` |
| `write-callout-eod-review` | `write-run-note-eod-review` | `‚Ä¶/YYYY-MM-DD/eod-review.md` | `cowork-eod-review` |
| `write-callout-tripwire-yellow` + `write-callout-tripwire-red` | `write-run-note-midday-tripwire` (severity in frontmatter) | `‚Ä¶/YYYY-MM-DD/midday-tripwire.md` | `cowork-midday-tripwire` |
| `write-callout-finance` | `write-run-note-finance` | `‚Ä¶/YYYY-MM-DD/finance.md` | `cowork-finance-snapshot` |
| `write-summary-weekly` | `write-run-note-weekly-review` | `‚Ä¶/YYYY-Www/weekly-review.md` | `cowork-weekly-review` |
| `write-summary-monthly` | `write-run-note-monthly-review` | `‚Ä¶/YYYY-MM/monthly-review.md` | `cowork-monthly-review` |

`write-run-note-finance` is the **reference implementation** the consumer-extensions doc points at. Its skill body is the most copy-pasteable starting point for a consumer-defined atomic-note writer (e.g. sprint-sync).

### 2.2 Six orchestrator SKILL.md rewires

The 6 existing orchestrators get their step lists updated to call the new sub-skill names:

- `cowork:bootstrap-vault` ‚Äî calls `cowork:scaffold-timeframes` (unchanged) + emits stub finance snapshot on first run via `write-run-note-finance`
- `cowork:morning-briefing` ‚Äî terminates in `write-run-note-morning-briefing` (was `write-callout-morning-briefing`)
- `cowork:midday-tripwire` ‚Äî terminates in `write-run-note-midday-tripwire` (was branched yellow/red callout writes)
- `cowork:eod-review` ‚Äî terminates in `write-run-note-eod-review` (was `write-callout-eod-review`)
- `cowork:weekly-review` ‚Äî terminates in `write-run-note-weekly-review` (was `write-summary-weekly`)
- `cowork:monthly-review` ‚Äî terminates in `write-run-note-monthly-review` (was `write-summary-monthly`)

No orchestrator step **count** changes (still ~5‚Äì7 steps each); only the final write-step's sub-skill reference + the read of `prompt_source` upstream of write.

### 2.3 ActivityFeed default-blueprint widening

`activity-feed@0.1.0 ‚Üí 0.2.0` MINOR. The only code delta is `_DEFAULT_BLUEPRINTS` adding the 6 new `type:` values. Daily/Weekly/Monthly Hubs + Today.md (all of which call ActivityFeed with default blueprints) auto-pick up run-notes with zero hub-side edits.

### 2.4 SpaceDailyDashboard 3rd panel

`daily@0.4.0 ‚Üí 0.5.0` MINOR. `space-daily-dashboard.js` grows a "Today's Cowork Activity" section under tasks + meetings. Implementation pattern (mirrors existing tasks + meetings panels):

```js
const getCoworkRuns = () => {
  const folder = `spice/cowork/daily/${moment(today).format("YYYY/MM-MMMM")}/${today}`;
  const pages = dv.pages(`"${folder}"`)
    .where(p => typeof p.type === "string" && p.type.startsWith("cowork-"))
    .sort(p => p.created_at, "asc");
  return pages.array();
};
```

Renders as compact bullet list ‚Äî each item: `{HH:mm} <orchestrator label> ‚Üí [[link]]`. Hidden when empty (matches the existing both-empty short-circuit pattern). Re-uses the same panel chrome (SVG icon + header + container styles).

### 2.5 `CoworkReadiness` CustomJS helper + Cowork.md embed

New helper at `helpers/cowork-readiness.js`, materialized to `ranch/scripts/cowork/cowork-readiness.js`. Renders a 4-row panel:

| Row | Source | Status renderings |
|---|---|---|
| Engagement | `vault-config.md` `engagements[]` first entry | ‚úì `<id>` / ‚úó "Not bootstrapped ‚Äî run cowork:bootstrap-vault" |
| Prompts | Vault listing of `spice/cowork/prompts/*.md` | "4/4 present" + per-prompt body-length indicator (empty stub vs customized) |
| MCP routing | Reads `spice/cowork/.routing-cache.json` (written by `check-vault-routing` runs) | ‚úì ready / ‚úó "missing:gmail" etc. + "last checked: <timestamp>" |
| Last runs | ActivityFeed query (scope=today, blueprints=cowork-*) per orchestrator | per-orchestrator "last run: 7:05 today" / "(never)" |

Embedded in `Cowork.md` as a new section block above the existing Engagements row:

```dataviewjs
await dv.view("ranch/views/customjs-guard", { class: "CoworkReadiness" });
```

`/cowork` slash command body gets a one-line update telling the user the hub now surfaces live readiness.

### 2.6 Six rule_fragments

One per new `type:` value, mirroring the existing `cowork-daily` / `cowork-weekly` / `cowork-monthly` fragment pattern. Each declares:
- `extends: _canonical-vocab`
- `required_frontmatter` (per Section 1 table)
- `path_pattern` regex matching the atomic-note location

### Error handling

- **`check-vault-routing` returns non-`ready`** ‚Üí orchestrator records failure to `.routing-cache.json` and exits. **No partial run-note is written.** Surfaces in `/cowork` readiness panel.
- **Write step fails partway** ‚Üí orchestrator retries the single MCP write call once, then bails. Frontmatter-only / empty-body notes are never produced.
- **Prompt body empty (stub state)** ‚Üí orchestrator emits a no-op note with `body: "(prompt body empty ‚Äî edit spice/cowork/prompts/morning-briefing.md)"` and a `warning: empty_prompt` frontmatter field. Lets the user *see* the wiring works before customizing.

### Idempotency / overlapping runs

- Re-run same day ‚Üí deterministic path, overwrite.
- Two jobs at 7:05 AM and 7:08 AM ‚Üí different file slugs, no contention.
- Daylight-savings / TZ edge ‚Üí `day:` from `cowork:date-context`, not local clock.

### Read-side cadence

- Daily / Weekly / Monthly Hubs + Today.md auto-refresh via Dataview's reactive query (no manual reload).
- SpaceDailyDashboard cowork panel uses the same `dv.pages()` pattern as tasks/meetings ‚Äî picks up new run-notes within a Dataview refresh cycle.
- `/cowork` readiness panel reads `.routing-cache.json` for routing status (cheap), live-queries ActivityFeed for last-run timestamps.

## Section 4 ‚Äî Onboarding, consumer extensions, migration

### Onboarding ritual (the "structured manner")

5-step checklist for connecting Claude Cowork to a sauce-installed vault ‚Äî surfaced both in `Docs/cowork-onboarding.md` AND mirrored live by the `/cowork` readiness panel:

| Step | Action | How readiness panel reflects it |
|---|---|---|
| 1 | Install sauce + subscribe to cowork blueprint | implicit (panel renders = blueprint installed) |
| 2 | Run `cowork:bootstrap-vault` to create `vault-config.md` + engagement | `Engagement: ‚úì accuris` |
| 3 | Customize the 4 prompt bodies (or leave empty ‚Äî orchestrators emit stub notes) | `Prompts: 4/4 present (N empty stubs)` |
| 4 | Connect Claude Cowork app: mount vault as MCP, register orchestrators as scheduled jobs (config snippet in doc) | `MCP routing: ‚úì ready` |
| 5 | First scheduled run lands ‚Üí readiness panel shows last-run timestamps | `morning-briefing: last run 7:05 today` |

The panel is idempotent ‚Äî it's the same panel a user opens on day 90 to sanity-check that runs are still landing. Onboarding is just "day 1's reading of the same panel."

### Consumer-extension pattern (sprint-sync as worked example)

Documented in `Docs/cowork-consumer-extensions.md`. Steps for a consumer adding their own scheduled job (e.g. `sprint-sync` against ADO):

1. Define a new `type:` value (e.g. `cowork-sprint-sync`) in a consumer-local rule_fragment (`.local/` override per landmine #22).
2. Author a skill body at `.claude/skills.local/cowork-sprint-sync/SKILL.md`. Use `write-run-note-finance/SKILL.md` (shipped in sauce) as the copy-paste reference template.
3. The skill writes an atomic note at `spice/cowork/daily/YYYY/MM-MMMM/YYYY-MM-DD/sprint-sync.md` with the canonical frontmatter shape.
4. Register `cowork:sprint-sync` (or any consumer-namespaced ID) as a scheduled job in Claude Cowork app.
5. Optional: add `cowork-sprint-sync` to ActivityFeed via a per-vault config override (or accept the default fallback ‚Äî ActivityFeed renders unknown types under a generic group).

The doc explicitly notes that `write-run-note-finance` ships in sauce as the canonical reference because finance is consumer-flexible (open-ended snapshot vs structured review).

### Migration from current accuris setup

| Current accuris job | Becomes | Migration action |
|---|---|---|
| `morning-daily-summary` @ 7:05 | `cowork:morning-briefing` @ 7:05 | Rename in Claude Cowork app. Output redirects to `spice/cowork/daily/.../YYYY-MM-DD/morning-briefing.md` (no more daily-note template writes). |
| `eod-daily-summary` @ 17:09 | `cowork:eod-review` @ 17:09 | Same. |
| `weekly-summary` @ Fri 04:03 | `cowork:weekly-review` @ Fri 04:03 | Output ‚Üí `spice/cowork/weekly/.../YYYY-Www/weekly-review.md`. |
| `morning-sprint-sync` @ 7:08 | Consumer-extension `cowork:sprint-sync` | Stays consumer-local; ADO MCP specifics out of sauce scope. Doc walks through. |
| `eod-sprint-sync` @ 17:04 | Consumer-extension `cowork:sprint-sync-eod` | Same. |

Legacy `Timestamps/Summary/2026-05-19/daily-summary-2026-05-19-tuesday.md` path goes silent post-migration. Old files remain (read-only history); new flow writes to the sauce-shape only.

### MCP setup recommendation

Current setup (`obsidian-mcp` + Claude Cowork project mounted at vault root) **stays**. Reasoning documented in the onboarding doc:

- `obsidian-mcp` understands Obsidian semantics (wikilinks, frontmatter, daily-notes folder); filesystem MCP would lose that.
- Mounting at vault root keeps Claude's working-directory matching the canonical `spice/cowork/...` paths the contract uses ‚Äî no path translation layer needed.
- The platform's `cowork:check-vault-routing` already probes `obsidian` as a backend; no new MCP plumbing.

What the doc *does* document: how to register scheduled jobs in Claude Cowork app with the right `engagement_id` argument, and how to confirm the vault path is correct.
