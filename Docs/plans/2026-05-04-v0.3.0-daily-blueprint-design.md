# Design — `daily` blueprint + core-plugin-settings schema (v0.3.0)

> [!abstract] Goal
> Ship the `daily` blueprint mirroring accuris's `Timestamps/` daily-note pattern verbatim, materialized at `beacon/daily/YYYY/MM-MMMM/YYYY-MM-DD-DayOfWeek.md`. Driven by Obsidian's **core Daily Notes plugin** ("Open today's daily note" command) plus the **Calendar community plugin** (browse-by-day UI). Adds an installer schema for writing core-plugin data files (`.obsidian/<id>.json`) — the first time the platform manages core-plugin settings, expanding landmine #12's allowlist by one path.

> [!info] Why this design exists
> v0.2.0 closed with a working blueprint methodology + a real shipped blueprint (`boards`). The user's stated next-cycle direction is "build a NEW blueprint that I'll use daily." Daily notes are the highest-frequency surface in any vault — they become the platform's anchor and the natural cross-reference target for every other module. Mirroring accuris's pattern (rather than designing fresh) preserves muscle memory + delivers the dashboard-driven cross-module surface that makes daily notes valuable.

## Decisions locked during brainstorming (2026-05-04)

> [!success] Approved choices
> 1. **Daily blueprint shipped alone in v0.3.0.** No sibling blueprints (meetings, to-do) in scope this cycle. The dashboard tolerates missing siblings (renders empty meetings + tasks sections; never errors). Future v0.3.x cycles add meetings + to-do; the dashboard auto-populates without daily-blueprint changes.
> 2. **Verbatim accuris daily pattern.** Path `beacon/daily/YYYY/MM-MMMM/YYYY-MM-DD-DayOfWeek.md`; frontmatter `{ created, tags: [<vault_identity_tag>, daily, YYYY/MM/DD], cssclasses: [wide] }`; body two dataviewjs blocks (`SpaceNavButtons` + `SpaceDailyDashboard`) + `## Notes` section.
> 3. **Entry-point UX is the core Daily Notes plugin** — built-in "Open today's daily note" command + Calendar community plugin for browse-by-day. NO custom slash command, NO custom Templater hotkey, NO smart-Today nav-button. The user already uses this UX in accuris; v0.3.0 mirrors it.
> 4. **Installer expands allowlist by ONE path** — `.obsidian/daily-notes.json` (core plugin's data file). Calendar plugin's data.json holds only weekly-notes + display prefs (NOT daily-related — Calendar reads `daily-notes.json` directly), so it's NOT added to the allowlist. v0.3.x can add Calendar's data.json later if display-pref defaults become valuable.
> 5. **New manifest schema field `core_plugin_settings[]`** declares which core-plugin data files to write + the merge shape. Distinct from the existing `external_plugins[]` (community-plugin presence check) and `templater_hotkeys[]` / `slash_commander_bindings[]` (per-template config writes).
> 6. **`SpaceDailyDashboard` ported with hardcoded paths** to `beacon/meetings` + `beacon/to-do` (graceful-empty when absent). YAGNI on per-consumer override; codify only when a consumer actually needs different paths.
> 7. **Workshop_version bump 0.6.0 → 0.7.0** (minor — additive contract: new schema field + new helper + allowlist expansion). Pre-approved by the brainstorming flow; reconfirm at plan-writing.
> 8. **2-stage cadence** mirroring v0.2.0 (S1 installer extension; S2 blueprint + deploy + manual smokes + tag).

> [!info] Inherited from prior cycles
> - JSON-everywhere (landmine #6).
> - Desktop-only (landmine #8).
> - Failure-loud posture (every error path Notice + history entry; full git fields per landmine #14).
> - Module-directory invariant under `beacon/` namespace (landmine #11; codified in installer per v0.2.0 S1).
> - Stub content-static (landmine #13).
> - `.obsidian/` ban-lift requires four safety mechanics (additive merge, backup-on-edit, malformed-JSON guard, failure-loud history) per landmine #12.

### Data flow at note-creation time

1. User runs the command **"Daily notes: Open today's daily note"** (Cmd+P → "Open today's daily note") OR clicks a date in Calendar plugin.
2. Daily Notes core plugin reads `.obsidian/daily-notes.json` for `folder` + `format` + `template` settings.
3. If today's note doesn't exist:
   - Plugin computes path: `<folder>/<format-applied-to-today>.md` → `beacon/daily/2026/05-May/2026-05-04-Monday.md`.
   - Plugin reads the template at `template` path → `Docs/Meta/Templates/Daily Note.md`.
   - Plugin writes a new file at the computed path with the template body.
   - Templater's "Trigger Templater on new file creation" hook fires → evaluates `<% %>` tokens (`tp.file.creation_date`, `tp.date.now`, etc.) in the template body → final note content materialized.
4. If today's note exists: plugin opens it (no template re-application).
5. The materialized note's two dataviewjs blocks render at first view:
   - `SpaceNavButtons` — pulls registry from `Docs/Meta/nav-buttons-registry.json` and renders the nav-bar.
   - `SpaceDailyDashboard` — looks for sibling content under `beacon/meetings` + `beacon/to-do`, renders empty until those blueprints land.
6. The `## Notes` section is the user's free-form daily journal.

> ```dataviewjs
> await dv.view("{{views_path}}/customjs-guard", { class: "SpaceDailyDashboard" });

>

>
> Substitutions at install time (lenient body sub):
> - `{{vault_identity_tag}}` → consumer's `platform-config.json:variables.vault_identity_tag` (e.g., `barebones`).

>
> Notes:
> - `module_directory: "daily"` — installer derives `{{module_directory}} → "beacon/daily"` per the v0.2.0 T1.2 mechanic.
> - `core_plugin_settings[].settings.folder` references `{{module_directory}}` (substituted at install via the per-blueprint overlay) — yields `"beacon/daily"`.
> - `core_plugin_settings[].settings.template` references `{{templates_path}}` (consumer's templates path, e.g., `Docs/Meta/Templates`) — yields `"Docs/Meta/Templates/Daily Note.md"`.

> - `customjs_classes: ["SpaceDailyDashboard"]` — declarative; the CustomJS plugin auto-scans the scripts folder + picks up the class. The list is informational/documentation.

## Installer extension (S1 scope)

### New manifest field: `core_plugin_settings[]`

Schema:
```json
{
  "core_plugin_settings": [
    {
      "id": "<plugin-id>",
      "settings": { /* shape varies per plugin */ }
    }
  ]
}
```

- `id` — the core plugin's identifier as used by Obsidian (`daily-notes`, `templates`, `bookmarks`, etc.). Maps to data file path `.obsidian/<id>.json`.
- `settings` — the JSON object to merge into the existing data.json at that path. Additive merge — never strip pre-existing keys; only set the keys this manifest declares. Recursive merge for nested objects.

### New helper: `applyCorePluginSettings(tp, mech, variables, history, git)`

Mirrors v0.1.3 helper posture:
1. **Bail early** if `mech.core_plugin_settings` absent or empty array.
2. **Per-entry processing:**
   a. Validate `entry.id` is a non-empty string.
   b. Resolve target path: `.obsidian/${entry.id}.json`.
   c. Read existing file (if exists). C4 malformed-JSON guard: if unparseable, record `event: "warning", step: "core_plugin_settings"` with `message: "<path> is malformed JSON"`; skip this entry.
   d. Substitute placeholders in `entry.settings` values via `substituteLenient` (the per-blueprint variables overlay carries `{{module_directory}}` + `{{templates_path}}` etc.).
   e. Compute the merged result: existing ∪ substituted-settings (additive merge; substituted-settings wins on key collision; recursive for nested objects).
   f. If merged result === existing (idempotent): record `event: "info", step: "core_plugin_settings"` with `action: "skipped_existing"`; continue.
   g. Backup-on-edit: write existing content to `<target>.beacon-backup`.
   h. Write merged result to target.
   i. Record `event: "info", step: "core_plugin_settings"` with `action: "applied"`, plugin_id, settings_keys, backup_path, full git fields, attempted_at.
3. **Failure-loud:** every error path records history + Notice + continues to next entry.

### Wire-up in `installItem`

After `applySlashCommanderBindings` (current last helper call), add:
```javascript
await applyCorePluginSettings(tp, mech, variables, history, git);
```

Order: `applyPreInstall` → files[] loop → post_install → applyRuleFragment → applyNavButtons → applyExternalPlugins → applyTemplaterHotkeys → applySlashCommanderBindings → **applyCorePluginSettings (NEW)**.

## Failure-loud posture

| Failure | Behavior | History |
|---|---|---|
| `core_plugin_settings[].id` missing or non-string | Skip entry; warning Notice | `warning, step: core_plugin_settings, reason: invalid_id` |
| `.obsidian/<id>.json` malformed JSON | Preserve file; skip entry; Notice | `warning, step: core_plugin_settings, message: malformed JSON` |
| Substitution fails (missing variable) | Skip entry; Notice | `error, step: core_plugin_settings, message: <substituteStrict error>` |
| Backup write fails | Skip target write (no half-update); Notice | `error, step: core_plugin_settings, action: backup_failed` |
| Target write fails | Notice; entry recorded | `error, step: core_plugin_settings, action: write_failed` |
| Daily Notes core plugin not enabled | (Detection deferred — out of scope) | Surfaced via post_install notice; user's responsibility |
| All v0.1.x + v0.2.0 hardenings | Carried forward | as before |

## Open trail (resolve at plan-writing time)

> [!question]- 1. Is `customjs_classes[]` purely informational, or does the installer do anything with it?
> The project blueprint declares `customjs_classes: ["ProjectNavButtons", ...]` but I believe the installer doesn't touch it (CustomJS plugin auto-scans the scripts folder). Verify during plan-writing — if it's load-bearing, the daily blueprint's declaration matters; if not, document as informational.

> [!question]- 2. How does Templater's new-file-template hook know which template to apply?
> The Daily Notes plugin reads `template` from its data.json + writes the template body to the new file. Templater's "Trigger Templater on new file creation" hook then evaluates `<% %>` tokens. But Templater's hook fires for ALL new files; how does it know to apply this specific template? Answer: Templater's hook re-evaluates the BODY of any new file, replacing `<% %>` tokens with their evaluated values. The Daily Notes plugin already wrote the template body; Templater just evaluates the tokens in-place. No separate "which template" decision. Confirm during plan-writing by inspecting Templater's behavior on a Daily Notes file.

> [!question]- 3. Should `applyCorePluginSettings` recursively merge nested objects?
> The daily-notes.json shape is flat (3 top-level keys). But future core plugins may have nested settings. Plan-writing should decide: shallow merge (replace nested objects wholesale) vs deep merge (recurse). Recommend shallow for v0.3.0 (YAGNI); upgrade to deep when a real plugin needs it.

> [!question]- 4. Should the post_install notice include the format string explanation?
> The user can change `folder` / `format` / `template` after install (Obsidian Settings UI). The notice could explain what gets written so the user knows what to expect. Or keep terse and let the user inspect Settings → Daily notes if curious. Recommend terse.

## Cross-cutting risks & landmines

> [!warning] Risks surfaced by this design
> 1. **Allowlist expansion is durable.** Every cycle adding a new `.obsidian/` path requires explicit user approval. v0.3.0 adds `.obsidian/daily-notes.json`. Future cycles must NOT silently grow the allowlist.
> 2. **Core-plugin data files have no schema guarantees.** Obsidian core plugins can change their data.json shape between versions. The four safety mechanics (additive merge + backup-on-edit + malformed-JSON guard + failure-loud) protect against most hazards, but a structural breaking change in Obsidian could still leave consumers in a stuck state. Mitigation: pin Obsidian version in `Docs/use.md` if a breaking change ships.
> 3. **Daily Notes plugin must be enabled before "Open today's daily note" works.** Post_install notice covers this; consumer-side responsibility. If the user expects the install to "just work" without enabling the core plugin, they'll hit the failure mode silently. Future cycle could add a check.
> 4. **Templater "Trigger Templater on new file creation" must be enabled.** Same caveat as the boards blueprint (v0.2.0 post_install notice mentions this). Consumer-side responsibility.
> 5. **The `format` field's path-with-subdirs is an Obsidian behavior** (date-format string includes `/` separators). Verify on first install that subdirectories are created correctly + the file lands at the expected nested path.
