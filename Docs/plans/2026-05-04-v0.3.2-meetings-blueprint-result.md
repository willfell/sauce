## The bracket-escape discipline FIX-2 — moment.format literal-overlap codified for blueprint authors

> [!success] What changed in FIX-2 (in-cycle blueprint-author-discipline fix)
> At Smoke A first attempt, clicking the "Meetings" nav-button landed the hub at the **corrupt path** `beacon/meetings/11ub1/2026-05-04-Meetings.md` instead of `beacon/meetings/hubs/2026-05-04-Meetings.md`.
>
> Root cause: the manifest declared `folder: "{{module_directory}}/hubs"`. `substituteForMomentFormat` (the v0.4.0 FIX-1 helper) wraps every substituted `{{...}}` value in brackets so `moment.format` treats it as a literal — but it does NOT bracket-wrap the non-substituted literal segments authored by the blueprint. After substitute, the string was `[beacon/meetings]/hubs`. `moment.format` then read `hubs` as: `h` → 12-hour-no-padding (current hour 11 → "11"), `u` → not a token → literal "u", `b` → not a token → literal "b", `s` → seconds-no-padding (current second 1 → "1"). Result: `11ub1`.
>
> Tactical fix: bracket-escape the literal `hubs` segment in the manifest manually: `"{{module_directory}}/[hubs]"`. After install-time substitute the resolved string is `[beacon/meetings]/[hubs]`; moment.format reads both bracketed groups as literals → `beacon/meetings/hubs`. ✓
>
> Why this didn't surface in v0.4.0's to-do (the previous canonical `runTemplaterTemplate` consumer): to-do's folder was `{{module_directory}}/YYYY/MM-MMMM` — the non-substituted literals are `/` and `-`, neither of which are moment format tokens. To-do's filename ended in `-[ToDo]` with manual brackets already in place. Meetings hub is the **first** `runTemplaterTemplate` consumer with a non-substituted literal *letter sequence* that overlaps moment tokens (`h` and `s` are both tokens; `u` and `b` are not, which is what made the corruption read like garbage rather than fail loudly).
>
> Documentation added to `Docs/how.md`: bracket-escape discipline note in the `runTemplaterTemplate` action-types callout. Authoring rule: "for any non-substituted literal letter sequence in folder/filename, bracket-escape it as `[literal]` if it could overlap moment tokens — when in doubt, bracket-escape." Carry forward as a v0.4.x candidate (see handoff): a more robust schema with explicit two-field `folder_prefix` (literal) + `folder_date_pattern` (moment-formatted) mirroring Obsidian Daily Notes plugin's config, once the bracket-escape gap surfaces a third time. Two times establishes a pattern; three times justifies the schema bump.

> [!info] What the mid-cycle v1 → v2 redesign bought us
> v1 was a sound design that smoke-tested clean — the to-do-style nav-button created a single dated meeting note, attendees-prompt fired, daily dashboard panel populated. But during T1.6 v1 smokes the user surfaced that the accuris meetings flow is fundamentally **hub-centric**: one Meetings Hub per day aggregates that day's individual meeting notes, with the hub itself spawning new meeting notes via an in-note button rather than via the platform's nav-button. v1's one-click-makes-meeting shape didn't match.
>
> The redesign cost ~1.5 hours (paused subagent-driven-development; invoked `de:brainstorming` to explore v2; invoked `de:writing-plans` to commit the v2 plan; resumed subagent-driven-development) but landed the platform on the right primitive for future cross-meeting-context features (e.g., MeetingsHub aggregation extending to weekly/monthly hubs in future cycles, per-meeting wikilink-back to today's daily, cross-day meeting search).
>
> The redesign methodology — "before we accept this module, ensure it follows accuris" — is itself a documented workflow pattern. Carry forward: smokes are the right gate for "does this match the user's mental model"; mid-cycle redesigns are acceptable when the user surfaces a misalignment, and the brainstorming → plan → execution flow handles them cleanly.

> [!warning] Surprise 2 (recovered): NewMeetingButton + MeetingsHubCards "unavailable" in customjs-guard
> After FIX-2 re-install, Smoke B (Spawn New Meeting) and Smoke C (MeetingsHubCards) both showed customjs-guard's "unavailable" Notice when the dataviewjs blocks rendered. Root cause: the Node-harness install writes via `fs` (not via Obsidian's vault adapter). The CustomJS plugin's auto-rescan watches Obsidian's adapter events, not raw fs events, so it didn't pick up the newly-materialized `meetings-hub-cards.js` + `new-meeting-button.js` files in its jsFolder. Fix: Cmd+R Obsidian to force CustomJS plugin re-init; on next render, both classes were available and both buttons worked. Documented as a smoke-prerequisite for any future cycle adding new CustomJS classes: "after Node-harness install, Cmd+R Obsidian before exercising any newly-introduced CustomJS class."

## Pacing rhythm — what worked / what to improve

### What worked

- **Per-task subagent dispatch on T1.1a (manifest + 2 templates) + T1.1b (2 CustomJS classes) + T1.8 (this writeup); master-driven mechanical T1.1c-T1.7 + T1.9-T1.10.** Two T1.1 subagents gave clean review boundaries: one focused on blueprint authoring (manifest schema + template body discipline), the other focused on verbatim accuris ports (3 surgical changes per file; otherwise byte-verbatim). Both halves of two-stage review (spec compliance + code quality) were trivially fast as a result — caught zero issues on T1.1a + T1.1b on first pass.
- **Mid-cycle redesign (v1 → v2) handled via skill transition.** Paused subagent-driven-development; invoked `de:brainstorming` to explore v2 (hub-centric primitives, cost estimate, comparison to v1 mechanics); presented v2 design; got user approval; invoked `de:writing-plans` for the v2 plan; resumed subagent-driven-development on the v2 plan. Same flow that v0.4.0's brainstorming → plan → execution used — works for redesigns, not just initial designs.
- **`de:systematic-debugging` Phase 1 prevented thrashing on the `11ub1` bug.** Root-causing took <2 minutes of token-by-token moment.format decomposition before any fix proposal. The Phase-1 gate ("no fixes without root cause") is what kept the fix from being "try wrapping `hubs` in brackets and see what happens" — instead it was "I know exactly why `h` becomes `11` and `s` becomes `1`, so the fix is to bracket-escape `hubs` as `[hubs]`."
- **Cleanup-via-AskUserQuestion gate at T1.5 surfaced the additional v1 smoke artifacts before destructive action.** Initial scan found 1 expected file; full scan found 4. User got full visibility on what would be deleted before approval; second cleanup round at FIX-2 (the bogus `11ub1/` tree) followed the same gate.
- **Verbatim accuris-port discipline.** Each CustomJS class started byte-verbatim from accuris and applied only the 3 (or 2) explicitly-documented surgical changes. Made the diff trivially reviewable; made the "did we faithfully port the behavior" question trivially answerable.

### What to improve

- **Plan-writers and brainstormers: when authoring a `runTemplaterTemplate` folder/filename string with non-substituted literal letter sequences, explicitly check for moment-token overlap.** For meetings hub, the `hubs` literal was visible in both the design doc and the plan; neither flagged the `h`/`s` token-overlap risk; the bug surfaced at first use in Smoke A. Carry forward as a brainstorming + writing-plans checklist item: "for `runTemplaterTemplate` folder/filename: are there any non-substituted literal letter sequences? If yes, do any letters overlap moment format tokens (`Y`, `M`, `D`, `d`, `h`, `H`, `m`, `s`, `S`, `a`, `A`, `e`, `E`, `Q`, `w`, `W`, `Z`, `X`, `x`, `g`, `G`, `T`, `k`, `o`)? If yes, bracket-escape them."
- **Post-Node-harness-install Cmd+R requirement should be a documented smoke prerequisite for any cycle adding new CustomJS classes.** Surprise 2 cost ~5 minutes of "wait, why is the button unavailable, the file is right there." Add to a future `Docs/use.md` smoke-checklist: "after `node platform/test/run-install.js`, if any new CustomJS classes were materialized, Cmd+R Obsidian before exercising them."
- **Future plans for blueprints that ship CustomJS classes should explicitly call out the helpers entries in the manifest.** v0.3.2 manifest's `helpers[]` was authored cleanly on first pass (subagent caught it from accuris's example), but the field is undocumented in `Docs/how.md`; first-time blueprint authors might miss it. Worth a `Docs/how.md` schema-doc addition in a future cycle.
