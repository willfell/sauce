# v0.4.2 (v2) Design — split-field schema + filename pattern + daily-nav (expanded scope)

> [!abstract] Goal
> v0.4.2 v1 (committed at `319ddf9`) shipped the split-field `runTemplaterTemplate` schema. v2 EXPANDS scope mid-cycle (per gotcha-6 mid-cycle redesign methodology, mirroring v0.3.2's v1→v2) to also land:
> 1. **Filename pattern consistency** — all date-routed daily artifacts adopt `<Name>-YYYY-MM-DD` (date as suffix). Includes meetings hub layout fix (date-routed under year/month, mirroring to-do).
> 2. **Daily nav** — `invoke_command` action type + SpaceNavButtons absorbs prev/next-day arrow row (skip-to-nearest-existing with grey-out) + adaptive multi-row layout (port from accuris space-nav-buttons.js).
> 3. **Mobile-aware visual design** — adopted as a NEW platform value: every rendered UI must be mobile + desktop responsive. Installer remains desktop-only (landmine #8 unchanged); visual rendering is the new explicit design contract.
>
> One workshop bump (0.8.1 → 0.9.0); one annotated tag `v0.9.0` at cycle close.

> Per gotcha-6: ACCEPT redesign rather than ship the wrong primitive. Same flow as v0.3.2 v1→v2 (paused SDD → invoke brainstorming → revised design → revised plan → resume SDD).

## Daily-nav — SpaceNavButtons absorbs prev/next + adaptive layout (Ask #3)

### nav-buttons mechanism 2.2.0 → 2.3.0 ADDITIVE

> [!info] What changes
> 1. **New action type `invoke_command`** — for "Today" button + future-flexible Obsidian command dispatching. Renderer dispatch ~10 LOC; `validateAndResolve` passthrough preserves `command_id`.
> 2. **SpaceNavButtons class enhanced** with three responsibilities:
>    - **Top arrow row** — prev/next-day arrows + adaptive layout
>    - **Skip-to-nearest-existing** — scans daily folder for existing notes; jumps to nearest in target direction
>    - **Hover styling** — port accuris's hover-darken (cosmetic parity with accuris)
> 3. **Adaptive multi-row layout** — port accuris's row distribution (lines 258-268): 2-row desktop / 3-row mobile. Mobile awareness via `app.isMobile`.
>
> The thin-renderer-over-registry purity is partially compromised for daily-nav specifically: SpaceNavButtons reads `.obsidian/daily-notes.json` at runtime to acquire daily folder + format. If that file is absent or malformed, top arrow row is silently skipped (graceful degradation = daily blueprint not installed).

### `invoke_command` action shape

> [!example] Manifest action shape
> ```json
> {
>   "id": "<button-id>",
>   "label": "<text>",
>   "icon": "<icon-key>",
>   "order": 50,
>   "action": {
>     "type": "invoke_command",
>     "command_id": "<obsidian-command-id>"
>   }
> }
> ```
>
> **Renderer dispatch (`space-nav-buttons.js`):**
> ```js
> if (type === "invoke_command") {
>   if (!action.command_id || !app.commands.commands[action.command_id]) {
>     new Notice(`nav-buttons: command not found "${action.command_id}" (from ${btn._source})`, 8000);
>     return;
>   }
>   app.commands.executeCommandById(action.command_id);
>   return;
> }
> ```
>
> **`validateAndResolve` passthrough:**
> ```js
> if (btn.action.type === "invoke_command" && btn.action.command_id) {
>   return btn;  // preserve command_id literal; no install-time validation (commands are runtime-only)
> }
> ```

### Daily-nav top arrow row

> [!example] Click-time logic in SpaceNavButtons
> ```js
> // ... after registry-driven button list rendering ...
>
> // Read daily blueprint settings; if absent, skip top arrow row.
> const dailyMeta = await readDailyNotesMeta(app);  // returns null if .obsidian/daily-notes.json absent/malformed
> if (!dailyMeta || !dailyMeta.folder || !dailyMeta.format) return;
>
> // Extract current note's date.
> const currentFile = dv.current();
> const dm = currentFile.file.name.match(/(\d{4}-\d{2}-\d{2})/);
> const currentDate = dm ? window.moment(dm[1], "YYYY-MM-DD", true) : window.moment();
>
> // Scan daily folder for all existing dailies; sort by extracted date.
> const allDailies = app.vault.getMarkdownFiles()
>   .filter(f => f.path.startsWith(dailyMeta.folder + "/"))
>   .map(f => ({ file: f, dm: f.name.match(/(\d{4}-\d{2}-\d{2})/) }))
>   .filter(x => x.dm)
>   .map(x => ({ ...x, m: window.moment(x.dm[1], "YYYY-MM-DD", true) }))
>   .filter(x => x.m.isValid())
>   .sort((a, b) => a.m.diff(b.m));
>
> // Find nearest-earlier and nearest-later relative to currentDate.
> const earlier = allDailies.filter(x => x.m.isBefore(currentDate)).pop();
> const later = allDailies.filter(x => x.m.isAfter(currentDate))[0];
>
> // Render arrow row at top of container with prev/next buttons. Greyed (opacity 0.4, cursor:default, no click) when target is null.
> ```
>
> Behavior: prev arrow opens nearest-earlier daily note; if none, button is greyed + disabled. Symmetric for next. **No file creation** — opens existing only.

### Adaptive layout

> [!example] Row distribution (port accuris lines 258-268)
> ```js
> const isMobile = app.isMobile;
> const rowCount = isMobile ? 3 : 2;
> const baseSize = Math.floor(buttons.length / rowCount);
> const remainder = buttons.length % rowCount;
> const rows = [];
> let idx = 0;
> for (let r = 0; r < rowCount; r++) {
>   const size = baseSize + (r < remainder ? 1 : 0);
>   if (size > 0) rows.push(buttons.slice(idx, idx + size));
>   idx += size;
> }
> ```
>
> Buttons distributed across N rows, balanced by count. Each row uses `flex: 1` per button + `flex-wrap: nowrap` so buttons stretch evenly. Text fits via Obsidian's default button styling.

### Mobile-aware visual design (NEW platform value)

> [!warning] Carries forward beyond this cycle
> Beacon's renderers (CustomJS classes) MUST be designed for both desktop and mobile from now on. This is a NEW value adopted at v0.4.2 brainstorming. Implications:
> - Adaptive layouts (column counts, row distributions, button stretch behavior)
> - Touch-friendly target sizes (≥40px tap area on mobile)
> - Mobile-detection via `app.isMobile`
> - No reliance on hover-only states for primary affordances (mobile has no hover)
>
> **Installer is unchanged** — landmine #8 (`require("fs")` desktop-only) still applies. Mobile install requires future re-architecture (Obsidian Sync delivery + vault-adapter reads). The visual contract is what changes.
>
> Carry forward to all future blueprint CustomJS classes (DailyNavBar, MeetingsHubCards, NewMeetingButton, SpaceDailyDashboard, future-blueprint classes). Each cycle should review its rendering for mobile-fitness.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Top arrow row renders incorrectly on existing notes (filename regex misses non-date-prefixed notes) | Regex `\d{4}-\d{2}-\d{2}` matches anywhere in filename; `Meetings-2026-05-04` matches at position 9; non-date notes fall back to `moment()` (current date) — top row still renders if daily-meta present, just rooted at today |
| Skip-to-nearest scans entire daily folder per click (perf) | `app.vault.getMarkdownFiles()` is in-memory; even 1000+ daily notes scan in <10ms |
| Daily blueprint not installed → top row renders broken | `readDailyNotesMeta` returns null for absent/malformed `.obsidian/daily-notes.json` → top row silently skipped |
| MeetingsHubCards filter changes from `startsWith` to `endsWith` — old meeting notes (date-prefixed) won't show up | Acceptable — old prefix-pattern files predate v0.4.2; future cycle could ship a one-time migration script for existing consumers; barebones is fresh + tmp-acc preserves deliberate-lag |
| Templater folder-templates miss new date-routed hub subfolders (`beacon/meetings/hubs/2026/05-May/`) | Templater folder-templates apply RECURSIVELY — registration on `beacon/meetings/hubs` covers all subfolders |
| Mobile testing: don't have a mobile device in CI | Test desktop adaptive layout in S4 T4.7 by resizing window; mobile-sim via Obsidian's dev panel; full mobile validation deferred to user-environment testing |
| daily install gating across S2/S3 boundary | RESOLVED: split daily bump. S2 lands 0.1.1 → 0.1.2 with format change only (depends_on unchanged >=2.0.0) → installs cleanly with nav-buttons 2.2.0. S3 lands 0.1.2 → 0.1.3 with Today button + depends_on >=2.3.0 → installs cleanly with nav-buttons 2.3.0. Each stage independently install-green. |
