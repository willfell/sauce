# v0.21.0 Consumer Bootstrap — design

> [!abstract] Goal
> Add a Node-based bootstrap orchestrator at `platform/bootstrap.js` so a fresh consumer vault can go from `git clone + cd <vault>` to "fully loaded, themed, configured, ready to use" with **one shell command**. Fetches all required community plugins from GitHub releases, vendors them into `.obsidian/plugins/<id>/`, walks the user through subscription configuration via an interactive wizard, then invokes the existing Node installer (`runInstall(vaultPath)`).

> [!info] Origin
> Surfaced post-v0.20.0 when the user attempted to roll out the v0.19.0 styling mechanism to the barebones consumer. Manual prereq install (Style Settings community plugin) blocked the rollout. The user requested: "running an installer orchestrates all of it with a command, then opening up the vault should consist of the configuration wanted from the user." This cycle delivers that.

## Section 1 — Scope + locked decisions

### Scope

- **In scope:** Node-based bootstrap orchestrator outside Templater. Interactive wizard via `@inquirer/prompts`. Foundational + declared plugin set fetched from GitHub releases with idempotent skip-if-present. First-run wizard generates `platform-config.json` + `platform-subscription.json` from scratch. Re-run wizard offers edit-existing-config menu. Atomic file writes with backup-on-edit. Failure-loud + idempotent-resume posture.
- **Out of scope:** Plugin version pinning. Multi-vault batch bootstrap. Auto-trigger on Obsidian open. Vendored `node_modules/` for airgapped install. Plugin update lifecycle beyond force-redownload.

### Locked decisions

- **Path:** B — bootstrap.js outside Templater. Pure Node CLI.
- **Plugin scope:** Foundational (templater-obsidian, customjs, dataview) + every `external_plugins[]` declared across subscribed mechanisms/blueprints. New `foundational_plugins[]` schema field at workshop manifest level.
- **Runtime:** Node. Reuses `runInstall(vaultPath)` (refactored from `install.js` CLI entrypoint).
- **Wizard depth:** Full wizard + edit-existing-config. Both first-run subscription generation and re-run subscription editing.
- **Prompt library:** `@inquirer/prompts` (first npm dep in workshop).
- **Plugin versioning:** Latest from GitHub; idempotent skip-if-present; force-redownload via wizard menu.
- **Allowlist expansion (landmine #12):** USER APPROVED 6 → 9 paths.

### Consumer flow (target UX)

```bash
git clone <consumer-repo> my-vault
cd my-vault
node ../beacon/platform/bootstrap.js   # workshop_relative_path resolved from platform-config.json
# (interactive wizard walks user through config/subscription if absent)
# (plugins fetched from GitHub releases + theme vendored + appearance + style-settings + files)
open -a Obsidian my-vault
# Cmd+R inside Obsidian once
# Done.
```

First run: ~30s. Subsequent runs: ~2s (skip-if-present).

## Section 2 — Architecture

### Schema addition: `foundational_plugins[]`

```jsonc
// platform/manifest.json
{
  "workshop_version": "0.21.0",
  "foundational_plugins": [          // NEW
    { "id": "templater-obsidian" },
    { "id": "customjs" },
    { "id": "dataview" }
  ],
  "mechanisms": [...],
  "blueprints": [...]
}
```

No changes to existing `external_plugins[]` shape. Bootstrap reads `foundational_plugins[]` (workshop-level) PLUS each subscribed item's `external_plugins[]` (mechanism/blueprint-level) and unions them into the install set.

### Plugin id → GitHub repo lookup

Single source of truth: `https://raw.githubusercontent.com/obsidianmd/obsidian-releases/master/community-plugins.json`. Officially maintained by Obsidian. Bootstrap fetches once per run, caches in memory, builds `id → {repo, name}` map. Avoids vendoring our own id↔repo mapping.

### First-run wizard flow (no `platform-config.json` yet)

1. Welcome banner + brief explainer.
2. `input` — vault path (default `process.cwd()`, validate dir exists + writable).
3. `input` — workshop relative path (default `../beacon`, validate exists + has `platform/manifest.json`).
4. `input` — vault display name (default basename of vault path).
5. `checkbox` — mechanisms to subscribe (default-checked: customjs-guard, nav-buttons, cards, beacon-button, styling; default-unchecked: validator, audit).
6. `checkbox` — blueprints to subscribe (default-unchecked; user opts in).
7. Confirm summary → atomic write `platform-config.json` + `platform-subscription.json`.
8. Compute plugin set (foundational ∪ external_plugins from subscribed items).
9. `confirm` — "Install N plugins from GitHub: [list]. Proceed?"
10. Fetch + vendor plugins (skip-if-present).
11. Invoke `runInstall(vaultPath)` (existing install.js entrypoint, refactored for re-import).
12. Final report + "Open Obsidian + Cmd+R."

### Re-run wizard flow (config exists)

1. Read existing config + subscription.
2. `select` — top-level menu:
   - **Just install plugins + run installer (skip-if-present)** — canonical re-run.
   - **Edit subscription** → mechanisms + blueprints multi-select with current values pre-checked → diff confirm → atomic write → fall through to install.
   - **Edit config** → workshop_relative_path / display name / vault path → atomic write → fall through.
   - **Force re-download plugins** → checkbox of currently-installed plugins → fetch overwrites → fall through.
   - **Quit**.
3. After any edit, fall through to the install step (steps 8-12 of first-run flow).

### Validation

- Path inputs check existence + writability.
- Workshop path checks for `platform/manifest.json`.
- Mechanism/blueprint checkbox dynamically reads workshop's `platform/manifest.json` for current catalogue (no hardcoded list — auto-updates as new mechanisms ship).
- All file writes atomic (write to `<file>.tmp` → rename) with `<file>.beacon-backup` for overwrites.
- Wizard state held in memory only until final confirm — Ctrl+C is safe at any prompt before the final confirm.

### File layout under `platform/`

```
platform/
├── bootstrap.js                        NEW — main wizard entry
├── bootstrap-lib/
│   ├── wizard.js                       NEW — first-run + re-run prompt flows
│   ├── fetch-plugin.js                 NEW — HTTPS GET + write to .obsidian/plugins/<id>/
│   ├── community-plugins-index.js      NEW — fetch + cache obsidian-releases index
│   └── community-plugins-merge.js      NEW — additive merge for community-plugins.json
├── install.js                          UPDATE — export runInstall(vaultPath); no behavior change
└── manifest.json                       UPDATE — foundational_plugins[]; workshop_version 0.21.0
```

### Reuses

- **HTTPS:** Node built-in `https.get` — zero external deps for the network layer. Wrap in promise.
- **JSON merge:** existing `_safeReadJson` / atomic-write patterns from install.js helpers.
- **Backup-on-edit:** `.beacon-backup` pattern from v0.1.3 plugin-data helpers.
- **Failure-loud history:** new `<vault>/Docs/Meta/bootstrap.history.json` ledger (separate from platform-installed.json).

## Section 4 — Failure modes + harness coverage

### Failure mode matrix

| Scenario | Behavior |
|---|---|
| **Network offline** at index fetch | Fail-loud: "Cannot reach raw.githubusercontent.com. Check your connection or retry." Exit 1. No partial state written. |
| **Network offline** mid-plugin-fetch | Fail-loud per plugin; partial state OK (plugins fetched before the failure stay on disk; skip-if-present resumes them on retry). Final report shows N succeeded / M failed with reason. |
| **GitHub 404** on plugin's manifest.json or main.js | Fail-loud per plugin; skip + continue. Reason logged. |
| **GitHub 404** on plugin's styles.css | Tolerated silently. Many plugins ship without styles.css. |
| **GitHub rate limit** (60/hr unauth) | Fail-loud with hint: "GitHub rate-limited. Wait ~60min or set `GITHUB_TOKEN` env var." Exit 1. |
| **Malformed JSON** in upstream community-plugins index | Fail-loud at index fetch step. |
| **Plugin id not in upstream index** | Fail-loud per plugin: "Plugin id `<id>` not found in obsidian-releases." Skip + continue. |
| **Workshop missing `node_modules`** | Detect `require('@inquirer/prompts')` failure. Print: "Run `cd <workshop-path> && npm install` first." Exit 1. |
| **Consumer config malformed** | Fail-loud at parse step. Offer "regenerate from scratch" wizard branch. |
| **`.obsidian/plugins/<id>/` exists but main.js corrupted** | Skip-if-present is naive — only checks manifest.json existence. Don't attempt repair. User runs force-redownload from re-run menu. |

### Harness coverage — `platform/test/run-bootstrap.js`

Mock the HTTPS layer (intercept `https.get` via small testing shim — pattern from existing harness mocking). 8 cases under selector `bootstrap`. TDD-first: write failing cases first, implementation makes them pass.

| Case | Setup | Expected |
|---|---|---|
| **BS1** fresh-vault first run | Empty `.obsidian/`, mocked HTTPS returns valid releases | Plugin files vendored; community-plugins.json populated; install ledger has expected entries |
| **BS2** idempotent re-run | BS1 state, mock untouched | Zero plugin downloads; runInstall idempotent (zero new history) |
| **BS3** force-redownload single plugin | BS1 state, force flag for `obsidian-style-settings` | Only that plugin re-fetched; others skip; `.beacon-backup` written |
| **BS4** styles.css 404 tolerated | Mock returns 404 for one plugin's styles.css | Plugin still installed; manifest+main.js present; styles.css absent; no error |
| **BS5** main.js 404 fails loud | Mock returns 404 for one plugin's main.js | That plugin skipped + reported; other plugins still installed |
| **BS6** unknown plugin id | Subscription declares `external_plugins[].id` not in index | Fail-loud per plugin; other plugins succeed |
| **BS7** community-plugins.json additive merge | Pre-existing array `["foo", "bar"]`, fetch installs `baz` | Final array `["bar", "baz", "foo"]` (sorted, deduped, additive); `.beacon-backup` written |
| **BS8** subscription wizard generates valid files | Mock prompts return canonical selections | platform-config.json + platform-subscription.json written + parse-validate clean |

### Harness invocation

`node platform/test/run-bootstrap.js` (no selector required for clean exit; mirrors v0.20.0 whole-suite-green posture).

## Section 6 — Risks + open questions

### Risks

- **R1 — npm posture lock-in.** Once `package.json` ships, every consumer effectively needs Node + npm. Mitigation: bootstrap.js is the ONLY consumer of npm; harnesses + install.js stay zero-dep. Documented as one-time setup.
- **R2 — GitHub rate limits.** Unauthenticated 60/hr per IP. Mitigation: cache the obsidian-releases index for the run; honor `GITHUB_TOKEN` env var; document in failure-mode message.
- **R3 — Plugin id ↔ repo drift.** Upstream removal/rename. Mitigation: per-plugin failure non-fatal; clear message; user can pin manually.
- **R4 — Wizard interrupted mid-flow.** Mitigation: atomic write at END of wizard; Ctrl+C safe at any prompt before final confirm.
- **R5 — Workshop self-bootstrap.** Workshop dogfoods itself. Mitigation: skip-if-present handles this; workshop self-install during S5 verifies.

### Open questions

- **Q1 — Where does `bootstrap.history.json` live?** Recommendation: `<vault>/Docs/Meta/bootstrap.history.json` (separate from platform-installed.json).
- **Q2 — Auto-run bootstrap on Obsidian open?** Recommendation: No — explicit terminal command only.
- **Q3 — Multi-vault batch bootstrap?** Recommendation: Out of scope.
- **Q4 — Vendor `node_modules/` for offline install?** Recommendation: No — out of scope for this cycle.
