# v0.50.4 — Empty `wiki/Wiki.md` on newly-created projects (design)

**Date:** 2026-05-16
**Card:** [[Create will note is broken]] (Projects Blueprint sub-board)
**Cycle target:** PATCH v0.50.3 → v0.50.4

## Symptom

User created a project called `test` via "+ New Project" on `spice/projects/Projects.md`. Result on disk:

```
spice/projects/test/
├── test.md             793 B   ✓ populated
├── Project Map.md      375 B   ✓ populated
├── test-board.md       388 B   ✓ populated
└── wiki/
    └── Wiki.md           0 B   ✗ empty
```

All other extra_files materialized correctly; only the wiki sidecar landed empty. Pre-existing projects' `wiki/Wiki.md` files (created by `applyWikiBackfill` at v0.50.3 install) are populated (~666–729 bytes each). The bug is specific to the **entity-create extra_files** path.

## Root cause

`platform/blueprints/project/manifest.json`'s project entity-create entry declares the wiki sidecar like this:

```json
"extra_files": [
  { "filename_pattern": "Project Map.md",           "body_template": "Template, Project Map.md" },
  { "filename_pattern": "{{prompts.slug}}-board.md","body_template": "Template, Project Board.md" },
  { "filename_pattern": "wiki/Wiki.md",             "body_template": "Template, Wiki Hub.md" }
]
```

The wiki entry embeds `wiki/` directly in `filename_pattern`. `_createExtra` in `entity-create.js` ensures only `xFolder` (the project root, `spice/projects/test`), never the dir component of `filename_pattern`. So when `app.vault.create("spice/projects/test/wiki/Wiki.md", content)` runs, Obsidian's vault.create encounters a missing parent dir and either:

- (a) auto-creates the missing dir + an empty file, then silently fails to write the body, or
- (b) throws — but Obsidian first leaves an empty file behind as a side-effect of dir-creation.

Either way the observable outcome is `Wiki.md` at 0 bytes. The throw (if any) is swallowed by the absence of a try-block around line 85's `for…of` loop in entity-create — but execution reached `vault.create`, so the loop didn't terminate at a prior step.

The mechanism's existing `subfolder` field (`_createExtra` lines 592–594) is designed for exactly this case: when a sibling field is set, `_createExtra` builds `xFolder = folder/subfolder` and `_ensureFolder` it before `vault.create`. The wiki extra_file doesn't use this field; it stuffs the subfolder into `filename_pattern` instead.

## Why the test harness didn't catch it

- **EC-24** (`run-entity-create.js:374`) tests the *happy path* via the `subfolder` field — confirms nested-path composition when `subfolder` is set. The wiki manifest entry doesn't use `subfolder`, so EC-24 doesn't exercise the failing code path.
- **WIKI-5** (`run-entity-create.js:795`) asserts the manifest's wiki entry maps `filename_pattern: "wiki/Wiki.md" → body_template: "Template, Wiki Hub.md"`. This is a *schema shape* assertion — it validates the registry has the expected fields, but doesn't run the materialization flow end-to-end.

Neither harness exercises the `filename_pattern`-with-embedded-slash case against a real `vault.create`.

## Fix options

### Option A — manifest fix only

Rewrite the wiki extra_file in `platform/blueprints/project/manifest.json`:

```json
{ "subfolder": "wiki", "filename_pattern": "Wiki.md", "body_template": "Template, Wiki Hub.md" }
```

**Pros:** uses the schema as designed. Minimal diff.
**Cons:** doesn't fix the underlying mechanism gap. A future blueprint author who embeds a subfolder in `filename_pattern` hits the same bug.

### Option B — mechanism fix only

Patch `_createExtra` to split `xPath` and ensure any dir component embedded in `filename_pattern`:

```js
const xPath = `${xFolder}/${filename}`;
// Ensure parent of xPath in case filename_pattern itself embeds a subfolder.
const parentDir = xPath.includes("/") ? xPath.substring(0, xPath.lastIndexOf("/")) : "";
if (parentDir && parentDir !== xFolder) await this._ensureFolder(parentDir);
```

**Pros:** fixes the class of bug, not just this instance. Forward-defense.
**Cons:** the `subfolder` field semantics become redundant (both paths work). Cosmetic schema overlap.

### Option C (chosen) — both

Do both. Manifest fix is the principled use of the schema; mechanism fix is forward-defense for future blueprint authors who don't read the schema doc.

**Order of operations matters:** apply the manifest fix first. Mechanism fix is purely additive (it only triggers when filename_pattern contains `/`, which the manifest fix removes). Together they leave the codebase in a state where both styles work, but the canonical style (used in-tree) is the `subfolder` field.

## Repair of existing empty file

The user's `spice/projects/test/wiki/Wiki.md` (0 bytes) is left behind by the buggy flow. `applyWikiBackfill` (in `install.js`) currently skips a project if `wiki/Wiki.md` already exists (it checks for *missing* files, not empty ones). Two options to repair:

- **A.** Have the user manually `rm` the empty file and re-run `sauce reinstall --vault <path>`; backfill will then create it.
- **B.** Patch `applyWikiBackfill` to treat 0-byte files as missing (overwrite with template content).

This cycle picks **A** (out of scope here; covered by the existing FIX-LATER #1 in the v0.50.3 hotfix handoff: "applyWikiBackfill should be unconditional"). The user can repair the single test/ project by hand after install.
