# v0.48.0 design — project kanban "+ Add a card" workstream fix + icons polish

## Background

### What v0.47.0 left undone

v0.47.0 closed 2026-05-15 fixing four headspace Phase 3 e2e defects (BUG-1 icons, BUG-2 `type:person`, BUG-3 project atlas body, BUG-4 `extra_files` bodies). During S11 manual smoke at headspace, the user clicked "+ Add a card" on a project's kanban board and surfaced a fifth issue:

- The kanban-plugin "+ Add a card" UI creates `spice/projects/<slug>/tasks/<TaskName>.md` and pastes `Template, Kanban Card.md` content into it.
- `Template, Kanban Card.md` is a Templater template — its `<%* %>` blocks run a workstream picker (reads atlas frontmatter `workstreams[]` via `tp.system.suggester`) and auto-promote the file from flat `tasks/<X>.md` to folder form `tasks/<X>/<X>.md` via `tp.file.move`.
- The kanban-plugin uses Obsidian's CORE template paste mechanism, NOT Templater. So the `<%* %>` blocks stay as raw text in the created file. Neither the workstream prompt nor the auto-promote fires.
- Project's manifest declares zero `templater_folder_templates[]`, so Templater's `trigger_on_file_creation` doesn't wake up on the new file either.

Per the v0.47.0 cycle's escalation gate, this finding was deferred to v0.48.0. Captured as FLN-i in `Docs/plans/2026-05-15-v0.47.0-result.md`.

### Factual correction to FLN-i (verified in this design session)

FLN-i states that v0.46.0 stripped `_createTaskNote` + `_createTaskBoard` (~184 LOC) from `ProjectNavButtons`. **This is incorrect.** Verified against `git show 6efa21f` (the v0.46.0 S6 strip commit):

The render-side wiring for "New Note" / "Open Board" / "Create Board" buttons on task-hub and task-note contexts is still intact at lines 567-625.
- **The "atlas missing helper buttons" framing is also misleading:** "+ Add Task" / "+ Add Task Board" buttons on the project atlas page were NEVER present pre-v0.46.0. The real BUG-5 is **one observable defect** (kanban "+ Add a card" doesn't run Templater post-paste). This design fixes that single defect by triggering Templater post-creation via a startup-registered listener.

## Architecture

Three moving pieces, smallest blast radius:

1. **`ProjectTaskCreateListener`** — new CustomJS class. Owns:
   - One `app.vault.on('create')` subscription, registered once per session (idempotent via class-level `_initialized` flag).
   - Path regex filter: `^spice/projects/[^/]+/tasks/[^/]+\.md$` (top-level files under `tasks/`, NOT nested files like `tasks/<X>/<X>.md` which would re-fire after `tp.file.move`).
   - Content guard: skip if file body has no raw `<%* ` text (idempotency; also handles the case where Templater has already processed the file).
   - Templater command invocation: `app.commands.executeCommandById('templater-obsidian:replace-in-file-templater')` after a brief settle (microtask queue).
   - Failure-loud: `new Notice(...)` if Templater command missing OR if file read fails.

2. **`Template, Project Task Create Listener.md`** — new Templater startup template. Body is a single `<%* customJS.ProjectTaskCreateListener.init(); %>` line. Registered in Templater plugin config's `startup_templates[]` field (note: this is the field name Templater actually uses, NOT `startup_templates_files`; verified via install.js:2378 seed defaults) via the installer (new aggregation helper, see below).

3. **`Template, Kanban Card.md`** — enhanced (not rewritten). The existing 3-strategy atlas frontmatter read + `tp.system.suggester` picker + `tp.file.move` auto-promote all stay. Added:
   - **"+ Create new workstream" sentinel:** suggester's `text_items` gains a `+ Create new workstream` entry, `items` gains a sentinel value `__create_new__`. If picked, follow-up `tp.system.prompt("Workstream name")` collects name → slugify → check id-collision against existing → `processFrontMatter(atlasNote, fm => fm.workstreams = [...(fm.workstreams || []), { id, name }])` → use new id as `workstreamValue`.
   - **Empty-workstream toast:** if `ws.length === 0`, show `new Notice("No workstreams defined on this project's atlas. Define one (or pick '+ Create new' next time) to enable workstream tagging on tasks.")` and skip the picker (preserves current silent-skip behavior at the workstreamValue level, but adds discoverability).

The kanban-plugin UI is completely untouched. The kanban-plugin's `new-note-template` setting still points at `Template, Kanban Card.md`. The kanban-plugin still creates `spice/projects/<slug>/tasks/<TaskName>.md` and pastes content into it. The difference: a moment later, our listener invokes Templater on that file, and Templater processes the `<%* %>` blocks the way it always would have if Templater had been the original creator.

## Components & contracts

### `platform/blueprints/project/helpers/project-task-create-listener.js` (NEW, ~60 LOC)

```js
class ProjectTaskCreateListener {
    constructor() {
        this._initialized = false;
        this._pathRegex = /^spice\/projects\/[^/]+\/tasks\/[^/]+\.md$/;
        this._templaterCommandId = 'templater-obsidian:replace-in-file-templater';
    }

    init() {
        if (this._initialized) return;
        this._initialized = true;

        app.vault.on('create', async (file) => {
            try {
                if (!file || file.extension !== 'md') return;
                if (!this._pathRegex.test(file.path)) return;

                // Defer a microtask to let kanban-plugin finish writing content
                await new Promise(r => setTimeout(r, 50));

                const content = await app.vault.read(file);
                if (!content.includes('<%* ') && !content.includes('<% ')) return;  // idempotency

                if (!app.commands.commands[this._templaterCommandId]) {
                    new Notice(`Templater command missing: ${this._templaterCommandId}. Task created without workstream prompt.`);
                    return;
                }

                // Open the file so it's the active file for Templater's replace-in-file
                await app.workspace.openLinkText(file.path, '');
                await app.commands.executeCommandById(this._templaterCommandId);
            } catch (e) {
                new Notice(`ProjectTaskCreateListener error: ${e.message}`);
                console.error(e);
            }
        });

        console.log('ProjectTaskCreateListener: subscribed to vault.create');
    }
}
```

Contract: `customJS.ProjectTaskCreateListener.init()` is the single public method. Idempotent across multiple calls. Subscribes once. Fails loud via `Notice`.

### `platform/install.js` (MODIFIED — NEW helper `applyTemplaterStartupTemplates`)

Pattern modeled on the existing `applyTemplaterFolderTemplates` helper at `install.js:3721` (handles `templater_folder_templates[]` aggregation + additive-merge into the Templater data.json's `folder_templates[]` field with malformed-JSON guard + history-failure-loud + backup-on-edit per landmine #12). The new `applyTemplaterStartupTemplates` is a direct parallel: same aggregation pattern, same data.json target, different source field (`templater_startup_templates[]`) and different target field (`startup_templates[]`).

```js
function applyTemplaterStartupTemplates(ctx) {
    const aggregated = [];
    for (const bp of subscribedBlueprints) {
        const entries = bp.manifest.templater_startup_templates || [];
        for (const entry of entries) {
            aggregated.push(substituteLenient(entry, ctx));
        }
    }
    const dataJsonPath = '.obsidian/plugins/templater-obsidian/data.json';
    // ... additive merge into data.startup_templates[] (preserve user entries)
    // ... per landmine #12 safety mechanics: malformed-JSON guard, backup-on-edit
}
```

Idempotency: existing entries (matching path) are not duplicated. User-added entries are preserved.
