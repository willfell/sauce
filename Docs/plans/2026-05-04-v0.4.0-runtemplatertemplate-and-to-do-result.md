## The codified `substituteForMomentFormat` helper — moment.format literal-escape automated

> [!success] What changed in FIX-1 (in-cycle installer fix)
> At Smoke A first attempt, the `runTemplaterTemplate` nav-button created the to-do file at the **corrupt path** `b0pmcon/to-0th/2026/05-May/2026-05-03-ToDo.md` instead of `beacon/to-do/2026/05-May/2026-05-04-ToDo.md`.
>
> Root cause: `validateAndResolve` substituted `{{module_directory}}` → `beacon/to-do` (literal text); the renderer then called `moment().format("beacon/to-do/YYYY/MM-MMMM")` which interpreted literal characters as moment format tokens (`e` → day-of-week numeric, `a` → am/pm marker, `do` → day-of-month with ordinal suffix), corrupting the literal prefix into garbage. The filename `YYYY-MM-DD-[ToDo]` survived intact because `[ToDo]` is bracket-escaped — moment's literal-escape syntax. The folder's literal prefix was NOT bracket-escaped; that was the gap.
>
> Fix: new helper `substituteForMomentFormat` next to `substituteLenient` (in `platform/install.js` ~line 645). Identical regex/lookup logic but wraps each substituted value in `[...]` so `moment.format` treats the value as a literal. `validateAndResolve`'s `runTemplaterTemplate` branch swapped its `substituteLenient` calls for `substituteForMomentFormat` for both `folder` and `filename`. R1 harness assertion updated to expect the bracketed form (`"[beacon/to-do]/YYYY/MM-MMMM"` instead of `"beacon/to-do/YYYY/MM-MMMM"`); harness stays green at 220/220.
>
> From "blueprint authors must remember to bracket-wrap their `{{module_directory}}`" to "the platform handles literal-escape automatically." Future `runTemplaterTemplate` consumers inherit the fix for free.
>
> The user's brainstorming framing — "settle these underlying critical decisions while we ship out these first modules, ensuring a sophisticated eco-system, few points of failure, and preventing long term tech debt" — is exactly what FIX-1 delivers. The bug surfaced AT FIRST USE in production-style smoke; the systemic fix prevents every future `runTemplaterTemplate`-consuming blueprint from re-encountering it.

> [!success] What the v0.4.0 foundations buy the platform going forward
> - **`runTemplaterTemplate` action type** — declarative date-routing without an intermediate-file flash (the prior `createFromTemplate` would write the template body to a transient location and rely on `<%* %>` self-routing to move itself; `runTemplaterTemplate` resolves the destination at click-time so the file lands in the right folder on first write).
> - **`applyTemplaterFolderTemplates` helper** — Templater-trigger-on-create independence: even when users create a file manually inside `beacon/to-do/` (right-click "New note"), the right template applies. Backstop for the nav-button-click path.
> - **Per-action-type architecture** — `validateAndResolve` and the renderer both branch on `action.type`. Adding the next action type (e.g., `invoke_command` for "Today" daily nav-button) is a localized change.
> - **Landmine #12 helper count grew without path count growing** — `applyTemplaterFolderTemplates` writes to the same `.obsidian/plugins/templater-obsidian/data.json` as `applyTemplaterHotkeys` (just to a different top-level field), so the allowlist remains at 3 paths. Helper count 3 → 4 reflects new behavior, not a new attack surface.

## Pacing rhythm — what worked / what to improve

### What worked

- **Per-task subagent dispatch on T1.1 / T1.3 / T1.4 / T2.1 / T2.7 (writeup); master-driven mechanical T1.2 / T1.5 / T1.6 / T2.2 / T2.3 / T2.8.** Same cadence as v0.1.3 + v0.2.0 + v0.3.0. Keeps master context light; reviewer-quality on the substantive code.
- **Two-stage review (spec compliance + code quality) caught real defects.** T1.1 had a race-tolerance gap in the renderer dispatch; T1.3 had a typo and a filter-asymmetry between applied + skipped paths. Both fixed in fresh implementer iterations rather than carrying as v0.4.x debt.
- **TDD-discipline harness extension on T1.4.** Subagent wrote FT1-FT5 + R1 first (red), then implemented the helper (green). Same pattern paid off again at FIX-1: when `substituteForMomentFormat` changed the resolved registry shape, R1 went red immediately, and the assertion update was the natural closing move.
- **Mid-cycle bug surfaced AT FIRST USE in manual smoke and was fixed in-stage.** Parallel to v0.3.0's T-FIX1 pattern. The `de:systematic-debugging` skill kicked in at the first smoke failure; root-causing the moment.format token-interpretation bug took ~5 minutes of evidence-gathering before any fix proposal — the Phase 1 gate (no fixes without root cause) prevented thrashing.
- **`nav_buttons` decision-deferral on the future "Today" daily nav-button.** Daily ships with empty `nav_buttons[]`; an `invoke_command` action type is on the v0.4.x candidate list (see handoff). The per-action-type architecture from v0.4.0 makes that addition trivial.

### What to improve

- **Plan-writers: when a task touches a file with both a `platform/.../source.js` and a materialized `Docs/Meta/.../same.js`, explicitly say "modify the source and re-materialize via self-install."** Surprise 3 wasted ~10 minutes recovering from the implementer's edit-the-materialized-file mistake. Cheap to fix at plan-writing time, expensive at implementer time.
- **Plan-writers: anticipate that any action type whose folder/filename strings flow through `moment.format` needs literal-escape protection.** Surfaced via FIX-1; codified going forward via `substituteForMomentFormat`. Plan-time checklist: "are any moment.format strings receiving substituted values?" → if yes, ensure substitution wraps in brackets (or the helper now does it automatically).
- **Future plans could either raise the harness floor when a helper has 5+ failure paths, or drop the floor and trust the TDD discipline.** Same observation as v0.3.0: spec floor of +30 with implementer landing at +48 is honest behavioral coverage, not padding — but the floor is slightly under-specified. Either resolution is fine.
