# v0.59.11 — Kanban Card name-collision fix (design)

**Date:** 2026-05-17
**Workshop version target:** v0.59.10 → v0.59.11 PATCH
**Blueprint:** project v1.13.5 → v1.13.6 PATCH
**Card:** [[If a task or sub-task item - is the same name as another task item - the templater note that gets made for it gets put to the root directory of the vault]]
**Workstream:** Projects Blueprint

## Symptom

When the user clicks "+ Add a card" on a project's kanban board with a title that already exists as a task note **anywhere in the vault** (different project, sub-task in same project, or any same-name `.md` matchable by Obsidian's link resolver), the newly-created Templater note orphans at vault root (or in the wrong project's `tasks/` folder) instead of being auto-promoted into `spice/projects/<correct-slug>/tasks/<title>/<title>.md`.

The workstream picker may also bind to the wrong project's atlas, so the new card carries a `workstream:` from a sibling project.

## Root cause (two-headed)

### (1) Source-board detection is title-substring-based, not path-based

`ranch/templates/Template, Kanban Card.md` :: `_findSourceKanbanBoard(title)` runs two strategies:

- **Strategy 1 (v0.51.1, cache-first):** `app.metadataCache.getBacklinksForFile(newFile)`. Returns backlinks indexed against the new file's path. Obsidian's link resolver maps `[[title]]` to the **first** matching basename. When two same-name `.md` files exist, the link resolves to the older one — so the new file has no incoming backlinks at template-run time. Cache lookup returns empty; fall through.
- **Strategy 2 (v0.56.2, vault-scan):** enumerates kanban-plugin boards; collects boards whose body contains the `[[<title>]]` substring; sorts by mtime descending. Both Project A's board (pre-existing link to Task A) and Project B's board (freshly-appended link by kanban-plugin) match the substring. mtime tiebreaker is fragile — recent edits to either board for unrelated reasons flip the winner.

Result: source board can resolve to the **wrong project** under collision.

### (2) Auto-promote silently skips on target-exists

Bottom of template:

```js
const existing = app.vault.getAbstractFileByPath(newTargetPath + ".md");
if (!existing) {
    await tp.file.move(newTargetPath);
}
```

When `newTargetPath` is occupied (most commonly: the same-name task already exists at `spice/projects/<slug>/tasks/<title>/<title>.md`), the move is skipped and the new file remains wherever Templater initially placed it — typically vault root.

## Fix design

### (A) Strategy 0 — sibling-board detection (new, runs FIRST)

Before Strategy 1, check whether `tp.config.target_file.path`'s parent directory contains exactly one kanban-plugin board file. If yes, that board IS the source.

This works because kanban-plugin v2.0.51's `createNewMarkdownFile(folder, title)` uses the clicked board's `new-note-folder` setting. For every project task-board in the sauce blueprint, `new-note-folder` is set to the board's own folder — so the new file lives next to its source board.

```js
// Strategy 0: directory-of-target sibling-board detection. Bypasses
// title-substring pitfalls when the new file is sibling to its source.
const newFileForStrategy0 = tp.config.target_file
    || tp.file.find_tfile(tp.file.path(true));
if (newFileForStrategy0) {
    const targetPath = newFileForStrategy0.path || "";
    const lastSlash = targetPath.lastIndexOf("/");
    const targetDir = lastSlash >= 0 ? targetPath.substring(0, lastSlash) : "";
    if (targetDir) {
        const siblingBoards = app.vault.getMarkdownFiles().filter(f => {
            if (!f.path.startsWith(targetDir + "/")) return false;
            if (f.path === targetPath) return false;
            // sibling only — exclude descendants
            const rest = f.path.substring(targetDir.length + 1);
            if (rest.includes("/")) return false;
            const cache = app.metadataCache.getFileCache(f);
            return cache?.frontmatter?.["kanban-plugin"] === "board"
                || f.path.endsWith("-board.md");
        });
        if (siblingBoards.length === 1) {
            return siblingBoards[0].path;
        }
        // siblingBoards.length === 0 or > 1 → ambiguous, fall through
    }
}
```

Falls through cleanly when:
- new file at vault root (no project context)
- multiple boards in same dir (ambiguous — let Strategy 1/2 resolve)

### (B) Auto-promote suffix disambiguation

When `newTargetPath` already exists, append `-2`, `-3`, ... to the filename **and the folder** until a free slot is found. Move the new file to the suffixed path. Show a Notice surfacing the disambiguation.

```js
// Auto-promote with name-collision disambiguation.
const sourceParts = activePath.split("/");
const projectsIdx = sourceParts.indexOf("projects");
if (projectsIdx >= 0 && projectsIdx + 1 < sourceParts.length) {
    const projectDir = sourceParts.slice(0, projectsIdx + 2).join("/");
    const fileName = tp.file.title;
    let chosenName = fileName;
    let suffix = 2;
    // Disambiguate against existing per-task folder + file. Loop is bounded
    // (suffix stops at 999) but in practice collisions are 0-or-1 deep.
    while (
        app.vault.getAbstractFileByPath(`${projectDir}/tasks/${chosenName}/${chosenName}.md`)
        && suffix <= 999
    ) {
        chosenName = `${fileName}-${suffix}`;
        suffix++;
    }
    const newTargetPath = `${projectDir}/tasks/${chosenName}/${chosenName}`;
    if (!app.vault.getAbstractFileByPath(newTargetPath + ".md")) {
        await tp.file.move(newTargetPath);
        if (chosenName !== fileName) {
            new Notice(`Task name "${fileName}" already exists in this project. Saved as "${chosenName}".`, 6000);
        }
    }
}
```

## Out of scope

- Source-board detection by **path-prefix matching against kanban-plugin board settings**. Would require parsing each board's `new-note-folder` to disambiguate Strategy 0 when sibling-boards is empty. Complex; not needed for the common case.
- Cross-vault sync race conditions (Obsidian Sync touch-bumps mtime). Not addressable in template-only code.
- Reform of the per-task-folder convention itself (e.g., adding a UUID suffix at creation time). Would require coordinated blueprint redesign across `project`, the kanban template, and the `ProjectTaskCreateListener` — out of PATCH scope.
