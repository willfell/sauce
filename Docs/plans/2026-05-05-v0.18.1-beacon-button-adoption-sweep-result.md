## In-cycle CFs — chronological

> [!example]+ CF-1 (commit 5daba3b; to-do 0.1.3 → 0.1.4): Today To-Do template trailing-space trim
> **Smoke trigger:** Smoke C. After clicking the renamed "To Do" nav-button on a daily note, the freshly created `beacon/to-do/2026/05-May/ToDo-2026-05-05.md` file's first task line was `- [ ] ` (with a trailing space). Templater + folder-template mechanics auto-applied the template body verbatim, including the trailing space. The space caused the empty checkbox to register as an open task in the daily dashboard's tasks panel queries.
>
> **Root cause:** Pre-existing latent defect from the to-do blueprint's original v0.4.0 authoring. The template's first task line had `- [ ] ` (space after `]`) intended as a "ready-to-fill" placeholder, but it registered as an open task in Dataview-style task queries (which check for the open-checkbox shape `- [ ]` followed by zero or more characters).
>
> **Fix:** Drop the trailing space → `- [ ]`. The line is still a usable empty checkbox visually, but no longer parses as an open task.
>
> **Versions:** to-do 0.1.3 → 0.1.4 (in-cycle re-process bump per gotcha 9; the installer's `subscribed.version === installed.version` short-circuit at `install.js:111` would skip re-process if version unchanged; bumping forces template re-deployment to barebones).
>
> **Files:** `platform/blueprints/to-do/templates/Today To-Do.md` + `platform/blueprints/to-do/manifest.json` + `platform/manifest.json` + `../barebones-beacon-poc/Docs/Meta/platform-subscription.json`.

## NEW v0.18.1 lessons (carry forward to v0.19.0+)

> [!info] BeaconButton adoption is fast
> 1. **Pure adoption migrations of an existing mechanism land in single-task subagent dispatches with zero spec/code review loops.** T1.1 (new-meeting-button) + T1.2 (workstream Add/Remove) each took one subagent dispatch with no rework. Mechanism API contract was QUOTED literally in the design doc (v0.6.0 lesson) → subagent had unambiguous instructions → clean first-pass. Pattern reusable for any future BeaconButton adoption (e.g., trips status mutation, project board card actions).

> [!info] Trailing-space template defect class
> 2. **Template-body invisible defects can survive multiple cycles undetected.** The Today To-Do template's `- [ ] ` (trailing space) had been latent since the v0.4.0 to-do blueprint authoring. It surfaced ONLY when the renamed nav-button forced the user to re-experience fresh file creation. Pattern: future blueprint authoring should grep for trailing whitespace in template bodies before merge (`grep -n " $" templates/*.md`). A possible v0.19.0 docs polish task: add a trailing-whitespace check to the helper-cases harness for all blueprint templates.

> [!info] Reserved-headroom-UNUSED precedent extended (4 data points)
> 3. **v0.13.0 + v0.14.0 + v0.18.0 + v0.18.1 (reserved-headroom-UNUSED-or-near-unused) precedent now has 4 data points.** v0.18.1 was almost-unused (1 CF instead of 0; CF was a latent template defect, not migration-driven). The "pure adoption / pure dedupe / pure label edit" cycle shape reliably lands clean. Decision rule reinforced: new mechanism / pattern extraction / pure adoption → expect clean; new UI surface → expect CFs.

> [!info] Combined cycles (BeaconButton sweep + label cleanup) are bundle-friendly
> 4. **When two unrelated micro-tasks share a release window, bundling them into one PATCH cycle is cheaper than two separate PATCH cycles.** v0.18.1 bundled the BeaconButton adoption sweep + nav-button label cleanup despite the surfaces being independent (BeaconButton calls in 2 helper classes; label edits in 2 manifests). Single commit, single tag, single set of harness runs, single set of smokes. The decision was made mid-brainstorm when the user surfaced the label cleanup as additional scope. Pattern: future PATCH cycles should opportunistically bundle small unrelated polishes if the manual smoke matrix doesn't grow significantly.
