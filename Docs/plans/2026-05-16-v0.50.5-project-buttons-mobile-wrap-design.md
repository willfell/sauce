# v0.50.5 — Project nav-buttons wrap on narrow widths (design)

**Date:** 2026-05-16
**Card:** [[Project Buttons in tasks can be cluttered on phone]] (Projects Blueprint sub-board)
**Cycle target:** PATCH v0.50.4 → v0.50.5

## Symptom

On the Obsidian mobile app, the `ProjectNavButtons` row renders 3–5 buttons on a single horizontal flex line. Long task or card names (e.g., `Frontmatter Default Doesn't Show`, `Convenience Functionality`) cause:

- Label text to truncate or squish inside buttons (no `white-space` rule + `flex: 1` + `min-width: 0` lets buttons shrink below their content width).
- The whole row to overflow the screen edge.

User-reported example path: `spice/projects/sauce/tasks/Convenience Functionality/board/Frontmatter Default Doesn't Show/Frontmatter Default Doesn't Show.md` — a `task-board-card` context already trimmed to 3 buttons (Card Board · Task: X · Project Hub). Even at 3 buttons, long labels make the row crowded.

## Root cause

`ProjectNavButtons.render()` at `platform/blueprints/project/helpers/project-nav-buttons.js` (~line 416–446) sets:

```css
container { display: flex; flex-wrap: nowrap; gap: 6px; }
button    { flex: 1; min-width: 0; /* + padding, border, etc. */ }
```

This forces all buttons onto one row and lets each shrink to fit the available space. When the total content width exceeds the container, every button shrinks proportionally; labels with no `white-space: nowrap` then break across lines inside each button, or get clipped by the inner span's natural overflow behavior.

## Fix

Three CSS edits in `ProjectNavButtons.render()`:

1. **Container:** `flex-wrap: nowrap` → `flex-wrap: wrap`. Allows buttons to flow to multiple rows when one row is full.
2. **Button:** `flex: 1` → `flex: 0 1 auto`. Each button takes its content width (the `auto` basis); allowed to shrink under flex pressure (the `1`) but not grow (the `0`).
3. **Label span:** add `white-space: nowrap` to the inline label `<span>`. Forces the label to stay on a single line; if the button's natural width exceeds the row's remaining space, the button wraps to the next flex row rather than the label wrapping inside.

Drop the now-unnecessary `min-width: 0` (paired with `flex: 1` so flex items could shrink below content width; obsolete with `flex: 0 1 auto`).

**Combined behavior:**
- Wide desktop window: everything fits on one row, looks identical to today.
- Narrow desktop window or mobile: buttons that don't fit drop to a second (or third) row; each button is sized to its content; no intra-button label wrapping.

## Why this approach

User picked this over three alternatives:

- **Mobile icon-only:** would lose information (some icons are confusable — board vs map). Less reversible. Rejected.
- **Conditional button trim by context:** would lose functionality on mobile. Higher complexity (per-context branches). Rejected for this cycle.
- **Multi-row split via row-count math (SpaceNavButtons pattern):** more complex, less responsive to actual content width. Rejected.

The wrap approach is the minimum-blast-radius option: pure CSS, no JS branching, no `app.isMobile` check, no manifest changes.

## Scope

In-scope for v0.50.5:
- `ProjectNavButtons.render()`'s main button row (lines ~416–446).

Out of scope (FIX-LATER candidates):
- AccentButton rows rendered below the nav row (Create Board / Bump Status / Set Teams / Set Products on `task-hub` and `task-note` contexts). They use a similar layout pattern and may also crowd on mobile — separate inspection cycle.
- `renderTaskNoteTiles` (different surface; cards, not button row).
- The workstream widget below the button row (single inline widget, unaffected).
- Other blueprints' nav-buttons (boards, daily, meetings, etc.) — separate cycles if their renderers show the same pattern.

## Test additions

`platform/test/run-helper-cases.js` already grep-asserts source strings in helper files. Add 3 sub-asserts:

- **PNB-WRAP-1:** `project-nav-buttons.js` contains `flex-wrap: wrap` (in the main button container, not the workstream widget which uses `gap: 8px` and no flex-wrap).
- **PNB-WRAP-2:** `project-nav-buttons.js` contains `flex: 0 1 auto` (the button's flex value).
- **PNB-WRAP-3:** `project-nav-buttons.js` contains `white-space: nowrap` (on the label span).

Source-string asserts are sufficient — no behavior to test beyond "the CSS values are present"; the runtime CSS interpretation is the browser's job, and the renderer test harness (`run-renderer.js`) doesn't simulate viewport widths anyway.
