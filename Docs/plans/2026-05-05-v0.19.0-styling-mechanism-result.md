## In-cycle CFs

> [!example]+ CF-1 (post-S2 dogfood) — prereq gate for new-state-materializing helpers

**Surface:** S3 barebones smoke A. Barebones lacks Style Settings community plugin. Per design Section 4 Smoke A pass-criteria: when prereq absent, NOTHING should be written. Observed pre-CF-1: helpers wrote `.obsidian/themes/Baseline/`, `.obsidian/appearance.json`, and `.obsidian/plugins/obsidian-style-settings/data.json` (the last creating a phantom plugin folder Obsidian had no awareness of).

**Root cause.** The existing `applyExternalPlugins` v0.1.3 helper merely emits a warning + continues — downstream helpers are unaware. This was fine for `applyTemplaterHotkeys` etc. because they READ existing files (graceful no-op via missing-target Notice). The new styling helpers ACTIVELY MATERIALIZE state from nothing: theme dir mkdirs, appearance.json fresh-writes, plugin data.json populates. A downstream prereq miss isn't recoverable post-hoc — must short-circuit at the top.

**Fix.** New shared helper `_externalPluginsSatisfied(tp, manifest)`:
- Returns `{ok: bool, missingIds: string[]}`.
- Reads `.obsidian/community-plugins.json` once per call; conservative on read/parse failure (`ok=false`, `missingIds = required[]`).
- Each of the three new helpers calls the gate at the top and emits `info/{theme_overwrite, appearance, style_settings}` + `action: "skipped_missing_prereq"` + `missing_plugin_ids: [...]` + return early when prereq missing. No mkdir, no fresh-writes, no plugin folder phantoms.

**Bumps.** styling@0.1.0 → 0.1.1 PATCH per gotcha 9 in-cycle re-process bump rule (lockstep edits to platform/manifest.json + platform/mechanisms/styling/manifest.json + Docs/Meta/platform-subscription.json + ../barebones-beacon-poc/Docs/Meta/platform-subscription.json — 4 files).

**Verification.** Post-CF-1 barebones install: 3 `skipped_missing_prereq` events; zero theme files, no appearance.json, no plugins/obsidian-style-settings/ folder. Workshop re-install: 4 `skipped_existing` events (sha256 match on theme files; structural-equal on appearance + Style Settings) — no behavioral change since prereq is satisfied in workshop. Helper-cases harness still 312/0 (existing styling cases include obsidian-style-settings in community-plugins.json by default; gate is a noop for tests).

**Plan-anticipated.** Plan T3.2 explicitly flagged this as a CF candidate: *"Acceptable IF they handle missing prereqs gracefully... If helpers SHOULD be gated, add a short-circuit at the top of applyStyleSettings reading community-plugins.json and skipping if obsidian-style-settings absent — record this as inline-CF if surfaces during smoke."* The CF closed exactly as the plan anticipated.

## NEW lessons (v0.19.0)

> [!tip]+ Lessons codified
> **(a) Helpers that materialize NEW state need stronger prereq contracts than helpers that READ existing state.** `applyTemplaterHotkeys` etc. degrade gracefully via missing-target Notices because they READ existing files. The styling helpers ACTIVELY MATERIALIZE state from nothing (theme dir + appearance.json + plugin data.json), so a downstream prereq miss isn't recoverable post-hoc — must short-circuit at the top. Pattern: any helper that mkdirs/writes-fresh-from-source needs an explicit prereq check, not just upstream warnings.
>
> **(b) Shared prereq gate `_externalPluginsSatisfied`** is now reusable across any future helper that materializes state and depends on a community-plugin prereq. Returns `{ok, missingIds}` shape; conservative on read/parse failure (ok=false). Future helpers (e.g., a hypothetical Templater-content-templates auto-promoter) would call the same gate.
>
> **(c) First-of-its-kind UI mechanism cycle pulled exactly 1 CF** (vs the 2-6 the v0.5.0/v0.11.0/v0.12.0/v0.6.0/v0.16.0/v0.17.0 precedent suggested for first-of-its-kind UI cycles). Stronger than expected because:
> - Helpers were structurally analogous to existing v0.1.3/v0.3.0 helpers (3-precedent posture mirror).
> - 15 TDD-first harness cases caught all functional bugs pre-smoke.
> - The only surface that smoke surfaced was the cross-helper prereq contract, which the plan T3.2 explicitly anticipated as a CF candidate.
>
> Pattern: when the design quotes the helper posture pattern AND the harness covers the spec literally AND the plan flags potential CF surfaces upfront, even first-of-its-kind cycles can land near-clean.
>
> **(d) `require("fs")` + `require("path")` per-helper local imports** is the canonical pattern for helpers that read from outside the vault adapter (workshop). Don't promote these to module-top — keep them local for clarity that THIS helper reaches outside `.obsidian/`. Reinforces v0.1.x landmine #8 (cross-vault filesystem reads need require("fs"); desktop only).
>
> **(e) Two backup-suffix conventions** now live in the codebase and the landmine doc:
> - `.bak` for file-content overwrite (vendored themes; v0.2.0 boards Option B precedent).
> - `.beacon-backup` for plugin-data additive merge (Templater/Slash Commander/core daily-notes/appearance/Style Settings).
>
> Mixing them would be confusing — codified explicitly in landmine #12 mechanic #2. Future helpers writing to `.obsidian/` should pick the suffix matching their merge posture.
>
> **(f) Three parallel subagents implementing independent helpers in one batched message** worked cleanly: applyVendoredThemes / applyAppearance / applyStyleSettings each implemented in isolation against the same harness target; master committed the combined work in a single S2 atomic commit. Pattern: when N independent helpers share posture + the harness pre-locks behavior + each subagent has a stable insertion point (e.g., "before `// pruneNavButtonsRegistry — drop`"), parallel dispatch saves wall-time vs sequential. Mirrors v0.18.0 T2.1-T2.7 batched dispatch precedent.
>
> **(g) The v0.18.0 lesson "in-cycle re-process bump rule at scale" hit again** — CF-1's prereq gate fix required bumping styling 0.1.0 → 0.1.1 because the workshop's installed.json had styling@0.1.0 stamped from S2 dogfood, and the installer's `subscribed.version === installed.version` short-circuit (install.js:111) would have skipped re-process. Lockstep edits to 4 files (workshop manifest + mechanism manifest + workshop subscription + barebones subscription). This is the **6th data point** in the gotcha 9 in-cycle re-process bump rule precedent (after v0.6.0 + v0.17.0 + v0.18.0 + v0.18.1). Promotion to landmine **#16** is a v0.19.x candidate (the post-v0.19.0 docs polish handoff has it queued).
>
> **(h) Reserved-headroom-UNUSED precedent did NOT extend** — v0.13.0 + v0.14.0 + v0.18.0 + v0.18.1 + v0.18.2 had been a 5-data-point clean precedent. v0.19.0 broke that streak with CF-1. Pattern: reserved-headroom-UNUSED is reliable for *bug fixes scoped to one existing helper / pure dedupe / pure adoption / pure label / pure docs cycles*. NEW mechanism cycles still pull at least 1 CF, even when functionally analogous to existing precedents. The CF surface is usually the **integration contract** (here: prereq gate across helpers) not the helper bodies themselves.
