# v0.22.0 Result — Inside-vault bootstrap + lean CLI

## Summary

Flips the consumer-entry surface from a hand-typed `node ../beacon/platform/bootstrap.js` (requires user to clone + npm-install first) to a remote curl one-liner that clones beacon into `<vault>/Beacon/`, npm-installs, runs the existing first-run wizard, and writes activation + CLI surface artifacts. After bootstrap, the user activates per shell via `source Beacon/Scripts/activate.sh` and operates the vault through a `beacon` CLI with four verbs (`bootstrap` / `update` / `status` / `wizard`).

## In-cycle CFs

**S3 CF defensive hardening** (commit `f56c7fc`) — 4 IMPORTANT issues surfaced by code-quality reviewer pre-close, fixed in-cycle:

| Issue | Root cause | Fix |
|---|---|---|
| IMP-1 stdin hang on `curl | bash` | `read -r ans` blocks when stdin is the curl pipe (non-tty) | Detect `[ ! -t 0 ]`; refuse-loud with guidance to use `--overwrite` or download-and-run pattern |
| IMP-2 silent .bak destruction | `rm -rf "$BAK"` before `mv` destroys prior backup on consecutive re-installs | Timestamp the backup (`Beacon.bak.YYYYMMDD-HHMMSS`); never collide, never destroy |
| IMP-3 tilde / relative path | `--vault ~/notes` (quoted) lands as literal; relative paths break later operations | Canonicalize via `cd && pwd -P` after flag parse |
| IMP-4 half-install on npm fail | git clone succeeds but npm install fails — Beacon/ left in broken state | `trap cleanup_partial EXIT` + `INSTALL_OK=1` immediately before exec |

All 4 surfaced via parallel-stage code-quality review; fixed surgically; harnesses unchanged green; manual smoke confirmed clean.

## In-cycle observations (NOT bugs; carry-forward candidates for v0.22.x)

1. **`install.sh --non-interactive` is bash-side only.** Doesn't propagate to the node wizard; first-run wizard prompts still fire. v0.22.x candidate: propagate flag through to node CLI + wizard auto-accept-defaults mode (enables CI/headless installs), OR document the constraint clearly.
2. **Fresh-clone Beacon/ reports `dirty` immediately after install.sh.** Likely `npm install --omit=dev` reconciles `package-lock.json` against the just-cloned tree. `beacon update --force` handles it cleanly. v0.22.x candidate: investigate package-lock determinism OR add post-install untracked files to a workshop-side `.gitignore`.

## NEW lessons (v0.22.0)

(a) **Per-shell source-activation matches Python venv / nvm / pyenv conventions and avoids shell-rc edit risk** — design's lowest-blast-radius choice; each new shell needs `source Beacon/Scripts/activate.sh`; no global PATH pollution; vault is fully portable. Trade-off: one extra command per shell.

(b) **Inside-vault `<vault>/Beacon/` (capital B) coexists cleanly with the lowercase `beacon/<module>/` blueprint-content namespace** — codified as second sanctioned top-level vault dir in CLAUDE ask-before-acting + landmine #18.

(c) **`bootstrapCtxFromArgs` is the right shape for the one verb that creates state** — bootstrap is special because it MUST tolerate missing platform-config.json (it CREATES it); a synthetic minimal ctx + `--vault` arg + bypass of `resolveContext` cleanly handles this without polluting other verbs' resolution. Pattern reusable for any future verb that bootstraps fresh state.

(d) **Bash-driver harnesses use PATH-prefixed shim scripts as the inline-synthetic-body posture** — extends v0.20.0 lesson (a) to the bash layer; `withShimEnv(setup, fn)` mirrors `withTempVault(setup, fn)` shape. Shim heredocs MUST be column-zero in source. `${process.execPath}` for node passthrough avoids shimDir-vs-system-PATH recursion. macOS `/bin/bash` (3.2) vs Linux `/usr/bin/bash` (5.x) — PATH must include `/bin` for spawnSync to find macOS bash.

(e) **`install.sh --non-interactive` is bash-side only** — does NOT propagate to the node wizard; first-run wizard prompts still fire. v0.22.x carry-forward.

(f) **Fresh-clone `Beacon/` reports git status `dirty` immediately after install.sh** — likely because `npm install --omit=dev` reconciles `package-lock.json` against the just-cloned tree. v0.22.x carry-forward.

(g) **Banner version reflects live manifest at clone time** — install.sh shows the manifest.json `workshop_version` at clone time. The next install after the v0.22.0 tag shows `v0.22.0`.

(h) **TDD baseline harnesses can ship plan-quoted bugs that surface only at S2/S3** — S1 baseline shipped `withTempVault({}, ...)` per plan-verbatim instruction; helper signature mismatch surfaced via spec reviewer; S1-fix-implementer fixed helper-side rather than retroactively rewriting plan-quoted case bodies. Pattern: when a TDD baseline has plan-quoted case bodies that reveal a helper-side gap, fix the helper (additive, backward-compatible) not the case bodies (preserves plan-of-record).

(i) **`set -euo pipefail` + `trap cleanup_partial EXIT` + `INSTALL_OK=1` immediately before exec** is the canonical bash posture for orchestrators that may leave half-installed state — cleanup runs unconditionally on any non-zero exit before `INSTALL_OK=1` is set; cleanup is a no-op once exec hands off.
