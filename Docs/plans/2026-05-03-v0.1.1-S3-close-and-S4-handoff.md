> [!info]- The post-quality-review correction
> Original kanban frontmatter declared `tags: ["{{vault_identity_tag}}", "board"]` per the design doc. Quality review verified that `install.js:290 substituteLenient` only reads from `paths.variables` (i.e., `platform-config.json:variables`). It does NOT consume:
> - `variants.json` (which has the per-consumer `vault_identity_tag` values)
> - The top-level `vault_identity` field of `platform-config.json`
>
> Net effect would have been: `{{vault_identity_tag}}` lands literal in barebones's materialized `boards/To-Do-Board.md` frontmatter, producing an invalid Obsidian tag.
>
> Decision (option A of four): drop the placeholder. Per-vault `boards/To-Do-Board.md` is already vault-scoped by location; an identity tag adds little. Smallest delta; zero installer change; no per-consumer config friction. Logged inline + new landmine #9 codifies the substitution-scope constraint.

Same as the v0.1.1-S2-close handoff — see that doc's "Cross-cutting non-negotiables" section. Adds one new entry as of 2026-05-03:

> [!warning] New since S2 close
> - **Landmine #9 — installer substitution scope.** Body substitution only consumes `platform-config.json:variables`. variants.json + top-level `vault_identity` are NOT in scope. If a blueprint's content needs vault-specific tokens, either drop the placeholder, add the matching key to every consumer's `variables` map, or commit to a future install.js extension that merges variants.
