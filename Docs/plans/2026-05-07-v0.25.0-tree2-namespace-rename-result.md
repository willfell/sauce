## NEW lessons (v0.25.0)

**(a) Tree 2 was the cheapest of the three rename cycles — single source-of-truth design pays off at rename-time.** install.js:250 (the line that prefixes `beacon/` to every blueprint's `module_directory` value) was a 1-character change. Blueprint manifests UNCHANGED — they already use bare `module_directory: "<name>"`. Lesson: when designing a multi-namespace platform, put the prefix application at ONE source-of-truth (the install pipeline), not in every consumer-facing manifest. Future namespace renames inherit this cheapness.

**(b) Fresh-start posture eliminates migration-helper complexity.** Without legacy consumer state to preserve, no `applyTemplaterFolderTemplates` migration helper, no `daily-notes.json:folder` migration helper, no manual sed in install.md upgrade section. Saved an entire CF class compared to v0.24.0's CF-3 (CustomJS jsFolder migration). Lesson: when locking namespace renames at brainstorm, ASK explicitly about consumer-state-preservation. The constraint dictates whether the cycle is mass-replace + reinstall (cheap, like v0.25.0) or mass-replace + migration-helpers (medium-cost, like v0.24.0).

**(c) `.beacon-backup` retention was conditional, not permanent.** v0.23.0 + v0.24.0 lessons preserved the suffix because consumer vaults had on-disk `.beacon-backup` files from prior installer runs. Once user adopts fresh-start, those constraints retire. Lesson: backup-suffix conventions are not platform-version-stable forever; they're stable until the project takes a brand-rebrand-AND-consumer-reset window. Document the conditional as part of the convention itself (now codified in landmine #12 history block).

**(d) Spec compliance was perfect; quality found 1 NIT.** Spec reviewer found ZERO gaps across all 15 design checks. Quality reviewer found 1 NIT (landmine #11 "Known violations (legacy, awaiting future cycles)" header was stale because the named violations had been RESOLVED in v0.5.0 + v0.2.0). Lesson reaffirmed (5th data point: v0.20.0 + v0.22.1 + v0.23.0 + v0.24.0 + v0.25.0): **never skip the two-stage subagent review.** Spec compliance and code quality are different functions; spec-only review would have shipped this NIT. Pattern: budget for ~1-3 quality NITs even on clean spec-compliant cycles.

**(e) Internal harness tmpdir labels + cards-mechanism filename + `BEACON_CARDS_FILE` constant survived the brand rename intentionally.** `mkdtempSync("beacon-caseXX-")` patterns in `run-helper-cases.js` + `run-bootstrap.js` are internal-only fixtures (not user-visible namespace references). `platform/mechanisms/cards/beacon-cards.js` filename + `'beacon-cards'` selector + `BEACON_CARDS_FILE` constant in `run-renderer.js` reference the literal mechanism filename which was NOT renamed (per user choice at brainstorm — Q&A explicitly excluded the mechanism file rename). Lesson: brand-rename sed scope discipline (v0.23.0 lesson b) extends to "what NOT to rename" — internal-only tokens that share a prefix-string with the renamed namespace must be enumerated upfront. Plan-level discipline: list them in a "PRESERVED (intentionally unchanged)" section in the design + plan.

**(f) All 8 blueprint manifest.json description + post-install message strings carried `beacon/<module>/` references.** These are consumer-visible at install time (via Notice on first install). Plan's call-site inventory must always grep `description` + `message` strings in manifests; not just code. The S2 mass sed swept these in correctly because the sed scope explicitly included `platform/blueprints/`, but it would have been a missed-site if the sed scope had been narrower. Lesson reaffirmed (v0.24.0 lesson b): sed scope MUST include source dirs comprehensively — including JSON manifest fields that hold human-readable strings.

**(g) `Docs/install.md` upgrade section is the canonical fresh-start migration recipe.** Under user's fresh-start stance, the recipe is `rm -rf <vault>/<old-namespace>` + `sauce update --force`. v0.23.0 (Tree 1) + v0.24.0 (Tree 3) + v0.25.0 (Tree 2) all follow the same upgrade-section pattern. Lesson: every Tree-rename cycle adds an upgrade section to install.md mirroring the previous cycle's shape. The v0.25.0 section ships with a `[!success] Sauce rebrand sequence COMPLETE` callout summarizing the namespace tripod.

**(h) Reserved-headroom precedent extends to 12 data points; fourth MIXED case.** Pure-rename cycles draw 1-3 review-surfaced CFs. Acceptance gates (manual smoke + user-environment validation) catch the residual class of issues that harness GREEN can't reach (per v0.24.0 lesson d). v0.25.0's combination of (1) single source-of-truth core change + (2) fresh-start posture + (3) two-stage subagent review pre-commit + (4) user-confirmed manual Daily Notes acceptance gate gave a clean 1-NIT cycle close.
