## The codified module-directory invariant — locked in installer

> [!success] What blueprint authors must declare (post-v0.2.0)
> Every blueprint manifest declares a top-level `module_directory: "<name>"` field (REQUIRED; bare directory name, no `beacon/` prefix). The installer (T1.1) enforces presence + cross-blueprint uniqueness; missing field produces `error/module_directory_missing` (skips install); collision produces `warning/module_directory_collision` (first-wins by topo order).
>
> The installer (T1.2) populates a per-blueprint variables overlay where `{{module_directory}}` resolves to the literal `beacon/<bare-name>` — so a blueprint's `files[]`, `nav_buttons[].action.target`, and content files can reference `{{module_directory}}` and get clean materialization at `<vault_root>/beacon/<bare-name>/...`.
>
> Mechanisms are exempt from the rule — they're shared infrastructure, not modules; their content continues to install under `Docs/Meta/...` per existing conventions.
>
> boards@0.1.0 is the first blueprint shipped under the rule. project@0.3.0 declares `module_directory: "projects-legacy"` as a stopgap (its `files[]` doesn't reference `{{module_directory}}`, so no materialization actually happens under `beacon/projects-legacy/` — the field exists only to satisfy T1.1's required-field check and flag the future migration).

## Surprises during execution

> [!warning] Surprise 1 (resolved): applyNavButtons didn't prune empty contributions
> Discovered during T2.8's first barebones smoke. project@0.3.0 declared `nav_buttons: []` (empty array) to retire its Board contribution. But `applyNavButtons` line 844 had an early-return `if (!manifest || !Array.isArray(manifest.nav_buttons) || manifest.nav_buttons.length === 0) return;` — meaning empty declarations never reached the registry-write path. Result: project's stale v0.2.1-era `id: "board"` button persisted in `nav-buttons-registry.json:contributions.project[]` indefinitely. Fix in `install.js` `applyNavButtons`: refactored the guard so empty/missing `nav_buttons[]` ACTIVELY prunes the prior contribution from the registry (records `info/nav_buttons` + `action: "pruned_empty_declaration"` history). Surfaces in T2.8 re-run (post-fix): only `boards.boards-board` remains in the registry; project key absent. Carried into the S2 part 1 commit `54e4af1` rather than queuing as a follow-up.

> [!warning] Surprise 2 (resolved): boards-board target stored as literal `{{module_directory}}/...` placeholder
> Discovered alongside Surprise 1 during the same barebones smoke. The boards manifest declares `nav_buttons[0].action.target: "{{module_directory}}/To-Do-Board.md"` (per the plan's spec). `validateAndResolve` substituted `template_source` (basename → full content path) but passed `target` through unchanged — so the registry stored the literal `{{module_directory}}/To-Do-Board.md`, which the renderer would not be able to resolve at click time. Fix in `validateAndResolve`: added `substituteLenient(btn.action.target || "", variables)` so the `{{module_directory}}` placeholder (and any future `{{xxx}}` in target) gets resolved using the per-blueprint variables overlay (T1.2 already populates `variables.module_directory` to `beacon/<bare-name>` for blueprint installs). Post-fix the registry stores `target: "beacon/boards/To-Do-Board.md"` — resolved literal; renderer dispatches without needing per-blueprint substitution context.

> [!info] Confirmation 1: T1.4's pre_install delete idempotent on re-runs
> First barebones run fired the pre_install delete (sha `88bd05d0...`; bak preserved). Forced fresh re-install (blanked `platform-installed.json`) saw `info/pre_install_delete_skip` event with `message: "file already absent"` instead of a second delete event. The `applyPreInstall` helper's existence check correctly distinguishes "file absent" from "file present, delete it" without producing duplicate deletes or rewriting the bak file.

> [!info] Confirmation 2: project's `module_directory: "projects-legacy"` stopgap works as designed
> T1.1's collision detector saw two blueprints with distinct `module_directory` values (`projects-legacy` vs `boards`) — no collision warning. T1.2's per-blueprint overlay set project's `{{module_directory}}` to `beacon/projects-legacy`, but project's `files[]` doesn't reference `{{module_directory}}` (its dests use `{{rules_path}}`, `{{templates_path}}`, `{{scripts_path}}`, `{{commands_path}}`) — so the substitution variable is set-but-unused. Project's content lands at the same paths as before (`Docs/Meta/Templates/...`, `Docs/Meta/Scripts/project/...`, etc.) — no behavior change. The `_comment` field in project's manifest documents the choice + future migration path.

## Pacing rhythm — what worked / what to improve

### What worked

- **Per-task subagent dispatch on the four installer-extending tasks (T1.1-T1.4) + the three boards-authoring tasks (T2.1-T2.3) + the result writeup (T2.12).** Master-driven mechanical tasks (T2.4-T2.6 manifest/subscription edits, T2.7-T2.8 harness invocations, T2.13 CLAUDE.md, T2.14 commit/tag/push). Same cadence as v0.1.3 + v0.1.2 — keeps master context light + spec/code-quality review through subagents on substantive code.
- **TDD red→green discipline on all four S1 mechanics.** Each subagent wrote failing cases first, observed the red light, then implemented. Total `run-helper-cases.js` sub-asserts grew 16 → 119 across T1.1-T1.4 (M1, M2, M3, M4, M5, O1, O2, O3, O4, O5, P1, P2, P3, P4 — 14 new cases).
- **Surfacing applyNavButtons + validateAndResolve bugs during T2.8 fixed them in S2 rather than deferring.** Both bugs were pre-existing (the empty-contribution prune issue would have bitten any prior cycle that retired a button; the literal-target issue was new to v0.2.0 because boards is the first blueprint to use a substitution placeholder in `target`). Caught at T2.8, fixed in `install.js`, re-ran T2.8 to verify both bugs resolved + no regression. Net result: a cleaner installer than the plan promised.
- **`module_directory: "projects-legacy"` stopgap turned out to be the right call.** It avoided forcing a project-blueprint-migration cycle into v0.2.0's scope while still satisfying T1.1's required-field check. The `-legacy` suffix is a deliberate flag for the future migration cycle.
- **Stub-dispatch + `gitState()` from v0.1.2 carried v0.2.0 cleanly.** Every history entry written this cycle has real `git_commit` shas + null tags (no tag at HEAD until v0.6.0 lands) + `git_dirty: true` (working tree active). Landmine #14's null-tolerance contract was respected by the harness assertion + the helper code.

### What to improve

- **Plan T2.10's manual Smoke B (delete + nav-button createFromTemplate path) was implicitly verified, not explicitly run.** The user confirmed Smoke A (kanban renders) + Smoke C (card creation routes correctly), and Smoke B's expected outcome is "the file gets recreated when you click the nav-button on a missing target." Since A worked from the post-install file (which the installer materialized via the same content) + the registry shows the resolved target post-fix, Smoke B's failure modes (renderer can't read template_source / can't write target) were ruled out indirectly. Future plans should specify whether implicit-verification is acceptable or whether each manual smoke must be run discretely. For v0.2.0 the implicit verification was acceptable because the failure modes were pre-ruled-out by harness assertions.
- **applyNavButtons bug should have been caught at the plan level.** The plan T1.1-T1.4 focused on NEW mechanics (module_directory + Option B + pre_install) but didn't foresee that retiring a contribution via empty `nav_buttons[]` would expose a pre-existing pruning gap. Future plans touching subscription-aware pruning should explicitly enumerate the contribution-retirement path as a test case. Carry forward as a writing-plans discipline note.
