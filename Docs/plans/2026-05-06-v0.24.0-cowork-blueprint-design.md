# v0.24.0 — Cowork blueprint + bundled `/beacon-cowork-setup` claude-code skill

## Why this cycle

User runs an "AI coworker" workflow today in their accuris vault: a scheduled job in **Claude Cowork** (the desktop app's project + scheduling feature) fires a tiny loader prompt that says "Use `accuris-obsidian` MCP. Execute `Cowork/prompts/prompt-morning-daily-summary.md`. Read it, follow it." The actual prompt — with `## MCP Routing`, `## Context Dependencies`, and `## Instructions` sections — lives in the vault. Schedule never edits; prompts iterate freely; vault is source of truth.

Beacon currently doesn't model this. Every consumer (accuris / ero / headspace) hand-rolls their own `Cowork/` directory. There's no canonical structure, no installable defaults, no skill that scaffolds a new consumer's cowork setup.

**v0.24.0 ships:**

1. A self-contained `cowork` blueprint at `beacon/cowork/` with hub + prompts library + time-routed summary directories.
2. A bundled claude-code slash command `/beacon-cowork-setup` that installs into `<vault>/.claude/commands/`. The skill interviews the user (role, cadence, context paths, MCP id) and produces tailored prompt notes + the loader prompt the user pastes into Claude Cowork.
3. Self-healing prompts — every prompt the skill emits includes a canonical `## Self-Heal Protocol` block + `## Iteration Log` sink so prompts iterate themselves over real failure modes (3-retry cap; user-intent escape hatch; git-reviewable changes).

**v0.24.0 explicitly does NOT ship** (each becomes a future cycle once cowork proves the ergonomics):
- `event-log` mechanism
- `stable-id` mechanism
- `schema-contract` mechanism
- Inbox / approval-queue blueprint
- Wiki blueprint
- `claude://` URL integration / obsidian-shellcommands wiring

## Scope and locked decisions

### Section 1 — Namespace

`beacon/cowork/` (lowercase, per landmine #11). User migrates `accuris/Cowork/` → `accuris/beacon/cowork/` post-v0.24.0 with a one-line edit to their existing claude-cowork scheduled-job loader (path string updated; MCP id unchanged).

**Rejected:** capital `Cowork/` at vault root. Would have required landmine #11 exemption + new "root-level blueprint" manifest field; substantial platform change for no compounding value.

### Section 6 — Bundled `/beacon-cowork-setup` skill

A claude-code slash command that lives at:
- **Workshop master:** `platform/blueprints/cowork/commands/beacon-cowork-setup.md`
- **Installed at consumer:** `<vault>/.claude/commands/beacon-cowork-setup.md`
- **Invocation:** user types `/beacon-cowork-setup` in claude-code at the vault directory.

**Pulls in 3 platform deltas:**
1. **NEW allowlist path** `.claude/commands/` (12th — landmine #12 expansion; user-approved at design close). Posture mirrors v0.21.0 plugin vendoring: atomic write + `.beacon-backup` on overwrite + path-traversal validator.
2. **NEW manifest mechanic** — `files[].target_root: "claude_commands" | "vault"` discriminator (default `"vault"`). Reuses existing file-write helper; no new helper. Helper count stays at 9. Discriminator-based routing chosen over a separate `claude_commands[]` field for minimum schema delta.
3. **NEW canonical platform variable** `cowork_mcp_id` — added to `CANONICAL_VARIABLES` set in `platform/bootstrap-lib/wizard.js`. Wizard prompts at first-run if cowork blueprint is selected: "Which MCP id should Cowork prompts route to? (e.g., accuris-obsidian)". Defensive augment-on-every-run for older configs (per v0.21.0 lesson f).

**Skill behavior (interview-driven):**

1. **Auto-detect** — read `<vault>/Docs/Meta/platform-installed.json`; list installed blueprints. Read claude-code MCP config; offer detected vault-MCP as default.
2. **Ask role / context** — "What do you do? What does this vault hold?"
3. **Ask cadences** — multiselect: daily morning / EOD / weekly / monthly.
4. **Per cadence: what to summarize** — open-ended; AI maps to canonical sections.
5. **Per cadence: context paths** — pre-checked from detected blueprints (meetings/notes, journal, to-do, daily, project); user adjusts.
6. **Output target** — defaults to beacon constants; user can override (warned).
7. **Confirm** — show a preview of each prompt-to-be-written; user approves.

**Skill writes** (idempotent — backup + replace if existing prompt files present; never silent overwrite):
- `beacon/cowork/prompts/prompt-<cadence>-<purpose>.md` — one per selected cadence, populated with the user's tailored Context Dependencies + Instructions, the constant Self-Heal Protocol, and an empty Iteration Log.

**Skill outputs** (printed to stdout, copy-paste-ready, one per cadence):

```text
─── LOADER PROMPT for Claude Cowork — paste into your scheduled job ───
Use the `<mcp-id>` MCP for ALL vault reads and writes. Do NOT use other vault
MCPs. If a referenced file is missing in this vault, treat that as a hard error
and report it; do not substitute a similarly-named file from another vault.
Execute `beacon/cowork/prompts/prompt-<cadence>-<purpose>.md`. Read that prompt
first (MCP Routing + Context Dependencies sections), then follow its instructions
end to end.
──────────────────────────────────────────────────────────────────────
```

**Constants (beacon ships) vs Dynamics (skill discovers / asks):**

| Constant | Dynamic |
|---|---|
| Directory layout `beacon/cowork/{prompts,dailies,weeklies,monthlies}/` | User's role / job context |
| Date routing `YYYY/MM-MMMM/` for dailies, `YYYY/` for week/month | Which cadences to enable |
| Filename pattern `Cowork-<Period>-<DateToken>.md` | What sections each summary captures |
| Six-section prompt shape (MCP/Context/Instructions/Output/Self-Heal/Iteration Log) | MCP id (accuris-obsidian / headspace-obsidian / custom) |
| Loader-prompt template | Vault paths to load as context |
| Frontmatter shape (`type`, `prompt_id`, `prompt_kind`, `output_target`, `created`) | Tone / voice / output formatting preferences |
| Self-Heal Protocol verbatim text | Custom user-added context paths |

**Re-runnable** — running `/beacon-cowork-setup` again detects existing prompt files and offers: replace (with `.bak`) / edit one prompt / cancel.

**Discoverability** — the Cowork hub ships a `[!tip]` callout: "Set up Claude Cowork: run `/beacon-cowork-setup` in claude-code at this vault."

### Section 7 — Self-Heal Protocol (constant block in every prompt)

Verbatim text the skill bakes into every prompt's `## Self-Heal Protocol` section:

```markdown
## Self-Heal Protocol

If executing this prompt fails or produces output that doesn't match the spec:

1. **Diagnose** — identify the root cause (missing file, ambiguous instruction,
   MCP error, malformed expected output, schema drift, etc.).
2. **Classify** —
   - *Prompt-clarity issue* (wording, stale path, missing fallback, loose
     output spec) → proceed to step 3.
   - *User-intent issue* (the prompt asks for X but X conflicts with what the
     user actually wants) → STOP. Append a `[!warning]` callout to the
     `## Iteration Log` describing the conflict. Do NOT write the summary
     output. Exit.
3. **Edit in place** — fix THIS prompt file (clarify phrasing, update stale
   path, tighten output spec, add a fallback branch for the failure mode).
4. **Log** — prepend to `## Iteration Log`:
   `- YYYY-MM-DD HH:MM: <what changed> — <why> — error: "<msg>"`
5. **Retry** — re-execute with the fix applied. Hard cap: **3 retries per
   run**. If still failing after 3, append a `[!failure]` entry to Iteration
   Log with the final error state and exit.
6. **Never** silently rewrite the user's `## Instructions` section to mean
   something semantically different. Edit phrasing / paths / output-spec
   freely; preserve user intent.

## Iteration Log
<!-- Self-heal appends entries here. Newest first. Reviewable in git. -->
```

**Why this works:**
- Prompts in `beacon/cowork/prompts/` are markdown files — git-tracked in any consumer that's a git repo. Every self-heal edit is reviewable in `git log`.
- 3-retry hard cap kills runaway loops.
- "User-intent" escape hatch prevents AI from silently changing what the user actually asked for; flips to halt + alert.
- Iteration Log gives transparent visibility — user can scrub what changed and revert in git.

### Section 10 — Risks + mitigations

| Risk | Mitigation |
|---|---|
| `.claude/commands/` allowlist expansion is the first allowlist surface OUTSIDE `.obsidian/`. Sets precedent for future claude-code-side beacon artifacts. | Codify in landmine #12: "claude-code-side allowlist subset is `.claude/commands/` only; future expansions (e.g., `.claude/skills/`, `.claude/settings.json`) require explicit user approval." |
| Self-heal protocol could iterate prompts in destructive directions over time. | 3-retry hard cap; `[!warning]` halt-and-alert on user-intent issues; all changes git-reviewable; user can revert via `git checkout`. |
| `/beacon-cowork-setup` skill body is non-trivial markdown. Drift between skill body and the canonical prompt shape (Section 3) could cause skill to emit invalid prompts. | Skill body QUOTES the prompt-shape sections literally (per v0.6.0 lesson — API-contract quoting). Harness extension: lint that the skill markdown contains the canonical prompt-shape headers. |
| Re-running the skill could clobber user-tailored prompts. | Idempotent posture: detect existing → offer replace-with-bak / edit-one / cancel. Never silent overwrite. |
| Claude Cowork desktop app behavior could change (new schedule format, new prompt syntax). | Beacon owns the IN-VAULT shape (prompts + loader template). Claude Cowork side is documented but external; any drift surfaces as a `Docs/use.md` doc-only patch in a future cycle. |
| Existing accuris/Cowork/ users (the user themselves) need a one-time migration. | Document migration in `Docs/use.md` "Adopting cowork blueprint into an existing accuris-style vault" — `mv accuris/Cowork accuris/beacon/cowork` + one-line edit to scheduled-job loader prompts. |
