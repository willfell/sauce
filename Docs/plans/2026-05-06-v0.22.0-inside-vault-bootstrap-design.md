# v0.22.0 — Inside-vault bootstrap + lean CLI

## Cycle goal

Flip beacon's consumer-entry surface from `node ../beacon/platform/bootstrap.js` (hand-typed, requires user to clone + npm-install first) to a single remote curl one-liner that clones the workshop into the consumer vault as `<vault>/Beacon/`, npm-installs, runs the existing first-run wizard, and writes activation + CLI surface artifacts. After bootstrap, the user activates per shell via `source Beacon/Scripts/activate.sh` and operates the vault through a `beacon` CLI with four verbs (`bootstrap`, `update`, `status`, `wizard`).

**Two layouts coexist** — the existing dev-sibling layout (`/Users/willfell/Documents/obsidian/sync/workshop/beacon/` with sibling POC vaults `barebones-beacon-poc/` + `accuris-beacon-poc/` referencing it via `workshop_relative_path: "../beacon"`) is unchanged. Only fresh consumer vaults bootstrapped via the curl one-liner get the new inside-vault layout.

**Multi-theme presets DEFERRED** to v0.23.0+ per user direction (post-v0.21.1 handoff recommended multi-theme as primary; user pivoted to bootstrap orchestration this cycle).

## Architecture (high level)

```
[USER]                                         [VAULT]                  [GITHUB]
  │                                                │                        │
  │ curl -fsSL .../install.sh | bash              │                        │
  ├────────────────────────────────────────────► curl pulls install.sh ◄──┤
  │                                                │                        │
  │   bash subshell:                               │                        │
  │     preflight (node? git? cwd?)                │                        │
  │     prompt vault path                          │                        │
  │     git clone --depth=1 ────────────────► <vault>/Beacon/ ◄────────────┤
  │     cd Beacon && npm install                   │                        │
  │     exec node platform/cli/beacon-cli.js bootstrap                     │
  │                                                │                        │
  │       cmd-bootstrap.js:                        │                        │
  │         phaseFirstRunWizard ──────────► <vault>/Docs/Meta/{config,sub}  │
  │         phaseFetchPlugins ──────────────────────────────────────────────┤
  │         phaseRunInstaller ───────────► <vault>/Docs/Meta/...            │
  │         phaseWriteActivation ────────► <vault>/Beacon/Scripts/{...}     │
  │                                                │                        │
  │   "Activate with: source Beacon/Scripts/activate.sh"                   │
  │                                                │                        │
  │ source Beacon/Scripts/activate.sh              │                        │
  ├──────────────────────────────────► PATH=$PATH:.../Beacon/Scripts        │
  │                                                │                        │
  │ beacon status / update / wizard                │                        │
  │   → cmd-*.js → walk cwd ancestors → load config → dispatch              │
```

`beacon` is vault-scoped: the CLI walks cwd ancestors looking for `Docs/Meta/platform-config.json`. From any subdirectory inside an activated vault, every verb is one short command.

### Implementation-shape decisions absorbed

- **Remote script: bash** (not node). The curl|bash idiom is universal; bash is the right layer to detect `node`/`git` missing.
- **CLI structure: per-verb files** under `platform/cli/cmd-*.js` (vs single 1000-LOC switch). Keeps each verb under 150 LOC; parallel-subagent-friendly per v0.21.1 lesson (a) — insertion points >200 LOC apart.
- **Color: ANSI escapes via `colors.js`** (vs adding `chalk`/`picocolors`). Keeps dep tree at the existing `@inquirer/prompts` only.
- **TTY detection**: `colors.js` falls back to no-color when stdout is not a TTY (CI / harness).
- **`beacon update` semantics**: `git fetch + git reset --hard origin/main` inside `Beacon/`. Failure-loud if working tree dirty; `--force` overrides. Re-`npm install` if `package.json` SHA changed. Re-run installer phase. Treats `Beacon/` as managed-not-edited (mirrors landmine #15 vendored-theme posture).

## Data flow — three contexts

### (1) Install-time

`curl -fsSL https://raw.githubusercontent.com/willfell/beacon/main/install.sh | bash`

```
curl pulls install.sh
  → bash subshell starts
    → preflight: node? git? cwd looks like a vault (.obsidian/) or empty dir?
    → prompt: "Vault path?" (default: cwd)
    → prompt: "Clone beacon into <vault>/Beacon? [Y/n]"
    → git clone --depth=1 https://github.com/willfell/beacon <vault>/Beacon
    → cd <vault>/Beacon && npm install --omit=dev
    → exec node <vault>/Beacon/platform/cli/beacon-cli.js bootstrap --vault <vault>
      → phaseFirstRunWizard (existing wizard; default workshop_relative_path="Beacon")
      → phaseFetchPlugins (existing)
      → phaseRunInstaller (existing)
      → phaseWriteActivation (NEW): generates <vault>/Beacon/Scripts/activate.sh + Scripts/beacon
    → final report: "Activate with: source Beacon/Scripts/activate.sh"
```

### (2) Activate-time (per shell)

```
source Beacon/Scripts/activate.sh
  → exports PATH="<abs-vault>/Beacon/Scripts:$PATH"
  → exports BEACON_VAULT="<abs-vault>"
  → prints: "beacon active. Try: beacon status"
```

### (3) CLI-time

```
beacon <verb> [args]
  → wrapper exec's node <workshop>/platform/cli/beacon-cli.js
    → walk cwd ancestors → find Docs/Meta/platform-config.json → resolves vault
      (fallback: $BEACON_VAULT env var if set; error otherwise)
    → load config + subscription + workshop manifest
    → dispatch:
        bootstrap → cmd-bootstrap.js (re-run from clean state; rare)
        update    → cmd-update.js   (git fetch + reset Beacon/, re-npm-install if pkg changed, re-run installer)
        status    → cmd-status.js   (read-only diff: workshop git head vs origin, sub'd vs installed versions, drift)
        wizard    → cmd-wizard.js   (delegate to existing runReRunWizard from bootstrap-lib)
```

The CLI is a thin dispatcher; heavy logic lives in `bootstrap.js` phases (refactored). `cmd-update.js` is the only verb with substantial new logic — `git fetch + reset --hard origin/main` plus the dirty-tree gate.

## Error handling (failure-loud across all surfaces)

| Surface | Failure | Behavior |
|---|---|---|
| `install.sh` | `node` missing | Print install hint (Homebrew / official installer link) + exit 1 |
| `install.sh` | `git` missing | Same posture |
| `install.sh` | Vault path does not exist | Prompt to create or pick a different one |
| `install.sh` | `<vault>/Beacon/` already exists | Prompt: "overwrite? back up as `Beacon.bak`?"; default `n` |
| `install.sh` | `npm install` fails | Surface stderr + exit 1; `Beacon/` left in place for retry |
| `cmd-update` | Working tree dirty inside `Beacon/` | Fail loud with `git status --short` output; require `--force` to override |
| `cmd-update` | `package.json` SHA changed | Re-run `npm install` automatically (no separate flag) |
| CLI dispatcher | cwd has no `Docs/Meta/platform-config.json` ancestor + no `BEACON_VAULT` env var | Print: "Not inside a beacon-managed vault. cd into one or set BEACON_VAULT"; exit 1 |
| `phaseWriteActivation` | Activation file write fails | Backup-on-edit posture (`.beacon-backup`); failure-loud entry in install history |

## Constraints carried forward (non-negotiable; from v0.21.1 handoff + CLAUDE.md)

- No personal content in workshop.
- JSON only (#6).
- Failure-loud posture (#12 mechanics for any new `.obsidian/` path; mirrored for new `Beacon/Scripts/` write paths).
- Substitution variables only from `platform-config.json:variables` (#9), with `{{module_directory}}` overlay for blueprints.
- **In-cycle re-process bump rule (LANDMINE #16)** — when reusing in-cycle staged work at version X, in-cycle revisions MUST bump X+1.
- **API-contract quoting (v0.6.0 lesson)** — existing mechanism / phase API contracts must be QUOTED literally in the plan doc (especially for the bootstrap.js phase refactor — quote each new phase signature).
- **Helpers-that-materialize-new-state need explicit prereq gates** (v0.19.0 CF-1; v0.21.1 broadened gate). `phaseWriteActivation` materializes new state — add an explicit prereq check (Beacon/ exists; vault path resolved).
- **Two backup-suffix conventions (v0.19.0 lesson e)** — `.bak` for file-content overwrite (vendored themes; install.sh `Beacon.bak` mirrors this); `.beacon-backup` for plugin-data additive merge.
- **Test isolation via inline synthetic bodies (v0.20.0 lesson a)** — `run-install-sh.js` mocks `git` + `npm` via `PATH`-prefixed shim scripts (inline synthetic-body posture extended to bash-driver harness).
- **Cross-doc line-citation drift (v0.20.0 lesson c)** — any cycle that touches a file cited as `<filename>:<lineno>` in any doc should grep all docs for that pattern.
- **Bootstrap network posture (LANDMINE #17, v0.21.0)** — bootstrap.js is the platform's only network gateway; install.js stays filesystem-only. v0.22.0 EXTENDS this: `install.sh` is the SECOND network gateway (curl + git clone + npm install); CLI verbs (`beacon update`) are the THIRD (git fetch). Update landmine #17 wording or add #18.
- **Canonical helper posture stack (v0.21.0 lesson d)** — atomic write + backup-on-overwrite + path-traversal validator. Apply to `phaseWriteActivation`.
- **Defer-then-validate-then-load wizard architecture (v0.21.0 lesson e)** — `install.sh` defers workshop-existence validation until AFTER the vault prompt + clone.
- **Defensive augment-on-every-run for new schema fields (v0.21.0 lesson f)** — additive-only augment; never overwrite user-supplied values. Apply to any new config field added by `phaseWriteActivation`.
- **Parallel-subagent dispatch (v0.21.1 lesson a)** — insertion points 800+ lines apart with literal "don't touch any other line" gating in prompts. S2 dispatches 4 cmd-*.js subagents in parallel.
- Module-directory invariant under lowercase `beacon/` (#11) is UNCHANGED — capital `Beacon/` is a new top-level platform-managed dir, not a blueprint module-directory.
- `.obsidian/` allowlist at 11 paths (#12 v0.21.1 expansion) — UNCHANGED in v0.22.0; CLI does not touch `.obsidian/`.
- Stub content-static (#13) — never re-edit any `Docs/Meta/Templater/platformInstall.js`. Preserved.
- `gitState()` best-effort (#14) — null-tolerant. Reused by `cmd-status.js`.
- Workshop dogfoods every release.
- Desktop-only installer (#8 v0.21.0 amendment).
- **Ask before:** workshop_version bump, force-push, history rewrite, tag deletion, modifying any consumer's stub body, editing `platform/installer-stub.js`, creating top-level vault directory (this cycle codifies `Beacon/` as sanctioned), adding 12th+ `.obsidian/` allowlist path, editing a consumer's `platform-installed.json` by hand.

## Reserved-headroom precedent

8 data points (7 UNUSED + v0.21.0 USED 5 CFs).

- curl/bash → node hand-off (argv plumbing, exit-code propagation)
- cwd-ancestor walk corner cases (vault inside vault; symlinked vault; trailing slash)
- Activation absolute-path edge cases (path with spaces; path with non-ASCII; symlinked workshop)
- Sync-exclusion doc feedback (which providers user actually uses)
- Dirty-tree handling on `beacon update` (what counts as dirty — untracked? unstaged? both?)
- Visual layer terminal-compat (different terminal apps render box-drawing chars differently)

Plan reserves `0.22.x` headroom for these.
