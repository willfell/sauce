# Design — v0.1.x validator-subsystem patch

> [!abstract] Goal
> Close the four T4.7 findings surfaced by v0.1.1's manual smokes. Ship `external_plugins[]` schema + Slash Commander dependency declaration, defensive `validate.js` input resolution, semantic `checkRequiredBlocks` rewrite against the `kind`/`must_call`/`via` rule schema, and `Validate.md` + `Audit.md` runner templates. **No `workshop_version` bump** — workshop stays at `0.4.0` so the next cycle (v0.1.2 git-init) tags the patched state as `v0.4.0`.

> [!info] Why this design exists
> v0.1.1 closed 2026-05-04 with T4.6 PASS and T4.7 DEFERRED. T4.7 surfaced four pre-existing v0.1.0-era latent bugs: (1) no slash-command UX for `/new-project` / `/audit`, (2) no runner UX for the validator/audit mechanisms, (3) `validate.js` input resolution broken against current Templater's `tp.file` wrapper, (4) `validate.js:checkRequiredBlocks` schema mismatch with the project rule's `kind`/`must_call`/`via` schema. The validator hook fires on every template execution today; the bugs are user-facing live. v0.1.x ships first (recommended sequencing v0.1.x patch → v0.1.2 → v0.2.0).

## §1 — Manifest schema delta (`external_plugins[]`)

> [!warning] New optional manifest field
> Lands on **both mechanism and blueprint manifests** uniformly so v0.2.0 reuses the schema unchanged for its Kanban plugin dependency. Each entry shape:
>
> ```json
> "external_plugins": [
>   { "id": "<obsidian-plugin-id>", "required": true, "reason": "<one-liner>" }
> ]
> ```

### Installer behavior (`install.js`)

After file copies + post_install steps, if the manifest's `external_plugins[]` is non-empty:

1. Read `.obsidian/community-plugins.json` (vault-relative; the file is an array of enabled plugin ids per Obsidian's runtime contract).
2. For each declared external plugin where `required: true`:
   - If the plugin id is **NOT** present in the array → Notice (`"<mechanism> requires plugin <id>: <reason>. Install + enable in Settings → Community plugins."`) + history entry `{ event: "warning", step: "external_plugins", name: <mech>, plugin_id: <id>, reason: <reason> }`.
   - If present → silent.
3. Read-only check. NEVER edit `.obsidian/community-plugins.json` (CLAUDE.md ask-before gate).
4. Failure-loud per the existing C4 hardening: malformed/unreadable `community-plugins.json` → Notice + history error + skip the check (do NOT abort install).

### Manifests that gain `external_plugins[]` this cycle

| Manifest | Add | Reason copy |
|---|---|---|
| `platform/mechanisms/validator/manifest.json` | `obsidian-slash-commander` (required:true) | `"Surface /validate slash command for ad-hoc validation."` |
| `platform/mechanisms/audit/manifest.json` | `obsidian-slash-commander` (required:true) | `"Surface /audit slash command for the audit walker."` |
| `platform/blueprints/project/manifest.json` | `obsidian-slash-commander` (required:true) | `"Surface /new-project slash command."` |

> [!warning] Open trail #1 — Slash Commander plugin id verification
> The `id` MUST match what Obsidian's `.obsidian/community-plugins.json` writes for Slash Commander. Likely candidates: `obsidian-slash-commander`, `slash-commander`. Lock from disk in S2.1 by installing Slash Commander in barebones first + reading the resulting `community-plugins.json`. Do NOT commit the manifest declarations until the id is locked from disk.

## §3 — `validate.js` code fixes (Findings #3 + #4)

### Finding #3 — defensive input resolution

Replace the existing `tpFileOrObj?.file ?? tpFileOrObj` line at the top of the exported function with a `resolveTFile` helper:

```js
function resolveTFile(input, app) {
  if (!input) return app.workspace.getActiveFile() || null;
  // shape: { file: TFile }
  if (input.file && input.file.path && typeof input.file.path === "string") return input.file;
  // shape: TFile directly (has TFile-shape: string path + basename)
  if (input.path && typeof input.path === "string" && input.basename) return input;
  // shape: tp.file wrapper (current Templater) — fall through to activeFile
  return app.workspace.getActiveFile() || null;
}
```

Used at function entry:

```js
const file = resolveTFile(tpFileOrObj, app);
if (!file) {
  result.violations.push({ rule: "internal", severity: "error", message: "validate: cannot resolve file" });
  return result;
}
```

Drop the stale `tp.user.validate(tp.file, "project")` USAGE comment. Replace with the working pattern (`tp.user.validate({ file: app.workspace.getActiveFile() })`).

### Finding #4 — semantic `checkRequiredBlocks` rewrite

```js
async function checkRequiredBlocks(ctx, gr, mr) {
  const required = [...(gr.required_blocks || []), ...((mr || {}).required_blocks || [])];
  const dvjsBlocks = [...ctx.body.matchAll(/```dataviewjs\n([\s\S]*?)\n```/g)].map(m => m[1]);
  for (const spec of required) {
    if (spec.when && !whenMatches(spec.when, ctx.fm)) continue;
    if (spec.kind === "dataviewjs" && spec.must_call && spec.via === "customjs-guard") {
      const className = spec.must_call.replace(/^customJS\./, "");
      const wrapperRe = new RegExp(
        `dv\\.view\\(\\s*["']Docs/Meta/Views/customjs-guard["']\\s*,\\s*\\{[^}]*class\\s*:\\s*["']${className}["']`
      );
      const matched = dvjsBlocks.some(b => wrapperRe.test(b));
      if (!matched) {
        ctx.result.violations.push({
          rule: "required_blocks.missing",
          severity: "error",
          message: `Missing required ${spec.kind} block calling ${spec.must_call} via ${spec.via}`,
        });
      }
    } else {
      ctx.result.violations.push({
        rule: "required_blocks.schema",
        severity: "warn",
        message: `Unknown required_blocks schema: ${JSON.stringify(spec).slice(0, 100)}`,
      });
    }
  }
}
```

Drops `spec.content` path entirely. `whenMatches` is the existing in-file helper (current line 132's regex eval, refactored into a function).

> [!info] Future-proofing note
> The `via` mechanism's view path is hard-coded as `Docs/Meta/Views/customjs-guard` for v0.1.x. When a second `via` mechanism ships, refactor to resolve the path from the via mechanism's manifest. Defer until that need exists.

## §11 — Cross-cutting risks

> [!warning] New risks surfaced by this design
> 1. **Slash Commander plugin id ambiguity.** Locking from disk in S2.1 mitigates; do NOT guess.
> 2. **`community-plugins.json` shape divergence.** Today it's a flat array of enabled plugin ids. Obsidian core could refactor (unlikely but possible). The C4-style malformed-JSON guard limits blast radius.
> 3. **Bootstrap-copy resync × 3 still in force this cycle.** The discipline is annoying and error-prone (landmine retired by v0.1.2). v0.1.x is the LAST cycle that pays this tax.
> 4. **`checkRequiredBlocks` regex assumes the canonical customjs-guard wrapper format.** A blueprint that uses `await dv.view('Docs/Meta/Views/customjs-guard', {class:'X'})` (single quotes vs double) or different whitespace would fail. The regex tolerates `["']` and `\s*` but the path string is hard-coded; document for future via mechanisms.
> 5. **Slash Commander is third-party.** If the user uninstalls it after install completes, slash commands stop working but no install warnings re-fire. Documented in `use.md` as user responsibility.
