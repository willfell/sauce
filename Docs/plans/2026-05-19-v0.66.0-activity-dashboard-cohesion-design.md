# v0.66.0 — Activity Dashboard Cohesion — Design

> Brainstorm origin: 2026-05-19 conversation, triggered by user smoke on the v0.64.x Activity panel. Mockups co-developed at `<headspace-vault>/activity-dashboard-mockups.md`.
>
> **Status:** design, awaiting user spec-review. No code touched.

## 1. Context

The v0.64.x cycle (4 PATCHes in one afternoon) shipped the cross-blueprint Activity panel as the third section of `SpaceDailyDashboard`. v0.64.3 closed with the panel functional but visually rough. User raised four concrete cohesion gaps during a smoke pass on the headspace + accuris vaults:

1. **Collapse indicator is the default browser triangle.** Foreign-looking next to the Lucide SVG icons used everywhere else in the dashboard. Worst at narrow widths.
2. **Cards are bare titles.** "Sauce", "Journal-2026-05-19", "Scratch-2026-05-19-10-24", "Template, Trip Atlas". No timestamp, no source context, no way to tell at a glance what changed.
3. **Project hub doesn't roll up.** This is FLN-v64-8 from the v0.64.x result doc. When the user edits any task inside `spice/projects/<slug>/tasks/...`, nothing surfaces in the Activity panel — only direct edits to the project hub `<slug>/<Name>.md` count.
4. **Two-level collapse is one tap too many.** Outer Activity `<details>` wraps inner per-blueprint `<details>`. User has to expand twice to see anything; on mobile it's worse.

The first three are visual-or-feature gaps. The fourth is interaction friction. Together they constitute a single "cohesion pass" on the panel — small enough for one MINOR cycle, large enough to be its own arc.

## 2. Decisions captured from brainstorm

Captured via 1:1 dialogue, mockups iterated in `<headspace>/activity-dashboard-mockups.md`:

| Axis | Decision | Reason |
|---|---|---|
| Scope | All four (visual cohesion, project roll-up, flatten collapse, mobile pass) | User selected all four in scoping question; they're tightly coupled in the render pass. |
| Per-card metadata | Timestamp (HH:MM) + type pill + source breadcrumb | First-line body preview rejected (cost: `dv.io.load` per card; not worth it). |
| Default state | All three sections (Tasks/Meetings/Activity) default OPEN | Matches current behavior; user prefers glance-and-scroll. |
| Kanban-board edits | Count as project activity | Moving cards on the kanban IS work on the project. Simpler rule. |
| Breadcrumb format | Count-only — "3 children touched" | Terse, fits cleanly, doesn't wrap awkwardly. |
| Approach | A — extend `activity-feed` + `cards` mechanisms additively | Smallest API surface; backwards-compat; reusable by future consumers (cowork hubs). |

## 3. Approach (A) recap

**Baselines on `main` HEAD as of 2026-05-19 (post-v0.65.0 cowork-scheduling close):** workshop 0.65.0, daily 0.6.0, activity-feed 0.2.0, cards 0.2.5.

- `activity-feed@0.2.0 → 0.3.0` (MINOR) — three additive opts: `rollUpRoots`, `flatGrouped`, `metaBuilder`. Default behavior unchanged for callers that don't pass them. v0.65.0's `_DEFAULT_BLUEPRINTS` widening (cowork run-note types) is preserved.
- `cards@0.2.5 → 0.2.6` (PATCH) — `meta` opt accepts function-form `(page, parentEl) => void` in addition to existing string form. Caller-driven meta rendering enables the time + pill + breadcrumb layout without HTML-string-concat fragility. Mirrors v0.2.4's subtitle-callback pattern.

- `workshop 0.65.0 → 0.66.0` (MINOR).

No mechanism is renamed. No file is deleted. No existing API contract changes.

## 4. API additions

### 4.1 `activity-feed@0.3.0`

Three new opts on `render(dv, opts)`:

```js
{
  // Existing opts unchanged: scope, blueprints, groupBy, limit, useStatusChangedAt,
  //                          asOf, includeMtime, getTitle, collapsible, colorByType, title.

  /**
   * Roll-up rules. When set, descendant pages of a matched root are coalesced
   * into a single virtual entry for the root file.
   *
   * Each rule:
   *   type:         string  — page.type that owns the root (e.g., "project", "trip")
   *   rootPath:     (page) => string|null  — derive the root file path from a CHILD page
   *                                          (e.g., for type:project-task at
   *                                          spice/projects/sauce/tasks/X/X.md,
   *                                          return "spice/projects/sauce/Sauce.md")
   *   childMatch:   (page) => boolean      — does THIS page count as a child of some root?
   *   exclude:      (page) => boolean      — opt-out filter (templates, etc.); default ()=>false
   *
   * Roll-up emits one synthetic page per root with:
   *   _isRollUp:        true
   *   _rollUpChildren:  number   — count of distinct child pages that matched the window
   *   _rollUpLatest:    page     — the most-recently-edited descendant (for tie-break sort)
   *   file:             page.file of the root file (so target / getTitle work normally)
   *   type:             root.type (so colorByType resolves)
   *   created_at:       max(root.created_at, latest child mtime) — used for sort
   *
   * Roll-up is applied AFTER the time-window filter and BEFORE the type-allowlist
   * filter, so the root's own `type:` is what determines whether it surfaces.
   */
  rollUpRoots: [
    { type, rootPath, childMatch, exclude? }, ...
  ],

  /**
   * When true (and groupBy === "blueprint"), render per-blueprint sections as
   * muted uppercase headers (text-only, no <details>) instead of nested
   * <details> elements. Cards render directly below each header.
   *
   * Mutually exclusive with collapsible:true at the group level. The OUTER
   * Activity <details> in SpaceDailyDashboard is unaffected.
   *
   * Default false (preserves v0.1.x behavior).
   */
  flatGrouped: boolean,

  /**
   * Per-card meta callback. Receives (page, parentEl) and renders into parentEl
   * directly. Called for every card; supersedes the v0.1.x `meta` string-builder.
   *
   * When provided, ActivityFeed delegates to BeaconCards with meta passed in
   * function form (requires cards@0.2.5+).
   *
   * Falls back to BeaconCards' default `(p) => p.type` string when omitted.
   */
  metaBuilder: (page, parentEl) => void,
}
```

### 4.2 `cards@0.2.6`

`BeaconCards.render` already accepts `meta` as `(page) => string`. v0.2.6 widens this so `meta` can also be `(page, parentEl) => void`. Detection: `meta.length >= 2` → function-form; render into supplied parent. Else string-form (current behavior, unchanged).

This mirrors the v0.2.4 subtitle-callback pattern (`subtitle` already supports `(parent: HTMLElement) => void` per the inline comment at `beacon-cards.js:19`). Same plumbing, different field.

```js
// Inside BeaconCards._renderRowCard / _renderStackedCard meta handling
const metaResult = ctx.metaFn(page);
if (typeof ctx.metaFn === "function" && ctx.metaFn.length >= 2) {
  // function form: caller renders into the meta container
  const metaEl = row.createEl("div");
  metaEl.style.cssText = /* same as string-form container */;
  ctx.metaFn(page, metaEl);
} else if (metaResult) {
  // string form: existing behavior
  const meta = row.createEl("div");
  meta.style.cssText = /* ... */;
  meta.innerHTML = metaResult;
}
```

## 5. Project roll-up query strategy

The rule for each rolled-up entity type:

```js
{
  type: "project",
  childMatch: (p) => {
    const path = p.file && p.file.path;
    return typeof path === "string" && /^spice\/projects\/[^/]+\//.test(path);
  },
  rootPath: (p) => {
    // p.file.path = "spice/projects/<slug>/<...descendant>"
    // Project hub canonical path = "spice/projects/<slug>/<HubName>.md"
    // We can't derive HubName from the slug alone (display name differs from
    // slug in some projects). Instead: find the page in the same <slug>/
    // directory with type === "project" and use its file.path.
    const m = String(p.file.path).match(/^spice\/projects\/([^/]+)\//);
    if (!m) return null;
    const slug = m[1];
    const hubs = dv.pages('"spice/projects/' + slug + '"')
      .where(pg => pg.type === "project")
      .array();
    return hubs.length > 0 ? hubs[0].file.path : null;
  },
  exclude: (p) => {
    // Template files at vault root or anywhere named "Template, *"
    const name = p.file && p.file.name;
    return typeof name === "string" && /^Template,/i.test(name);
  },
},
{
  type: "trip",
  childMatch: (p) => {
    const path = p.file && p.file.path;
    return typeof path === "string" && /^spice\/trips\/[^/]+\//.test(path);
  },
  rootPath: (p) => {
    const m = String(p.file.path).match(/^spice\/trips\/([^/]+)\//);
    if (!m) return null;
    const slug = m[1];
    const hubs = dv.pages('"spice/trips/' + slug + '"')
      .where(pg => pg.type === "trip")
      .array();
    return hubs.length > 0 ? hubs[0].file.path : null;
  },
  exclude: (p) => {
    const name = p.file && p.file.name;
    return typeof name === "string" && /^Template,/i.test(name);
  },
}
```

**Roll-up algorithm in `ActivityFeed._query`:**

After the existing time-window + type-allowlist filters, but before sort + slice:

```js
if (rollUpRoots && rollUpRoots.length > 0) {
  // Bucket descendants by computed rootPath
  const buckets = new Map();  // rootPath → { root, children: [] }
  const survivors = [];
  for (const p of filtered) {
    let consumed = false;
    for (const rule of rollUpRoots) {
      if (rule.exclude && rule.exclude(p)) { consumed = true; break; }
      if (!rule.childMatch(p)) continue;
      const rootPath = rule.rootPath(p);
      if (!rootPath) continue;
      // Skip if this page IS the root (no self-rollup)
      if (rootPath === p.file.path) continue;
      if (!buckets.has(rootPath)) {
        // Resolve the root page object once
        const rootPage = dv.page(rootPath);
        if (!rootPage) { consumed = true; break; }
        buckets.set(rootPath, { root: rootPage, children: [] });
      }
      buckets.get(rootPath).children.push(p);
      consumed = true;
      break;
    }
    if (!consumed) survivors.push(p);
  }

  // For each bucket, emit a synthetic page (or merge with root if root already
  // surfaces in survivors).
  for (const [rootPath, { root, children }] of buckets) {
    // De-dup: if root is already in survivors, decorate it in place.
    // Dataview pages are Proxy objects; in-place property assignment works
    // because the Proxy traps `set` and the new keys land on the underlying
    // page. Read access in the renderer goes through the same Proxy.
    const existing = survivors.find(s => s.file && s.file.path === rootPath);
    if (existing) {
      existing._isRollUp = true;
      existing._rollUpChildren = children.length;
      existing._rollUpLatest = pickLatest(children);
    } else {
      // Hand-rolled synthetic page (codified BeaconCards synthetic-page pattern,
      // see cards@0.2.0 docs in beacon-cards.js:34-40). Explicit object — do NOT
      // Object.assign on the Proxy, which copies the proxy's enumerable own
      // properties (often empty) rather than the resolved frontmatter.
      const latest = pickLatest(children);
      const synthetic = {
        file: root.file,
        type: root.type,
        title: root.title,
        aliases: root.file && root.file.aliases,
        created_at: latest && latest._tsValue ? latest._tsValue : root.created_at,
        status_changed_at: root.status_changed_at,
        _isRollUp: true,
        _rollUpChildren: children.length,
        _rollUpLatest: latest,
      };
      survivors.push(synthetic);
    }
  }
  filtered = survivors;
}
```

`pickLatest(children)` uses the same tsKey selector as the main query — `status_changed_at` if `useStatusChangedAt`, else `created_at`, falling back to `file.mtime.toISO()` when neither is set.

**Edge case — root not allowed in `blueprints[]`:** the root's `type:` (e.g., `project`) must be in the allowlist OR no card surfaces. SpaceDailyDashboard's `_DEFAULT_DASHBOARD_BLUEPRINTS` already includes `project` + `trip`. If a future caller drops these from the allowlist but keeps the rule, the rollup silently produces no output — acceptable; documented.

**Edge case — root has no `type: project`:** the project hub's frontmatter must declare `type: project` for the rollup target to land in the right group. FLN-v64-8 hypothesis was that this is the actual root cause for some projects. The design doc for v0.66.0 includes a one-time audit (§ 9) of the four consumer vaults to confirm all project hubs are correctly typed.

### 6.2 Per-card meta line — what `_renderActivityMeta` builds

```js
_renderActivityMeta(p, parentEl) {
  parentEl.className = "sauce-meta";

  // Timestamp — prefer created_at (HH:MM), fall back to file.mtime
  const ts = p.created_at || (p.file && p.file.mtime);
  if (ts) {
    const m = window.moment(ts);
    if (m && m.isValid()) {
      const time = parentEl.createEl("time");
      time.textContent = m.format("h:mm A");
    }
  }

  // Type pill
  const t = p && p.type ? String(p.type) : null;
  if (t) {
    const pill = parentEl.createEl("span");
    pill.className = "sauce-pill";
    const dot = pill.createEl("span");
    dot.className = "sauce-pill-dot";
    dot.style.background = (this._BLUEPRINT_COLORS && this._BLUEPRINT_COLORS[t]) || "var(--color-base-50)";
    const label = pill.createEl("span");
    label.textContent = t;
  }

  // Roll-up breadcrumb
  if (p && p._isRollUp && typeof p._rollUpChildren === "number" && p._rollUpChildren > 0) {
    const bread = parentEl.createEl("span");
    bread.className = "sauce-bread";
    bread.textContent = "· " + p._rollUpChildren + " " + (p._rollUpChildren === 1 ? "child" : "children") + " touched";
  }
}
```

## 8. Edge cases + landmines

| Case | Behavior | Risk |
|---|---|---|
| Asset with `type: project` directly in `spice/projects/` (no subdir) | Surfaces as own card; no roll-up triggered (no descendants). | None — same as today. |
| Project hub lacks `type: project` frontmatter | Roll-up's `rootPath` returns null → children are NOT consumed → they fall through to survivors with their own (non-allowlisted) `type:` → silently dropped by allowlist. User sees nothing for that project. | Mitigated by audit step § 9 confirming all hubs are typed correctly. |
| Two project hubs in same `<slug>/` dir | `dv.pages(...).where(p => p.type === "project").array()[0]` picks one nondeterministically. Shouldn't happen in practice. | Logs a console.warn in the rule's `rootPath` if `hubs.length > 1`. |
| File deleted between query and roll-up `dv.page()` call | `dv.page(rootPath)` returns null; child is consumed (no survivor) but no synthetic added. Net: child silently lost. | Acceptable race-condition window (sub-second). User just won't see that activity until next render. |
| `metaBuilder` throws | BeaconCards' meta render path needs try/catch wrap, else single bad card aborts whole panel render. | Add try/catch in v0.2.5 BeaconCards meta callsite. |
| Time format on non-English locale | `moment().format("h:mm A")` honors locale settings. Fine. | None. |
| User has hundreds of project-task edits today | Roll-up coalesces ALL into one card; count grows to "47 children touched". | Acceptable — that's what the user asked for. |

**Landmine implications:**

- **Landmine #11** (`spice/<module>/` invariant): unchanged. Cohesion lives in the daily blueprint's already-owned directory.
- **Landmine #12** (allowlist): the new CSS snippet `.obsidian/snippets/sauce-daily-dashboard.css` is sauce-namespaced; covered by existing snippets allowlist entry. No allowlist expansion.
- **Landmine #1 / #2** (customjs cold-load TDZ): the new rendering paths use existing customjs-guard wiring; no new TDZ exposure.
