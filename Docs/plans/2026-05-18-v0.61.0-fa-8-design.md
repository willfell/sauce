## Why now

FA-2..FA-7 (closed v0.53–v0.59.1) shipped canonical cross-ref keys on every entity-shaped blueprint: `people:`, `projects:`, `teams:`, `products:`, `trips:`, `meetings:`. Every blueprint that mentions one of those entity types now emits a plural-typed wikilink list using the canonical key. The data is uniform. The reverse query is mechanical.

This is the payoff. A single Dataview query, parameterized by the canonical key, produces "every note that mentions this entity" for any of the six entity types. The mechanism that wraps it becomes universal — `BacklinkPanel.render(dv, {entityType: "team"})` works the same way on a Team page as `{entityType: "trip"}` works on a Trip atlas.

Without FA-2..FA-7, this cycle would have needed per-blueprint shims to bridge legacy keys. With them, the mechanism is a thin (~200 LOC) wrapper over a uniform reverse query.

4. **Replacing PeopleRendering.renderMentionList on Person.md** — that helper provides scope-specific (meetings / daily) curated rendering. BacklinkPanel is ADDITIVE on Person.md — a 3rd panel below the existing two, giving universal cross-blueprint coverage. Future cycle may consolidate.
5. **Replacing ProjectReferencedByCards on Project.md** — that helper uses Dataview `outlinks` reverse-lookup (any `[[link]]` anywhere in the body). BacklinkPanel is ADDITIVE on Project.md — uses canonical `projects:` frontmatter reverse-lookup (the FA-spec canonical pattern). Both stay; future cycle may consolidate.
6. **Group-by-month rendering** — `groupBy: "month"` opt declared but minimally implemented (lists in month buckets); rich month rollup deferred to FA-9.
7. **Existing notes don't auto-get the new embed.** Entity-create's `body_template`/`inline_body` controls the embed on NEW notes only; existing entity notes need manual addition. The frontmatter-alignment migration verb is for keys, not body. A future "template-resync" verb is its own brainstorm.
8. **Universal CSS-file ship.** BacklinkPanel renders via existing `customJS.BeaconCards.render(dv, opts)` — no separate CSS asset. Cards layout reuse keeps the visual language consistent with existing `Mentions`/`Referenced By` panels.

### Class: `BacklinkPanel`

Single public method `render(dv, opts)`. Opts validated literally — unknown values fall back to defaults with a `new Notice(...)`.

```js
class BacklinkPanel {
  /**
   * Render a backlinks panel for the current entity page.
   * @param {object} dv  — Dataview API in dataviewjs scope
   * @param {object} opts
   * @param {string} opts.entityType  — "person" | "project" | "team" | "product" | "trip" | "meeting"
   * @param {string|null} [opts.groupBy="none"]  — "type" | "month" | "none"
   * @param {number} [opts.limit=25]
   * @param {string} [opts.sortBy="created_at"]  — "created_at" | "title"
   */
  render(dv, opts) { ... }
}
```

Internals:
- `_ENTITY_TYPE_TO_KEY = { person: "people", project: "projects", team: "teams", product: "products", trip: "trips", meeting: "meetings" }`
- `_reverseQuery(dv, entityType)` returns the pages array via `dv.pages().where(p => p[key]?.some(l => l.path === current.path))`
- `_renderEmpty(dv)` emits "No mentions yet" placeholder (italic, dimmed)
- `_renderGroupedByType(dv, pages)` partitions by each page's `type:` frontmatter, emits H4 section per group + BeaconCards row layout per group
- `_renderGroupedByMonth(dv, pages)` partitions by `created_at` YYYY-MM, emits H4 month sections
- `_renderFlat(dv, pages)` emits a single BeaconCards row-layout panel

## Risk + mitigations

| Risk | Mitigation |
|---|---|
| BacklinkPanel.render throws in dataviewjs scope on a fresh consumer with no migrated notes | Empty-state ("No mentions yet") + try/catch around reverse query — degrade gracefully to Notice |
| BeaconCards depends_on `>=0.2.5` not satisfied at consumer | Workshop manifest declares 0.2.5; pre-existing consumers already on 0.2.4 will get the 0.2.5 bump on next install; for headspace verified at 0.2.5 below |
| Existing Person/Project pages stay un-embedded (the body comes from template at note creation time; existing pages are immune) | Documented as out-of-scope #7. Headspace smoke uses NEW notes to verify behavior; existing pages get the panel on user re-template or v0.62.0 catch-up cycle |
| Dataview reverse-query is O(N) per entity page | Existing PeopleRendering / ProjectReferencedByCards already do this — pattern is established + performance is acceptable for typical vault sizes |
| Cards-version drift across blueprints (some at 0.2.4, some at 0.2.5) | Verified — all 5 entity blueprints already depends_on cards >=0.2.4 (compatible with 0.2.5). No bump needed on cards itself |
