## Surprises during execution

> [!warning] Surprise 1 (resolved): Audit-walker exposure was wrong in design + plan
> The design (§2) + plan (T2.2.2) both assumed the audit walker was exposed as `tp.user.auditWalker()` (camelCase, no argument). T2.1 discovery surfaced the actual Templater contract: filename `audit-walker.js` → `tp.user["audit-walker"]` (hyphen preserved; bracket notation required) AND `module.exports = async function (tp)` requires the `tp` argument. Both wrong on two axes. Fixed in T2.1 before T2.2 wrote the `Audit.md` body. Locked in `T2.1-discovery.md:§2`.

> [!warning] Surprise 2 (the v0.1.3 trigger): Templater Template Hotkeys registration is required
> The design + plan + initial `Docs/use.md` Slash Commander section all assumed Templater auto-generates per-template `Templater: Insert <name>` commands for every `.md` file in the templates folder. **False.** Templater's command palette only exposes 4 fixed base commands; per-template commands ONLY appear after explicit registration in Templater's "Template Hotkeys" section (`.obsidian/plugins/templater-obsidian/data.json:enabled_templates_hotkeys[]`). Without registration, Slash Commander's command picker shows none of the three "Insert" commands; mapping fails.
>
> Verified empirically by reading the barebones Templater config + the user's screenshot of Slash Commander's command picker showing only 4 base Templater commands.
>
> Captured in `T2.1-discovery.md:§8` + the use.md section now has a "Why step 2 is required" warning callout.
>
> **This was the trigger for the user's "no manual plugin configurations" posture decision** — having documented the manual setup ritual, the conclusion was: this is a platform gap, the installer should automate it, not document around it. Spawns v0.1.3.

> [!warning] Surprise 3 (locked plugin id): `slash-commander`, NOT `obsidian-slash-commander`
> Design open-trail #1 left this open with two candidates. T2.1 disk-read locked it: the community-plugin slug is the un-prefixed form. All three manifest declarations + use.md + this writeup cite the locked id. Future cycles importing additional community plugins should follow the same disk-read discipline.

## Follow-up cycle — v0.1.3 plugin-data automation

**Cycle nickname (working):** v0.1.3 plugin-data automation
**Trigger:** Surprise 2 above; user decision 2026-05-04 to eliminate manual plugin-config from the consumer onboarding flow.
**Design doc target:** `Docs/plans/2026-05-04-v0.1.3-plugin-data-automation-design.md` (to be brainstormed + filed in a separate session via `de:brainstorming`).

> [!example]- Scope sketch (committed at this writing)
> **New manifest schema fields** (additive, optional, applied to mechanism + blueprint manifests):
> - `templater_hotkeys[]` — list of templates this manifest needs registered in Templater's Template Hotkeys (each entry: `{ template: "<filename.md>" }`).
> - `slash_commander_bindings[]` — list of `{ keyword, command_id }` pairs to surface as slash commands in Slash Commander.
>
> **Two new installer helpers** (mirror `applyExternalPlugins` posture — read-merge-write idempotent, C4 malformed-JSON guard, failure-loud Notice + history, additive merge only — never strip user entries):
> - `applyTemplaterHotkeys(tp, manifest, history)` — read `.obsidian/plugins/templater-obsidian/data.json`, merge entries into `enabled_templates_hotkeys[]`, write back. Idempotent; backup-on-edit (`<file>.beacon-backup`).
> - `applySlashCommanderBindings(tp, manifest, history)` — same pattern against Slash Commander's data.json; schema TBD via T1.x discovery (read the data.json after the user has made one binding manually so we have a reference shape).
>
> **`.obsidian/` ban lift, scoped:** new CLAUDE.md non-negotiable allowing edits ONLY to `.obsidian/plugins/templater-obsidian/data.json` + `.obsidian/plugins/slash-commander/data.json`. Everything else in `.obsidian/` stays ask-before-acting. New landmine entry codifying the allowlist + the safety mechanics (additive merge, backup-on-edit, malformed-JSON guard).
>
> **Harness extension:** `run-install.js` mock already has fs-backed adapter; need to verify writes to `.obsidian/plugins/<id>/data.json` are captured + asserted in tests. New test cases for idempotent merges + malformed-data fallback + concurrent-Obsidian-runtime considerations (Obsidian may write data.json back; document that re-runs are safe).
>
> **Manifest declarations:** validator + audit + project gain `templater_hotkeys[]` (each declares the corresponding runner template) + `slash_commander_bindings[]` (each declares the corresponding `/<keyword>` → `obsidian-templater:insert-<Name>` mapping). Schema: TBD pending Slash Commander data.json shape.
>
> **Doc collapse:** `Docs/use.md` Slash Commander setup goes from the current 4-step manual flow to:
> 1. Install Slash Commander once (Settings → Community plugins → Browse → Install + Enable). Irreducible per Obsidian API limitation.
> 2. Run the platform installer. Templater Template Hotkeys + Slash Commander bindings auto-applied.
> 3. Reload Obsidian (Cmd+R / Ctrl+R or restart). Plugin caches re-read; slash commands live.
>
> **T2.6 deferred smokes executed** as v0.1.3's S2 acceptance gate (against the automated flow).
>
> **Non-goals (carry into v0.1.3 design):**
> - Companion Obsidian plugin (would absorb both responsibilities via official APIs; large scope expansion; future cycle if ever).
> - Generic `plugin_config[]` schema for arbitrary plugin data merges (YAGNI; only Templater + Slash Commander today; v0.2.0's Kanban plugin may extend).
> - Auto-installing Slash Commander (Obsidian's plugin install API is not exposed to scripts — physics, not platform debt).
> - Auto-reloading Obsidian (no API path; would require a notice telling the user to reload).

> [!info] Sequencing
> Recommended cycle order revision: v0.1.0 ✅ → v0.1.1 ✅ → v0.1.x ✅ → **v0.1.3 (plugin-data automation, NEW)** → v0.1.2 (git-pull + thin stub bootstrap) → v0.2.0 (boards blueprint).

## Pacing rhythm — what worked / what to improve

### What worked

- **Stage-level checkpoints + bundled implementer dispatches.** S1 used `de:subagent-driven-development`; S2 used `de:executing-plans`. Both kept the master agent in flow, surfaced to the user only at planned boundaries (T2.1 plugin id, T2.6 smokes — which we then chose to defer, T2.8 close).
- **Disk-read discipline for external facts.** T2.1 read `community-plugins.json` from disk + `audit-walker.js:1` directly — both surfaced design-draft errors early (`obsidian-slash-commander` → `slash-commander`; `auditWalker()` → `["audit-walker"](tp)`). Fixed before any commit.
- **Triple-bump protocol (landmine #10) for harness re-runs.** Bumping validator + audit + project to `-test` versions across 9 sites forced re-processing without bumping canonical mechanism versions; restored cleanly in T2.4. Test mechanic stayed surgical.
- **Failure-loud + C4 malformed-JSON guard reused as the template** for `applyExternalPlugins`. New helper inherited the posture without bespoke design.
- **Brainstorming the v0.1.3 cycle the moment Surprise 2 surfaced** rather than papering over the gap with thicker docs. Honoring "doc the gap" vs "fix the gap" as a real choice.

### What to improve

- **Design open trails should call out third-party tool runtime contracts more explicitly.** The design assumed Templater auto-generates per-template commands; should have been an explicit open-trail item to verify before locking the use.md flow. v0.1.3 design will inventory every Obsidian plugin contract assumption upfront.
- **Plan T2.2.2 hardcoded the wrong audit-walker function name and missed the `tp` argument.** T2.1 caught it but only because we re-read `audit-walker.js`. Plans that prescribe wrapper-template syntax should require the writer to verify the wrapped function's actual exposure before committing the plan.
- **Cycle scope creep proximity.** Surprise 2 + the user's posture decision genuinely could have justified pulling automation into v0.1.x as Stage 3. Choosing Option 1 (new follow-up cycle) was right but borderline; for next cycle, codify a "scope-creep checkpoint" where mid-cycle posture changes get a structured decision rather than an ad-hoc one. Option for v0.1.3 design: include a "scope-creep posture" section noting what's in/out + what would justify a new cycle vs an in-cycle stage.
