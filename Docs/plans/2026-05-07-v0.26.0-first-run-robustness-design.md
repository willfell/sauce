## P0-1 — wizard "Edit subscription" double-wrap fix

> [!warning] Symptom
> On a configured vault, the Edit-subscription branch leaves the written `platform-subscription.json` with nested entries shaped `{name: {name: "X", version: "Y"}, version: "0.0.0"}` instead of flat `{name: "X", version: "Y"}`. Subsequent installs emit 16 `[object Object]` skip notices and exit 1. Bricks `sauce wizard` post-install today.

**Site:** `platform/bootstrap-lib/wizard.js:330-373` (Edit-subscription branch). Existing extraction at lines 334-339 + `_buildSubscriptionEntries` at line 370-371 LOOKS correct in isolation; the wrap must be happening either in inquirer's checkbox return path (objects leaking through), in a downstream merge step in `bootstrap.js` / `cmd-bootstrap.js` / `installItem`, or via legacy stale subscription state. Implementation plan will reproduce the bug, quote the exact failing call shape per v0.6.0 lesson, then apply the minimum coercion to keep selections as bare-string `value`s and writeback as flat `{name, version}` objects exactly once.

**Fix shape:** coerce existing-config entries to bare strings before passing to inquirer's `checkbox()`; re-wrap once on submit via `_buildSubscriptionEntries(stringArray, manifestEntries)`.

**Test:** new harness case `caseC15WizardEditSubscriptionFlatWrite` in `run-cli.js`. Sets up a temp vault with pre-existing object-shaped subscription entries (or pre-existing nested-double-wrapped entries reproducing the symptom); runs Edit-subscription with `selectedMechs = ["customjs-guard", "nav-buttons"]`; asserts every entry in the written file is shape `{name: <string>, version: <string>}` with no nested objects.

## P0-2 — `scaffoldFoundationalPluginData` helper

> [!info] Architecture
> NEW helper at `platform/install.js`. Mirrors v0.19.0 styling-mechanism + v0.21.1 convenience-mechanism posture: failure-loud + idempotent skip-if-present + atomic write (tmp+rename) + path-traversal validator on `id` + history entries on every action.

**Hardcoded plugin-defaults registry (Templater-only at v0.26.0):**

```js
const FOUNDATIONAL_PLUGIN_DEFAULTS = {
  "templater-obsidian": (variables) => ({
    templates_folder: variables.templates_path || "ranch/templates",
    trigger_on_file_creation: true,
    enable_folder_templates: true,
    folder_templates: []
  })
};
```

Other foundational plugins (`customjs`, `dataview`) intentionally absent — they don't need data.json defaults at install time. Future additions go in this registry.

**Behavior:**

1. Read foundational plugin ids from `manifest.foundational_plugins[]` AND `manifest.external_plugins[]` (union; either schema field can drive scaffolding — exact resolution decided in plan).
2. For each id appearing in `FOUNDATIONAL_PLUGIN_DEFAULTS`:
3. Verify `.obsidian/plugins/<id>/` directory exists; if absent, skip with Notice + history `action: "skipped_missing_plugin_dir"`. Plugin code must be installed first via Obsidian's community-plugins flow (mirror posture from `applyExternalPlugins`).
4. If `.obsidian/plugins/<id>/data.json` ALREADY exists: skip with history `action: "skipped_already_present"` (downstream additive helpers like `applyTemplaterHotkeys` / `applyTemplaterFolderTemplates` handle merge cases).
5. If absent: write the registry-resolved default object via atomic tmp+rename (no `.sauce-backup` since there's no prior file to back up). History `action: "scaffolded"`.
6. On unknown plugin id (no registry hit): silent no-op (no history entry — registry is opt-in).

**Insertion in `installItem`:** between `applyExternalPlugins` (`platform/install.js:619`) and `applyTemplaterHotkeys` (`platform/install.js:620`). Order: validate prereq → scaffold defaults → apply additive helpers.

```text
  await applyExternalPlugins(tp, mech, history, git);
  await scaffoldFoundationalPluginData(tp, mech, variables, history, git);  // NEW v0.26.0
  await applyTemplaterHotkeys(tp, mech, variables, history, git);
  ...
```

**Test:** NEW harness selector `scaffold-foundational` with 5 cases TDD-first per BB1-BB6 / FF1-FF8 / VT1-VT5 / HK1-HK5 / CP1-CP5 precedent.

> [!example]- Harness cases SF1–SF5
> - **SF1** absent-data.json triggers scaffold-write (Templater registry resolved with `templates_path` → "ranch/templates"; written file matches expected shape).
> - **SF2** present-data.json skips scaffold (no overwrite; existing arbitrary user content preserved verbatim).
> - **SF3** plugin-dir absent skips with `skipped_missing_plugin_dir` history entry.
> - **SF4** unknown plugin id (no registry hit) skips silently (zero history entries).
> - **SF5** `templates_folder` substitution from `variables.templates_path` (exercises the variables-aware default function).

Helper-cases harness 387 → ~392 sub-asserts (+5 net minimum).

### NEW landmine #19

```text
#19 — Platform-managed directory names are lowercase

All directories materialized by sauce installer logic under pantry/, ranch/,
spice/ MUST be lowercase. Mixed-case or TitleCase directory names cause
macOS APFS case-collision risk + path-canonicalization drift across
case-sensitive/case-insensitive filesystems.

EXCEPTIONS (do not "fix" these):
- MM-MMMM/ date-routed folders (e.g., 05-May/) — moment.format default;
  user-facing date display, not a directory-naming choice.
- assets/themes/<ThemeName>/ — vendored theme directory; preserve the
  vendor's chosen case verbatim.
- User-facing NOTE FILENAMES (Projects.md, Trips.md, Meetings-<date>.md,
  Thursday-<date>.md, Journal-<date>.md) — file naming, not directory naming.
- Historical doc paths in Docs/plans/ and Docs/prompts/ — preserve
  cycle-time accuracy (path-migration-2026-05-05 precedent).

Recovery from violation: rename via git mv + sed sweep + harness baseline
update. Single atomic commit per the v0.24.0/v0.25.0 mass-rename pattern.
```
