# v0.63.0 design — to-do blueprint: All-To-Dos + Migrate dialog

## Background

### What's there today

`to-do@0.2.0` is the simplest blueprint in the platform: one per-day note `spice/to-do/YYYY/MM-MMMM/ToDo-YYYY-MM-DD.md` with a nav-button row at the top, a `## Tasks` section with a single `- [ ]` placeholder, and a `## Notes` section. One nav-button (`todo-today`, label "To Do") creates/opens today's note via Templater. No hub note, no helpers, no entity-create wiring. Headspace consumer has real to-do data going back to April 2026 — tasks are vanilla Obsidian markdown checkboxes (`- [ ]` unchecked, `- [x]` checked).

### What this cycle adds

Two operator-visible features:

1. **All-To-Dos aggregator** — a single note `spice/to-do/All-ToDos.md` that lives outside the date-routed folders and shows every uncompleted task from prior days (excluding today), grouped by date with source-note links. Live Dataview view — always-current. Reachable via new nav-button `All To-Dos` on every to-do note, and via direct link.
2. **Migrate-to-tomorrow dialog** — a new nav-button `Migrate` on the daily to-do note opens a custom modal listing today's unchecked tasks. User checks items + clicks Migrate; selected tasks (plus any indented child lines: URLs, sub-bullets, sub-checkboxes) are removed from today and appended to tomorrow's note, creating tomorrow's note from template if it doesn't yet exist.

A small third piece for navigation symmetry: an inline `← Back to Today's To-Do` AccentButton on the All-To-Dos note, rendered below the nav-button row by a new `ToDoHubActions` CustomJS helper.

## Decisions (Q1-Q5 from brainstorm)

| # | Decision | Choice |
|---|---|---|
| Q1 | What does the All-To-Dos note contain? | All uncompleted tasks, grouped by date — live Dataview view |
| Q2 | Include today's uncompleted tasks in the All-To-Dos view? | Exclude today (backlog view of past days) |
| Q3 | Where do the two new buttons live on the daily to-do? | Platform nav-buttons in the existing nav row — both registry-driven |
| Q4 | Migration target day? | Always tomorrow; create tomorrow's note if missing |
| Q5 | What happens to a migrated task on today's note? | Removed from today (true move) |
| (sub) | Back-button target on All-To-Dos? | Today's to-do; create if missing |
| (sub) | Indented child lines under a migrated task? | Travel with the parent |

## Architecture

Three independent surfaces designed as one cohesive cycle — all three sit on platform infra that already exists (nav-buttons `invoke_command` action, `customjs_startup_scripts[]`, AccentButton mechanism). No platform mechanism changes; this is a pure blueprint cycle.

### Architecture overview

1. **All-To-Dos aggregator** — install-time materialized note at `{{module_directory}}/All-ToDos.md` (mirrors scratch's `Scratch Hub.md → spice/scratch/Scratch.md` pattern). Body: standard nav-button row + inline `ToDoHubActions` AccentButton row + a single dataviewjs block running `ToDoAllList`. `ToDoAllList` is a pure-render CustomJS class that uses Dataview's `pages('"spice/to-do"').file.tasks.where(t => !t.completed)`, groups results by source-file basename (`ToDo-YYYY-MM-DD`), excludes today, renders headings desc with source-note wikilinks.

2. **Migration dialog** — new nav-button `todo-migrate` uses action type `invoke_command` (existing infra v0.4.2) with `command_id: "sauce:to-do-migrate"`. The command is registered by a customJS startup script `ToDoMigrateInit` (same registration pattern as v0.49.0's `ProjectTaskCreateListenerInit`). The command instantiates `ToDoMigrateModal` (a plain-DOM overlay, not an Obsidian `Modal` subclass — avoids constructor-fetch gymnastics inside the customJS sandbox; matches the overlay pattern other helpers in the codebase already use). The modal parses today's `## Tasks` section into top-level-task blocks (each = unchecked `- [ ]` line + immediately following indented children until the next top-level line), renders one checkbox per top-level block, and on submit applies a pure-function migration to today + tomorrow's content strings before writing both via `vault.modify`.

3. **Back-to-Today inline button** — `ToDoHubActions.render(dv)` clears `dv.container`, then renders one `customJS.AccentButton` row with label "← Back to Today's To-Do". onClick invokes the same Templater command the `todo-today` nav-button uses, so the "create today's if missing" semantics are identical.

### Why nav-buttons (not AccentButtons) for All-To-Dos + Migrate

Per Q3: registry-driven nav-buttons are the lower-churn path. They surface on EVERY note in the module-directory automatically (no template-body edits), they live in `manifest.json` declaratively, and the renderer already handles ordering + icon resolution. The only reason to use AccentButtons would be visual distinction; the user explicitly chose nav-row consistency.

### Why the back-button IS an AccentButton (not a nav-button)

The "Back to Today" button is a per-note local action that only makes sense on the All-To-Dos hub note. Making it a global nav-button would surface it on every to-do note, where it's redundant (the existing "To Do" nav-button already creates/opens today's). Inline AccentButton scoped to the hub note is the right surface — mirrors `ScratchLeafActions`.

### New file 2 — `platform/blueprints/to-do/helpers/todo-hub-actions.js`

CustomJS class `ToDoHubActions`. Materialized to `{{scripts_path}}/to-do/todo-hub-actions.js`. ~30 LOC. Contract:

- `render(dv)` — clears `dv.container`, renders one centered AccentButton row with label `← Back to Today's To-Do`, icon a back-arrow SVG (vendored inline, mirrors `ScratchLeafActions`). onClick: calls `app.commands.executeCommandById("templater-obsidian:Today To-Do")` to invoke the same Templater template the `todo-today` nav-button uses. If that command id resolution fails (template not registered), falls back to opening `spice/to-do/<today YYYY>/<today MM-MMMM>/ToDo-<today YYYY-MM-DD>.md` directly via `openLinkText` (the file will be created with a placeholder if Templater is offline).
- Dual-fire guard — same `__todoHubRenderGen` counter pattern Scratch uses.
- Embed guard — `if (dv.container.closest(".markdown-embed")) return;` so embedded views don't double-render.

### New file 3 — `platform/blueprints/to-do/helpers/todo-all-list.js`

CustomJS class `ToDoAllList`. Materialized to `{{scripts_path}}/to-do/todo-all-list.js`. ~50 LOC. Contract:

- `render(dv)` — clears container, then:
  1. Compute today via `window.moment().format("YYYY-MM-DD")`.
  2. Query: `dv.pages('"spice/to-do"').where(p => /^ToDo-\d{4}-\d{2}-\d{2}$/.test(p.file.name) && p.file.name !== ` + "`ToDo-${today}`" + `)`.
  3. For each page, extract the date suffix from filename + collect uncompleted tasks via `p.file.tasks.where(t => !t.completed)`.
  4. Group by date desc; render `## YYYY-MM-DD` heading per group + `- [ ]` bullets per task with trailing `   (from [[ToDo-YYYY-MM-DD]])` source link.
  5. If no uncompleted tasks across all dates: `dv.paragraph("No backlog — all prior to-dos completed.")`.
- Dual-fire guard + embed guard same as ToDoHubActions.

### New file 4 — `platform/blueprints/to-do/helpers/todo-migrate-modal.js`

CustomJS class `ToDoMigrateModal`. Materialized to `{{scripts_path}}/to-do/todo-migrate-modal.js`. ~180 LOC including overlay styling. Contract:

- `open()` (instance method) — gates on `app.workspace.getActiveFile()?.basename.match(/^ToDo-\d{4}-\d{2}-\d{2}$/)`. If gate fails, `new Notice("Open today's to-do note first.")` + return. Otherwise reads active file content, calls `ToDoMigrateModal.parseTasks(content)`, builds the overlay DOM:
  - Backdrop `<div>` covering viewport (rgba dim).
  - Modal `<div>` centered, scrollable, with header "Migrate to-dos to <tomorrow YYYY-MM-DD>", body listing one row per top-level unchecked task (checkbox + task text + greyed preview of children indented below), footer with `[Cancel]` `[Migrate N tasks]` buttons.
  - Keyboard handlers: Esc → close; Enter on Migrate button → submit.
  - First checkbox auto-focused on open.
- `submit()` — re-reads active file (handles concurrent edits per Section 4), matches selected blocks by top-line exact-string + parser-line-index. For any failed matches, accumulates a "could not be matched" count to include in the result Notice. Computes tomorrow's path. If tomorrow's file is missing: writes a freshly-rendered template body via `vault.create` (template body duplicated inline in this helper — see FLN-todo-1). Reads tomorrow's content, calls `ToDoMigrateModal.applyMigration(today, tomorrow, [indices])`, writes both files via `vault.modify`. Notice on success.
- **Static helpers** (pure functions; testable in Node without Obsidian):
  - `ToDoMigrateModal.parseTasks(content)` — returns `Array<{ topLine: string, childLines: string[], startIdx: number, endIdx: number }>` for all unchecked top-level tasks under the `## Tasks` heading. Returns `[]` if no `## Tasks` heading.
  - `ToDoMigrateModal.applyMigration(todayContent, tomorrowContent, selectedIndices)` — returns `{ today: string, tomorrow: string }`. Removes selected blocks (top-line + children) from today's `## Tasks` section; appends them verbatim to tomorrow's `## Tasks` section (after any existing tasks, including the template's empty `- [ ]` placeholder).

### New file 5 — `platform/blueprints/to-do/helpers/todo-migrate-init.js`

CustomJS class `ToDoMigrateInit`. Materialized to `{{scripts_path}}/to-do/todo-migrate-init.js`. ~15 LOC. Contract:

- `invoke()` — called by customjs plugin on startup (per `customjs_startup_scripts[]` wiring v0.49.0 already shipped). Calls `app.commands.addCommand({ id: "sauce:to-do-migrate", name: "Sauce: Migrate to-dos to tomorrow", callback: () => new customJS.ToDoMigrateModal().open() })`. Idempotent — guarded by `this._registered` flag.

## UX & error-handling

### Notices

| Trigger | Message |
|---|---|
| Migrate command invoked outside a `ToDo-*` file | "Open today's to-do note first." |
| Active file is a `ToDo-*` but parser finds no `## Tasks` heading | "No `## Tasks` section found in this file." |
| Active file has no unchecked tasks | Modal opens with "Nothing to migrate." body; Migrate button disabled. |
| Migration succeeds | "Migrated N task(s) to ToDo-<YYYY-MM-DD>." (If some tasks couldn't be matched due to mid-flight edits: "Migrated N of M; K could not be matched (note may have been edited).") |
| Vault write fails | "Migration failed: <error.message>." Modal stays open for retry. |
| Dataview unavailable on All-To-Dos | `dv.paragraph("All-To-Dos requires the Dataview plugin.")` |

### Modal mechanics

- Plain-DOM overlay (backdrop + modal `<div>` appended to `document.body`).
- First checkbox auto-focused on open.
- Esc closes; outside-click on backdrop closes (Cancel-equivalent).
- Tab cycles through checkboxes + Cancel + Migrate.
- Space toggles current checkbox (native).
- Migrate-button label updates live with selected count; disabled when count = 0.
- Migrate-button disabled during async vault writes; re-enabled if write fails.

### Parser invariants

- Top-level task = line matching `^- \[ \] ` (unchecked only). `- [x]` is skipped including any children.
- Child = any line starting with ≥1 space or tab, following a top-level task line, terminated by next top-level line or by a blank line followed by content at column 0.
- A task with zero children is valid (most common case).
- Multiple consecutive children (URLs, sub-bullets, nested sub-checkboxes) all travel together.

### Idempotency & user customization

- `All-ToDos.md` install: skipped if file already exists. User can customize freely; subsequent installs preserve their edits.
- Tomorrow's file creation: only if missing. If present, append-only.
- Tomorrow's `## Tasks` section: migrated blocks appended at end of section, before next heading or EOF. Existing tasks (including the template's `- [ ]` placeholder) are preserved.
- Re-clicking Migrate on already-migrated tasks → no idempotency guard; duplicates are accepted as user intent.

`run-todo-modal.js` sub-asserts:

- **TM-1** — `parseTasks` extracts only unchecked top-level tasks from a fixture (3 lines: unchecked, checked, unchecked → returns 2 blocks).
- **TM-2** — `parseTasks` correctly groups indented children with their parent (unchecked task + 2 indented URL lines → 1 block with `childLines.length === 2`).
- **TM-3** — `applyMigration(today, tomorrow, [idx])` returns `{ today, tomorrow }` with tomorrow's `## Tasks` section containing the migrated lines verbatim (parent + children, in order).
- **TM-4** — `applyMigration` removes exactly the selected blocks from today (other blocks untouched, line order preserved for survivors).
- **TM-5** — `applyMigration` against an empty-tomorrow fixture (just `## Tasks\n- [ ]\n`) inserts after the empty placeholder, not before it.

The parser + applier live as static methods on `ToDoMigrateModal`. The harness `require()`s the helper file in Node — both methods are written to be pure (no `app`, no `customJS`, no `window` references in those two methods) so Node loads them cleanly.

## FIX-LATER-NOTEs (anticipated; captured at design time)

- **FLN-todo-1** — `todo-migrate-modal.js` duplicates the `Today To-Do.md` body inline when creating tomorrow's file (so it doesn't depend on Templater being awake at migrate-submit time). Future refactor: share a single body template via a new platform primitive (e.g., a `_body_template.md` partial referenced by both `Today To-Do.md` and the migrate helper) so they can't drift. Low priority — duplication is ~6 lines.
- **FLN-todo-2** — Anticipated: the parser is line-index-based; if the user inserts a brand-new task between modal-open and Migrate-submit, the indices may shift. Submit-time re-parse handles this, but we accept that a manually-edited intervening task line could cause a "could not be matched" warning. Document; don't guard.
