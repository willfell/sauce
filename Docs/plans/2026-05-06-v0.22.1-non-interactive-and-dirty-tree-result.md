## Post-smoke critical finding (NEW v0.22.1 lesson before all others)

**The macOS APFS case-collision is NOT just a git-status cosmetic issue — it actively breaks Obsidian runtime.** Surfaced when the user tried to open the daily note in scratch/4 and got *"Failed to create daily note. Folder 'beacon/daily' not found."*

Root cause: APFS resolves `beacon` ≡ `Beacon` to the same physical inode (so the FS-level mkdirs succeed inside `Beacon/`). But Obsidian's vault index is case-sensitive at the app layer — it indexes whatever casing it walks (`Beacon/daily`), and lookups for the configured `folder: "beacon/daily"` fail.

Affected blueprints (all that materialize content under their `beacon/<module>/` namespace): Daily Notes, Boards (Kanban), Meetings (hubs/notes), Finance, Journal, To-Do, Trips, Projects. Effectively **all 8 blueprints are functionally broken on macOS APFS** until the rename ships.

The v0.22.1 `.gitignore` fix masks the symptom for `git status` but does NOT fix the runtime breakage.

**Decision (user-locked 2026-05-06):** ship v0.22.1 as designed (PATCH; flag-propagation + dirty-tree mask + harness fixes); document the runtime breakage prominently in `Docs/install.md`; queue rename of workshop clone dir to non-colliding name as PRIMARY recommendation for v0.23.0. User is exploring creative beacon-themed names (`beak`, etc.); options brainstorm goes into the post-v0.22.1 handoff doc.

## NEW lessons (v0.22.1)

(a) **`.gitignore` patterns must be root-anchored when masking install-time-generated artifacts that share names with content namespaces elsewhere in the tree.** Bare-name patterns (`boards/`) match at any depth. The next mechanism / blueprint cycle landing nested files under `Docs/Meta/Scripts/<mech>/...` or `platform/blueprints/<bp>/<bare-name>/...` would silently lose tracking. Root-anchoring (`/boards/`) preserves the dirty-tree masking at workshop root (where the case-collision materializes) without collateral. Surfaced by S2 code-quality reviewer (CF-1). Pattern reusable for any future workshop-side `.gitignore` extension that targets install-time artifacts.

(b) **`phaseFirstRunWizard` short-circuits on existing config** — `--overwrite` to install.sh wipes `Beacon/` but preserves `Docs/Meta/platform-{config,subscription}.json`. The wizard only runs when both files are absent; on re-bootstrap, the existing config is read verbatim and any new flags are ignored. Implication for v0.22.1+ smokes: testing the new `--mechanisms=all` / `--blueprints=all` overrides requires a fresh-vault smoke target, not a `--overwrite` re-bootstrap. v0.22.x candidate: `install.sh --reset-config` flag that ALSO removes consumer-side platform-config.json (out of v0.22.1 scope; not user-requested).

(c) **`BEACON_REPO_URL=file:///<path>` is a serviceable smoke transport** when the workshop dev environment doesn't have anonymous-read access to the workshop's GitHub remote. install.sh line 7 already supports the env var; no install.sh change needed. Future v0.22.x quality work: document `file://` URL pattern explicitly in install.md "smoke testing" section (currently only `BEACON_REPO_URL` is mentioned implicitly).

(d) **Reserved-headroom precedent now 9 data points; one MIXED case introduced (S2 CF-1).** Prior: 7 UNUSED + v0.21.0 5 USED + v0.22.0 1 USED. v0.22.1 expected UNUSED per design but landed 1 CF from code-quality review — so the bucket gains a "review-surfaced CF on otherwise-clean cycle" subcategory. Decision rule refinement: expected-clean cycles can still pull 1 review-surfaced CF; the two-stage spec-then-quality reviewer dispatch (v0.20.0 lesson d) is the load-bearing surface. Cycles that skip the quality reviewer would have shipped this latent bug into v0.22.1 production.

(e) **Two-stage subagent review (v0.20.0 lesson d) reaffirmed at scale.** S2 spec reviewer found ZERO issues; S2 quality reviewer found one IMP (root-anchoring) + one NIT (CSV `all,foo` ambiguity). Pattern: spec compliance gates "did you do what the plan said," quality gates "is what you did sound." Spec-only review would have shipped the latent silent-ignore bug. Cheap insurance.

(f) **Plan baseline-count drift is harmless when commit-message reports actual reality.** Plan T1.9 predicted "install-sh 13/2" but actual was 11/3 (the I7 marker-exists assertion correctly PASSES while only the flag-includes assert FAILS). Commit message used the actual count. Spec reviewer noted the drift as a defensible reality-over-plan adjustment. Pattern: when TDD harness predicted counts diverge from observed, prefer reality in the commit message; the plan stays as-of-design-time.
