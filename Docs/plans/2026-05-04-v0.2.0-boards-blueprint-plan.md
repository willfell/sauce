### Task 1.3 — Implement Option B content overwrite in `files[]` loop (subagent)

**Files:**
- Modify: `platform/install.js:265-296` (the `for (const f of mech.files || [])` loop in `installItem`)

**Behavior:**
- Before writing `dest`, read its current content (if exists).
- If current content === substituted source → skip the write (idempotent; no history event needed beyond the existing flow).
- If current content ≠ substituted source AND current content is non-empty:
  - Compute SHA-256 of current content (`prior_sha`) and substituted source (`new_sha`).
  - Write current content to `<dest>.bak` (overwrite-on-edit; one-deep; no rotation).
  - Write substituted source to `dest`.
  - Push history entry: `{ event: "replace", step: "file_overwrite", name: mech.name, dest, prior_sha, new_sha, bak_path: "<dest>.bak", git_commit, git_tag, git_dirty, attempted_at }`.
- If `dest` doesn't exist OR is empty (0-byte) → write substituted source as fresh; existing flow.
- The mechanic applies ONLY to `files[]`-declared content (platform contract). Runtime-generated files (card notes, user-created notes) are NOT touched by this mechanic; they're created by Templater / nav-button actions, not by `installItem`.

**Step 1 — Write the failing test (TDD red):**

Add cases to `run-helper-cases.js`:

```javascript
// Case A: dest content matches source → no replace event (idempotent)
// Case B: dest content differs from source, prior is non-empty → .bak written + replace event
// Case C: dest content differs, prior is 0-byte → no .bak; just write; no replace event (treated as fresh)
// Case D: dest doesn't exist → write fresh; no replace event
```

Each case scratch-vaults a fake blueprint with one content file, primes the dest to the case's prior-state, runs the installer, then asserts:
- Final dest content matches expected.
- `<dest>.bak` exists or absent per case.
- History contains `replace` event with correct sha fields per case OR doesn't.

**Step 2 — Run + fail (4 new cases all red).**

**Step 3 — Implement the overwrite mechanic in the files loop:**

Around `install.js:295` (`await adapter.write(destPath, substituted);`), wrap with the read-compare-backup-write logic. Use `crypto.createHash("sha256")` from Node's stdlib for sha computation. Use `app.vault.adapter.read(destPath)` for the current content (consistent with installer's other adapter reads); use `adapter.exists(destPath)` for the existence check.

Pseudocode:
```javascript
const exists = await adapter.exists(destPath);
let priorContent = null;
if (exists) {
  try { priorContent = await adapter.read(destPath); } catch { /* leave null */ }
}
if (priorContent !== null && priorContent === substituted) {
  // identical; skip the write entirely
} else {
  if (priorContent !== null && priorContent.length > 0) {
    const priorSha = sha256(priorContent);
    const newSha = sha256(substituted);
    await adapter.write(`${destPath}.bak`, priorContent);
    await adapter.write(destPath, substituted);
    history.push({
      event: "replace",
      step: "file_overwrite",
      name: mech.name,
      dest: destPath,
      prior_sha: priorSha,
      new_sha: newSha,
      bak_path: `${destPath}.bak`,
      git_commit: git.commit, git_tag: git.tag, git_dirty: git.dirty,
      attempted_at: new Date().toISOString(),
    });
  } else {
    await adapter.write(destPath, substituted);
    // existing fresh-write flow; no history event
  }
}
```

**Step 4 — Run + pass (4 cases all green).**

**Implementer notes:**
- The `.bak` is for recovery, not for diff narrative. One-deep, overwrite-on-edit, no rotation. Same posture as the v0.1.3 `applyTemplaterHotkeys` `.beacon-backup` mechanic.
- Use `app.vault.adapter` reads/writes (not `fs.promises`) for consistency with existing installer flow.
- `prior_sha` + `new_sha` are full sha256 hex strings. Length will balloon in `installed.json`; accept it (audit-grade information).
- The `replace` event count appears in `run-install.js`'s "New history entries" block — verify after each barebones run.
- Failure of any step (e.g., `.bak` write throws) records `event: "error", step: "file_overwrite"` and skips the dest write. Don't half-update.

### Task 1.4 — Implement `pre_install[]` schema with `type: "delete"` action (subagent)

**Files:**
- Modify: `platform/install.js` (add new helper `applyPreInstall(tp, mech, variables, history, git)`; wire-up in `installItem` BEFORE the `files[]` loop)

**Behavior:**
- Read `mech.pre_install || []` after T1.3's content-overwrite mechanic but BEFORE T2.x's content materialization. Actually correction: pre_install runs FIRST, before any other helper. New invocation order in `installItem`: `applyPreInstall` → `files[]` loop (with Option B) → `post_install` → `applyRuleFragment` → `applyNavButtons` → `applyExternalPlugins` → `applyTemplaterHotkeys` → `applySlashCommanderBindings`.
- Each `pre_install[]` step has shape `{ type: "delete", path: "<dest-relative-path>", reason: "<why>" }`.
  - `path` is substituted via `substituteStrict` (path semantics — fail loud on missing variables).
  - If file exists: read it for sha (audit), write to `<path>.pre_install_bak` (one-deep, overwrite-on-edit), delete original via `adapter.remove(path)` or equivalent. Record `event: "delete", step: "pre_install_delete"` history entry with `name: mech.name, path, prior_sha, bak_path, reason, git fields, attempted_at`.
  - If file doesn't exist: record `event: "info", step: "pre_install_delete_skip"` history entry with reason "file already absent". Idempotent.
  - If file is a directory: record `event: "warning", step: "pre_install_delete_skip"` with reason "target is a directory; pre_install delete is single-file only"; continue.
- Currently `type: "delete"` is the only supported action. Plan does NOT add `type: "rename"` / `type: "move"` (YAGNI).
- Unknown `type` value: record `event: "warning", step: "pre_install_unknown_type"`; skip.
