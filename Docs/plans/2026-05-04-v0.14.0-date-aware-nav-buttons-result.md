## Outcome summary

> [!abstract] What landed
> When the user clicks an RTT nav-button on a date-routed file (e.g. a daily note `Monday-2026-05-09.md` or a journal entry `2026-05-03-Journal.md`), the dispatched template now writes its target file under THAT file's date — not today's. From a daily for `2026-05-09`, "ToDo" creates `ToDo-2026-05-09` under `beacon/to-do/2026/05-May/`. From a journal for `2026-05-03`, "Meetings" creates the hub for `2026-05-03`. Files without a date in the filename fall back to today (the v0.4.2 behavior). Three blueprints (to-do, meetings, journal) inherit this transparently — zero manifest edits, zero template edits.

> [!abstract] Why this matters
> The platform's RTT (`runTemplaterTemplate`) primitive shipped in v0.4.0 and was hardened in v0.4.2 with the split-field schema; until v0.14.0 it always resolved `targetDate` against `window.moment()` (now). For a journaling/retrospective workflow this was wrong: clicking "ToDo" from yesterday's daily was supposed to surface yesterday's ToDo, not create today's. v0.14.0 makes the dispatcher **date-aware** at the call site without touching any consumer blueprint. The fix is purely mechanism-side, additive, and free for every current and future RTT consumer.

## Lessons (NEW)

> [!info]- 1. Helper-extraction returning a string is cleaner than passing a moment instance
> The helper `_resolveActionDate(dv)` returns a `'YYYY-MM-DD'` **string**, not a moment instance. The caller (the RTT branch) reconstructs the moment via `window.moment(targetDateStr, 'YYYY-MM-DD')` and continues with the existing `.format(...)` chain. This is cleaner than returning a moment for two reasons:
>
> - **Symmetry with the regex.** The regex extracts a string `'2026-05-09'`; returning that string keeps the helper's data-shape consistent with what it actually parsed. Wrapping in `moment(...)` would force the helper to know about both the present (today fallback path) AND the past-extraction path producing the same shape.
> - **Stub-friendliness for harness.** DA1 and DA2 just assert string equality — no moment-instance comparison gymnastics. Future renderer harness cases that exercise `_resolveActionDate` get the same simple shape.
>
> **Takeaway:** for helpers that bridge a regex-extracted source and a downstream formatter, return the **lowest-common-denominator string**, not the formatted-target type. Caller does the cheap reconstruction; harness gets cheap assertions.

> [!info]- 2. Regex precedent reuse — same pattern, two code paths, zero divergence
> The regex `/(\d{4}-\d{2}-\d{2})/` was first introduced in v0.4.2 at L107-122 of `space-nav-buttons.js` for the daily-nav top-row's "skip to nearest existing" logic. v0.14.0 reuses the IDENTICAL regex for `_resolveActionDate`. No divergence. No "almost the same but slightly different" pattern. The two call sites share a single grammar invariant: **filename pattern `<Name>-YYYY-MM-DD` puts the date as a hyphen-separated suffix; capturing that date is `\d{4}-\d{2}-\d{2}` anywhere in the filename.**
>
> Why this matters: gotcha 8 (filename pattern invariant from v0.4.2) is now load-bearing for TWO renderer code paths. Future renderer features that need to extract a date from a filename should reuse the same regex verbatim — not reinvent. If the invariant ever changes (e.g. a future ISO-with-time format), there's exactly one regex to update + a clear set of call sites to audit.
>
> **Takeaway:** the filename-pattern invariant earns its weight when reused. Two call sites is the bare minimum for "this is a load-bearing invariant rather than an incidental shape."

> [!info]- 3. Zero-config / always-on default reinforced as platform posture
> v0.14.0 makes date-aware dispatch the **only** behavior — there is no opt-in flag, no manifest field, no mode toggle. Every RTT consumer (to-do / meetings / journal today; trips / finance tomorrow) gets it for free, transparently. This mirrors v0.4.2's `invoke_command` decision (no per-consumer knob; a literal `command_id` and the runtime fires it) and v0.5.0's openLink target-substitution decision (no flag; install.js always substitutes `action.target`).
>
> The platform value: **when a behavior is always-correct in every plausible context, ship it as the default with no knob**. Knobs add manifest schema, harness coverage, doc burden, and the risk of consumer-side mis-configuration. The discipline check: if the answer to "is there ANY plausible RTT consumer that would want today-not-active-file's-date?" is no, the knob is wrong.
>
> **Takeaway:** when adding cross-cutting renderer behavior, default to no-knob always-on. Reserve manifest fields for behavior that genuinely diverges across consumers.
