# v0.57.0 — FA-5 Cowork migration (result)

Cowork alone — heaviest timeline migration per umbrella design §C Wave 3. `cowork@0.7.0 → 0.8.0` MINOR + `workshop_version 0.56.2 → 0.57.0` MINOR. Controller-direct cadence (~6 file edits). Whole-suite preflight green at tag.

## What shipped

**Note template canonical-vocab adoption (3 templates):**
- `content/Daily Note.md` — `tags: [cowork-daily, daily]` → `tags: [daily]`; `created:` (YYYY-MM-DDTHH:mm:ss) → `created_at:` (YYYY-MM-DDTHH:mm:ssZ ISO+TZ).
- `content/Weekly Note.md` — `tags: [cowork-weekly, weekly]` → `tags: [weekly]`; `created:` → `created_at:` ISO+TZ.
- `content/Monthly Note.md` — `tags: [cowork-monthly, monthly]` → `tags: [monthly]`; `created:` → `created_at:` ISO+TZ; **`month_iso: "YYYY-MM"` renamed to canonical `month: "YYYY-MM"`** (resolves the cross-blueprint `month:` clash with finance — both now use the same canonical key); `month_label:` retained as friendly free-form display.

**Hub + prompt static `created_at:` (5 hubs + 4 prompts):**
- `content/Cowork.md`, `content/Daily Hub.md`, `content/Weekly Hub.md`, `content/Monthly Hub.md`, `content/About Cowork.md` — gain static `created_at: "2026-05-17T15:03:00-06:00"` so workshop-dogfood install satisfies the new `extends:` contract.
- `content/prompts/{morning-briefing,eod-review,weekly-review,monthly-review}.md` — same static `created_at:` injection. Prompts retain their own `updated:` + `updated_by:` semantics.

**Rule fragments (12 of 13 declare `extends: _canonical-vocab`):**
- All cowork rule_fragments — Cowork.md hub, active-threads, active-projects, vault-config, Daily Hub / Weekly Hub / Monthly Hub / About Cowork, daily/weekly/monthly note glob, prompts — declare `extends: "_canonical-vocab"` (FA-1 substrate; inherits canonical `type:` regex + `created_at:` ISO+TZ requirement).
- **One exception:** the `.claude/skills/cowork/*/SKILL.md` fragment retains its `name:` + `description:` schema and does NOT extend `_canonical-vocab` — different frontmatter shape. Documented in the manifest + asserted in helper-cases (FA5-EXTENDS-3) + cowork-smoke (v0.57.0 shape).
- **Daily/weekly/monthly fragments drop `required_frontmatter.created`** — covered by extends.
- **Daily fragment drops `required_tags: [{ tag: "cowork-daily" }]`** — template no longer emits.
- **Monthly fragment adds canonical `month:` validation:** `{ required: true, type: "string", matches: "^\\d{4}-\\d{2}$" }`. `month_label:` retained as free-form `{ required: true, type: "string" }`.

**Manifest version + description + post_install:**
- `cowork@0.7.0 → 0.8.0` MINOR. Description appended with FA-5 narrative. post_install Notice updated to v0.8.0 + suggests `sauce migrate-frontmatter --vault <v> --blueprint cowork --apply` for existing consumer-vault content.

**Helpers — no functional changes:**
The 4 helpers (`cowork-{daily,weekly,monthly}-hub-cards.js` + `cowork-daily-actions.js`) already read canonical-shaped keys:
- `cowork-monthly-hub-cards.js` reads `p.month_label` — still works (we keep `month_label`).
- `cowork-weekly-hub-cards.js` reads `p.week_label`, `p.week_start`, `p.week_end` — unchanged.
- `cowork-daily-hub-cards.js` reads `p.file.name` + body snippet — no frontmatter touch.
- `cowork-daily-actions.js` delegates frontmatter emission to `Daily Note.md` template — template change covers.

Net: helpers are unchanged this cycle. Plan-author note: design §C Wave 3 mentioned helper edits but inspection showed they're either no-ops or already correct.

## Consumer-vault migration: DEFERRED to next deploy round

Per the round-11 (FA-3 v0.55.0 ship) + round-13 (deploy round v0.52.1→v0.56.0) pattern, this cycle ships the workshop tag without running `sauce migrate-frontmatter --blueprint cowork --apply` across the 4 consumer vaults. Two reasons:

1. **Brew sauce is at v0.56.2.** The newly-tagged v0.57.0 is not yet brew-released; the auto-PR on `willfell/homebrew-sauce` requires merge (round 13 squashed multiple PRs into one direct main bump).
2. **Migration spec v0.57 may need additive `month_iso → month` rule.** Existing cowork-monthly notes in consumer vaults carry `month_iso: "YYYY-MM"` (from v0.7.0 templates). To populate the new canonical `month:` key, the migration rules need a new entry. Today's `platform/migrations/v0.53-frontmatter.json` carries a `{ from: "month", to: "month_label" }` rule that handles pre-v0.7.0 content but NOT the post-v0.7.0 `month_iso:` case. A v0.57 migration-rules append (`from: "month_iso", to: "month"` scoped to `type: cowork-monthly`) is the cleanest fix; can land in a follow-up patch or in the next deploy round.

The next deploy round (whenever the user picks up the cowork-migration apply step) will:
1. Merge the v0.57.0 tap PR (or direct main bump on `willfell/homebrew-sauce` Formula/sauce.rb).
2. `brew upgrade sauce` to v0.57.0 across machines.
3. Bump consumer subscription pins: cowork@0.7.0 → 0.8.0 at all 4 vaults; workshop_version pin 0.56.2 → 0.57.0.
4. (Optional) Add the `month_iso → month` rule to `platform/migrations/v0.53-frontmatter.json` (or create `v0.57-frontmatter.json` if the cmd supports multi-file).
5. `sauce migrate-frontmatter --vault <v> --blueprint cowork --apply` × 4 vaults.
6. `sauce reinstall --vault <v>` × 4 vaults — deploys v0.57.0 cowork templates (the 3 note templates + 5 hubs + 4 prompts).
7. `sauce audit --frontmatter-alignment --blueprint cowork` reports zero `legacy_key_used` post-migration.

## FIX-LATER notes

1. **FLN-FA5-1 — `month_iso → month` migration rule missing.** `platform/migrations/v0.53-frontmatter.json` has `from: "month", to: "month_label"` for very-old cowork-monthly content but not the modern `from: "month_iso", to: "month"`. Consumer-vault existing cowork-monthly notes (post-v0.7.0 shape with `month_iso:` + `month_label:`) won't get `month:` populated by migration. Two options for the deploy round: (a) append the rule to `v0.53-frontmatter.json`, OR (b) add a backfill rule that derives canonical `month:` from filename (YYYY-MM.md → `month: "YYYY-MM"`) for cowork-monthly path-matched files. Option (b) is more robust (handles both pre-v0.7.0 and post-v0.7.0 cases). Deferred.

2. **FLN-FA5-2 — Hub + prompt static `created_at:` is the install-time date, not the historical creation date.** Used `2026-05-17T15:03:00-06:00` (FA-5 ship time) as the static date for the 5 hubs + 4 prompts. Semantically: "this hub's canonical creation timestamp was established when FA-5 codified the canonical contract." This satisfies the rule_fragment `extends:` contract but doesn't preserve a "real" creation date. Consumer-vault content gets migration-backfilled from mtime (more accurate). Acceptable trade-off for static-materialized files.

3. **FLN-FA5-3 — SKILL.md fragment intentionally does NOT extend `_canonical-vocab`.** Different frontmatter schema (name + description vs type + created_at). Documented in helper-cases (FA5-EXTENDS-3) + cowork-smoke (v0.57.0 shape). Future canonical-vocab extension for Claude skills would need its own shared rule template (e.g., `_skill-vocab.json`). Out of scope this cycle.

4. **FLN-FA5-4 — `frontmatter_branch[]` extends-propagation gap (FLN-FA3-1).** Cowork has no `frontmatter_branch[]` rule_fragments so unaffected. The FA-3 known gap (extends doesn't propagate into branches) doesn't impact FA-5.

5. **FLN-FA5-5 — Workshop subscription does NOT include cowork.** Workshop dogfood install never materializes cowork content. Cowork test coverage rides on `run-cowork-smoke.js` + `run-helper-cases.js` static asserts. Pre-existing gap (carried forward from v0.45.0 self-contained design); not regressed.
