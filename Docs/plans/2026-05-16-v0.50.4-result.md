# v0.50.4 — Empty `wiki/Wiki.md` on newly-created projects (result)

**Date:** 2026-05-16
**Cycle:** PATCH v0.50.3 → v0.50.4
**Cadence:** controller-direct (sub-agent-driven would have been overkill)
**Sauce-pipeline round:** 3

## What shipped

1. **`entity-create@0.3.2`** — `_createExtra` now ensures the parent directory of `xPath` whenever `filename_pattern` itself embeds a subfolder (the case `"wiki/Wiki.md"` was hitting). Pure additive forward-defense; existing happy paths (no embedded slash) are unchanged. See [`entity-create.js`](../../platform/mechanisms/entity-create/entity-create.js) `_createExtra` near line 597.

2. **`project@1.10.3`** — the wiki extra_file in [`platform/blueprints/project/manifest.json`](../../platform/blueprints/project/manifest.json) rewritten from `{ "filename_pattern": "wiki/Wiki.md", … }` to `{ "subfolder": "wiki", "filename_pattern": "Wiki.md", … }` — schema-canonical form. The entity-create mechanism's existing `subfolder` field is the intended slot for nested-path destinations.

3. **Test coverage** — `run-entity-create.js` grew from 46 → 48 pass:
   - **EC-39** drives `_createExtra` against the stub vault with `filename_pattern: "wiki/Wiki.md"` and asserts (a) `createFolder` was called for the embedded subfolder and (b) the file landed with the expected body content.
   - **EC-40** asserts the in-tree project manifest's wiki extra_file uses the canonical `subfolder` field.
   - **EC-2** updated for the version bump (`0.3.1` → `0.3.2`).
   - **WIKI-5** rewrote its lookup predicate from `filename_pattern === "wiki/Wiki.md"` to a body-template-based search (stable across the canonicalization).

## Root cause (brief)

`vault.create(path, content)` against a missing parent dir in Obsidian doesn't throw cleanly — it auto-creates the missing dir but the body content is silently dropped, leaving a 0-byte file. The mechanism's `_ensureFolder` was only being called on `xFolder` (= `folder + optional subfolder`), never on the dir component of `filename_pattern`. Two extra_files in the project manifest had `filename_pattern` without a slash (succeeded); the wiki extra_file had `filename_pattern: "wiki/Wiki.md"` (empty file).

Full diagnosis in [`2026-05-16-v0.50.4-empty-wiki-on-new-project-design.md`](./2026-05-16-v0.50.4-empty-wiki-on-new-project-design.md).

## FIX-LATER notes

Same as the v0.50.3 hotfix handoff (carried forward, none new this cycle):
- **FLN-1.** `applyWikiBackfill` should be unconditional (currently gated by version-newer-than-installed in `installItem`; doesn't repair 0-byte files either). Future cycle: move it parallel to `materializeSkills` so it runs every install, AND treat 0-byte Wiki.md as missing.
- **FLN-2.** Two-tier release validation (tap merge + brew upgrade + consumer reinstall + smoke) should be codified in `/sauce-pipeline` as a mandatory Phase C step.
- **FLN-3.** EC-39's vault stub is permissive — its `vault.create` doesn't enforce parent-dir-exists like real Obsidian does. A stricter mock that throws when the parent is missing would make the regression louder. Considered for a future test-harness cycle.

## Risks (none materialized)

- Mechanism change is gated (only fires when `filename_pattern` contains `/`).
- Manifest change is a same-output re-encoding (the resulting `xPath` is identical; only the path through `_createExtra` differs).
