## Why this cycle

The `module_directory: "projects-legacy"` stopgap was acknowledged at v0.2.0 close as layout debt — project blueprint v0.3.0 dropped its Board contribution to make room for the new boards blueprint, but its remaining files were left at their v0.1.x pre-invariant locations under `boards/planning/...` rather than ported to `beacon/projects/...`. v0.5.0 pays down the stopgap: every file the project blueprint materializes now lives under `beacon/projects/<slug>/`, mirroring accuris's planning-board pattern transcribed into Beacon's vocabulary. The cycle also pulls forward the accuris hub pattern (atlas + Structure + Board + Tasks tightly linked via project-context nav-buttons) so the experience matches a daily-use vault rather than a v0.1.x scaffold.

The "iron everything out before moving on" framing extended scope inline: five CF (carry-forward) polish concerns surfaced at S2 close were sequenced as sub-stages CF-1..CF-5 rather than deferring to v0.5.x. The result is one BREAKING bump from 0.3.0 to 1.0.0 with full landing rather than a multi-cycle migration.

## Surprises during execution

> [!warning]- Surprise A (resolved): Kanban silent-falls-back to vault root when target folder absent
> First barebones Smoke B FAILed because `_createProject` only mkdir'd `beacon/projects/<slug>/`, not the `tasks/` subfolder. When Kanban tried to create a new card at `beacon/projects/test-alpha/tasks/Setup.md`, the missing `tasks/` folder caused Kanban to silently fall back to vault root with no error surfaced.
>
> Tactical fix: in-cycle bump project 1.0.0 → 1.0.1; `_createProject` now mkdirs both `<slug>/` and `<slug>/tasks/` upfront. Re-test PASS.
>
> Carry forward as a pattern: any class-driven project layout creation should pre-create the FULL expected directory tree, not just the project root. Kanban's silent-fallback is a class of bug where the failure mode is invisible — preventive mkdir is the only safe posture.

> [!warning]- Surprise B (resolved): `_openAsKanban` race produced blank panes
> The accuris-ported `_openAsKanban` method called `leaf.openFile(file)` then `leaf.setViewState({type: "kanban", ...})`. When invoked from the Open Board button, `setViewState` raced with file-body load and produced a blank Kanban view. Workaround: opening the same file from the file explorer worked fine (Obsidian's view-detection takes a different path).
>
> Tactical fix: in-cycle bump project 1.0.1 → 1.0.2; simplified `_openAsKanban` to just `app.workspace.openLinkText` — Kanban auto-detects `kanban-plugin: board` frontmatter and takes over the leaf without explicit `setViewState` transitions.
>
> Carry forward: prefer letting Obsidian's plugins auto-detect view types from frontmatter rather than forcing `setViewState` transitions. The platform's openLink action_type (CF-1, S2.3) is the same posture — let Obsidian decide.

> [!warning]- Surprise C (resolved): `Docs/Meta/Templater/Templates` is the wrong path; consumers use `Docs/Meta/Templates`
> Initial `_createProject` hardcoded `Docs/Meta/Templater/Templates` for tpl reads, mirroring an accuris convention that doesn't apply to Beacon. Beacon's `platform-config.json` sets `templates_path: Docs/Meta/Templates` (no `Templater/` subdirectory; that's only for user scripts).
>
> Tactical fix: replaced hardcoded path in ProjectNavButtons + Project Board.md + Task Board.md with `{{templates_path}}` token; installer's `substituteLenient` resolves it per-consumer.
>
> Carry forward: never hardcode platform paths in helper / template content; use `{{xxx}}` tokens that the installer substitutes from each consumer's variables map (landmine #9). Pre-flight should always read both consumer's `platform-config.json:variables` to know the actual substitution targets.

> [!warning]- Surprise D (resolved): Dataview `#project` tag matches sub-tags
> Initial Projects.md used `dataview FROM "beacon/projects" AND #project` for the All Projects listing. The atlas was tagged `project` AND `project/<slug>`; the Structure note was tagged ONLY `project/<slug>`. Dataview's `AND #project` matched BOTH (sub-tags inherit from parent tag).
>
> Tactical fix: ported accuris's `PlanningBoardProjects` → `ProjectsHubCards` class which uses `etags.includes("#project")` for EXACT-match filtering. Subsumed CF-3 into CF-2 — one class swap addresses both concerns (card UI + atlas-only filter).
>
> Carry forward: when filtering by tag in Dataview, use `etags.includes("#tag")` for exact-match; the `FROM #tag` form matches sub-tags by inheritance. The `etags` array is the per-page list of tags AS WRITTEN, not the inheritance-expanded set.

> [!warning]- Surprise E (resolved): Renderer `openLink` action_type was dead code in installer; needed validateAndResolve branch
> SpaceNavButtons renderer already had `if (type === "openLink") app.workspace.openLinkText(action.target, ""); return;` since v2.0.0 — but `install.js`'s `validateAndResolve` had no `openLink` branch, so `{{module_directory}}` placeholders in `target` were never substituted. The literal placeholder reached the registry; renderer would dispatch on the literal string `"{{module_directory}}/Projects.md"` and Obsidian would fail to find that note.
>
> Tactical fix: `install.js` gained a 9-line `openLink` branch in `validateAndResolve` (S2.3, `9b555b9`); harness R6 case verifies the round-trip.
>
> Carry forward: any new action_type added to the renderer dispatcher should ALSO have a `validateAndResolve` branch in `install.js` for at-install-time substitution. Document this as a pair-invariant in `Docs/how.md`.

> [!warning]- Surprise F (resolved): icon name `project` (singular) didn't match renderer's registered `projects` (plural)
> v1.2.0 manifest declared `icon: "project"` for the new Projects nav-button entry; the renderer's ICONS map registered the briefcase SVG under key `projects` (plural). Result: button rendered with the fallback "P" letter rather than the icon glyph.
>
> Tactical fix: in-cycle bump project 1.2.0 → 1.2.1; icon string corrected to `"projects"`.
>
> Carry forward: cross-reference manifest `icon` names against `space-nav-buttons.js`'s ICONS map keys at design time. Worth documenting the canonical icon-name list in `CLAUDE.md` or `Docs/how.md` so future blueprint authors don't round-trip on naming.

## Pacing rhythm — what worked / what to improve

### What worked

- **Parallel subagent dispatch on T2.3 + T2.4 + T2.5 + T2.6 (four fully disjoint files).** Zero merge conflicts; all four returned spec-clean on first review. Gave a ~10x throughput vs serial dispatch and isolated the subagent-execution risk to the per-file scope.
- **Stage-close AskUserQuestion gates caught the icon-letter regression (Surprise F) before commit.** S2.3 close gate fired with "screenshot showing the icon" prompt; user confirmed the "P" fallback letter; in-cycle 1.2.0 → 1.2.1 patch landed in the same gate cycle without polluting `9b555b9`'s scope.
- **Per-smoke confirmations isolated each surprise to a single fix-cycle.** Smoke B → 1.0.0→1.0.1; Smoke D → 1.0.1→1.0.2; CF-1 verify → 1.2.0→1.2.1. If smokes had been batched at end-of-S2, the context-switching cost between failed-smoke debug + remaining-smokes execution would have been ~3x.
- **The "iron everything out before moving on" extension model (CF-1..CF-5 inline) preserved cycle continuity while honoring user feedback.** Five polish concerns absorbed without scope creep into v0.5.x; one BREAKING bump landed end-to-end.

### What to improve

- **Pre-flight should have surfaced the `templates_path` mismatch (Surprise C).** Workshop AND barebones BOTH use `Docs/Meta/Templates` (no `Templater/` subdir); accuris uses `Extras/Templates`. The plan-writing phase didn't read the actual `platform-config.json:variables` values to know the correct convention. Bake into future plan-writing: always read both consumer's `platform-config.json:variables` to know the actual substitution targets BEFORE specifying paths in template / helper content.
- **Plan-writing under-counted in-cycle bumps.** Plan expected single project bump 0.3.0 → 1.0.0; actual was six bumps to 1.2.2. Each in-cycle surprise required a bump per gotcha 9 to force re-process. Future plan-writing should reserve "1.0.0 → 1.x.y headroom" for in-cycle iteration rather than spec'ing an exact target version.
- **Renderer ICONS map should be discoverable (Surprise F).** The "P" fallback cost a round-trip; documenting the canonical icon-name list in `CLAUDE.md` or `Docs/how.md` would prevent it. Ideally the renderer logs available ICONS keys when an unknown icon is requested, so the diagnostic is at the failure point rather than requiring source-read.
- **Pair-invariant for action_type schema work needs codification.** Surprise E's "renderer-has-it-but-installer-doesn't" pattern would be caught at design time if `Docs/how.md` carried a "new action_type checklist" requiring both renderer dispatch AND validateAndResolve branch.
