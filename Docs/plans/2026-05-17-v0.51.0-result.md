## What shipped

A clickable status widget on every project hub. Renders the current status as a colored chip + "Updated YYYY-MM-DD" label; click the chip ‚Üí custom overlay with 7 status buttons (matching the entity-create overlay aesthetic); pick any ‚Üí writes `status` + `status_changed_at` via `app.fileManager.processFrontMatter` ‚Üí dataview auto-rerenders the chip.

Closes the missing piece of v0.39.0's `ProjectActionButtons` Tier-1 design that never shipped (v0.39.0 S7 was never executed; v0.46.2 later cleaned up the dangling manifest references). The `status` frontmatter field has existed since v0.39.0 S5; the Projects.md hub has filtered/grouped by status since the same era. v0.51.0 lands the in-vault UI to change it.

**Bundle scope (intentionally narrow):** Just Bump Status. v0.39.0's Tier-1 bundle also included Set Teams + Set Products; both deferred to a follow-up cycle when needed.

## Why a new class, not an extension of `ProjectWorkstreamManager`

Single responsibility per class is the established sauce pattern. `ProjectWorkstreamManager` is workstream-specific (renders workstream cards + parses the kanban board for per-workstream progress bars). Adding status logic into it would make the name lie and tangle two independent concerns. The new widget is a sibling, registered alongside in the manifest, invoked from a separate dataviewjs block in `Template, Project.md`.

The new `## Status` H2 section sits between the nav-buttons row and the existing `## Workstreams` section ‚Äî user picked "inline with workstreams row" during brainstorming; interpreted as visually adjacent / same area.

## User smoke step

After `brew upgrade sauce` + tap PR merge + `sauce reinstall --vault <headspace>` + Obsidian Cmd-R:

1. Open any project hub (e.g. `spice/projects/sauce/Sauce.md`). Expect a new `## Status` section above `## Workstreams` showing a colored chip (e.g. `[in-progress ‚ñº]`) and an "Updated YYYY-MM-DD" label if `status_changed_at` is set.
2. Click the chip. Expect a centered overlay with 7 vertically-stacked status buttons (current status shown with `(current)` and muted styling).
3. Pick a different status. Expect the overlay to close and the chip to re-color within a second or two (Obsidian's metadata cache refresh + dataview re-render).
4. Open the project file's frontmatter (Cmd-J / properties pane) and confirm `status:` and `status_changed_at:` reflect the new values.
5. Test cancel paths: click overlay backdrop (should close); press Escape (should close); click Cancel button (should close).
