## Stage S2 — validator predicate additions + audit fixtures + cowork:check-vault-routing extension

> [!info] Subagent-dispatchable: YES (validator + audit + cowork-skill split cleanly into 3 parallel implementers)

### S2.1 — Validator predicate handlers

`platform/mechanisms/validator/`: add ADDITIVE predicate handlers:
- `min_length` — list-type frontmatter values with at least N entries.
- `items_schema` — discriminated-union per-item validator. Schema:
  ```json5
  {
    "items_schema": {
      "discriminator": "<field_name>",
      "by_type_source": "engagement-types/<type>.json#required_fields",  // path resolver
      "common_required": { /* same as required_frontmatter shape, applied to every item */ }
    }
  }
  ```
- The `by_type_source` resolver reads the referenced manifest at audit time, extracts `required_fields[]` array, treats each entry as a `required_frontmatter`-style assertion.

Mechanism version: `validator@0.1.1 → 0.1.2` (MINOR — additive predicates).

### S2.2 — Audit rule-fragment for cowork

Add new `rule_fragments[]` entry to cowork manifest (already getting touched in S1):

```json5
{
  "target": "cowork",
  "fragment": {
    "scope": { "path_glob": "spice/cowork/context/vault-config.md" },
    "required_frontmatter": {
      "type":         { "required": true, "type": "string", "equals": "cowork-vault-config" },
      "updated":      { "required": true, "type": "string", "matches": "^\\d{4}-\\d{2}-\\d{2}$" },
      "updated_by":   { "required": true, "type": "string" },
      "engagements":  {
        "required": true,
        "type": "list",
        "min_length": 1,
        "items_schema": {
          "discriminator": "type",
          "by_type_source": "engagement-types/<type>.json#required_fields",
          "common_required": {
            "id":   { "type": "string", "matches": "^[a-z][a-z0-9-]+$" },
            "type": { "type": "string" }
          }
        }
      }
    }
  }
}
```

## Subagent dispatch protocol (cross-stage)

Stages with `Subagent-dispatchable: YES` follow this protocol:

1. **Main thread:** preps the stage context (reads relevant design + plan sections, surveys current state, decides files-to-touch list).
2. **Main thread:** dispatches one or more implementer subagents via Agent tool with `subagent_type: "general-purpose"` (or named persona if a fitting one exists). Each implementer gets a self-contained prompt quoting design + plan sections LITERALLY.
3. **Main thread:** waits for implementer return; reads diffs via `git diff`; runs harnesses.
4. **Main thread:** if harnesses green → commit + push. If failures → either fix-loop OR dispatch a second implementer with specific failure context.
5. **Main thread:** updates TaskList, advances to next stage.

**Two-stage review at S2/S5/S6 close** per CLAUDE.md anti-patterns:
- Spec reviewer subagent — does the implementation match the design/plan/spec?
- Quality reviewer subagent (e.g., `de:code-reviewer`) — code-quality, idempotence, error-handling, naming.

Both reviewers must pass before stage commit lands on `origin/main`.
