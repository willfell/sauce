### Root cause analysis — issue A (template collision)

Both blueprints declare `{{templates_path}}/Daily Note.md` as a destination in their `files[]`:

- `platform/blueprints/daily/content/daily-template.md` → ships `SpaceNavButtons` + `SpaceDailyDashboard` + `<!-- COWORK_CALLOUTS -->` marker + `## Notes`.
- `platform/blueprints/cowork/content/Daily Note.md` → ships Templater preamble (day/friendly/created_at) + cowork-flavor frontmatter (`type: cowork-daily`, `day:`, `day_label:`, `created_at:`, `tags: [daily]`) + `SpaceNavButtons` + `CoworkHubNav` + `# <% friendly %>` H1 + `## Notes`.

Install order (cowork loads after daily because of `cowork depends_on daily >=0.2.5`) means cowork's copy wins. The cowork copy lacks the `SpaceDailyDashboard` dataviewjs block AND lacks the `<!-- COWORK_CALLOUTS -->` marker that cowork's own orchestrator skills (morning-briefing, eod-review, midday-tripwire) require. Daily-note creation in both consumer vaults silently lost the dashboard somewhere in the cowork-self-contained cycle (v0.45.0 / v0.7.0).

### Root cause analysis — issue C (frontmatter visibility)

Obsidian's `propertiesInDocument` setting in `.obsidian/app.json` defaults to `"hidden"` (verified against `/Users/willfellhoelter/notes/legacy/{accuris,headspace}/.obsidian/app.json`, both `"hidden"`). The convenience mechanism already manipulates app settings via the workshop-level `app_settings` field + `applyAppSettings` helper (added v0.26.1 P1-2). Currently `app_settings` ships three keys (`alwaysOpenInNewTab`, `defaultViewMode`, `livePreview`); we add a fourth.

#### `activity-feed.js` — TWO NEW OPTIONS

```js
// Public API additions to render(dv, opts):
//
//   asOf:           string|moment (default: window.moment())
//                   Anchors the time window. When set, _resolveTimeWindow
//                   computes startIso/endIso around this anchor instead of now.
//                   Format: anything window.moment() accepts (ISO string,
//                   "YYYY-MM-DD", moment object).
//
//   includeMtime:   boolean (default: false)
//                   When true, _query's predicate is OR'd: a page matches
//                   if EITHER its tsKey (created_at: or status_changed_at:)
//                   falls in [startIso, endIso] OR its file.mtime moment
//                   falls in the same range. Catches "edited today but
//                   created earlier" notes.

render(dv, opts) {
  // ... existing option normalization ...
  const asOf = safeOpts.asOf;          // NEW
  const includeMtime = safeOpts.includeMtime === true;  // NEW
  // ...
  timeWindow = this._resolveTimeWindow(scope, asOf);
  // ...
  pages = this._query(dv, blueprints, timeWindow, useStatusChangedAt, includeMtime, limit);
  // ...
}

_resolveTimeWindow(scope, asOf) {
  const useMoment = typeof window !== "undefined" && window.moment;
  if (useMoment) {
    const anchor = asOf ? window.moment(asOf) : window.moment();
    if (scope === "today") return {
      startIso: anchor.clone().startOf("day").format(),
      endIso:   anchor.clone().endOf("day").format()
    };
    if (scope === "week") return {
      startIso: anchor.clone().startOf("isoWeek").format(),
      endIso:   anchor.clone().endOf("isoWeek").format()
    };
    if (scope === "month") return {
      startIso: anchor.clone().startOf("month").format(),
      endIso:   anchor.clone().endOf("month").format()
    };
    return null;
  }
  // Native fallback path: existing logic, but Date is constructed from asOf
  // when provided. asOf is parsed via new Date(asOf) — accepts ISO + YYYY-MM-DD.
  // ... (mirror existing native branch with anchor instead of now)
}

_query(dv, blueprints, timeWindow, useStatusChangedAt, includeMtime, limit) {
  const start = timeWindow.startIso;
  const end   = timeWindow.endIso;
  const tsKey = useStatusChangedAt ? "status_changed_at" : "created_at";

  const inTsWindow = (p) => {
    const tsRaw = p[tsKey];
    if (!tsRaw) return false;
    const ts = String(tsRaw);
    if (/^\d{4}-\d{2}-\d{2}$/.test(ts)) {
      return ts >= start.slice(0, 10) && ts <= end.slice(0, 10);
    }
    return ts >= start && ts <= end;
  };

  const inMtimeWindow = (p) => {
    if (!includeMtime) return false;
    if (!p.file || !p.file.mtime) return false;
    // file.mtime is a luxon DateTime in Dataview. Compare via .toISO().
    const mIso = (typeof p.file.mtime.toISO === "function")
      ? p.file.mtime.toISO()
      : String(p.file.mtime);
    return mIso >= start && mIso <= end;
  };

  const chain = dv.pages()
    .where((p) => {
      if (!p) return false;
      if (blueprints.indexOf(String(p.type)) < 0) return false;
      return inTsWindow(p) || inMtimeWindow(p);
    })
    .sort((p) => { /* existing sort */ }, "desc")
    .slice(0, limit);

  return Array.isArray(chain) ? chain : (chain && typeof chain.length === "number" ? Array.from(chain) : []);
}
```

## 3. Critical mechanic — ActivityFeed `asOf` + `includeMtime` enhancement

### Why this is the load-bearing change

The dashboard panel must reflect the **daily note's date**, not the wall-clock date. A user opening `Wednesday-2026-05-13.md` (a past daily note) expects to see activity from May 13, not today (May 19). Today's `ActivityFeed._resolveTimeWindow(scope)` hard-anchors to `window.moment()`.

### Design choice — extend ActivityFeed, do NOT fork into the dashboard

Three options were considered:

1. **SpaceDailyDashboard queries directly, ignoring ActivityFeed.** Pro: minimal blast radius on ActivityFeed. Con: duplicates the cross-blueprint scan logic; future widgets (weekly dashboard, monthly dashboard) re-invent it.
2. **SpaceDailyDashboard pre-filters pages, passes them to ActivityFeed as an `extPages` opt.** Pro: dashboard owns the date logic. Con: forks ActivityFeed's contract (it currently owns querying); adds a coupled-but-leaky API.
3. **ActivityFeed grows `asOf` + `includeMtime` knobs (chosen).** Pro: ActivityFeed remains the single cross-blueprint query engine; the dashboard becomes a thin caller. Future widgets reuse the same engine with different anchors. Pro: backwards-compatible — existing callers (cowork hubs + Today.md) keep working with zero changes. Con: two more options on the API surface.

### Mtime semantics

Dataview exposes `page.file.mtime` as a Luxon `DateTime`. The predicate compares `mtime.toISO()` against `[startIso, endIso]`. This is consistent with how `_query`'s existing `tsKey` branch handles ISO-with-TZ strings.

**Edge case — DST + TZ.** `window.moment().startOf("day")` and luxon's ISO emit are both local-tz; both serialize to `±HH:MM` offsets. The comparison is string-lexicographic on the same TZ basis. Cross-TZ vault sync could cause edge-of-day misses but the existing `_query` already has this fragility for `created_at:`; we're not worsening it.

**Edge case — Dataview not yet hydrated.** If `page.file.mtime` is missing (race during vault open), the page is excluded from the mtime predicate. The created_at predicate path is unaffected. Acceptable — the dashboard re-renders on first user interaction.

**Edge case — bulk reformatting bumps mtimes.** Acknowledged. If a user runs a vault-wide reformat, the next day's dashboard will be flooded. Mitigation: `groupBy: "blueprint"` keeps the flood organized, and the `limit` opt (default 50) caps total output. We accept the trade-off because the user's card explicitly asks for "added or updated."
