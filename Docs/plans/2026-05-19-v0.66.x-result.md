# v0.66.x result — Activity Dashboard Cohesion arc

## What shipped (final state at v0.66.2)

**User-visible:**
- `SpaceDailyDashboard` sections (Tasks / Meetings / Activity) now render with a Lucide chevron-right SVG that rotates 90° on `<details[open]>`. The default browser triangle is gone.
- Titles dropped the "Today's" prefix — just **Tasks (N)** · **Meetings (N)** · **Activity (N)**.
- Default state: Tasks **open** · Meetings **open** · Activity **closed**. Activity is review-y; tap to dive in.
- Activity panel groups (project / scratch / journal / trip / etc.) render as **muted uppercase headers with colored dots** instead of nested `<details>` toggles. Single-tap reveal at the outer-section level only.
- Each Activity card has a meta line: `HH:MM time · type pill · optional "N children touched" breadcrumb`. Type pills are outline-style (colored dot + lowercase name in a rounded pill).
- **Project + trip rollup** — any edit anywhere under `spice/projects/<slug>/**` or `spice/trips/<slug>/**` surfaces ONE card for the hub. Breadcrumb counts the descendants touched today. Clicking the card opens the hub, not a child. Kanban-board moves count as activity (per design decision). `Template, *` files are excluded.
- Smart title resolver chain extended: **`name:` → `title:` → `aliases[0]` → first outline heading → filename**. The new `name:` branch handles project hubs whose filenames are literally `Project.md` (the project blueprint stores the real name under `name:`). Synthetic rollup pages now carry both `name` and `title` so the resolver can use either.
- **Scratch title wiring fixed** — the `+ New Scratch` dialog's optional title prompt now actually lands in frontmatter (`title: "{{prompts.title}}"`). Pre-v0.66.2 scratches that have no title still fall through to filename; new scratches created from v0.66.2 onward carry the dialog value.
- Mobile pass — `@media (max-width: 480px)` shrinks paddings + meta-line gap; meta line wraps cleanly under narrow titles.

**Architecture:**
- `cards@0.2.5 → 0.2.6` PATCH — `meta` opt accepts function-form `(page, parentEl) => void` callback (arity-2 detection mirrors the v0.2.4 subtitle-callback pattern).
- `activity-feed@0.2.0 → 0.3.0` MINOR — three additive render-opts: `rollUpRoots[]` (coalesce descendants under a root file), `flatGrouped` (muted-header layout in place of nested `<details>`), `metaBuilder` (caller-driven per-card meta rendering). All additive; v0.1.x/v0.2.x callers (cowork hubs) get prior behavior unchanged.
- `activity-feed@0.3.0 → 0.3.1` PATCH (v0.66.1) — synthetic rollup page now carries `name` alongside `title`, so downstream resolvers can read project-blueprint name.
- `daily@0.6.0 → 0.7.0` MINOR — `SpaceDailyDashboard` adds 7 new helpers (`_renderSection`, `_renderActivityMeta`, `_buildRollupRules(dv)`, `_CHEVRON_SVG`, `_BLUEPRINT_PILL_COLORS`, `_ROLLUP_RULES`, `_escapeHtml`); three section blocks rewritten to use `_renderSection`; ships a new `sauce-daily-dashboard.css` snippet materialized via the v0.41.0 `applySnippets` plumbing.
- `daily@0.7.0 → 0.7.1` PATCH (v0.66.1) — title prefixes dropped; Activity default-closed; `_resolveTitle` gains `p.name` branch before the existing chain.
- `scratch@0.4.0 → 0.4.1` PATCH (v0.66.2) — `new_entity_buttons[].frontmatter_template` adds `"title": "{{prompts.title}}"`. One-line declarative fix.

### v0.66.2 — PATCH (tagged 2026-05-20T02:47Z)
**Title:** Scratch title wiring (entity-create frontmatter_template gap)
**Versions:** workshop `0.66.1 → 0.66.2` · scratch `0.4.0 → 0.4.1`
**Tap PR:** [#72](https://github.com/willfell/homebrew-sauce/pull/72) merged ~02:48Z

User reported: "scratch just shows the titles from the [note name], not the frontmatter for the title that's specified in the dialog" — and predicted (correctly) it was a mechanism bug.

Root cause: scratch's `new_entity_buttons[0].prompts[]` collected a `title` value from the "Scratch title (optional)" dialog, but `frontmatter_template` only emitted `type`, `created_at`, `day` — never referenced `{{prompts.title}}`. The dialog input was captured into `ctx.prompts.title` then discarded at frontmatter-render time. Existed since v0.3.0 of the scratch blueprint (entity-create migration). Manifest description claimed the title "lands in `title:` frontmatter" but the wiring was simply absent.

Fix: one-line declarative — add `"title": "{{prompts.title}}"` to `frontmatter_template`. Empty prompt (title is optional) lands as `title: ""` which `_resolveTitle` correctly falls through. `ScratchDayList` (day-hub cards) + `SpaceDailyDashboard` activity panel (via the v0.66.1 `p.title` branch) both already read `p.title` — no downstream changes needed.

Stage commits: `9678cb9` PATCH source + manifest + SHC-S1 pin · `817e785` dogfood byproducts.

## FIX-LATER-NOTEs captured

- **FLN-v66-1 (Important, deferred):** 7 legacy-shape projects across accuris-sauce + ero-sauce silently don't surface in the rollup. Their hubs use `type: structure` (5 in accuris: Denali Doc/Media Migration, Document DB Migration, NR Drop Filter Exec/Migration) or `type: project-board` (2 in ero: NEPA Rag, Request for Proposal Trigger Flow) — pre-modern conventions. The rollup's `rootPath` callback queries `dv.pages(...).where(p => p.type === "project").array()`; with no `type: project` hub in those slug dirs, it returns null and the children fall through to the allowlist, which drops them silently. **Behavior is benign** (no error, no false card) but the user gets zero signal that those projects had activity. Fix candidates: (a) widen the rollup's hub-type predicate to accept `project | project-board | structure`; (b) ship a migration helper that audits + bumps legacy hubs to `type: project`. Track for a future cycle.
- **FLN-v66-2 (Suggestion):** `_renderFrontmatter` in entity-create emits `title: ""` when the optional title prompt is blank. Acceptable today (resolver chain falls through to filename) but cosmetically noisy. Worth a conditional-emission feature in entity-create — e.g., `"title": "{{prompts.title?:omit}}"` syntax — so optional empty values drop the line entirely.
- **FLN-v66-3 (Observation — process):** Per-stage subagent dispatch (Subagent-Driven Development from superpowers) worked cleanly for S1-S5. Three implementers ran in series with full task context each. None hit BLOCKED; all reported DONE with sensible spec deviations (S1: implementer caught + fixed an S2-blocking gap where `platform/manifest.json` cards catalogue wasn't bumped; S2: implementer adapted fake-DV shim to add `.array()`; S5: implementer adapted cowork-helper regex from JS files to content `.md` files since that's where cowork uses ActivityFeed). Controller stayed thin: dispatch → verify diff → commit handed off in implementer prompt → move on.
- **FLN-v66-4 (Observation — process):** Three subagent stages, two PATCH cycles, one user-smoke round between each major surface = ~3-4 hours wall time for the full arc. The 4-PATCH v0.64.x arc on the same day took similar time. PATCH-driven smoke convergence is the right velocity for visual UI cycles on a single-developer single-machine workflow.
- **FLN-v66-5 (Important — design gap):** The scratch v0.66.2 bug existed since v0.3.0 of the scratch blueprint (entity-create migration). The manifest description claimed the title "lands in `title:` frontmatter" but the wiring was simply absent — and no test asserted it. Worth a v0.66.3 follow-up adding an SHC sub-assert: `for each blueprint with new_entity_buttons[].prompts[].key K, assert K appears in frontmatter_template as {{prompts.K}}` — catches similar mis-wirings across all entity-create-using blueprints (meetings, people, projects, scratch, finance, etc.).
- **FLN-v66-6 (Observation):** The workshop's `sauce update --force` git-resets `pantry/` against `origin/main` BEFORE running the installer. This means subscription edits must be **committed to `origin/main` first** or they get blown away. Discovered S7 the hard way; once committed via `chore(workshop): subscription lockstep`, the installer picked up the new pins. Worth documenting in `Docs/how.md` for future cycle-runners.
- **FLN-v66-7 (Important — visible UX bug, not closed):** Every Activity card displays `12:00 AM` regardless of the underlying note's `created_at`. Reproed at accuris-sauce close: `test-activity` scratch has `created_at: "2026-05-19T21:00:04-06:00"` in frontmatter (9:00 PM), but the card meta line reads `12:00 AM`. Hypothesis: Dataview parses ISO-with-time frontmatter into a Luxon `DateTime` object before exposing it on `p.created_at`; `window.moment(luxonObj)` then receives the object, doesn't know how to coerce it, and silently falls back to "today at midnight" (or `Invalid Date` which `.format("h:mm A")` still renders as `12:00 AM` in some moment versions). Fix candidates: (a) detect Luxon DateTime via duck-type (`typeof tsRaw.toFormat === "function"`) and call `tsRaw.toFormat("h:mm a")` directly; (b) call `String(tsRaw)` first so moment always gets a string; (c) coerce native Date via `tsRaw.toISOString()`. Small PATCH; recommend a v0.66.3 if user finds the wrong time bothering, otherwise next-cycle polish.

## Lessons

1. **Subscription lockstep is the foot-gun.** Workshop has its own subscription that pins versions; bumping the workshop catalogue without bumping the workshop's own subscription means the installer skips the blueprints (subscription < workshop catalogue). The v0.65.0 commits caught this with a dedicated S10.0 "subscription lockstep" commit; v0.66.0 had to learn it again at S7. Worth a pre-flight CLI check (`sauce status` could surface "workshop manifest = X but subscription pins = Y; mismatch will skip blueprints").
2. **The audit step paid off but partially.** S0.5 hypothesized FLN-v64-8's root cause (legacy hubs lacking `type: project`); the audit confirmed 7 dirs DO fall in that bucket. Net: the rollup works for 34/41 project dirs across vaults; 7 silently don't surface. Better than the v0.65.0-and-earlier state where ZERO project descendants surfaced. Future cycle should close the 7-vault gap.
3. **PATCH-on-PATCH is fine when smoke surfaces real issues.** v0.66.x was three releases in ~2 hours because two rounds of smoke revealed distinct problems (visual cohesion in v0.66.1; declarative manifest gap in v0.66.2). Sauce's tagging + brew + reinstall loop is fast enough that "ship, smoke, PATCH" is cheaper than batching fixes into a single bigger MINOR.
4. **The `name:` vs `title:` distinction.** Different sauce blueprints store the user-meaningful name in different frontmatter fields. Projects use `name:`; scratches (when wired) would use `title:`; people use `name:`. The v0.66.1 fix added `p.name` to the resolver chain BEFORE `p.title` so both work. Worth canonizing in `Docs/how.md`: "if your blueprint writes user-meaningful display name, prefer `name:` (matches projects/people convention) UNLESS you have a structural reason to use `title:` (matches book/article semantics)."
