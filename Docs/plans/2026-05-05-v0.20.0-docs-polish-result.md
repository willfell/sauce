## NEW lessons (v0.20.0)

> [!example]+ Five new platform values codified

**(a) Test isolation via inline synthetic bodies decouples harness tests from blueprint churn.** T4.0 root cause was a 12-cycle-old `fs.readFileSync` coupling to a deleted blueprint file (`kanban-board.md`, deleted in v0.2.0 boards cycle). The whole-suite renderer harness fataled for 12 cycles; selector-based runs (`empty`, `malformed`, `unknown-action`, `beacon-cards`, `date-aware`, `finance`, `beacon-button`) sidestepped it as workaround. Fix: inline the synthetic body string + sentinel `'synthetic://kanban-board'` template_source. The sentinel reads as test-only at a glance (the `synthetic://` URI prefix is impossible to collide with real `vault.adapter.read` paths) and is reusable for any future test fixture decoupled from on-disk state. **Pattern:** when a harness test reads a blueprint asset for fixture content, prefer an inline synthetic body over `fs.readFileSync` of a real path — the test's intent is opaque content, not coupling to the blueprint's evolution.

**(b) Helper-cases as a generic lint host.** Adding a per-file violation walker (`caseTW1TemplatesNoTrailingWhitespace`) without new selector machinery extends harness reach cheaply. The case follows the established pattern (`async function caseXxxYyy()` + `assertTrue` per assertion + final summary `console.log`) — same as caseSS5 / caseM6 / caseR6 etc. Two-level `readdir` walk (outer `<bp>` → inner `<subdir>` → file ext filter) handles the `content/` + `templates/` + `helpers/` (skipped via .md filter) layout naturally. **Pattern reusable** for any future blueprint-asset content lint: frontmatter-shape audit, embed-link resolution audit, manifest schema audit, etc. The walk skeleton + `assertTrue`-per-file + JSON-quoted violation dump can be promoted to a shared `walkBlueprintFiles(extensions, validator)` helper if a second consumer surfaces.

**(c) Cross-doc line-citation drift is a real maintenance class.** Landmine #10's stale `install.js:111` citation (drifted to `:223` over 12 cycles as the installer grew) was caught by the S2 quality reviewer, NOT by spec compliance review (because spec compliance only checked landmine #16's quote, not adjacent landmines). The fix was a 1-line edit but the lesson is the maintenance discipline: **future cycles that cite specific source lines should mark them as drift candidates** AND any cycle that touches the cited file should grep all docs for `<filename>:<lineno>` references and update in lockstep. Candidate harness extension: lint Docs/*.md for `[a-z\.]+\.js:\d+` patterns and assert each line still exists at that line in the cited file.

**(d) Two-stage subagent review (spec then quality) yields different signal at low cost.** Spec compliance found ZERO issues at both S1 and S2. Quality reviewer found 2 real cross-doc nits (count bug + landmine #10 drift) at S2 + 2 cosmetic nits at S1. The two-stage flow is cheap insurance: spec-only review would have shipped both real bugs. **Pattern:** even for "trivial" doc-only stages, run both reviewers — spec compliance answers "did they do what we asked?", quality compliance answers "is the result actually correct in context?". The latter catches drift, count bugs, cross-doc inconsistencies that the implementer subagent has no way to verify against (their context is just the task body).

**(e) Master-driven S3 (cycle close) is the right shape.** Version bumps + status-snapshot updates + result writeups + tag creation are judgment-heavy + approval-gated; subagent dispatch adds friction without value. Pattern reaffirms v0.18.2 lesson c (master-driven cycles can be very fast for mechanical edits). **Decision rule:** dispatch subagents for tasks with clear surface + action + verification (S1 + S2). Master-drive tasks with judgment calls (writing result writeup, choosing CLAUDE.md insertion shape) + approval gates (version bump, tag push) + small mechanical edits where dispatch overhead exceeds the work.
