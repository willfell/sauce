# v0.17.0 Design — Finance widget UIs (workshop 0.16.0 → 0.17.0 MINOR; finance 0.1.4 → 0.2.0 MINOR)

> [!abstract] Outcome
> Ship in-place editor widgets on every finance entity page so users never edit frontmatter manually. Closes the foundational ask from v0.16.0 smoke time ("no user should ever manually edit a frontmatter") and addresses the YAML auto-parse / boolean-string / NBSP landmine class at the SOURCE — widget DOM inputs bypass every YAML edge case we hit in v0.16.0.

## Section 2 — `FinanceFrontmatter` shared helper (architectural heart)

> [!example]+ Quoted API contract (per v0.6.0 lesson — "API-contract quoting")
> ```js
> class FinanceFrontmatter {
>     /**
>      * Mutate frontmatter on a TFile. Wraps app.fileManager.processFrontMatter.
>      * @param {TFile|string} fileOrPath - TFile or vault-relative path
>      * @param {(fm: object) => void | Promise<void>} mutator - mutates fm in place
>      * @returns {Promise<void>}
>      */
>     async update(fileOrPath, mutator) { ... }
>
>     /** Read-only frontmatter snapshot via metadataCache. Returns null if file missing. */
>     read(fileOrPath) { ... }
>
>     /** Coerce a frontmatter value to boolean: true | "true" | "TRUE" → true; else false. */
>     isTruthy(v) { ... }
> }
> ```
>
> **Quoting / coercion responsibilities (one home, one discipline):**
> - Date-shaped values (`YYYY-MM`, `YYYY-MM-DD`) — caller writes strings; helper does NOT auto-quote (Obsidian serializer preserves strings as quoted YAML when ambiguous).
> - Booleans — written as proper JS `true`/`false`; Obsidian serializes as bare `true`/`false`.
> - Arrays of objects — written as JS arrays; Obsidian serializes block-form (avoids the NBSP/inline-flow trap entirely; v0.16.0 CF-4 root cause).
> - NBSP avoidance — widget inputs are DOM `<input>` values; never round-trip through user-typed YAML strings.
>
> **Why `processFrontMatter` (not raw text):**
> - Official Obsidian API since 1.4; handles missing-frontmatter creation, preserves trailing body verbatim, atomic write.
> - Bypasses every YAML edge case we hit in v0.16.0 (auto-parse-to-Date, inline-flow NBSP, boolean→string).
> - One file changed, one helper, one place quoting discipline lives.
>
> **Reusability beyond v0.17.0:** any future blueprint with frontmatter mutation (Trip status, Project state, etc.) wraps the same helper.

## Section 3 — Widget UX anatomy (hybrid pattern)

> [!info] Shared visual contract for every editor widget
> ```
> ┌─ <WidgetName>-root ─────────────────────────────────┐
> │  [+ Add <Item>]                                     │  ← action button (accent tone)
> │  ┌─────────────────────────────────────────────┐   │
> │  │ Col 1     │ Col 2  │ Col 3  │ ...     │ ✕  │   │  ← header row (muted)
> │  ├─────────────────────────────────────────────┤   │
> │  │ value     │ value  │ value  │ ...     │ ✕  │   │  ← clickable row → edit modal
> │  │ value     │ value  │ value  │ ...     │ ✕  │   │
> │  └─────────────────────────────────────────────┘   │
> │  (empty state: "No <items> yet. Click + Add.")     │
> └─────────────────────────────────────────────────────┘
> ```
>
> **Modal anatomy** (mirrors NewBudgetButton/NewPaycheckButton overlay precedent — same backdrop, dialog, button row, Esc/Enter handling):
> ```
> [Add <Item>]   ←  heading flips to "Edit <Item>" on row click
> ┌──────────────────────────────────┐
> │ <Field 1>       [______________] │
> │ <Field 2>       [______________] │
> │ ...                              │
> │ <validation status>              │
> │         [Cancel]      [Save]     │
> └──────────────────────────────────┘
> ```

> [!info] Per-widget specifics
> | Widget | Fields | Validation | Cross-file effect |
> |---|---|---|---|
> | `BudgetCategoriesEditor` | `name` (text) · `planned` (number) · `actual` (number) | name required; numbers ≥ 0 | none — single-file mutation |
> | `PaycheckExpensesEditor` | `item` · `amount` · `category` · `paid` (toggle) · `url` (optional) | item + amount required; amount ≥ 0; url valid-or-empty | none — single-file mutation |
> | `InvoiceTimeLogEditor` | `date` · `start` (time) · `end` (time) · `description` (text) | date required; end > start | **propagate**: total_hours = sum; sibling Invoice atlas's `hours` = total_hours; `amount` = rate × hours |
> | `InvoiceControls` | `rate` (number, $/hr inline) · `submitted` (button) | rate ≥ 0 | none — single-file mutation; submit-toggle sets/clears `submitted_date` to today |

> [!warning] Critical disciplines (apply v0.16.0 lessons proactively)
> - **Embed dedup** — every widget block opens with `if (dv.container.closest(".markdown-embed")) return;`. Suppresses widget UI when entity is rendered inside `![[...]]`.
> - **Mobile-aware (gotcha 7)** — rows wrap; inputs use `min-width: 0` + `flex: 1`; modal `max-width: 90vw`; min-tap-target 44px.
> - **Per-row delete** — explicit `✕` button with confirm Notice; no undo (consistent with Obsidian-native delete UX).
> - **Hours auto-compute** — `hours = (end_minutes - start_minutes) / 60` rounded to 2 decimals; surfaced inline in the modal as the user adjusts start/end (read-only computed field).
> - **Cross-file propagation** (InvoiceTimeLogEditor only) — after Time-Log frontmatter mutation, second `processFrontMatter` call on sibling Invoice atlas. Two-file write surface; non-atomic — accept trade-off for v0.17.0; surface inconsistency as a Notice if the second write fails.

> [!example]- BudgetNavButtons (mirror InvoiceNavButtons)
> ```js
> class BudgetNavButtons {
>     async render(dv) {
>         if (dv.container.closest(".markdown-embed")) return;
>         const previous = dv.container.querySelector(":scope > .bnb-root");
>         if (previous) previous.remove();
>         const path = dv.current()?.file?.path || "";
>         const m = path.match(/^beacon\/finance\/budgets\/(\d{4}-\d{2})\//);
>         if (!m) return;
>         const month = m[1];
>         const targets = [
>             { label: "Budget",      target: `beacon/finance/budgets/${month}/Budget-${month}.md` },
>             { label: "Budgets Hub", target: `beacon/finance/budgets/Budgets.md` },
>             { label: "Finance Hub", target: `beacon/finance/Finance.md` }
>         ].filter(t => t.target !== path);
>         // ... render buttons (same shape as InvoiceNavButtons)
>     }
> }
> ```
> `PaycheckNavButtons` is symmetric: `paychecks` regex + Paycheck/Paychecks Hub/Finance Hub targets.

> [!info] Hub button styling polish
> Lives entirely inside `finance-hub-cards.js`'s `_renderHubActionRow` + `_renderAreaLinkRow` methods. Visual-only:
> - Stronger visual hierarchy: action row uses accent-tone buttons with consistent padding + larger min-tap-target; area-link row uses secondary tone with "Open <Area>" label + chevron icon.
> - Group spacing: 16px between rows; 12px button gap within row.
> - Responsive: rows wrap on narrow viewports; min-tap-target 44px.
