> [!abstract] Why this matters
> Second cycle (after v0.6.0 trips) to ship a "per-entity ecosystem" blueprint with substantial CustomJS surface. Generalized the per-entity sub-folder pattern from invoice-only to ALL three sub-areas (CF-3) for layout symmetry. > [!example]- CF-2 (commit 5abb94d): date-tolerant FinanceStatus + entity-page status badges + Time-Log embed dedup
> **Smoke trigger:** post-CF-1, user surfaced six issues in one batch:
> 1. Status badge missing on entity NOTE pages (only on hub cards)
> 2. Card titles showed ISO timezone strings (e.g., `2026-05-01T00:00:00.000-06:00`) instead of `2026-05`
> 3. Past-month budget showed "Planning" instead of "Done"
> 4. Inline `= this.month` Dataview expressions on Invoice atlas returned `-`
> 5. Time-Log embedded via `![[Time-Log-2026-05]]` duplicated SpaceNavButtons + InvoiceNavButtons
> 6. Auto-promoted board card showed no nav buttons (root-cause was likely Cmd+R timing; CF-2 propagation resolved)
>
> **Single root cause for #2/#3:** YAML auto-parsing. Obsidian/Dataview parses `budget_month: 2026-05` as a Date|Luxon DateTime object; `String(month)` returns ISO timezone instead of "2026-05"; `moment(String, "YYYY-MM", true)` strict parse fails on the ISO; status falls back to Planning + title shows ISO. Same for `pay_period_start` and `month` on invoice.
>
> **Fix:** introduced three new helpers on FinanceStatus class:
> - `_toMoment(val)` — accepts Luxon DateTime (Dataview), JS Date, OR string; returns a moment instance
> - `formatDate(val, fmt)` — coerces any of the above to a formatted string
> - `renderBadge(dv, type)` — renders a colored status pill onto a Dataview container (palette mirrored from BeaconCards: success/warn/muted)
>
> All three *Cards classes updated to use `customJS.FinanceStatus.formatDate(...)` for title + range + submitted-date renders. All three NewXButton classes updated to: (a) quote ALL date frontmatter values on creation (`"${month}"` instead of `${month}`) to prevent YAML auto-parse to Date; (b) embed `customJS.FinanceStatus.renderBadge(dv, type)` dataviewjs block at top of each generated entity body. NewInvoiceButton replaced `= this.month` with literal substituted `${month}` in the table. Time-Log body wraps nav-button blocks in `dv.container.closest(".markdown-embed") != null` check so they suppress when embedded but still render when Time-Log is opened directly.
>
> **Versions:** finance 0.1.1 → 0.1.2.

> [!example]- CF-4 (commit 473e2c7): defensive truthy paid-expense check
> **Smoke trigger:** user added `expenses: [{item: X, paid: true}, {item: Y, paid: true}]` to a paycheck and status still showed Planning. After DevTools inspection (`f.expenses` returned `["expenses:  - item: X    paid: true  - item: Y    paid: true"]`), root cause traced to (a) inline-flow YAML with embedded U+00A0 NO-BREAK SPACEs (NBSPs) — likely copy/paste artifact from a chat/markdown renderer that converted spaces to NBSPs — caused YAML parser to interpret the entire content as a single quoted string item; (b) even with valid inline-flow, `paid: true` parser path can produce string `"true"` instead of boolean `true` depending on Obsidian YAML internals.
>
> **Fix:** finance-status.js paycheck branch + paychecks-cards.js subtitle now use shared `isPaid(e)` accepting `e.paid === true` OR `typeof e.paid === "string" && e.paid.toLowerCase() === "true"`. Strict-true still works; string-"true" tolerated. Did NOT auto-fix the NBSP-corrupted file — surfaced to user with DevTools query, rewrote file directly with their consent, demonstrated working state.
>
> **Versions:** finance 0.1.3 → 0.1.4.

## NEW v0.16.0 lessons (carry forward to v0.17.0+)

> [!warning] Frontmatter parsing landmines (codify in Docs/landmines.md candidate cycle)
> 1. **YAML date auto-parsing** — any frontmatter value matching `YYYY-MM-DD` or `YYYY-MM` gets auto-parsed by Obsidian/Dataview into Date|Luxon objects. Quote on creation (`"${month}"`) OR centralize date-tolerant access via a `_toMoment(val)` helper that accepts Luxon-DateTime / JS-Date / string. **Both are recommended together** — quote on the WRITE side; tolerant parse on the READ side.
> 2. **Frontmatter parser variance for booleans** — boolean values from inline-flow YAML may be parsed as string `"true"` instead of boolean `true`. Centralize coercion via shared isTruthy-style helper.
> 3. **NBSP gotcha** — manual frontmatter editing surfaces invisible-character traps (U+00A0 NO-BREAK SPACEs from copy/paste through chat/markdown renderers). Reinforces the user's "no manual frontmatter editing" thesis. Widget-driven mutation (queued for v0.17.0) sidesteps both YAML-block-vs-flow AND invisible-whitespace classes of bug.
>
> [!info] Composability patterns (likely reusable across future blueprints)
> 4. **renderBadge() pattern** — status pill rendered on entity NOTE pages (not just hub cards) via dataviewjs block embedded by the New<X>Button creation helper. Palette mirrored from BeaconCards. Likely generalizable to any entity blueprint with a status concept.
> 5. **Embed dedup via closest(".markdown-embed")** — when a note is embedded via `![[X]]`, its dataviewjs blocks should suppress duplicated nav-buttons. The check `dv.container.closest(".markdown-embed") != null` works inline at the top of the embedded note's nav-button block.
> 6. **Per-entity sub-folder symmetry** — when one sub-area uses an entity-ecosystem layout (Invoice's per-month folder), all sub-areas should mirror it (Budget/Paycheck per-period folder). Surface during DESIGN, not after smoke; CF-3 was avoidable.
>
> [!info] Cycle-shape patterns (refinement of established lessons)
> 7. **First-of-its-kind blueprint expects CFs that refine the per-entity layout itself** — not just visual polish. CF-3's per-entity sub-folder change moved every entity's filesystem location. This is more invasive than v0.6.0's CFs (which adjusted in-page UX without moving files). Plan for layout-drift CFs in future entity-ecosystem cycles.
