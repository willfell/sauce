# v0.18.2 Design — Project Map sub-tasks bug fix (workshop 0.18.1 → 0.18.2 PATCH; project 1.3.4 → 1.3.5 PATCH)

> [!abstract] Goal
> Sub-task cards (Task Board Cards under a project's task) currently surface in the project Map's task list as "Unassigned." Sub-tasks belong to a task, not a project — they should never appear on a project Map. Add two predicates to the existing `dv.pages().where(...)` filter in `ProjectWorkstreams` to exclude them. Belt-and-suspenders frontmatter+layout signal.

---

## Bug

Per `blueprints i want.md` line 68-69, carried since v0.5.0 project port:

> I create a project, create a task on a board, create a note for that task on the board, so it's sub-task, this sub-task gets shown within the projects Map section as unassigned, sub-tasks should never be included in the list of tasks for a project Map.

---

## Root cause

`platform/blueprints/project/helpers/project-workstreams.js:50-51` does a recursive `dv.pages` over the project directory, then a thin filter:

```js
const projectPages = dv.pages(`"${projectDir}"`)
    .where(p => p.file.path !== filePath && !p.file.name.endsWith("-board") && !p.file.name.endsWith("- Map"));
```

The recursion picks up sub-task cards living at `<projectDir>/tasks/<Task>/board/<Card>.md` (or post-promote `<projectDir>/tasks/<Task>/board/<Card>/<Card>.md`). Sub-task cards (Task Board Card.md) set `source_board` in frontmatter, which makes the downstream classifier at `:62-64`:

```js
} else if (page.source_board !== undefined || page.workstream !== undefined) {
    unassigned.push(info);
}
```

push them into the "Unassigned" bucket.

---

## Fix

### Filter shape (locked)

```js
const projectPages = dv.pages(`"${projectDir}"`)
    .where(p =>
        p.file.path !== filePath &&
        !p.file.name.endsWith("-board") &&
        !p.file.name.endsWith("- Map") &&
        !p.task_parent &&
        !p.file.path.includes("/board/")
    );
```

### Why two predicates (belt-and-suspenders)

| Predicate | Signal | Source-of-truth |
|---|---|---|
| `!p.task_parent` | **Frontmatter** — universal sub-\* identifier | `Task Board Card.md` L23 (`task_parent: <% taskParent %>`) AND `Task Note.md` L3 (`task_parent: {{TASK_PARENT_PATH}}`). Top-level Kanban Card does NOT set `task_parent`. |
| `!p.file.path.includes("/board/")` | **Layout** — sub-task cards live under a task's `board/` folder | Per `Task Board.md` L20 `new-note-folder` config + Task Board Card auto-promote regex `^beacon/projects/.../tasks/.../board/...$` |

Either alone closes the bug. Both together are robust against a future template authoring slip OR a layout reorg that drops one signal.

### Sub-notes (Task Note.md) — also defensively excluded

Task Note has `task_parent` but no `source_board`, so today it's already filtered by the downstream `:62-64` classifier. Adding `!p.task_parent` makes the exclusion explicit at the query stage. A user who manually adds a `workstream:` to a sub-note would today see it surface incorrectly under that workstream's section; the new filter prevents that. Defense-in-depth.

### Edge cases

- **Empty `task_parent: ` (regex miss in `Task Board Card.md` L7)** — Dataview reads as empty string. `!p.task_parent` is `!""` = `true` → still excluded. ✓
- **Manually adding `task_parent` to a top-level task** — that task gets hidden from Map. Acceptable behavior reversal: `task_parent` becomes the canonical "this is a sub-\*" marker. Reversible.
- **Pre-existing projects with sub-tasks already created** — they disappear from Map immediately after re-render. No migration needed. ✓
