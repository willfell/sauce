## NEW lessons (v0.24.0)

**(a) Tree-rename cycles must grep `path.join` multi-arg patterns explicitly.** Bare `s|Docs/Meta|ranch|g` literal sed misses `path.join(dir, "Docs", "Meta")` (multi-arg form). Two such cases surfaced in v0.24.0 (run-bootstrap.js:82 + install.js:3926). Pattern: any future Tree N rename design should include a `grep -rE 'path\.join.*"<oldpart>"\s*,\s*"<oldsubpart>"'` scan as a numbered pre-commit step.

**(b) Tree-rename sed scope must include source dirs, not just infrastructure code.** v0.24.0 plan T2.11 scoped sed to `install.js + bootstrap + cli + install.sh`. Did NOT include `platform/mechanisms/` or `platform/blueprints/`. CF-2 caught it post-harness-RED. Pattern: Tree N rename design must explicitly enumerate sed targets across ALL source trees, including templates + jsdoc examples that reference runtime paths.

**(c) Foundational plugin data.json migrations are a Tree-rename concern.** When the runtime plumbing tree moves (Tree 3 = ranch), every `.obsidian/plugins/<id>/data.json` whose schema includes a path field referencing the old tree needs migration. CustomJS's `jsFolder` was the only one in v0.24.0 — but every future foundational + allowlisted community plugin should be audited at design-time. Pattern: Tree-rename design MUST list every `.obsidian/plugins/*/data.json:<field>` that holds a path value.

**(d) Acceptance-gate-surfaced CF distinct from review-surfaced CF.** v0.22.1 + v0.23.0 surfaced CFs at two-stage subagent review (pre-commit). v0.24.0 CF-3 surfaced ONLY at the manual smoke (post-commit). Two-stage review can't catch user-environment-dependent issues like stale `.obsidian/plugins/<id>/data.json` config. Pattern: acceptance gate is load-bearing for Tree-rename cycles even after harness GREEN.

**(e) git rename detection swallowed staging step.** `git add -A platform/ Docs/...` post-`git mv` staged ONLY the rename moves (100% similarity). The actual content modifications (sed Docs/Meta → ranch in mechanism + blueprint sources) didn't stage until a follow-up `git add -u` ran. S2 split into 2 commits (494d29e renames + b2bdada modifications). Pattern: after `git mv` of a wide-touching rename, ALWAYS run `git add -u` separately to stage modifications, then verify `git diff --cached --stat` before committing.

**(f) Surgical helper migration preserves user customization.** `applyCustomJsSettings` only overwrites `jsFolder` when current value is `Docs/Meta/Scripts` (legacy v0.23.x stale value) OR absent/empty. Any other value (user-customized) is preserved + skipped via `info/skipped_user_customized` history. Pattern reusable for any future migration helper that needs to update one specific stale value while preserving user intent.

**(g) Helper-once-per-install-run vs helper-per-item.** Existing helpers (applyHotkeys / applyCommunityPluginData / applyCorePluginSettings) run inside `installItem` per mechanism/blueprint. CustomJS jsFolder is platform-wide (not per-item) — so applyCustomJsSettings runs ONCE at top-of-install before per-item iteration. Pattern: when a setting is platform-wide (driven by `variables.*` rather than per-item manifest fields), put the helper at top-level install entry, not inside installItem.

**(h) Stub body changes are rare events; new md5 must be captured durably.** v0.24.0's stub body change was the FIRST since v0.1.2 (~22 cycles). New md5 `ea23aa812503bfca66359d3b2b239ba8` is captured in landmine #13 + install.md. CLAUDE.md status snapshot also pinned. Pattern: any future stub body change must update all 3 surfaces atomically.
