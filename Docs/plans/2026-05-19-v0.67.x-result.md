## What shipped (final state at v0.67.3)

- **Activity smart default-open (v0.67.0):** opens by default when `activityCount > 0` (existing non-empty gate).
- **Section icon vertical alignment (v0.67.1):** Tasks/Meetings/Activity icons now baseline-aligned with their title text via `.sauce-section-icon { display: inline-flex; align-items: center }` + inner `svg { display: block }`.
- **"children touched" → "notes touched" wording (v0.67.0):** breadcrumb language now user-meaningful.
- **Daily template simplified (v0.67.3):** removed the COWORK_CALLOUTS marker line (the v0.67.0 segmented-accent companion remained user-visible across syntax attempts: `%% %%` → `[//]: # (...)`); removed the `## Notes` heading; appended a `---` horizontal-rule divider after the dashboard. Activity now uses single-color `accent: "grey"` (new CSS rule `[data-accent="grey"] { border-left: 4px solid var(--text-faint) }`), matching the user's preference for a subtle grey throughout.

## NEW FLNs surfaced during this arc

- **FLN-v67-1 (Minor — test quality):** AF-V067-RUC-4 relies on a fixture's `created_at: "2026-05-19T08:00:00Z"` falling within the asOf window to trigger the decorated-existing-survivor branch. The branch identity is comment-only; a future maintainer who changes the timestamp could silently flip RUC-4 to test the synthetic branch instead. Add a pre-condition assert `// root IS in windowed set` for self-documentation.
- **FLN-v67-2 (Important — platform learning):** `_renderTodoBadge`'s initial `Array.isArray(p.file.tasks)` guard would have silently no-op'd on every real Dataview page (DataArray Proxy, not native array). Caught by S2 code-quality review; fixed in v0.67.0 commit `353e082` via `.where()` with `.filter()` fallback. Codebase has prior precedent for DataArray-aware checks (existing `getTasks()` + `cowork-readiness.js`). **Worth a platform-level doc note in `Docs/how.md` or CLAUDE.md** — the `Array.isArray()` foot-gun is recurring.
- **FLN-v67-3 (Process — plan-template improvement):** Mid-cycle integration-smoke FAILs caused by `depends_on` bumps preceding catalogue bumps (v0.67.0 S5/S6 window) weren't documented as expected failure modes. The S6 implementer initially misdiagnosed them as pre-existing. Future plans with mid-cycle depends_on bumps should explicitly note this expected failure window.
- **FLN-v67-4 (Minor — UX nuance, OBSOLETE by v0.67.3):** raw-page-count vs coalesced-card-count mismatch in `byBlueprint`. Mooted because segmented accent is retired in v0.67.3; if re-enabled in a future cycle, this nuance should be re-evaluated.
- **FLN-v67-5 (Minor — cosmetic):** `.sauce-bread` selector declared twice in `sauce-daily-dashboard.css` (line 83 pre-existing one-liner + line 86 v0.67.0 block). Both work; CSS allows re-declaration. Worth consolidation in a future polish PR.
- **FLN-v67-6 (Minor — S4 hygiene):** S4 implementer used `git add -A` and swept in `.claude/scheduled_tasks.lock`. Fixed in v0.67.0 commit `0907c81` via untrack + .gitignore. Pattern improvement: implementer prompts should remind to stage specific files, not the whole tree.
- **FLN-v67-7 (Important — operational workflow):** Consumer vault subscription pins do not auto-bump when running `sauce update --force` against a newer workshop catalogue. v0.67.0 consumer reinstall required manual pin edits. v0.67.1/.2/.3 worked smoothly because of mid-execution pre-bumps. Worth shipping a `sauce update --bump-pins` flag (or auto-bump when `subscription.workshop_version < workshop.workshop_version`) in a future cycle.
- **FLN-v67-8 (NEW — UX learning):** Obsidian comment-syntax invisibility claims are unreliable. v0.64.1's claim that `%% %%` is "invisible in BOTH source AND reading mode" was wrong — it's visible in source AND in live-preview-with-cursor-on-line. v0.67.2's claim that `[//]: # (...)` is invisible in BOTH Reading and Live Preview was also wrong — Obsidian renders link-ref defs as visible text in Live Preview. **Lesson:** in Obsidian, NO inline marker syntax is truly invisible across all editor modes. Don't ship templates with user-visible anchor markers; if cowork-style anchoring is needed, position the marker at EOF (where the user doesn't typically scroll) OR use a fallback-only mechanism (search by heading, append at EOF).
- **FLN-v67-9 (Important — architectural):** `_getActivityCount` now duplicates ActivityFeed's rollup logic (`_buildRollupRules(dv)` invocation + survivors+rolledUpRoots dedupe). Cross-mechanism drift risk: if `activity-feed`'s rollup semantics change, `_getActivityCount` must update in lockstep or counts diverge from rendered cards. Worth factoring an exposed helper `ActivityFeed.computeBlueprintCounts(dv, opts) → {total, byBlueprint}` in a future activity-feed minor.
- **FLN-v67-10 (Important — DOM coupling):** Drill-in placement walks 2 levels up from `parentEl` to the card root (`parentEl.parentElement.parentElement`). Fragile against `cards@*` DOM restructures — if BeaconCards changes its row/left/meta layout, drill-in lands somewhere wrong. Cleaner long-term fix: `cards@0.2.7` MINOR adding an explicit `afterBody: (page, parentEl) => void` hook (the deferred Approach B from the v0.67.0 brainstorm). Now demonstrably needed.

## Lessons

1. **Obsidian comment-syntax invisibility is unreliable.** Two attempts (v0.64.1 `%% %%`; v0.67.2 `[//]: # (...)`) both shipped under the wrong belief that the syntax was invisible in Live Preview. Neither was. The right answer was to retire the marker entirely (v0.67.3). General rule: when templating in Obsidian, assume the user can see any text you write — including "comment" markers. Position vestigial markers at EOF (out of typical scroll path) OR fall back to heading-search anchoring.
2. **Subagent-Driven Development cadence works for typed implementer roles.** v0.67.0's 8 stages with sonnet+haiku model selection by signal (file count × integration complexity) shipped cleanly. The DataArray-vs-Array gotcha in S2 was caught by the code-quality reviewer (test mocks would have passed). Lesson: when mocking Dataview proxies, mocks should match the runtime type structure, not the test convenience structure.
3. **PATCH cadence drives convergence.** 3 PATCHes in 9 hours after v0.67.0 each closed real visible-on-screen issues. The user smoke → brew + reinstall → smoke loop is fast enough that batching multiple issues into one bigger PATCH would have meant longer feedback windows. 4 small releases is the right velocity for visual-UI cycles on a single-developer single-machine workflow.
4. **Consumer subscription pre-bump is the operational accelerator.** v0.67.0's reinstall required manual pin edits per FLN-v67-7. v0.67.1/.2/.3 worked smoothly because of mid-execution pre-bumps. Worth automating — see FLN-v67-7 for the candidate `sauce update --bump-pins` flag.
5. **DOM coupling earns explicit hooks the second time.** v0.67.0 deferred cards@0.2.7's `afterBody` hook (Approach A vs B; YAGNI). v0.67.1's drill-in DOM-walk fix exposes the coupling concretely. Now demonstrably worth shipping cards@0.2.7 with the explicit hook (FLN-v67-10).
