## Summary

Resolves the user-facing macOS bug that v0.22.1 surfaced post-smoke: `Beacon/` (capital B; v0.22.0+ inside-vault layout) case-aliases the lowercase `beacon/<module>/` namespace on APFS, breaking Obsidian's vault index for every blueprint runtime entry point. v0.22.1 `.gitignore` masked the `git status` symptom but did NOT fix runtime breakage — Daily Notes returned *"Folder 'beacon/daily' not found."* on every macOS install.

The fix renames the workshop clone dir to a non-aliasing name (`pantry/`). Brainstorm pivoted through three theme proposals (Lighthouse → Gravity/Magma → **Sauce**) before locking; the final theme bundles a full project-brand rebrand (CLI binary, env vars, ASCII banner, runtime banner, GitHub repo URL).

## NEW lessons (v0.23.0)

(a) **`.beacon-backup` suffix preserved across rename.** Landmine #12 mechanic #2 backup-suffix is the canonical convention. Renaming to `.sauce-backup` would silently miss prior backups in consumer vaults. Pattern: backup-suffix conventions are platform-version-stable; brand-rename does NOT cascade to backup-file names. Codified in S2 commit-message + design Section 12.

(b) **sed regex word-boundary discipline for brand-rename.** Bare `s|beacon|sauce|g` would have corrupted `beacon-button` mechanism name + `BeaconButton` / `BeaconCards` class names. Mitigation: scope sed to specific anchored patterns (`s|`Beacon/|`pantry/|g`, `s|beacon status|sauce status|g`, `s|env\.BEACON_VAULT|env.SAUCE_VAULT|g`, etc.); never bare-substitute generic words. Pattern reusable for any future brand-rename or mass-substitution cycle.

(c) **GitHub repo rename + permanent redirect via SSH-cache invalidation.** SSH backend can take ~30s to propagate the rename. Old URL `git@host:user/old.git` redirects to new URL automatically once propagated. Both URLs return identical refs post-propagation. The `git remote set-url` step + verification was needed to confirm authentication via the renamed URL works. v0.22.x users with `BEACON_REPO_URL` shell-rc exports need to update to `SAUCE_REPO_URL` (documented in install.md upgrade recipe).

(d) **Multi-cycle naming destination locked at brainstorm time** prevents revisiting naming each cycle. v0.23.0's 6-round brainstorm (Lighthouse → Gravity/Magma → Sauce) was cheap; locking the full theme triple (sauce + pantry + spice + ranch) means v0.24.0 + v0.25.0 march toward a known target. Pattern: brainstorm full naming sequence even if shipping in stages — the marginal cost of locking 3 future names alongside 1 current name is negligible compared to re-brainstorming each cycle.

(e) **Reserved-headroom: USED 3 review-surfaced CFs (10th data point; second MIXED case after v0.22.1).** Two-stage subagent review caught all three pre-commit, no S3-smoke regressions. Pattern reaffirms two-stage spec-then-quality review as load-bearing for wide-touching mechanical-rename cycles. Spec-only review would have shipped CF-3 (quality-class issue: stale comment URL with the old GitHub repo).

(f) **`platform/visual/banner.js` is a runtime-CLI brand surface.** Future brand-prose audits must include both `install.sh` ASCII banner AND `platform/visual/banner.js` runtime banner. Pattern: any brand-name string emitted to user-facing stdout from CLI verbs lives in this file. Design + plan checklists for future brand cycles should explicitly list it.

(g) **Pre-existing working-tree drift should be excluded from v0.X.Y atomic commits.** `Docs/Meta/Scripts/cards/beacon-cards.js` had cards@0.2.2 → 0.2.3 outline-pill content drift in the tree at session start (pre-v0.23.0). Quality reviewer caught it; deferred to a separate v0.23.x dogfood-resync commit. Pattern: when staging an atomic-rename commit, only stage files that the cycle touches; leave dogfood drift for a separate housekeeping commit.

(h) **No-migration policy for unreleased private repos** simplifies many cycles. User-locked: "no real vaults are utilizing." This eliminated migration-tooling design from v0.23.0 scope (no `beacon migrate` verb, no auto-rename inside `beacon update`). Manual `mv Beacon pantry` + `sed` recipe in install.md upgrade section is sufficient. Pattern: when the project hasn't been distributed externally, design BREAKING migrations as user-side documented recipes rather than tool-side automation.
