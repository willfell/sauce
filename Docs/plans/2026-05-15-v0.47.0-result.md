# v0.47.0 — icons mechanism extraction + entity-create defect cleanup (CLOSED 2026-05-15)

## Summary

v0.46.0 shipped the entity-create mechanism + 7-site big-bang migration. Phase 3 e2e validation in the headspace consumer vault (same day) surfaced four real defects that the workshop dogfood + 15-harness preflight didn't catch. v0.47.0 fixes all four in one bundled MINOR cycle and uses BUG-1 as the trigger to extract a canonical icon-resolution mechanism that DRYs three pre-existing patterns (entity-create string-passthrough bug, AccentButton raw-SVG contract, nav-buttons hardcoded ICONS map).

**Mechanism extraction (Lego principle 5 ≥2-rule):**
- NEW `icons@0.1.0` — Lucide kebab → SVG resolver. Two-tier resolution: Tier 1 ~21 vendored SVGs (15 byte-for-byte from nav-buttons + 6 new plus-variants); Tier 2 Obsidian `setIcon()` runtime fallback.
- `entity-create@0.1.1 → 0.2.0` — render() consumes `customJS.Icons.resolve`; JSON-schema tightens `body_template` + `extra_files[].body_template` to basename-only.
- `nav-buttons@2.6.2 → 2.7.0` — inline ICONS const deleted; consumes Icons.

**Bug fixes:**
| # | Severity | Bug | Fix |
|---|---|---|---|
| BUG-1 | HIGH | Kebab names rendered as literal text in entity-create buttons | icons mechanism + entity-create consumer migration |
| BUG-2 | HIGH | person notes created without `type: "person"` | One-line people manifest patch + forward-defense `_validateTypeFieldConvention` rule in install.js |
| BUG-3 | CRITICAL | project atlas body empty (runtime resolved bare basename against vault root) | NEW `_resolveBodyTemplatePath` in install.js auto-prepends `{{templates_path}}/` on bare basenames |
| BUG-4 | CRITICAL | extra_files Map + Board bodies empty | Same fix as BUG-3 (applies to `extra_files[].body_template` via same helper) |

- `sauce reinstall --vault <headspace>` exit 0; 80 new history entries; 4 content overwrites; clean run.
- `sauce audit --entity-create --vault <headspace>` → zero findings (5 aligned, 2 unverifiable [pre-existing v0.46.x FIX-LATER, not regression]).
- Registry-level fixes verified: `ranch/entity-create-registry.json` project entry now stores `body_template: "ranch/templates/Template, Project.md"` + matching extra_files paths; all 3 template files exist on disk. person entry stores `frontmatter_template.type: "person"`.
- All 7 entity-create flows now register their entries cleanly (was 6/7 in v0.46.2 due to FLN-f project skip).
- Workshop preflight green across 15 harnesses; +20 sub-asserts vs v0.46.2 baseline.

## Plan vs reality

### S7 plan-baseline drift — validator rule's gating predicate

The plan + design specified the rule should gate on `rule_fragments[*].when.frontmatter.type === "<bp-singular>"`. Investigation in S7.1 confirmed **no existing blueprint declares `when.frontmatter.type`** — every blueprint's rule_fragment targets via `scope.path_glob` instead. Two blueprints (scratch's `Scratch-2*.md` and `Scratch-Day-*.md` fragments) DO declare `required_frontmatter.type.equals` predicates, but that's a different predicate path.

**Resolution:** implemented the rule as literally specified. At v0.47.0 the rule is forward-defense — fires on zero current blueprints (gate trivially passes). BUG-2 is fixed independently by the people manifest patch. If a future blueprint adopts the `when.frontmatter.type` pattern, the rule fires and blocks install. Captured in CLAUDE.md Status (live) entry; no design re-litigation.

### S5 code review — two mid-stage documentation corrections (S5b)

S5 code review surfaced two latent documentation gaps:
- (I1) `installed.json` embed-site comment didn't warn that raw manifest values (bare basenames) are stored intentionally — embedding resolved paths there would cause the audit walker to produce double-prefixed paths like `"ranch/templates/ranch/templates/Foo.md"`. Comment added inline at install.js:404-411.
- (I2) `_resolveBodyTemplatePath` helper comment claimed "schema rejects them at validate time", but the JSON schema is not loaded by any runtime validator — the helper itself is the operative enforcement. Comment amended.

Both are documentation-only; bundled into S5b commit per the user's "two commits if mid-stage correction" guidance.

### Harness lockstep surfaced at every Icons-touching stage

S2, S3, and S5 all required minimal harness-pin lockstep updates beyond the plan's nominal 2-file scope (entity-create source + manifest):
- S2: `run-entity-create.js` EC-2 version pin + `run-renderer.js` R-EC-ROW source-string assertion.
- S3: `run-renderer.js` Icons instance wired into renderer eval scope (otherwise tests fatal on `customJS.Icons.resolve` undefined).
- S5: `run-entity-create.js` extractFn extension to pull `_resolveBodyTemplatePath` into the sandbox.

All updates were load-bearing (preflight goes red without them) and minimal (no new test cases, no refactors). Each surfaced as DONE_WITH_CONCERNS from the implementer subagent; spec reviewers confirmed scope was proportionate. Pattern matches v0.44.0's FLN-5 plan-author miss class — worth codifying as a checklist item for future Icons-style mechanism extractions: any consumer migration likely needs sandbox/test-pin updates.

### Sub-assert count vs plan

Plan targeted +16 sub-asserts whole-suite; delivered +20 (+25%). Breakdown:
- run-renderer.js: +7 R-EC-ICON-1..7 (per plan).
- run-entity-create.js: +6 EC-33..38 (per plan).
- run-helper-cases.js: +6 TYPE-1/2/3 × 2 asserts each (plan target was +3; each TYPE case naturally needs 2 asserts — behavioural return + history side-effect). Overshoot is more thorough coverage, not scope creep.

### Pacing

10 stage commits (S1..S9 + S5b) vs 12-stage plan; S11 manual smoke and S12 cycle close are the remaining 2 stages. ~5 subagent dispatches (S1, S2, S3, S5; plus 4 spec reviewers + 4 code-quality reviewers) for the 4 subagent-pacing stages; the other 6 stages were controller-direct per the Q6 hybrid pacing.

## NEW FIX-LATER-NOTEs

- **FLN-a (Important, deferred):** S1 code review flagged `_tier1` getter is invoked three times per `resolve()` hit. The cache (`__cache`) makes each call cheap, but a local capture would clarify intent. Plan kept S1 scope tight; bundle into a future Icons-touching cycle.
- **FLN-b (Minor):** S1 reviewer flagged double-underscore `__cache` is inconsistent with sauce convention (every other mechanism uses single underscore). Rename → `_cache` in a future cleanup pass.
- **FLN-c (Minor):** S2 reviewer flagged `plusIcon` is 16×16 while all Tier 1 SVGs are 15×15 — 1px visual inconsistency between resolved and fallback paths. Either delegate to `Icons.resolve("plus")` or shrink plusIcon literal.
- **FLN-d (architectural, low priority):** AccentButton's raw-SVG contract retained in v0.47.0. A future callsite-simplification cycle could widen AccentButton's `opts.icon` to accept kebab names (auto-resolve internally), letting the 17 existing callers drop their inline SVG literals.
- **FLN-e (S5 architectural):** JSON schema for entity-create's `new_entity_buttons[]` is documentation-only — no runtime validator loads it. If a future blueprint slips a path-shaped value into a manifest, `_resolveBodyTemplatePath`'s defensive pass-through is the only guard. A schema-driven validator gate in install.js would catch the violation upstream with a clearer error.
- **FLN-f (plan ordering for Icons-style extractions):** Every consumer migration in this cycle (S2, S3, S5) required minimal harness-pin lockstep updates not anticipated in the plan's nominal 2-file scope. For future plan authoring: add explicit "expect ~1-3 harness pin updates per migration; treat as load-bearing not scope creep" guidance.
- **FLN-g (validator rule scope):** The Layer 2 type-field rule fires on zero current blueprints because none declare `when.frontmatter.type`. A broader interpretation (gate on `required_frontmatter.type.equals` too) would activate the rule on scratch's existing fragments. Deferred as a follow-on broadening cycle.
- **FLN-h (Obsidian-side smoke):** This cycle's headspace re-validation was CLI-/registry-level (resolved-paths-on-disk verified; audit zero findings). The actual click-through-in-Obsidian smoke for the 7 entity-create buttons was not performed; the underlying mechanisms are proven sound but the gold-standard runtime check is deferred to the user's day-of-use.
- **FLN-i (BUG-5, surfaced post-S11, queued for v0.48.0):** Per-project kanban-board task-creation flow does not prompt for workstream. Root cause is a pre-existing v0.46.0 design gap: that cycle stripped `_createTaskNote` + `_createTaskBoard` (~184 LOC) from `ProjectNavButtons` with the intent that the kanban-plugin's "+ Add a card" UI + `Template, Kanban Card.md`'s `<%* %>` workstream picker (via `tp.system.suggester`) would replace those flows. The Kanban plugin uses Obsidian's CORE template paste mechanism, which does NOT execute Templater syntax. Project's manifest declares zero `templater_folder_templates[]` entries, so no Templater trigger fires on file creation in `spice/projects/<slug>/tasks/`. Net effect: the `<%* %>` block stays as raw text in the created task note, never prompts. User-observable symptoms (surfaced during v0.47.0 S11 smoke): (1) no workstream prompt on task-card creation; (2) no "+ Add Task"/"+ Add Task Board" buttons on the project atlas page (those were the helper buttons that previously called `_createTaskNote`/`_createTaskBoard` directly). v0.47.0 explicitly did NOT cause this regression — install didn't touch project-nav-buttons.js (source bytes unchanged), didn't touch Templater config (data.json last modified May 13 at headspace), and the documented BUG-1..4 fixes are all verified clean. Design options for v0.48.0: (a) extend `entity-create` with a `pick-from-frontmatter` prompt type so a `+ New Task` entity-create entry can read the atlas's `workstreams[]` field; (b) restore `_createTaskNote` + `_createTaskBoard` in `ProjectNavButtons` with entity-create-style declarative wiring; (c) register a `templater_folder_templates[]` entry on project (requires verifying Templater's folder-pattern support for the `<slug>` segment).
