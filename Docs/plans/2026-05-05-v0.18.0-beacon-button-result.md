## Outcome summary

> [!abstract] What landed
> First new mechanism in many cycles. Promoted the duplicated outline-accent action button pattern (12 call sites with subtle drift) into a shared `BeaconButton` mechanism. Visual surface IDENTICAL by design — pure dedupe + lock-the-pattern.
>
> **NEW mechanism `beacon-button@0.1.0`:**
> - Single class `BeaconButton` with one method `render(parent, opts) → HTMLButtonElement`.
> - opts: `{label, icon, onClick, flex?, disabled?, tone?}`.
> - `tone` reserved for future additive tones; only "accent" valid at v0.18.0; silently falls back on unknown.
> - Hover handlers honor `btn.disabled` (no swap when disabled).
> - `flex: true` appends `flex: 1; min-width: 0` for row-fill use.
> - Caller may mutate `btn.disabled` post-render.
>
> **S1 — project migration (4 sites):**
> - `project-nav-buttons.js` lines 717 / 733 / 737 / 765 migrated to `customJS.BeaconButton.render(...)`.
> - Private `_renderActionButton` method (lines 354-362) DELETED.
> - All 4 sites use `flex: true` (preserves prior row-fill behavior).
>
> **S2 — finance migration (8 sites):**
> - NewBudgetButton / NewPaycheckButton / NewInvoiceButton (3) — inline use, NO flex.
> - BudgetCategoriesEditor "+ Add Category" + PaycheckExpensesEditor "+ Add Expense" + InvoiceTimeLogEditor "+ Add Entry" (3) — inline use.
> - InvoiceControls Save (preserves disabled-state mutation pattern via BeaconButton's `btn.disabled`-aware hover handlers) + Mark/Unmark Submitted (2) — inline use.
>
> **CF-1 (in-cycle, surfaced in Smoke E):**
> - InvoiceControls Save now also recomputes `fm.amount = round(rate * hours, 2)` so rate changes immediately propagate to the summary strip without requiring a Time-Log re-edit.
> - `refreshSaveEnabled` extended: Save is enabled when EITHER input != saved rate OR stored amount != computed (rate × hours). Lets users re-sync stale amounts (created with rate=0 then rate updated later) without changing the rate value.
> - Removed misleading "Amount recomputes on next Time-Log edit" caveat note.
> - Pre-existing UX gap from v0.16.0/v0.17.0; Smoke E surfaced it; user opted in-cycle CF over deferral.

## In-cycle CFs — chronological

> [!example]+ CF-1 (commit dc56814 combined with S2; finance 0.2.6 → 0.2.9): InvoiceControls Save UX — amount recompute + drift detection
> **Smoke trigger:** Smoke E. User opened `beacon/finance/invoices/2026-08/Invoice-2026-08.md` with `rate: 175`, `hours: 2`, `amount: 0`. Time-Log had 2 hours logged. User expected $350 (= 175 × 2) but saw $0.
>
> **Root cause:** Pre-existing v0.16.0/v0.17.0 design — InvoiceControls's Save onClick only wrote `fm.rate = value`; it did not recompute `fm.amount`. The cross-file `amount = rate × hours` propagation only fired from `InvoiceTimeLogEditor`'s `_addFlow` / `_editFlow` / `_deleteFlow`. Users who set rate AFTER adding time-log entries got stuck with `amount = 0` (the value computed at entry-add time when rate was still 0). The widget even had a caveat note ("Amount recomputes on next Time-Log edit") acknowledging the gap.
>
> **Additional UX trap surfaced during CF-1 first-pass:** the original `refreshSaveEnabled` only disabled Save when input == saved rate. So a user with `rate=175, amount=0` (already saved at 175) couldn't trigger Save without changing rate to a different value first. CF-1 first-pass missed this — second pass fixed it.
>
> **Fixes (CF-1 first-pass + refinement, atomic):**
> (a) Save onClick now also computes `fm.amount = Math.round(value * h * 100) / 100` where `h = Number(fm.hours || 0)`. Rate changes propagate to amount immediately.
> (b) `refreshSaveEnabled` extended: Save is enabled when input is valid (`>= 0`) AND (input != saved rate OR stored amount != computed). Stale-amount entities can be re-synced at the saved rate without manipulating rate input.
> (c) Misleading "Amount recomputes on next Time-Log edit" caveat note removed (the v0.18.0 behavior makes it incorrect).
>
> **Versions:** finance 0.2.6 → 0.2.7 (S2 atomic) → 0.2.8 (CF-1 first-pass) → 0.2.9 (CF-1 refinement; in-cycle re-process bump per gotcha 9 because `same-version-with-different-content` install was being skipped — barebones got the CF-1 first-pass file, then I edited again at 0.2.8, the installer skipped because subscribed.version === installed.version → bumped to 0.2.9 forced re-process).
>
> **Manifest description:** updated finance to call out the new behavior + drift-detection logic.

## NEW v0.18.0 lessons (carry forward to v0.19.0+)

> [!warning] In-cycle re-process bump rule reinforced (gotcha 9 stress test)
> 1. **CF-1 surfaced the in-cycle re-process bump rule for the first time at scale.** When iterating on CF-1 inside the same finance@0.2.8 staging, the second-pass edit didn't propagate to barebones until I bumped 0.2.8 → 0.2.9. The installer's `subscribed.version === installed.version` short-circuit (install.js:111) skips re-process when the version hasn't changed, even if the file content has. **Pattern:** any in-cycle revision after the first install must bump version. The 3-bump stack (S2 atomic 0.2.7 → CF-1 first 0.2.8 → CF-1 refinement 0.2.9) is the load-bearing example. Future first-of-its-kind mechanism cycles + concurrent CF iterations should plan for this.

> [!info] Mechanism-promotion-from-duplicated-pattern as a cycle shape
> 2. **Mechanism promotion cycles are reliably-clean dedupe cycles.** v0.18.0 was the first new mechanism in many cycles; it shipped with one in-cycle CF (UX surface, not migration regression). The dedupe migration itself was clean — 12 sites migrated via 8 subagent dispatches in parallel (T2.1-T2.7 in one batch + T1.13 standalone). Subagents can be parallelized when they touch independent files with no shared state. The v0.18.0 plan's "expect single-stage clean / 0-1 CFs" call panned out — the only CF was UX-driven, not from the migration itself.

> [!info] Harness stub maintenance after mechanism dependency growth
> 3. **When a mechanism becomes a dependency of existing helpers, the harness stub map needs the new entry.** v0.17.0 finance harness stubs lacked `BeaconButton` until I added it (after T2.7 surfaced FF4 fatal). Pattern: any new mechanism that other CustomJS classes call needs a corresponding stub in `makeFinanceCustomJsStub` (and any other helper-load stub) at the moment of migration. Add the stub at the same time as the mechanism's first cross-class consumer.

> [!info] Reserved-headroom-UNUSED precedent extended (3+ data points)
> 4. **v0.13.0 + v0.14.0 + v0.18.0 (reserved-headroom-UNUSED-or-near-unused) precedent now has 3 data points.** v0.18.0 was almost-unused (1 CF instead of 0; CF was UX-driven, not migration-driven). Mechanism promotion + non-UI dedupe cycles are reliably clean. Visual + UX cycles still pull 2-6 CFs (v0.5.0 / v0.11.0 / v0.12.0 / v0.6.0 / v0.16.0 / v0.17.0 — 6-data-point first-of-its-kind UI cycle precedent). Decision rule reinforced: new mechanism / pattern extraction → expect clean; new UI surface → expect CFs.

> [!info] Tone palette slot reservation as future-proofing pattern
> 5. **Reserving an opt slot at API launch is cheaper than additive opts later.** BeaconButton ships `tone` opt at v0.18.0 with only "accent" valid; future tones (neutral / danger) become additive without API break. If `tone` had been omitted, adding it later would require all callers to opt in OR a deprecated default. Pattern reusable for future mechanisms.

> [!info] Cross-file UX coupling discovered late (CF-1 root cause analysis)
> 6. **Pre-existing UX gaps surface during smoke testing of derivative work.** v0.18.0 didn't change InvoiceControls's Save logic — it only changed how the Save button RENDERS. But the user's first real-world test ("set rate after adding time entries → expect amount to update") exposed an underlying limitation in v0.16.0/v0.17.0's design (Save only writes rate; cross-file amount propagation only fires from Time-Log entry mutations). Lesson: even pure dedupe cycles can surface latent UX bugs in the migrated surface. Plan for one UX-CF when migrating user-visible widgets.
