# v0.47.0 — icons mechanism extraction + entity-create defect cleanup + type-field validator rule

## Summary

v0.46.0 shipped the `entity-create@0.1.0` mechanism + 7-site big-bang migration. v0.46.1/0.46.2 patched two FIX-LATER items. Phase 3 e2e validation in the headspace consumer vault (2026-05-15) surfaced four real defects that the workshop dogfood + 15-harness preflight did not catch — three live render-time / install-time bugs and one frontmatter omission.

v0.47.0 fixes all four defects in one bundled MINOR cycle, and uses BUG-1 as the trigger for extracting a canonical icon-resolution mechanism that DRYs three pre-existing patterns: entity-create's broken `spec.icon` string passthrough, AccentButton's raw-SVG contract, and nav-buttons' hardcoded vendored ICONS map.

## Why MINOR (not v0.46.3 PATCH)

The four defects are individually PATCH-shaped (manifest edits + 1 installer auto-prepend). Bundling a NEW mechanism extraction into the same cycle bumps it to MINOR — consistent with every prior new-mechanism cycle in sauce history (v0.32.0 platform-claude, v0.40.0 scratch, v0.41.0 convenience, v0.46.0 entity-create).

The icon resolver could in principle have shipped as a private helper inside entity-create.js (Q2 option A — minimal patch). The principled fix is option (C), a separate mechanism — three pre-existing icon-resolution patterns (entity-create kebab-bug, AccentButton raw-SVG, nav-buttons ICONS map) satisfy Lego principle 5's ≥2-consumers-at-extraction-time rule. Each consumer migration in this cycle ratifies that principle.

## The Lego thesis applied to icons

**Concern:** "Given a Lucide kebab name, return SVG HTML."

**Pre-existing duplication (the ≥2-consumers check):**

1. `platform/mechanisms/nav-buttons/space-nav-buttons.js:62-78` — hardcoded `ICONS` map (~16 entries, each ~250 chars of vendored SVG) + `fallbackIcon(label)` letter helper.
2. `platform/mechanisms/entity-create/entity-create.js:46-50` — passes `spec.icon || plusIcon` to AccentButton, where `spec.icon` is a kebab name string and `plusIcon` is one vendored fallback SVG. (This is the broken site — it expects names to resolve themselves, which they don't.)
3. `platform/mechanisms/accent-button/accent-button.js:42` — `btn.innerHTML = opts.icon + ...`. AccentButton's contract is raw-SVG-only. 17 call sites pass real SVG; only entity-create passes a name (incorrectly).

**Extraction satisfies all six Lego principles:**

1. *One mechanism = one cross-cutting concern* — icon resolution, nothing else.
2. *Manifest is the API surface* — kebab name strings are already in blueprint manifests (`new_entity_buttons[*].icon` + `nav_buttons[*].icon`); no schema change required.
3. *Mechanisms own their materialization* — `icons.js` owns the kebab → SVG translation; no other code knows the SVG vocabulary exists.
4. *Versioning is concern-scoped* — adding `icons@0.1.0` doesn't churn any blueprint manifest; it churns two mechanism consumers (entity-create + nav-buttons) in this cycle.
5. *Extraction beats anticipation* — three existing patterns demonstrate the duplication today. Both v0.47.0 consumers (entity-create + nav-buttons) migrate in-cycle, satisfying the ≥2-consumers-at-extraction-time sub-clause.
6. *Blueprints are self-contained* — blueprints continue to write kebab names in their own manifests; the mechanism is the shared resolver.

## Mechanism design — `icons@0.1.0`

**Layout:**
```
platform/mechanisms/icons/
  manifest.json
  icons.js                 (CustomJS class: Icons — single resolve(name) method)
  README.md
```

**Manifest schema added to blueprints:** none. icons is a runtime-only helper; blueprints have nothing new to declare.

**API:**

```js
customJS.Icons.resolve(name)
// → string | null
// Returns SVG HTML for the given kebab name, or null if not resolvable.
// Caller is responsible for fallback handling on null.
```

**Two-tier resolution:**

1. **Vendored map (Tier 1):** ~20-21 entries pre-shipped in icons.js as a frozen object literal. Union of:
   - The existing 15 entries in `space-nav-buttons.js`'s ICONS map (board, daily, todo, meetings, summary, projects, planning, plus, journal, trips, trip, finance, people, scratch, briefcase).
   - The 6 new kebab names introduced by entity-create blueprints (`users-plus`, `user-plus`, `folder-plus`, `wallet-plus`, `file-plus`, `pencil-plus`).

   The `plus` entry is shared (already in nav-buttons map; doubles as entity-create's `plusIcon` fallback).

2. **`setIcon` fallback (Tier 2):** for names not in the vendored map, `Icons.resolve` creates a throwaway `<div>` element, calls Obsidian's `setIcon(el, name)`, captures `el.innerHTML`, returns it. Tier 2 is the long-tail Lucide name pathway.

**Why two-tier:**
- Vendored Tier 1 keeps the workshop test harness deterministic — DOM-icon assert can stat the exact SVG bytes for known names without an Obsidian runtime.
- Tier 2 keeps future consumers from hand-vendoring a new SVG every time a new Lucide name enters a manifest.

**Failure modes:**
- Tier 1 miss + Tier 2 returns empty string (Obsidian's `setIcon` no-ops on unknown name) → `resolve()` returns `null`. Each call site handles `null` with its own fallback (entity-create falls back to `plusIcon`; nav-buttons falls back to its existing letter-fallback helper).
- `setIcon` undefined (e.g., test harness without Obsidian) → Tier 2 catches the `ReferenceError`, returns `null`. Tier 1 still works under jsdom.

**Vendored SVG provenance:** lifted verbatim from `space-nav-buttons.js:62-78`. No re-design; pure dedupe.

## Migration 1 — `entity-create.js` `render()`

`platform/mechanisms/entity-create/entity-create.js:38-51`:

```diff
 class EntityCreate {
     async render(dv, opts) {
         if (dv.container.closest(".markdown-embed")) return;
         const { instance } = opts || {};
         const spec = await this._loadSpec(instance);
         if (!spec) { dv.paragraph(`EntityCreate: no spec for "${instance}"`); return; }

         const plusIcon = `<svg xmlns="http://www.w3.org/2000/svg" width="16" height="16" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round"><path d="M12 5v14"/><path d="M5 12h14"/></svg>`;
+        const resolved = spec.icon ? customJS.Icons.resolve(spec.icon) : null;
         customJS.AccentButton.render(dv.container, {
             label: spec.label,
-            icon: spec.icon || plusIcon,
+            icon: resolved || plusIcon,

```

**Effect:** kebab names are resolved through Icons; unknown names fall back to `plusIcon`. The `plusIcon` literal is retained as the deterministic ultimate fallback (avoids depending on Icons working at all for the basic create-button surface).

**Consumer count after this migration:** 1 of 2 in-cycle.

## BUG-3 / BUG-4 — convention-shift on `body_template`

**Root cause recap:** the installer's `resolveEntityCreateEntry` (`platform/install.js:1664-1666`) runs `substituteLenient` on `body_template`. When the value is a bare basename like `"Template, Project.md"`, substitution is a no-op. The registry stores the bare basename. At runtime, `_readBody` calls `app.vault.getAbstractFileByPath(subbed)` which resolves against vault root, finds nothing (real installed location is `ranch/templates/Template, Project.md`), returns `""`. Body comes out empty.

**Convention-shift fix (Q3 option B):** the installer now auto-prepends `{{templates_path}}/` when the substituted result has no `/` separator. Manifest writers continue specifying basenames; the installer is the single source-of-truth for resolution.

**Installer delta** (`platform/install.js:1660-1686` `resolveEntityCreateEntry`):

```diff
 if (typeof entry.body_template === "string") {
-  resolved.body_template = substituteLenient(entry.body_template, variables);
+  const substituted = substituteLenient(entry.body_template, variables);
+  resolved.body_template = _resolveBodyTemplatePath(substituted, variables);
 }
 ...
 if (Array.isArray(entry.extra_files)) {
   resolved.extra_files = entry.extra_files.map((ef) => {
     ...
     if (typeof ef.body_template === "string") {
-      out.body_template = substituteLenient(ef.body_template, variables);
+      const substituted = substituteLenient(ef.body_template, variables);
+      out.body_template = _resolveBodyTemplatePath(substituted, variables);
     }
     return out;
   });
 }
```

**New helper** (added inside `install.js`):

```js
// Convention: body_template values are blueprint-template basenames.
// If the substituted value has no path separator, prepend templates_path.
// Pre-existing values containing '/' pass through untouched (defensive;
// no v0.47.0 manifest uses them but the helper does not break old fixtures).
function _resolveBodyTemplatePath(value, variables) {
  if (typeof value !== "string" || !value) return value;
  if (value.includes("/") || value.includes("\\")) return value;
  const templatesPath = variables.templates_path || "ranch/templates";
  return `${templatesPath}/${value}`;
}
```

**JSON-schema rule** (`platform/mechanisms/entity-create/schema/new-entity-buttons.json`):
- `body_template`: pattern `^[^/\\]+\.md$` — basenames only. Same on `extra_files[].body_template`.
- Validator rejects any path-shaped value at the deep validator Layer 2 stage.

**Audit-walker stance:** unchanged. `platform/audit/entity-create-walker.js:200-227` already resolves `body_template` against `ranch/templates/<basename>` and emits `dead_path` MEDIUM when the file is missing. The walker passed in v0.46.x because basename + ranch/templates/ join exists on disk — the bug is purely a runtime resolution mismatch in `entity-create.js` `_readBody`. After v0.47.0 the registry stores the full vault-relative path; the walker's join semantics still produce the same result. No audit-walker delta required; `audit@0.2.1` stays put.

**Manifest impact:** project's existing manifest values (`"Template, Project.md"` etc.) already match the basename-only schema. No on-disk manifest change for the values themselves — the registry-output change is what fixes the bug. Project gets a PATCH bump because the registry it produces changes shape.

## AccentButton API stance

**Unchanged in v0.47.0.** The contract stays as `opts.icon` = raw SVG HTML. icons mechanism produces pre-resolved SVG; AccentButton consumes it as-is.

**Why not widen:**
- No caller is broken without widening.
- Widening adds two code paths (detect SVG vs name) and a runtime dispatch per render call.
- Migrating the 17 existing callers is mechanical but pure cosmetic — defer until a callsite-simplification cycle.

**Captured for future:** if a future cycle wants AccentButton to accept names, the change is purely additive (`if (typeof opts.icon === "string" && !opts.icon.startsWith("<")) { opts.icon = customJS.Icons.resolve(opts.icon); }`).

## Escalation gates (STOP and re-scope)

- **S1 `setIcon` fallback misbehaves headlessly.** If Tier 2 falls over under jsdom-free preflight, vendor the long tail OR document Tier 2 as runtime-only and assert Tier 1 coverage of every kebab name referenced in any manifest. Do not silently break preflight.
- **S5 convention-shift catches an `extra_files[].body_template` with `/`.** No v0.47.0 manifest uses this shape, but if a future manifest does (e.g., during this same cycle's edits), do not silently strip — surface the value, ask for re-scope.
- **S11 surfaces a 5th bug not in the catalogue.** STOP. Do not auto-iterate. Capture in a follow-up handoff and re-scope.
- **Any consumer of nav-buttons` ICONS map relies on private behavior that Icons doesn't reproduce.** If S3 surfaces a regression in a hub render, revert S3, ship S1+S2+S5/S7/S8 as-is. Icons mechanism still gets its ≥2-consumer claim from a future cycle.
