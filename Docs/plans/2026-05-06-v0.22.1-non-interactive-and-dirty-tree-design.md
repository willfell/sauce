## 2. Architecture

> [!info] Three layers of flag plumbing
> Bash flags currently land at `install.sh` and stop there. v0.22.1 forwards them through `exec node beacon-cli.js bootstrap ...`, into `cmd-bootstrap.js`, and into `runFirstRunWizard({nonInteractive, defaults})`. The wizard already has a `nonInteractive` branch (added in v0.21.0) — v0.22.1 just reaches it from the bash side and improves its defaults to match the interactive path.

```
curl ... | bash -s -- --non-interactive --mechanisms=all --vault=/abs/vault
   ↓
install.sh
   • parse flags (NEW: --vault=PATH equals form per IMP-5)
   • preflight (node + git)
   • git clone --depth=1 → <vault>/Beacon/
   • npm install --omit=dev
   • exec node Beacon/platform/cli/beacon-cli.js bootstrap \
         --vault <vault> \
         [--non-interactive] \
         [--mechanisms=<csv|all|none>] \
         [--blueprints=<csv|all|none>]                         (NEW: forwarding)
   ↓
beacon-cli.js dispatcher
   • routes "bootstrap" verb → cmd-bootstrap.run(ctx, args)
   ↓
cmd-bootstrap.js
   • parse args (NEW: --non-interactive + --mechanisms + --blueprints)
   • runBootstrap({ vaultPath, nonInteractive, defaults: { mechanisms, blueprints } })
   ↓
bootstrap.js phaseFirstRunWizard
   • runFirstRunWizard({ nonInteractive, defaults, ... })
   ↓
wizard.js nonInteractive branch
   • selectedMechs:
       defaults.mechanisms === "all"  → manifestMechs.map(m=>m.name)
       defaults.mechanisms === "none" → []
       Array.isArray(defaults.mechanisms) → defaults.mechanisms (CSV-split)
       undefined → DEFAULT_MECHANISMS_CHECKED  (NEW: was just ["customjs-guard"])
   • selectedBlueprints: same shape (undefined → [] preserves prior semantics)
```

### 2.1 Why these defaults

The interactive wizard's checkbox starts with 5 mechanisms checked (`DEFAULT_MECHANISMS_CHECKED` = `customjs-guard`, `nav-buttons`, `cards`, `beacon-button`, `styling`) and 0 blueprints. A user pressing Enter through every interactive prompt produces this set. `--non-interactive` should produce the **same** set — predictability over convenience. Override path is `--mechanisms=all` for users who want the full platform without prompts.

### 2.2 Why `.gitignore` over directory rename

User-locked during brainstorm. Cheap PATCH; preserves v0.22.0 `Beacon/` capital-B convention; documents the macOS APFS case-insensitivity edge case rather than designing around it.

**Symptom (real, smoke-confirmed):** On `/Users/willfell/scratch/2/Beacon/` (fresh v0.22.0 install on macOS APFS), `git status` shows untracked:

```
?? Scripts/         (phaseWriteActivation outputs)
?? boards/          (blueprint module_dir; collides with beacon/boards/)
?? finance/
?? projects/
?? trips/
```

**Root cause:** macOS APFS is case-insensitive by default. `Beacon/` and `beacon/<module>/` (landmine #11 invariant) resolve to the same directory. Blueprint `module_directory` mkdirs land **inside** the workshop clone instead of at vault root.

**Fix:** Workshop-side `.gitignore` adds entries for every blueprint's bare module name. Linux/case-sensitive installs continue to materialize at `<vault>/beacon/<module>/` correctly (separate from `<vault>/Beacon/`); the gitignore entries are no-ops for them. The v0.22.0 result writeup's speculation that the dirty state was `npm install --omit=dev` reconciling `package-lock.json` was wrong — the actual cause is the case-collision.

> [!warning] Why we are not renaming `Beacon/`
> Renaming the inside-vault clone dir (e.g. `BeaconWorkshop/`) would be BREAKING for v0.22.0 curl-bootstrapped vaults. Single user (the developer); only `scratch/2/` is affected today. But: any user who already saw v0.22.0 documentation, who has a tag-pinned `v0.22.0` install, or who has activated via the existing `Beacon/Scripts/activate.sh` would need migration. Cost outweighs benefit when `.gitignore` is sufficient.

### 3.3 `platform/cli/cmd-bootstrap.js` (existing — extend)

**Today (line 22):** hardcodes `nonInteractive: false`.

**v0.22.1:** parse args after the `--vault VALUE` already-handled prefix; pass `nonInteractive` + `defaults` through:

```js
function parseFlags(args) {
    let nonInteractive = false;
    let mechanismsArg, blueprintsArg;
    for (let i = 0; i < args.length; i++) {
        const a = args[i];
        if (a === "--non-interactive") nonInteractive = true;
        else if (a.startsWith("--mechanisms=")) mechanismsArg = a.slice(13);
        else if (a.startsWith("--blueprints=")) blueprintsArg = a.slice(13);
    }
    return { nonInteractive, mechanismsArg, blueprintsArg };
}

function resolveListArg(arg) {
    if (arg === undefined) return undefined;
    if (arg === "all") return "all";
    if (arg === "none" || arg === "") return [];
    return arg.split(",").map(s => s.trim()).filter(Boolean);
}
```

`run(ctx, args)` parses, builds `defaults: { mechanisms, blueprints }`, passes to `runBootstrap`.

## 4. Data flow + error handling

### 4.1 Bash → node argv

Forwarding rules:

| User passes | Bash forwards |
|---|---|
| `--non-interactive` | `--non-interactive` |
| `--mechanisms=all` | `--mechanisms=all` |
| `--mechanisms=daily,journal` | `--mechanisms=daily,journal` |
| `--mechanisms=` (empty) | `--mechanisms=` (forwarded; node parses to `[]`) |
| (omitted) | (no forwarding) |

The bash arg-parser stores in env-like locals `NON_INTERACTIVE=0|1`, `MECHS_ARG=<string>|""`, `BLUEPRINTS_ARG=<string>|""`. The exec line conditionally appends each flag only when set.

### 4.2 Node CSV parsing

`resolveListArg(arg)`:
- `undefined` → `undefined` (caller falls through to default)
- `"all"` → literal string `"all"` (wizard branch handles)
- `"none"` or `""` → `[]`
- otherwise → `arg.split(",").map(s=>s.trim()).filter(Boolean)`

### 4.3 Wizard CSV-vs-manifest mismatch

CSV entries naming non-existent mechanisms/blueprints are silently dropped via `_buildSubscriptionEntries`'s `_findEntry` returning `null`. **Decision:** preserve this posture (consistent with v0.21.0 `runFirstRunWizard` defaults handling). A future cycle may add user-facing warnings; out of v0.22.1 scope.

### 4.4 Failure surfaces

| Failure | Behavior |
|---|---|
| `--vault=` empty equals form | install.sh exits non-zero with "vault path required" message |
| `--mechanisms=foo` (foo not in manifest) | Wizard subscription drops "foo" silently; downstream install proceeds with valid subset |
| `--non-interactive` on a vault with existing `platform-config.json` | runBootstrap detects `cfgPreExisted=true` (existing v0.22.0 capture); re-run-wizard menu would normally fire interactively — in nonInteractive mode, skip the menu and proceed with re-install of subscribed-but-not-yet-installed entries (matches `cmd-update --force` semantics). **Plan stage will lock the exact branch.** |

|---|---|
| Adding `--mechanisms` / `--blueprints` flags pushes scope toward MINOR | Pure additive at CLI level. No new manifest schema field. No new mechanism. PATCH stays valid. |

| C14 (wizard default-set regression guard) requires loading the workshop manifest | Use the workshop's own `platform/manifest.json` (DEFAULT_MECHANISMS_CHECKED is hardcoded in wizard.js, not manifest-derived — so the assertion is purely against the wizard's constant). |
