# v0.18.0 Design — BeaconButton mechanism (workshop 0.17.0 → 0.18.0 MINOR)

> [!abstract] Cycle goal
> Promote the outline-accent action button pattern — currently duplicated across 12 call sites in `project` and `finance` blueprints with subtle cssText drift — into a shared `beacon-button` mechanism. **First new mechanism in many cycles.** Visual surface stays IDENTICAL by design; this is a pure dedupe + lock-the-pattern cycle so future blueprints can call `BeaconButton.render(...)` instead of copy-pasting 8 lines of inline cssText + hover handlers.

---

## Why now

> [!info] Pattern maturity + drift signal
> The outline-accent action button pattern was first introduced in `project` v1.x (`_renderActionButton` private method). It was copy-pasted into `finance` v0.2.x for `NewBudget/Paycheck/Invoice` triggers + 3 widget "+ Add X" buttons + InvoiceControls Save/Submit during v0.17.0's five in-cycle CFs. Drift is already real:
>
> - `project-nav-buttons.js:354` includes `flex: 1; min-width: 0;` (row-fill use)
> - `new-budget-button.js:18` omits `flex` / `min-width` (inline use)
> - `invoice-controls.js` Save button has bespoke disabled-state styling
>
> Promoting now (a) dedupes ~50+ LOC across 12 call sites, (b) locks the design before more blueprints copy-paste it, and (c) gives the next cycle's first-of-its-kind blueprint (Finance migration helper, styling, etc.) a uniform action-button primitive to call.

## API contract (LOCKED)

> [!example] Canonical signature — quote-verbatim from this design
> ```javascript
> // BeaconButton.render(parent, opts) -> HTMLButtonElement
> const btn = customJS.BeaconButton.render(actionRow, {
>   label: "New Project",         // required, string
>   icon: plusIconSvgHtml,        // required, inline SVG HTML string
>   onClick: async () => { ... }, // required, function
>   flex: true,                   // optional, default false (adds `flex: 1; min-width: 0`)
>   disabled: false,              // optional, default false (initial disabled state)
>   tone: "accent"                // optional, default "accent" (ONLY valid value at v0.18.0; reserved slot)
> });
> btn.disabled = true;            // caller may mutate after render; hover handlers respect btn.disabled
> ```

> [!info] Implementation rules
> - Inline cssText baked from canonical shape at `project-nav-buttons.js:354-362`.
> - Hover handlers check `btn.disabled` before swapping colors (no swap when disabled).
> - `flex: true` appends `flex: 1; min-width: 0;` to base cssText.
> - `tone` switch hard-coded to accent values for v0.18.0; slot reserved for future `"neutral"` / `"danger"` (additive).
> - Caller passes `icon` as inline SVG HTML string (matches every existing call site verbatim — no key-to-svg map).
> - Returns `HTMLButtonElement` (callers can mutate `disabled` post-render — InvoiceControls Save uses this).

> [!warning] Decisions explicitly NOT in v0.18.0 (reserved for future additive cycles)
> - **Tone palette extension** (`neutral` / `danger`) — slot reserved; YAGNI until a second tone is needed.
> - **Icon-as-key resolution** — caller-passes-svg-string is the only supported shape. Key map can be added additively in v0.18.x.
> - **Sizing knob beyond `flex`** — no `width` / `padding` / `radius` opts; if a future caller needs a different shape, that's a new mechanism (or new tone), not a knob explosion.
> - **`setDisabled(btn, bool)` helper** — caller mutates `btn.disabled` directly; mechanism's hover handlers honor it.

## Architecture

> [!info] Mechanism layout (mirrors `cards` mechanism)
> ```
> platform/mechanisms/beacon-button/
> ├── manifest.json    # name + version 0.1.0 + path; depends_on customjs-guard
> └── beacon-button.js # class BeaconButton { render(parent, opts) { ... } }
> ```
>
> Materializes at consumer-side `Docs/Meta/Scripts/beacon-button/beacon-button.js` (CustomJS scan folder; class auto-registers post Cmd+R).
>
> **No customjs-guard wrapper needed.** Callers are inside other CustomJS classes (project-nav-buttons, finance helpers), not dataviewjs blocks — `customJS.BeaconButton` reachable directly. The customjs-guard pattern (v0.11.0 CF-1 lesson) applies only to dataviewjs callers.

## Reserved-headroom expectation

> [!success] Closer to v0.13.0/v0.14.0 clean-cycle than v0.17.0 5-CF shape
> - **First-of-its-kind mechanism** — yes (first new mechanism in many cycles).
> - **First-of-its-kind UI cycle** — NO. Visual surface stays IDENTICAL by design.
> - 6-data-point UI-cycle CF precedent (v0.5.0 / v0.11.0 / v0.12.0 / v0.6.0 / v0.16.0 / v0.17.0) does NOT apply directly — those cycles introduced new UI; this cycle dedupes existing UI.
> - v0.13.0 / v0.14.0 reserved-headroom-UNUSED precedent applies (zero novel UX surface).
> - **Realistic CF expectation:** 0-1 inline CFs. Most likely failure mode is preserving subtle styling drift (e.g., NewBudgetButton's missing `flex` was deliberate for inline use; mechanism must not force `flex: true`).
> - Plan reserves 0.18.x headroom but commits to single-stage clean as primary outcome.
