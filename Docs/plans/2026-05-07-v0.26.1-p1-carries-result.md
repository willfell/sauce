- Backup-failure aborts modification (no writes) + history error
- Atomic write: `fs.writeFileSync(target+".tmp", body)` then `fs.renameSync` to target
- Additive shallow merge: `{ ...existing, ...workshopManifest.app_settings }` — platform-as-overrider for declared keys (NOT first-wins)
- History entry on success under `step: "app_settings"` with `keys_written` array

## NEW lessons (v0.26.1)

> [!info] (a) — Plugin id pre-flight probe is a reusable pattern for any future foundational_plugins[] expansion
> Design's best-guess `obsidian-paste-url-into-selection` mismatched the upstream canonical id `url-into-selection`. The S1 pre-flight probe (Task 1.0) caught the mismatch before manifest commit. Pattern: any future cycle adding to `foundational_plugins[]` should run `node -e "const idx=require('./platform/bootstrap-lib/community-plugins-index'); idx.fetchIndex().then(map => …)"` against each best-guess label to confirm canonical id BEFORE the manifest edit. Saved a CF that would have landed as a 404 fetch error in S3 acceptance smoke. Codify as a step in `Docs/how.md` for the next plugin-baseline cycle.

> [!info] (b) — `applyAppSettings` is the canonical workshop-level helper template for any future cycle that needs a single declared knob in `.obsidian/<file>.json`
> The helper structure (workshop-level call ONCE per install run; tp.app.vault.adapter.getBasePath try/catch; `.obsidian/` dir absent skip; malformed-JSON guard; backup-on-edit only-if-existed; backup-failure aborts; atomic tmp+rename; additive shallow merge with platform-as-overrider; failure-loud history) is now reusable verbatim for any future workshop-level setting that lives in a single `.obsidian/<file>.json`. Pattern: when a future cycle needs to ship one or two declared keys in a NEW `.obsidian/<file>.json`, mirror this helper structurally (the v0.24.0 `applyCustomJsSettings` was the previous reference; v0.26.1 is now equally canonical).

> [!info] (c) — Two-stage subagent review continues to catch IMPORTANT findings (7 data points reaffirmation)
> S2 spec reviewer found ZERO gaps (PASS first try). S2 quality reviewer found 4 IMPORTANT (I-1 re-run wizard auto-add not wired; I-2 BS15 stdout leak; I-3 atomic-write tmp orphan; I-4 AS5 doesn't cover backup-fails-aborts). Applied I-1 + I-2 in S2 review fix; deferred I-3 + I-4 to v0.26.x carries. Spec-only review would have shipped both I-1 and I-2 latent. Reaffirms 7 data points (v0.20.0 + v0.22.1 + v0.23.0 + v0.24.0 + v0.25.0 + v0.26.0 + v0.26.1). Spec compliance gates "is what's done what was specified"; quality review gates "is what's done sound." Skipping either ships latent bugs. Cheap insurance.

> [!info] (d) — Parallel-session git race during S1 commit
> S1 implementer's commit raced with a parallel session's `docs(queue): wire v0.24.0 cowork blueprint into endless-loop cycle queue` commit. The other session won the lock and absorbed the staged 3 harness files into its tree (correctly, since git stages share a single working tree but commits have separate authorship). Result: harness baselines landed on origin/main BUT under a misleading commit message. Subsequent parallel-session activity reverted the stale v0.24.0 cowork docs (`c1b4a1b`) and added cowork brainstorm + Reviews-blueprint brainstorm grounded in current state — interleaving with v0.26.1 commits but not blocking. **Pattern:** when the user is using `/remote-control` orchestration with multiple sessions, git lock contention is real and can produce intermixed commits. Acknowledged in result writeup; no history rewrite. Future cycles where parallel-session work is concurrent should consider serializing commits via worktree isolation OR adding a session-pause checkpoint before commit. Surfaces a v0.26.x candidate: `Docs/how.md` section on multi-session workflow + git-lock contention recovery.

> [!info] (e) — Plan baseline-count drift continues to be harmless when actual reality dominates
> Plan predicted bootstrap 49/0 + cli 34/0 + helper-cases 422/0 post-S2. Actual: bootstrap 58/0 + cli 42/0 + helper-cases 422/0. The bootstrap + cli drift came from S1 implementer's actual sub-assert counts being more thorough than plan estimates (BS14 has 8 sub-asserts not 3; C16-C22 have more verbose assertions). Per v0.22.1 lesson f, plan baseline-count drift is harmless when commit messages report actual reality. Reaffirmed.

> [!info] (f) — Defer-to-carry trade-off on backup-fails-aborts harness coverage
> Quality reviewer's I-4 flagged that AS5 covers "backup written before edit" + "backup body matches prior content" but NOT "backup-FAILURE aborts new-content write". The implementation IS sound (control flow walks `copyFileSync` → catch → `history.push` → return BEFORE `writeFileSync`), but uncovered by harness means a future refactor could silently break the invariant. Surfaced as v0.26.x candidate (AS6 case using `chmod` on the .obsidian/ dir or stubbed `fs.copyFileSync` to throw). Pattern: when control flow correctness is not asserted by harness, the contract is verified by code-review-time inspection only — surfaces fragility under refactoring. Treat as a real gap, not a NIT.

> [!info] (g) — Wizard auto-add scope question resolved by user (NOT design ambiguity)
> Quality reviewer's I-1 flagged that the design wording for "wizard auto-adds convenience" is ambiguous on first-run-only vs first-run-AND-re-run. The implementer chose first-run only as the conservative default; the quality reviewer flagged the soundness inconsistency (re-run users would still get the cmd-status warn but never the auto-heal); user resolved at quality review time = wire it into re-run too. Pattern: when design wording is genuinely ambiguous and the implementation has to make a call, the spec reviewer should ALWAYS surface as a question for user resolution rather than approving the implementer's call. Quality reviewer caught this; bumped to user; user answered "wire both". Final outcome correct.

> [!info] (h) — Reserved-headroom precedent extends to 14 data points (4th MIXED case)
> Cycle classification: MIXED (review-surfaced 2 IMPORTANT findings caught + fixed pre-close; no in-cycle CFs from acceptance smoke). Precedent: clean cycles (v0.13.0, v0.14.0, v0.18.0, v0.18.1, v0.18.2, v0.20.0, v0.21.1) — pure additive / mechanical-rename / scoped-bug-fix; mixed cycles (v0.22.1, v0.23.0, v0.26.0, v0.26.1) — review-surfaced or smoke-surfaced 1-3 issues, all caught + fixed pre-close; first-of-its-kind cycles — 4-6 CFs from extensive smoke iteration. v0.26.1 lands in MIXED — review caught I-1 + I-2 pre-S2-close; acceptance smoke caught zero new issues. Pattern: bundled multi-feature PATCH cycles where each feature has clear precedent (applyAppSettings ~ applyCustomJsSettings; cmd-help ~ existing CLI verbs; depends_on adds ~ landmine #16 lockstep) reliably MIXED, not first-of-kind, even when the bundle is broad. Extension of design phase reading-current-cycle-of-known-helper-precedent: when ALL features in a bundle have clear precedent, reserved headroom is small.
