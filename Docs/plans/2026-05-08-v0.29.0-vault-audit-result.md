## What shipped

### NEW `sauce audit` CLI verb

6th verb in `platform/cli/sauce-cli.js` VERBS registry, slotted between `migrate` (#5) and `help` (#7). Invoked as:

```
sauce audit [--vault PATH] [--blueprint NAME] [--output-file PATH] [--no-untracked-check] [--quiet]
```

- Default invocation: walks the cwd as a sauce vault (refuses if `<cwd>/ranch/platform-installed.json` missing â€” exit 2).
- `--vault <path>`: explicit target vault. Same validation gate.
- `--blueprint <name>`: filters audit to one installed blueprint (helpful for incremental cleanup sessions).
- `--output-file <path>`: writes markdown report to that path AND prints a one-line summary to stdout. Otherwise full report goes to stdout (pipeable).
- `--no-untracked-check`: skips top-level untracked-dir scan.
- `--quiet`: exit-code only, no output (for CI / scripting).

Exit codes:
- `0` â€” zero violations + zero untracked dirs
- `1` â€” one or more violations / untracked dirs found
- `2` â€” invocation error (vault is not a sauce vault, can't read filesystem, missing platform-installed.json, output-file parent dir missing)

## Surfaces hit (CFs)

### CF-1 â€” BLUEPRINT_MODULE_DIRS singular-vs-plural map (S2; commit `7df0394`)

**Surface:** Walker uses blueprint name `"project"` to navigate to `<vault>/spice/project/...` but the real vault directory is `spice/projects/` (plural). Test fixtures put both blueprint name and vault dir as `project` (singular) so harness GREEN didn't surface the gap; CF surfaced when running the walker against real consumer `accuris-sauce` for the first time during S2 plan-time verification.

**Root cause:** The blueprint `module_directory` field (per landmine #11) is declared in each blueprint manifest at the workshop, but it's NOT propagated to consumer's `ranch/platform-installed.json` at install time. The walker has no canonical lookup table for blueprint-name â†’ vault-dir.

**Fix:** Hardcoded `BLUEPRINT_MODULE_DIRS = { project: "projects" }` map at `walker.js:58` plus `blueprintToDir(bpName)` helper that returns the override or the blueprint name as-is. Currently only `project` maps differently; other 4 audited blueprints have name == directory (trips/people/meetings/daily). The map is documented inline as a v0.29.1 carry: replace with a `module_directory` field on each blueprint entry in `ranch/platform-installed.json`, populated by `install.js` at install time. Doing so would let the walker read directly from installed metadata â€” no hardcoded map.

### CF-2 â€” blueprintName() normalizer for object-shape installed.blueprints[] (S2; commit `7df0394`)

**Surface:** Walker treats `installed.blueprints` as `["trips", "project", ...]` (string array) â€” the shape used by test fixtures. Real consumer `ranch/platform-installed.json` uses `[{name, version, installed_at}, ...]` â€” object array. Walker's `for (const bp of installed.blueprints) walk(spice/${bp}/...)` was producing paths like `spice/[object Object]/...` against real vaults.

**Root cause:** Test fixtures and real consumer state diverged on this shape. The fixture-shape choice was a convenience in S1 RED ("just a string is enough to drive the walker") that didn't match the production shape.

**Fix:** Added `blueprintName(entry)` normalizer at `walker.js:69`:
```js
function blueprintName(entry) {
  if (typeof entry === "string") return entry;
  if (entry && typeof entry === "object" && typeof entry.name === "string") return entry.name;
  return null;
}
```
Both shapes now drive the walker correctly. This same gap surfaced again in `report.js` at S3 dogfood (CF-5) â€” confirming the lesson that fixture-shape convenience defers a real-shape integration debt.

### CF-3 â€” async withTempVault sync `finally` raced async fn (S2; commit `7df0394`)

**Surface:** `run-audit.js` `withTempVault(fn)` did `try { return fn(dir); } finally { fs.rmSync(dir, ...); }` â€” but case bodies are async (`async (dir) => { ... await runAudit(...) ... }`), so the sync `finally` ran BEFORE the async body resolved, deleting the tmp dir before the audit could read it.

**Root cause:** Helper signature predates async case bodies. Selector dispatcher at `run-audit.js:30` already does `await caseAU<N>()` so the async-ness is intentional everywhere except this helper.

**Fix:** `withTempVault` now `async`, `await`s `fn(dir)`, and the cleanup runs after the awaited body resolves. Pattern matches `withTempVault` in `run-migrate.js`.

### CF-4 â€” `[]` empty-array YAML recognition (S2; commit `7df0394`)

**Surface:** `writeNote` helper emitted `key: []` as `key:` (no value) when the test passed an empty array. Walker's minimal YAML parser saw `key:` with no value and parsed it as `null`, which then failed `required_frontmatter.cssclasses.contains: ["wide", "cards"]` checks because `null` â‰  `[]`.

**Root cause:** Sauce templates emit `key: []` (literal empty list) for empty arrays â€” that's the canonical Sauce shape. The audit YAML parser is tighter than full YAML and treated `key:` as null without considering that `[]` could be a literal.

**Fix:** Two-sided: (1) `writeNote` writes `key: []` literal for empty arrays; (2) walker `_coerceScalar` recognizes `^\[\s*\]$` (with whitespace tolerance per S2.10 I-8) and returns `[]`. This minimal special-case avoids dragging in a full YAML lib for v0.29.0; full extraction to a shared frontmatter parser at `platform/lib/frontmatter.js` is the v0.29.1 carry (also from v0.28.0 NIT-1).

### CF-5 â€” report.js blueprint-name normalization for real consumer shape (S3; commit `ddb8d6f`)

**Surface:** S3.2 dogfood smoke against `/Users/willfellhoelter/notes/sauce/accuris-sauce` produced report header `**Blueprints installed:** [object Object], [object Object], [object Object], ...` â€” render-broken text from JS implicit `.toString()` on objects.

**Root cause:** Same fixture-vs-real-shape gap as CF-2 but in a different code path. Walker had been patched (CF-2) but `report.js`'s own `readInstalledMeta` helper duplicated the read logic and was assuming string-array. Test cases use string-array fixtures so the harness didn't surface the gap; only S3 dogfood against a real consumer did.

**Fix:** Mirror `walker.js:69`'s normalizer in `report.readInstalledMeta`:
```js
const list = (parsed.blueprints || []).map(b =>
  typeof b === "string" ? b : (b && b.name) || null
).filter(Boolean);
```
Both shapes render correctly:
- Test fixture (string array): `Blueprints installed: trips, project, ...`
- Real consumer (object array): `Blueprints installed: boards, daily, meetings, people, project, ...`

The duplication-on-read is a sibling-helper smell â€” both walker.js and report.js do the same normalize. v0.29.1 carry: extract a shared `blueprintName(entry)` helper (likely co-located with the DRY frontmatter parser at `platform/lib/frontmatter.js` or its own `platform/lib/installed.js`).

Generalize: **two implementations of the same predicate engine is fine when their I/O shapes meaningfully differ; extract only after both pipelines have driven real data and their shared core is stable**.
