**Status:** design
**Date:** 2026-05-17
**Workshop tag target:** `v0.51.1` (PATCH)
**Blueprint version target:** `project@1.11.0 → 1.11.1` (PATCH)
**Off-board hotfix** (mirrors v0.50.1/2/3 chain): no card on `sauce-board.md` In Planning. Surfaced via brainstorm with Context7 research on Templater + obsidian-kanban APIs (`/sauce-pipeline-10`).

## Problem statement

`platform/blueprints/project/templates/Kanban Card.md`'s `_findSourceKanbanBoard(title)` helper (added in v0.49.2 to fix the vault-root-placement symptom) detects the source kanban board by **synchronously reading every kanban-plugin file's body** and substring-matching `[[<title>]]`. Quote from the helper's comment:

> v0.49.2 fix: detect source board by SCANNING THE VAULT for kanban-plugin boards that just got a link to the new card's title.

This is reliable but:

- **O(N) file reads** on every "+ Add a card" click — N = total kanban-plugin board files in the vault.
- **Substring match on raw body bytes** — robust enough today, but couples the helper to kanban-plugin's exact `[[<title>]]` append format. Any future kanban-plugin change (e.g., adding metadata around the link, escaping titles with brackets) would break it silently.
- **Bypasses Obsidian's indexed backlinks**, which are the canonical "what links to this file?" source-of-truth maintained by Obsidian's `MetadataCache`.

## Research findings (via Context7)

**Templater (`/silentvoid13/templater`):**
- `tp.config.run_mode`: 0=CreateNewFromTemplate, 1=AppendActiveFile, 2=OverwriteFile, 3=OverwriteActiveFile. Kanban-plugin's `templaterPlugin.append_template_to_active_file(template)` triggers mode `1`.
- `tp.config.active_file`: "the active file at the time Templater was launched (may be null)." In kanban's flow it IS the new card (kanban opens the new file as the active leaf *before* invoking Templater — empirically confirmed in v0.49.1 superseded fix).
- `tp.config.target_file`: TFile of the target note where the template is inserted. In append flow, this is the new card. Useful as a TFile handle for further API calls.
- `tp.hooks.on_all_templates_executed`: post-substitution callback. Out of scope for this PATCH (Option 3 in `/sauce-pipeline-10` brainstorm).

**obsidian-kanban (`/obsidian-community/obsidian-kanban`):**
- Documented append behavior: card line `- [ ] Implement user authentication` becomes `- [ ] [[Implement user authentication]]` immediately when the card is created. The board file is modified **before** Templater is invoked.
- No documented "source board" handle is passed to the template. Kanban's only contract with Templater is `append_template_to_active_file(template)`.

**Obsidian `MetadataCache`:**
- `app.metadataCache.getBacklinksForFile(file)` returns Obsidian's indexed backlinks for a `TFile`. Updated automatically when Obsidian re-indexes a modified file (event-driven).
- Identity-based (resolves `[[title]]` to the actual `TFile` Obsidian considers the canonical target) — more robust than substring-on-raw-bytes when titles collide across the vault.

## Design — Hybrid: cache-first, vault-scan fallback

User-approved (Q1 in `/sauce-pipeline-10`): try the cache path first; fall back to the v0.49.2 vault-scan if the cache returns zero results.

Rationale: kanban modifies the board file just before invoking Templater. Obsidian's `MetadataCache` typically catches up in the same event-loop tick — but "typically" is not "always" under load or on large vaults. A cold-cache miss would silently regress to v0.49.1's empty-source-board symptom. The vault-scan fallback is the safety net.

### Algorithm

```javascript
async function _findSourceKanbanBoard(title) {
    if (!title) return "";

    // --- Cache-first path: query indexed backlinks on the new file ---
    const newFile = tp.config.target_file
        || tp.file.find_tfile(tp.file.path(true));
    if (newFile) {
        try {
            const backlinks = app.metadataCache.getBacklinksForFile(newFile);
            const sources = backlinks?.data ? Object.keys(backlinks.data) : [];
            const cacheCandidates = [];
            for (const srcPath of sources) {
                const srcFile = app.vault.getAbstractFileByPath(srcPath);
                if (!srcFile || !srcFile.stat) continue;
                const cache = app.metadataCache.getFileCache(srcFile);
                const isKanban = cache?.frontmatter?.["kanban-plugin"] === "board"
                    || srcPath.endsWith("-board.md");
                if (isKanban) {
                    cacheCandidates.push({ path: srcPath, mtime: srcFile.stat.mtime || 0 });
                }
            }
            if (cacheCandidates.length > 0) {
                cacheCandidates.sort((a, b) => b.mtime - a.mtime);
                return cacheCandidates[0].path;
            }
        } catch (_) { /* cache path failed; fall through to vault scan */ }
    }

    // --- Vault-scan fallback (v0.49.2 behavior; cold-cache safety net) ---
    const candidates = [];
    const files = app.vault.getMarkdownFiles();
    for (const f of files) {
        const cache = app.metadataCache.getFileCache(f);
        const isKanban = cache?.frontmatter?.["kanban-plugin"] === "board"
            || f.path.endsWith("-board.md");
        if (!isKanban) continue;
        try {
            const body = await app.vault.read(f);
            if (body.includes(`[[${title}]]`)) {
                candidates.push({ path: f.path, mtime: f.stat?.mtime || 0 });
            }
        } catch (_) { /* ignore unreadable */ }
    }
    if (candidates.length === 0) return "";
    candidates.sort((a, b) => b.mtime - a.mtime);
    return candidates[0].path;
}
```

### Behavior matrix

| Scenario | Cache returns | Path taken | Outcome |
|---|---|---|---|
| Warm cache, single source board | `{boardPath: [ref]}` | Cache path | Same as v0.49.2, no IO |
| Warm cache, multiple source boards (link in 2+ boards) | `{boardA: [...], boardB: [...]}` | Cache path, mtime tiebreak | Same as v0.49.2 |
| Cold cache (race with kanban's append) | `{}` empty | Falls through to vault-scan | Same as v0.49.2 |
| Title collision (two notes with same title) | Only the resolved-target's backlinks | Cache path | **MORE robust than v0.49.2** (identity-based) |
| `getBacklinksForFile` throws | (exception) | Falls through to vault-scan | Same as v0.49.2 |
| Non-kanban-board source (e.g., `[[card title]]` in a daily note) | Returns source, but `isKanban` filter excludes | Cache path returns `[]`, falls through OR returns valid board | Same as v0.49.2 |

The fallback is byte-identical to v0.49.2's helper — no behavior regression possible on the unhappy path.

## Risks

1. **Cache hit but returns wrong board.** Mitigation: same mtime-tiebreak as v0.49.2; identity-based resolution is *more* robust on title collision, not less.
2. **`getBacklinksForFile` API shape changes in a future Obsidian release.** Mitigation: try/catch around the cache attempt; any throw falls through to vault-scan.
3. **`tp.config.target_file` is null in some flow we haven't characterized.** Mitigation: `tp.file.find_tfile(tp.file.path(true))` fallback within the cache attempt; if both null, the entire cache block is skipped and vault-scan runs.
4. **Test coverage is source-string asserts only.** Same posture as v0.49.2's KC-1..3 — Templater bodies aren't statically analyzable. Manual smoke at headspace (after brew deploy) is the runtime gate.
