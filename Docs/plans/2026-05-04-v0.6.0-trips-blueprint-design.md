# v0.6.0 Trips Blueprint — Design

> [!abstract] Cycle goal
> Ship `trips@0.1.0`, a per-trip "ecosystem" blueprint mirroring `../headspace/boards/trips/`. Each trip is its own folder under `beacon/trips/<slug>/` containing an atlas + 5 baseline section notes (Flights, Stay, Packing List, To Do, Notes). Hub at `beacon/trips/Trips.md` lists trips as BeaconCards-row-layout cards (upcoming first, past below). "New Trip" action button on the hub scaffolds the directory tree from a name + dates + location dialog. Per-trip nav buttons auto-discover sibling sections at render time so user-added sections (e.g. "Honorees", "Day Before") get nav coverage with zero manifest edits.

## Locked decisions

> [!success]- D1 — Layout: flat per-trip directory mirroring headspace
> Each trip lives at `beacon/trips/<slug>/` with files directly inside (NOT nested under `<slug>/sections/...`). Mirrors headspace exactly. Sub-folders under `<slug>/` are per-entity instances within the module's one directory — **landmine #11 invariant unchanged**, no new sub-module-directory concept introduced. Project blueprint precedent (`beacon/projects/<slug>/`) already established this pattern.

> [!success]- D2 — Baseline scaffold: atlas + 5 sections
> "New Trip" creates 6 files:
> 1. `<TripName>.md` — atlas (frontmatter: type=trip, created, start_date, end_date, location, attending, tags=[trip], cssclasses=[wide])
> 2. `Flights.md` — passengers + outbound/return tables + checklist
> 3. `Stay.md` — stay details placeholder
> 4. `Packing List.md` — Clothes / Toiletries / Electronics / Documents
> 5. `To Do.md` — pre-trip checklist
> 6. `Notes.md` — free-form notes
>
> Per-trip board explicitly **deferred** to a future cycle. User picked "Headspace baseline" (not "+ per-trip board") at brainstorm time.

> [!success]- D3 — New Trip prompt: name + dates + location
> Overlay dialog collects all four fields upfront. Atlas frontmatter pre-populated. Mirrors `_promptForProjectName` overlay pattern but extends with native `<input type="date">` for the two date fields + a text input for location. Empty dates allowed (sort handles missing dates gracefully).

> [!success]- D4 — Per-trip nav-button discovery: auto-discover siblings
> `TripNavButtons` scans `<slug>/` for all `.md` files at render time, renders one button per sibling (current file context-hidden). User-added sections (e.g. "Honorees" added in los-cabos-style trip) get nav coverage automatically. Atlas detection uses frontmatter `type: trip` (explicit; not name-based heuristic).

> [!success]- D5 — Hub view: TripsHubCards (BeaconCards row layout via thin wrapper)
> `TripsHubCards` is a thin wrapper class (~150 LOC) that calls `customJS.BeaconCards.render(dv, items, opts)` from inside `dv.view("Docs/Meta/Views/customjs-guard", { class: "TripsHubCards" })`. Mirrors v0.11.0 + v0.12.0 wrapping pattern (canonical: ProjectNotesCards / ProjectReferencedByCards / MeetingsHubCards). Direct `customJS.BeaconCards.render()` from a dataviewjs block is forbidden (landmines #1/#2 cold-load TDZ).
>
> Card content per item: icon (plane SVG); title (atlas filename); subtitle (location); meta ("MMM D - MMM D, YYYY" date range or "MMM D, YYYY" single date or "No date set"). Sort: upcoming-first (sortDate ≥ today, asc) then past (sortDate < today, desc); no-date trips → bottom of past. `parseDate` helper ported verbatim from headspace `trips-board.js` (handles Dataview DateTime + string + multiple format attempts).

## Helper specs

### TripsHubCards (~150 LOC)

> [!example]- Behavior
> 1. Query `dv.pages('"beacon/trips" AND #trip')` filtered to `f.file.folder !== "beacon/trips"` (excludes the Trips.md hub itself).
> 2. For each page: extract `start_date` / `end_date` / `date` / `location` via `parseDate` (verbatim port from headspace `trips-board.js`).
> 3. Sort: upcoming (sortDate ≥ moment().startOf('day'), asc) ++ past (sortDate < today, desc); no-date → bottom of past.
> 4. Build BeaconCards items array; each item: `{ title, subtitle, meta, icon, onClick }`.
>    - title: `trip.file.name`
>    - subtitle: `trip.location || ""`
>    - meta: date display string ("Feb 25 - Mar 1, 2026" / "Feb 25, 2026" / "No date set")
>    - icon: plane SVG
>    - onClick: `app.workspace.openLinkText(trip.file.path, "")`
>    - opacity reduced for past trips (BeaconCards `dim` flag if available; else inline style override)
> 5. Empty state: `dv.paragraph("*No trips yet. Click 'New Trip' on the hub to get started.*")`
> 6. Delegates: `await customJS.BeaconCards.render(dv, items, { layout: "row" })` — guarded against TDZ via the customjs-guard view-script entry point.

### TripNavButtons (~400 LOC)

> [!example]- Context detection
> ```
> trips-hub      ← path == beacon/trips/Trips.md
> trip-atlas     ← path matches beacon/trips/<slug>/*.md AND frontmatter.type === "trip"
> trip-section   ← path matches beacon/trips/<slug>/*.md AND NOT trip-atlas
> non-trip       ← otherwise (no render)
> ```

> [!example]- Per-context render
> | Context | Render |
> |---|---|
> | trips-hub | `_renderActionButton("New Trip", plusIcon, _onNewTripClick)` opens overlay dialog |
> | trip-atlas | Section buttons auto-discovered from `<slug>/*.md` minus self; each button = `app.workspace.openLinkText(siblingPath, "")` |
> | trip-section | Atlas button (back to `<slug>/<TripName>.md`) + sibling section buttons (auto-discovered, current-section-hides) |

> [!example]- _promptForTripDetails dialog
> Overlay (mirrors `_promptForProjectName` skeleton):
> - **Name input** (text, required) — slugify preview "Slug: beacon/trips/<slug>/" + collision check vs `app.vault.getAbstractFileByPath('beacon/trips/<slug>')`
> - **Start date input** (`<input type="date">`)
> - **End date input** (`<input type="date">`)
> - **Location input** (text)
> - Cancel / Create buttons; Enter submits; Esc cancels
> - Returns `{ name, slug, start_date, end_date, location } | null`

> [!example]- _createTrip flow
> 1. mkdir `beacon/trips/<slug>/` if absent
> 2. Read 6 templates from `{{templates_path}}/Template, Trip *.md`
> 3. Substitute `{{NAME}}` / `{{SLUG}}` / `{{DATE}}` (created `YYYY-MM-DD HH:mm`) / `{{START_DATE}}` / `{{END_DATE}}` / `{{LOCATION}}`
> 4. Create 6 files (atlas as `<NAME>.md`; sections as fixed names `Flights.md` / `Stay.md` / `Packing List.md` / `To Do.md` / `Notes.md`)
> 5. `app.workspace.openLinkText(atlasPath, "")` — open atlas
> 6. `new Notice("Created trip: <NAME>")`
>
> Failure-loud: missing template → `new Notice("Template missing: <path>")` + null return; existing folder → dialog status message + retry; partial-write → continue (best-effort, per project blueprint precedent).

## Risks + mitigations

> [!warning]- R1 — Dialog with date inputs is the first project-style overlay extended beyond name-only
> Mitigations: native `<input type="date">` (no custom date picker); empty dates allowed (sort tolerates missing); mobile width compat (full-width inputs, 44px min height per gotcha 7).

> [!warning]- R2 — Auto-discover races against `app.vault` mid-creation
> Mitigation: `TripNavButtons` re-renders on dataview re-fire; new files settle into the next render. The dialog's `_createTrip` `await`s each `app.vault.create`; the open-atlas step runs after all 6 writes complete.

> [!warning]- R3 — Trip atlas filename literal match could collide with vault-root or unrelated files
> Mitigation: detection uses `path.startsWith("beacon/trips/")` + frontmatter `type: trip` (NOT name match). Non-beacon-trips files with `tags: trip` are NOT picked up by TripsHubCards (path filter is primary).

> [!warning]- R4 — Sort logic with mixed Dataview DateTime + string frontmatter
> Mitigation: `parseDate` helper ported VERBATIM from headspace `trips-board.js` (handles Dataview DateTime, ISO string, M/D/YYYY, MM/DD/YYYY, fallback `moment(str)` parse).

> [!warning]- R5 — Cmd+R prerequisite (gotcha 2 + v0.13.0 + v0.14.0 reinforcements)
> Mitigation: T2.10 explicitly requires Cmd+R BEFORE smokes A–F. Two new CustomJS classes + 2 new ICONS entries trigger BOTH conditions.

> [!warning]- R6 — Workshop_version bump + tag are user-gated
> Mitigation: T2.1 gates approval before commit; T2.17 gates approval before tag.

## Out of scope (deferred to future cycles)

> [!info]- Explicitly NOT in v0.6.0
> - **Per-trip Kanban board.** Could mirror project blueprint's task-board pattern with a "Create Board" action button on the trip atlas. Carry as v0.6.x candidate.
> - **Workstream tagging.** Project-blueprint-specific concept; trips don't need it.
> - **Hierarchical sub-tasks.** Project-blueprint-specific; trips are flat.
> - **Trip archival / lifecycle states.** Trips' "past" status is implicit (sortDate < today); no explicit `status: archived` flow.
> - **Bulk import from existing headspace trips.** One-shot manual migration is the user's call; not platform mechanic.
> - **Per-trip frontmatter validation rules.** Validator mechanism could ship a `trips/_rules.json` fragment; not v0.6.0.
