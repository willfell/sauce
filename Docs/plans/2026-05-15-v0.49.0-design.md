# v0.49.0 design — L2 listener registration + BUG-7 + BUG-8

## Background

### What v0.48.0 left undone

v0.48.0 closed 2026-05-15 shipping the `ProjectTaskCreateListener` customjs class + Templater startup_templates registration + applyTemplaterStartupTemplates installer helper, intended to fix BUG-5 (kanban "+ Add a card" doesn't run Templater post-paste). Workshop preflight passed across 15 harnesses. Headspace re-validation at S10 surfaced THREE distinct issues:

- **BUG-6 (CRITICAL):** the v0.48.0 listener never subscribed at headspace. Cmd-R reload of Obsidian produced no `[ProjectTaskCreateListener] subscribed to vault.create` console log. The Templater `startup_templates[]` mechanism did not activate the listener at boot. Net effect: BUG-5 remains observable at headspace; v0.48.0's mechanism is dormant.
- **BUG-7 (HIGH):** "Create Board" AccentButton on a project task note (e.g., `tasks/<X>/<X>.md`) is gated `if (ctx.context === "task-hub")` at `project-nav-buttons.js:597`. Pre-existing behavior since at least v0.39.0; user's `Sauce` project demonstrates the per-task-board flow worked at some point but the gate keeps the button hidden in `task-note` context.
- **BUG-8 (cosmetic but pervasive):** the load-bearing sentinel `<!-- entity-create:<id> -->` injected by `injectAccentButtonBlock` (install.js) is visible in source/edit mode of all 7 entity-create-using hub files. Hidden in reading mode. User finds the visible HTML comment ugly; wants it gone in BOTH modes.

### S0 verification gate retrospect (v0.48.0 lesson)

v0.48.0's S0 gate empirically verified the Templater command id (Step 1.2) but DEFERRED Steps 1.3 (startup_templates fires at boot) and 1.4 (kanban race timing) per the user's defensive-hedge choice (option C: bump _postCreateDelayMs 50→200 + skip live smoke). The 200ms hedge turned out to be irrelevant because the listener never registered at all — the actual failure was earlier in the chain. v0.49.0 elevates S0 to a HARD-FAIL gate before any implementation, scoped specifically to verifying customjs `startupScriptNames[]` actually fires at boot at headspace.

## Architecture

Three independent fixes designed as one cohesive cycle since all three address post-v0.46.0-modularization rough edges that surfaced during v0.48.0 manual smoke.

### Architecture overview

1. **L2 listener registration** — customjs plugin's `startupScriptNames[]` field replaces v0.48.0's Templater `startup_templates[]` as the listener-init mechanism. The `ProjectTaskCreateListener` class itself is unchanged — only the registration vector changes. customjs is bound to its own plugin-load lifecycle which is more reliable than Templater's startup_templates (which empirically failed at headspace).

2. **BUG-7 — widen Create Board gate** — `project-nav-buttons.js:597` currently gates the "Open Board"/"Create Board" AccentButton block on `ctx.context === "task-hub"`. Widen to include `task-note`. The underlying `_createTaskBoard` helper takes `projectDir + taskFolder` which are both present in task-note context. Path construction at line 598 (`${projectDir}/tasks/${ctx.taskFolder}/board/${ctx.taskFolder}-board.md`) works correctly in both contexts.

3. **BUG-8 — sentinel relocation** — Move entity-create's load-bearing sentinel from outside-block HTML comment `<!-- entity-create:<id> -->` to inside-block JS comment `// entity-create:<id> — installer-managed; do not delete this comment`. The marker becomes invisible in BOTH source AND reading modes. Touches all 7 entity-create-using hub source files + install.js's `injectAccentButtonBlock` regex.

### Belt-and-suspenders for L2

Keep v0.48.0's Templater startup_templates entry in project manifest as a SECONDARY backstop. Both registration paths call `customJS.ProjectTaskCreateListener.init()`; the listener's `_initialized` flag makes the call idempotent. If Templater startup ever fires (it didn't at headspace), it's a no-op; if customjs startup fires (the new primary), it's the working path. No removal of v0.48.0's wiring needed; the orphaned data.json entry remains harmless.

## Components & contracts

### NEW manifest schema field `customjs_startup_scripts[]` (per-blueprint)

Each blueprint can declare which CustomJS class names should be auto-invoked at customjs plugin boot. Array of strings — each is the basename (no extension) of a script under the blueprint's `helpers/` dir whose class has an `invoke()` method.

```json
// platform/blueprints/project/manifest.json
"customjs_startup_scripts": ["ProjectTaskCreateListenerInit"]
```

### NEW init script `helpers/project-task-create-listener-init.js`

Three-line bootstrap:

```js
class ProjectTaskCreateListenerInit {
    invoke() {
        customJS.ProjectTaskCreateListener.init();
    }
}
```

Materialized to `ranch/scripts/project/project-task-create-listener-init.js`. The customjs plugin's startup-script invoker calls `customJS.ProjectTaskCreateListenerInit.invoke()` at plugin init time. The `invoke()` method delegates to the existing v0.48.0 listener's `init()` — full reuse, zero changes to the listener class.

### NEW installer helper `applyCustomJsStartupScripts` in install.js

Same posture as v0.48.0's `applyTemplaterStartupTemplates`:

- additive-merge (preserve user entries already in customjs `startupScriptNames[]`)
- malformed-JSON guard (never overwrite a customjs/data.json we can't parse)
- backup-on-edit (write `<target>.sauce-backup` before live write)
- failure-loud history (every failure path emits Notice + history entry)
- idempotent first-wins (if already in array, skip)

Targets `.obsidian/plugins/customjs/data.json`'s `startupScriptNames[]` array. Aggregates each subscribed item's `customjs_startup_scripts[]`.

`.obsidian/plugins/customjs/data.json` is **already in landmine #12 allowlist** (added in v0.24.0 for `applyCustomJsSettings`). No allowlist amendment needed.

Wired into `installItem` at install.js:991 (the per-item helper-call sequence) immediately after `applyTemplaterStartupTemplates`:

```js
await applyTemplaterFolderTemplates(tp, mech, variables, history, git);  // NEW v0.4.0
await applyTemplaterStartupTemplates(tp, mech, variables, history, git); // NEW v0.48.0
await applyCustomJsStartupScripts(tp, mech, variables, history, git);    // NEW v0.49.0
```

### BUG-7 widening — project-nav-buttons.js (1 line)

`platform/blueprints/project/helpers/project-nav-buttons.js:597`:

```diff
-            if (ctx.context === "task-hub") {
+            if (ctx.context === "task-hub" || ctx.context === "task-note") {
```

The `Open Board` / `Create Board` AccentButton block (lines 597-625) starts firing in task-note context too. The boardPath construction at line 598 (`${projectDir}/tasks/${ctx.taskFolder}/board/${ctx.taskFolder}-board.md`) works correctly in both contexts since `ctx.taskFolder` is set by both context detections (lines 47-55 of project-nav-buttons.js).

Note: this preserves v0.46.0's design (the `_createTaskBoard` helper remained intact at `:192`); the gate widening just exposes the existing functionality on more contexts.

### BUG-8 sentinel relocation — 7 hub source files + 1 installer regex change

**Source-file change pattern** (one example — `platform/blueprints/project/content/Projects.md`):

```diff

 ```dataviewjs
+// entity-create:project — installer-managed; do not delete this comment

 ```
```

|---|---|---|

**install.js `injectAccentButtonBlock` regex change:**

The current implementation (per v0.46.0 S2) searches for the outside-block HTML comment `<!-- entity-create:<id> -->` and injects an AccentButton dataviewjs block immediately after. The new implementation searches for the inside-block JS comment `// entity-create:<id>` within an existing dataviewjs block and ENSURES the injection target is that block (idempotent re-detection on subsequent installs).

Pseudocode change:

```js
// OLD (v0.46.0):
const marker = `<!-- entity-create:${id} -->`;
const idx = body.indexOf(marker);
// ... inject AccentButton block AFTER marker ...

// NEW (v0.49.0):
const sentinel = `// entity-create:${id}`;
const blockRegex = new RegExp(
  '```dataviewjs\\s*\\n[^`]*?' + escapeRegex(sentinel) + '[^`]*?```',
  'g'
);
const match = blockRegex.exec(body);
// If match found: skip (already injected — idempotent).
// If not found: see "Open implementation detail" below for resolution.
```

**Open implementation detail:** without the outside-block HTML comment as the locator, install.js needs a different way to know WHERE in the hub source file to inject the AccentButton block on first install. Options:

- (A) Source files ALREADY have the block (pre-injected during the v0.49.0 S5 source edit); installer only verifies the block exists + skips. **Recommended.** Hub source files already need to be edited for the sentinel relocation; might as well include the full dataviewjs block. Installer becomes idempotent-verify-only.
- (B) Use a different outside-block marker that's also invisible (e.g., a Markdown reference-style link with a special target). More complex; adds new failure modes.

**Choice: A.** S5 hub-source edits include the full dataviewjs block with the inside-block sentinel comment as the first line. The installer's `injectAccentButtonBlock` becomes `verifyAccentButtonBlock` — confirms the block exists (by sentinel substring search) + emits a warning if missing. No injection needed at install time; the source-file owns the block.

This significantly simplifies the installer: no string-injection logic, no idempotency state machine. The trade-off: subsequent blueprints adding entity-create entries must hand-author the dataviewjs block in their hub source file (rather than relying on installer auto-injection). Acceptable since entity-create entries are already declared per-blueprint in the manifest.

### Migration concern (existing consumer hubs)

Existing consumer hubs (at headspace + other vaults) currently have:

- The OLD outside-block HTML comment `<!-- entity-create:<id> -->`
- The dataviewjs block with `await customJS.EntityCreate.render(...)` (already injected by v0.46.0+ installer)

After v0.49.0 install, the installer overwrites the hub source file from `files[]`. The new hub source has:

- NO outside-block HTML comment (deleted)
- The dataviewjs block with the inside-block sentinel comment (new shape)

Backup-on-edit per landmine #12 covers any user-customized hub content (`.sauce-backup` written before overwrite). Users who customized their hub bodies will have their changes in the `.sauce-backup` file; they can manually merge.

### Escalation gates (STOP and report)

- **S0 fails** (customjs startupScriptNames doesn't fire): re-design listener registration. Options: hand-rolled `workspace.onLayoutReady`, L3 dataviewjs auto-load via Cowork.md, OR Templater folder_template with subfolder triggering. Update design doc, restart from S1.
- **S2 surfaces unexpected installer-helper interaction** with existing `applyCustomJsSettings` helper (already writes to customjs/data.json): reconcile the two helpers' write ordering to avoid trampling each other's state. May require helper consolidation.
- **S5 hub edits break a blueprint's existing dataviewjs render**: revert the BUG-8 fix for that blueprint; ship BUG-8 fix for only the verified-passing hubs in v0.49.0; defer the broken one.
- **S9 surfaces NEW bug (BUG-9, BUG-10)**: capture in `Docs/prompts/2026-05-15-v0.50.0-<brief>-handoff.md`. Decide with user: close v0.49.0 + roll new bugs into v0.50.0 OR roll back if the new bug is a v0.49.0 regression.
